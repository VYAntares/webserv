0

Démarrage du serveur

ConfigLoader vérifie et parse le fichier .conf, puis un ServerHandler est créé pour chaque paire (adresse, port) de chaque bloc server.

Notre main() est minimal : tout objet s'auto-enregistre. Chaque listen du fichier de conf représente un serveur qui écoute sur une paire adresse:port — d'où la double boucle. Un port occupé est loggé mais n'empêche pas les autres serveurs de démarrer (try/catch par listen). Les signaux (SIGINT, SIGPIPE...) sont installés plus tard, au début de handle_events().

main.cpp — notre code, en entier
int main(int argc, char **argv) { try { ConfigLoader cl(argc, argv); // lexer → parser → validator Config c = cl.getConfig(); for (size_t i = 0; i < c.cfg.size(); i++) { for (size_t j = 0; j < c.cfg[i].listen.size(); j++) { try { new ServerHandler(c.cfg[i].listen[j], c.cfg[i]); } catch (std::exception &e) { std::cerr << e.what() << std::endl; // un port raté ≠ tout s'arrête } } } EventLoop::instance()->handle_events(); // la boucle Reactor EventLoop::destroy(); // nettoyage complet au Ctrl-C } catch (std::exception &e) { std::cerr << e.what() << std::endl; return 1; // config invalide → refus de démarrer } return 0; }

ConfigLoader vérifie le lancement (argc, extension .conf, fichier régulier via stat(), non vide), puis enchaîne les trois étages : le Lexer tokenize ({ } ; word EOF), le Parser construit les structs, le Validator vérifie la cohérence et propage les directives du server vers ses locations. Détails complets dans les fiches de la section 04.

1

ServerHandler — les sockets d'écoute

Un ServerHandler par paire (adresse, port). Il crée son socket et s'enregistre lui-même dans l'EventLoop en ACCEPT_EVENT.

Le flow classique socket → bind → listen, avec trois options qui comptent : SO_REUSEADDR (redémarrage rapide sans "Address already in use"), O_NONBLOCK (un accept() ne doit jamais figer la boucle) et FD_CLOEXEC (les enfants CGI ne doivent pas hériter du socket d'écoute). En cas d'échec en cours de route, le constructeur ferme le fd avant de re-throw (RAII).

↓

↓
fcntl(O_NONBLOCK + FD_CLOEXEC)non-bloquant, et invisible pour les CGI
↓
bind(addr du .conf)l'adresse vient de la config (htons pour le port)
↓

↓

srcs/handlers/ServerHandler.cpp — constructeur + createSocket()
ServerHandler::ServerHandler(addrport addrs, const Server& server) : _server(server), _fd(-1) { try { _fd = createSocket(); bindAddress(_fd, addrs); if (listen(_fd, SOMAXCONN) == -1) throw std::runtime_error("listen() failed:" + std::string(strerror(errno))); } catch (...) { close(_fd); // RAII : pas de fd orphelin throw; } EventLoop::instance()->register_handler(this, ACCEPT_EVENT); } int ServerHandler::createSocket() { int fd = socket(AF_INET, SOCK_STREAM, 0); if (fd == -1) throw std::runtime_error("socket() failed:" + std::string(strerror(errno))); int opt = 1; if (setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)) == -1) { close(fd); throw std::runtime_error("setsockopt() failed: " + std::string(strerror(errno))); } if (fcntl(fd, F_SETFL, fcntl(fd, F_GETFL) | O_NONBLOCK) == -1) { close(fd); throw std::runtime_error("fcntl() failed: " + std::string(strerror(errno))); } fcntl(fd, F_SETFD, FD_CLOEXEC); // l'enfant CGI ne doit pas hériter du socket d'écoute return fd; }
socket() setsockopt() fcntl() bind() listen() htons()
⚠️

Multi-port = plusieurs ServerHandler, un seul epoll. Notre default.conf écoute sur 8080 et 8079 (contenus différents). Chaque ServerHandler vit dans la même _table de l'EventLoop — c'est l'exigence « un seul poll() pour toutes les I/O, listen inclus ». L'adresse vient de la config : getaddrinfo() l'a résolue au parsing (Parser::resolveHost), le ServerHandler ne fait que la binder.

2

Le Reactor — EventLoop

Pattern Reactor (Schmidt, 1995) : un Initiation Dispatcher central (EventLoop, singleton) et des Concrete Event Handlers derrière une interface unique (IEventHandler).

L'EventLoop ne connaît aucun type concret. Il ne voit que des IEventHandler* : c'est le polymorphisme qui appelle le bon code (ServerHandler::handle_accept, ClientHandler::handle_input, CGIReadHandler::handle_input...) sans un seul switch sur le type. Ajouter un nouveau handler = implémenter l'interface + s'enregistrer. EventLoop ne change jamais.

includes/core/IEventHandler.hpp — le contrat + les EventType
// Valeurs en octal, reprises du papier Schmidt — un seul bit chacune, // testables au & et combinables au | : enum EventType { ACCEPT_EVENT = 01, READ_EVENT = 02, WRITE_EVENT = 04, TIMEOUT_EVENT = 010, SIGNAL_EVENT = 020, CLOSE_EVENT = 040 }; class IEventHandler { public: virtual int getFd() const = 0; // pure : tout handler expose son fd virtual int handle_accept() { return 0; } virtual int handle_input() { return 0; } virtual int handle_output() { return 0; } virtual int handle_close() { return 0; } virtual int handle_timeout() { return 0; } virtual time_t getLastActivity() const { return 0; } // 0 = jamais timeout virtual ~IEventHandler() {} };
srcs/core/EventLoop.cpp — handle_events(), le cœur du Reactor
void EventLoop::handle_events() { signal(SIGINT, handle_sigint); signal(SIGTERM, handle_sigint); signal(SIGQUIT, handle_sigint); signal(SIGPIPE, SIG_IGN); struct epoll_event events[MAX_EVENTS]; // MAX_EVENTS = 128 while (!g_stop) { int n = epoll_wait(_epfd, events, MAX_EVENTS, 500); // 500ms : g_stop + timeouts for (int i = 0; i < n; i++) { HandlerEntry* entry = static_cast<HandlerEntry*>(events[i].data.ptr); IEventHandler* h = entry->handler; int ret = 0; try { if (entry->type == ACCEPT_EVENT) ret = h->handle_accept(); else { if (events[i].events & (EPOLLIN | EPOLLHUP | EPOLLERR)) ret = h->handle_input(); if (ret != -1 && events[i].events & EPOLLOUT) ret = h->handle_output(); // EPOLLERR/EPOLLHUP "nus" : fermer, sinon busy loop if (ret != -1 && (events[i].events & (EPOLLHUP | EPOLLERR)) && !(events[i].events & (EPOLLIN | EPOLLOUT))) ret = -1; } } catch (std::exception& e) { std::cerr << "[handler error] fd=" << h->getFd() << ": " << e.what() << "\n"; ret = -1; // on sacrifie le handler, jamais la boucle } catch (...) { ret = -1; } if (ret == -1) { remove_handler(h); // epoll_ctl(DEL) + retrait de la table delete h; // le destructeur ferme le fd } } checkTimeOut(); // hors du if(n>0) : tombe aussi quand le serveur est inactif } }
💡

Pourquoi ev.data.ptr pointe sur un HandlerEntry et pas sur le handler : epoll nous rend un void* — il faut savoir quoi appeler. HandlerEntry { IEventHandler* handler; EventType type; } garde le type d'enregistrement à côté du handler. La _table (std::map<IEventHandler*, HandlerEntry*>) permet à modify_handler/remove_handler de retrouver l'entrée en O(log n).

ACCEPT_EVENT : un client frappe. handle_accept() retourne toujours 0 — un échec d'accept est transitoire, retourner -1 détruirait le socket d'écoute et le port ne répondrait plus jamais. Piège Linux : le fd client n'hérite pas de O_NONBLOCK du socket d'écoute, il faut le poser explicitement.

srcs/handlers/ServerHandler.cpp — handle_accept()
int ServerHandler::handle_accept() { struct sockaddr_in client_addr; socklen_t len = sizeof(client_addr); int client_fd = accept(_fd, (struct sockaddr*)&client_addr, &len); if (client_fd == -1) return 0; // transitoire — surtout pas -1 // O_NONBLOCK n'est PAS hérité du socket d'écoute (Linux) if (fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK) == -1) { close(client_fd); return 0; } fcntl(client_fd, F_SETFD, FD_CLOEXEC); try { new ClientHandler(client_fd, _server, client_addr); // s'enregistre en READ_EVENT } catch (std::exception& e) { std::cerr << "accept: " << e.what() << std::endl; close(client_fd); } return 0; }

EPOLLIN sur un client : UN seul recv() de 4096 octets max (jamais de boucle, jamais d'errno — epoll re-notifiera en level-triggered). Les octets sont donnés au HttpParser, qui accumule et avance sa machine à états. On ne regarde que le résultat : COMPLETE → router, ERROR → ErrorHandler.

srcs/handlers/ClientHandler.cpp — handle_input()
int ClientHandler::handle_input() { _lastActivity = time(NULL); char buf[4096]; ssize_t n = recv(_fd, buf, sizeof(buf), 0); if (n <= 0) return -1; // déconnexion ou erreur → destruction std::string data(buf, n); _parser.runParsing(data); if (_parser.getState() == HttpParser::COMPLETE) _handleComplete(); // → Router → réponse → WRITE_EVENT else if (_parser.getState() == HttpParser::ERROR) _handleError(); // → ErrorHandler → WRITE_EVENT return 0; // requête incomplète : epoll rappellera }

EPOLLOUT sur un client : UN seul send() depuis l'offset _sent — la réponse part en plusieurs notifications si elle est grosse. Une fois tout parti : soit keep-alive (_reset() + retour en READ_EVENT, avec re-parse immédiat du reliquat pour le pipelining), soit return -1 et l'EventLoop nous détruit.

srcs/handlers/ClientHandler.cpp — handle_output()
int ClientHandler::handle_output() { _lastActivity = time(NULL); const char* data = _response.c_str() + _sent; size_t left = _response.size() - _sent; ssize_t n = send(_fd, data, left, MSG_NOSIGNAL); // pas de SIGPIPE if (n <= 0) return -1; _sent += static_cast<size_t>(n); if (_sent >= _response.size()) { if (_keepAlive) { _reset(); // delete _rh, clear, parser.reset() EventLoop::instance()->modify_handler(this, READ_EVENT); // Pipelining : une requête entière peut déjà être dans le buffer // du parseur — aucun octet ne re-déclenchera handle_input, donc // on re-parse le reliquat maintenant. std::string empty; _parser.runParsing(empty); if (_parser.getState() == HttpParser::COMPLETE) _handleComplete(); else if (_parser.getState() == HttpParser::ERROR) _handleError(); return 0; } return -1; // Connection: close → EventLoop détruira ce handler } return 0; // pas tout parti : epoll re-notifiera EPOLLOUT }
🔄

Le cycle complet côté Reactor : handle_accept → READ_EVENT → handle_input (× N recv) → parse COMPLETE → route → buildResponse → modify_handler(WRITE_EVENT) → handle_output (× N send) → keep-alive (retour READ_EVENT) ou return -1 (destruction). Et à chaque tour de boucle, checkTimeOut() chasse tout handler muet depuis plus de 30 secondes — clients comme CGI.

3

HttpParser — la machine à états

Stateful, incrémental, borné. Il survit entre les recv() et transforme le flux TCP fragmenté en HttpRequest structurée.

Le pipeline du projet tient en une ligne de commentaire de HttpParser.hpp : HttpParser → HttpRequest → Router → ARequestHandler. Le parser est stateful (il accumule entre les événements), la HttpRequest est de la donnée pure, le Router choisit, le handler construit. Les états : R_HEADERS → R_BODY ou R_CHUNKED → COMPLETE / ERROR.

srcs/http/HttpParser.cpp — runParsing(), la boucle d'états
void HttpParser::runParsing(std::string& buffer) { _req.error = (_errorCode != 0) ? _errorCode : 200; _buffer += buffer; // accumulation inter-recv // Boucle plutôt que simple switch : un seul recv() peut livrer les // headers ET tout le body d'un coup (petite requête). Sans boucle, // le body déjà présent resterait non consommé → requête figée. State before; do { before = _state; switch (_state) { case R_HEADERS: treatHeader(); break; // cherche \r\n\r\n, max 32 Ko (431) case R_BODY: treatBody(); break; // Content-Length octets (413 drainé) case R_CHUNKED: readChunked(); break; // dé-chunke pour le CGI default: break; // COMPLETE, ERROR : rien à faire } } while (_state != before && _state != COMPLETE && _state != ERROR); }

headerParser() découpe la première ligne (méthode, URI, version — validées par checkFirstLine(), qui décode aussi les %XX), range les headers dans une map, puis fait deux choses décisives : résoudre la limite de body de la bonne location (avant de lire le body !), et détecter Transfer-Encoding: chunked ou le boundary multipart. Le détail de chaque état est dans les fiches de la section 03.

Ce que le Router reçoit — includes/http/HttpRequest.hpp
struct HttpRequest { int error; // 200, ou 400/413/431... flagué par le parser std::string uri; // décodée, query comprise std::string body; // déjà dé-chunké si besoin std::string method; std::string version; // HTTP/1.1 ou HTTP/1.0 std::map<std::string, std::string> headers; Multipart mp; // formulaires + fichiers uploadés bool isMultipart; std::string boundary; };
⚠️

Le piège des requêtes fragmentées est réglé par l'architecture : le HttpParser est un membre du ClientHandler — il survit entre les recv(). Chaque handle_input() ne fait qu'ajouter des octets et relancer la machine à états. Rien n'est parsé deux fois, rien n'est perdu, et une requête coupée au milieu d'un header attend simplement le prochain EPOLLIN (ou le timeout 30s).

4

Router — l'aiguillage central

Une fonction statique, une cascade de garde-fous, et trois issues : un handler synchrone, NULL (CGI asynchrone), ou un ErrorHandler.

Router::route() est appelé par _handleComplete() avec la requête, le Server, l'adresse du client (pour REMOTE_ADDR) et le this du client en tant qu'IResponseSink*. La location est choisie par longest prefix match (findLocation(), partagée avec le parser), sur le chemin sans query string.

srcs/http/Router.cpp — route(), la cascade complète
ARequestHandler* Router::route(const HttpRequest& req, const Server& server, const std::string& peerAddr, IResponseSink* sink) { if (req.error != 200) return new ErrorHandler(server, req.error, ""); // le parser a déjà jugé // matcher la location sur le chemin SANS la query string : avec elle, // "/upload?x=1" ne matchait jamais "/upload" et retombait sur "/" std::string uriPath = req.uri.substr(0, req.uri.find('?')); const Location *loc = bestRouteFound(req.uri, server); if (!loc) return new ErrorHandler(server, 404, ""); if (!methodImplemented(req.method)) return new ErrorHandler(server, 501, ""); if (!methodAllowed(req.method, loc)) return new ErrorHandler(*loc, 405, ""); // la redirection s'applique avant tout accès au filesystem if (loc->return_path.first != -1) return new StaticHandler(req, *loc, ""); std::string path = resolvePath(loc, uriPath); if (path.empty()) // normalizePath : sortie de root (/../..) return new ErrorHandler(*loc, 403, ""); // upload : si un dossier de stockage est configuré, c'est lui qui reçoit if (req.method == "POST" && !loc->upload_store.empty()) { if (req.isMultipart == true) path = loc->upload_store; else { /* upload_store + basename de l'URI */ } } if (!fileExist(path, req.method)) return new ErrorHandler(*loc, 404, ""); if (forbiddenAccess(path, req.method)) return new ErrorHandler(*loc, 403, ""); std::string interpreter = isCgi(uriPath, loc); if (!interpreter.empty() && (req.method == "GET" || req.method == "POST")) { CGIHandler cgi(req, path, interpreter, peerAddr, loc, sink); return NULL; // asynchrone : réponse via onCgiDone() } if (req.method == "POST" && req.isMultipart == true) return new MultipartHandler(*loc, req, path); return new StaticHandler(req, *loc, path); }
💡

Trois sorties, trois modes : un ARequestHandler* → réponse synchrone, construite tout de suite ; NULL → CGI asynchrone, le client passe en masque epoll vide et attend onCgiDone() ; un ErrorHandler* → même circuit que le statique, avec les error_page de la config. Le détail du longest-prefix est dans la fiche routing.

5

Les handlers de réponse

Tous héritent d'ARequestHandler : un seul buildResponse() pour tout le monde. Static, Multipart, Error — et le circuit CGI à part.

StaticHandler — GET, POST, DELETE, autoindex, redirect. Son constructeur est un aiguillage : un return configuré part en redirection ; un dossier demandé sans slash final déclenche un 301 (comme nginx), avec autoindex il liste le dossier (throwList() via opendir/readdir), sinon 403. Ensuite GET lit, POST écrit, DELETE unlink.

srcs/http/StaticHandler.cpp — le constructeur-aiguillage
StaticHandler::StaticHandler(const HttpRequest& req, const Location& loc, const std::string& path): _req(&req), _loc(&loc), _path(path) { _ncode = 200; if (loc.return_path.first != -1) { handleReturn(loc.return_path); // redirection configurée (301/302 + Location) return; } if (isDir(path) && req.method != "POST") { std::string uriPath = req.uri.substr(0, req.uri.find('?')); if (uriPath[uriPath.length() - 1] != '/') { const std::pair<int, std::string> toreturn(301, uriPath + '/'); handleReturn(toreturn); // /files → 301 /files/ (comme nginx) } else if (loc.autoindex == 1) throwList(); // listing HTML du dossier else _ncode = 403; } else if (req.method == "GET") handleGet(); // ifstream + getType (MIME) else if (req.method == "POST") handlePost(); // ofstream, 201 si création else if (req.method == "DELETE") handleDelete(); // unlink → 204, ou 404 setErrorPage(loc); }
stat() ifstream ofstream unlink() opendir() readdir()

Le circuit CGI — quatre classes, zéro blocage. Contrairement au statique, le CGI ne construit rien tout de suite : CGIHandler (orchestrateur) crée le CGIProcess (pipe ×2, fork, chdir, execve) et enregistre deux nouveaux handlers dans l'EventLoop : CGIWriteHandler pousse le body dans le stdin du script, CGIReadHandler collecte sa sortie jusqu'à l'EOF, la parse (Status:, Content-Type:, Set-Cookie:) et livre la réponse au client via IResponseSink::onCgiDone().

srcs/handlers/CGIHandler.cpp — l'orchestrateur
CGIHandler::CGIHandler(const HttpRequest& req, std::string& path, std::string& interpreter, const std::string& peerAddr, const Location* loc, IResponseSink* sink) { _process = new CGIProcess(req, path, interpreter, peerAddr); // fork + exec + pipes CGIWriteHandler* wr = new CGIWriteHandler(_process->getWriteFd(), req.body); EventLoop::instance()->register_handler(wr, WRITE_EVENT); CGIReadHandler* rd = new CGIReadHandler(_process->getReadFd(), _process->getPid(), loc, sink); EventLoop::instance()->register_handler(rd, READ_EVENT); if (sink) sink->onCgiStart(rd); // pour pouvoir se détacher si l'un meurt avant l'autre }
🔀

Toute la mécanique est détaillée dans les 4 fiches de la section 05 : le mécanisme fork/execve/pipe, les variables d'environnement, EOF, chunked et exit codes, l'intégration dans l'event loop. L'essentiel côté architecture : les pipes CGI sont des IEventHandler comme les autres, dans le même epoll, avec le même timeout 30s.

ErrorHandler — la sortie de secours universelle. Construit avec un BaseBlock (Server ou Location : les deux héritent), un code et un body optionnel. Il mémorise l'error_page configurée ; c'est buildResponse() qui, au moment de sérialiser, chargera la page custom — ou générera <h1>404 Not Found</h1> si elle manque.

6

Construction de la réponse

Un seul buildResponse() pour tous les handlers — hérité d'ARequestHandler. Puis modify_handler(WRITE_EVENT) et epoll prend le relais.

Chaque handler a rempli _ncode, _type, _body (et éventuellement _location pour un redirect, _cookie pour le bonus sessions). buildResponse() sérialise le tout : status line, headers conditionnels, Content-Length exact, Connection selon le keep-alive décidé par le client. C'est aussi lui qui applique le fallback des pages d'erreur.

srcs/http/ARequestHandler.cpp — buildResponse()
std::string ARequestHandler::buildResponse() { // erreur sans body fourni → page custom (error_page) ou page générée if (!_noBody && isError(_ncode) && _location.empty() && _body.empty()) { if (_errorpage.length() > 0) getErrorPage(); else _body = "<html><body><h1>" + itos(_ncode) + " " + getReason(_ncode) + "</h1></body></html>"; } std::ostringstream oss; oss << "HTTP/1.1 " << _ncode << " " << getReason(_ncode) << "\r\n"; if (!_location.empty()) oss << "Location: " << _location << "\r\n"; // redirections 301/302 if (!_type.empty()) oss << "Content-Type: " << _type << "\r\n"; // MIME via getType() if (!_cookie.empty()) oss << "Set-Cookie: " << _cookie << "\r\n"; // bonus sessions (CGI) oss << "Content-Length: " << _body.size() << "\r\n" << "Connection: " << (_keepAlive ? "keep-alive" : "close") << "\r\n" << "\r\n" << _body; return oss.str(); }
srcs/handlers/ClientHandler.cpp — _handleComplete(), la décision keep-alive
void ClientHandler::_handleComplete() { HttpRequest req = _parser.getReq(); // HTTP/1.1 est keep-alive par défaut, HTTP/1.0 est close par défaut. std::map<std::string, std::string>::const_iterator it = req.headers.find("Connection"); if (it != req.headers.end()) _keepAlive = (it->second == "keep-alive"); else _keepAlive = (req.version == "HTTP/1.1"); _rh = Router::route(req, _server, _peerAddr, this); if (!_rh) { // CGI → asynchrone, silence radio epoll _keepAlive = false; EventLoop::instance()->modify_handler(this, static_cast<EventType>(0)); return; } if (req.method == "HEAD") { // HEAD = GET sans body _rh->stripBody(); _keepAlive = false; } _rh->setKeepAlive(_keepAlive); _response = _rh->buildResponse(); EventLoop::instance()->modify_handler(this, WRITE_EVENT); // "réveille-moi pour écrire" }
📋

Après le build : modify_handler(this, WRITE_EVENT) fait un epoll_ctl(EPOLL_CTL_MOD) — le fd est déjà enregistré (ADD à l'accept), on change seulement le masque. La réponse est cachée dans _response et _sent avance à chaque EPOLLOUT : pas de rebuild à chaque send.

7

Fermeture — une seule convention

Personne ne se close lui-même en plein traitement : on retourne -1, et l'EventLoop applique toujours le même rituel de destruction.

La convention return -1 centralise toute la destruction au même endroit de handle_events() : remove_handler() (epoll_ctl DEL + retrait de la table), puis delete — et c'est le destructeur de chaque handler qui ferme son fd et fait ses adieux spécifiques. Même circuit pour un client en Connection: close, une déconnexion brutale, un timeout ou une exception.

handler retourne -1fin normale, erreur, ou exception attrapée
↓
remove_handler(h)epoll_ctl(DEL) + delete du HandlerEntry — plus d'événements fantômes
↓
delete hle destructeur ferme le fd et nettoie (detachSink, kill CGI, 504...)
srcs/handlers/ClientHandler.cpp — le destructeur
// Si un CGI est encore en cours, on prévient son CGIReadHandler que nous // n'existons plus : sans ça, il appellerait onCgiDone() sur un objet détruit. ClientHandler::~ClientHandler() { if (_cgiRead) _cgiRead->detachSink(); delete _rh; close(_fd); std::cout << "[client " << _peerAddr << " fd=" << _fd << "] disconnected\n"; }

Au Ctrl-C : g_stop fait sortir la boucle, EventLoop::destroy() appelle le destructeur qui delete tous les handlers restants (fermant chaque fd, tuant chaque CGI en vol) puis ferme _epfd. Valgrind confirme : zéro fuite, zéro fd ouvert, même en plein trafic.

⚠️

Pourquoi personne ne fait close(fd) en plein traitement : le fd resterait dans epoll (événements fantômes sur un fd recyclé par accept !) et l'objet continuerait d'exister. La règle « un fd = un propriétaire = son destructeur » élimine les double-close et les use-after-close par construction. Détails dans la fiche timeouts & déconnexions.

ConfigLoader → ServerHandler → EventLoop → ClientHandler → HttpParser → Router → StaticHandler / CGI / ErrorHandler → buildResponse → send → delete. Maîtriser ce flux, c'est maîtriser notre webserv.
