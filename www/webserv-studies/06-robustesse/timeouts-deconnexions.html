Timeouts & déconnexions
— checkTimeOut() et la vie des fd

Un client qui n'envoie jamais la fin de sa requête, un script CGI qui boucle, un navigateur fermé brutalement : sans stratégie temporelle, chacun laisserait un fd zombie dans epoll. Notre réponse : un horodatage _lastActivity par handler, une chasse périodique dans EventLoop::checkTimeOut(), et une seule convention de destruction pour tout le monde.

01
Le problème
Trois fantômes qui s'accumulent sans timeout

« A request to your server should never hang indefinitely. »

— webserv.pdf, IV.1 Requirements
Sans timeout, qui reste coincé ?
Le client « slowloris »ouvre une connexion, envoie GET /index.ht… puis plus rien. Le parser attend son \r\n\r\n pour toujours, le fd reste dans epoll.
Le CGI muetslow.py dort 60 secondes : sans timeout, le client attend indéfiniment sa réponse et le processus enfant vit sa vie.
Le keep-alive dormantun navigateur garde la connexion ouverte après sa dernière requête, puis part déjeuner. Des centaines de fd occupés pour rien.

Multiplié par un stress test, chacun de ces cas épuise la limite de fd du processus — et un serveur sans fd disponible ne peut plus accept() personne.

02
getLastActivity()
Un horodatage par handler, exposé par l'interface
Le contrat dans IEventHandler

L'interface IEventHandler déclare virtual time_t getLastActivity() const { return 0; }. La convention : retourner 0 = immortel. ServerHandler n'override pas cette méthode — un socket d'écoute ne doit jamais être timeouté. ClientHandler et CGIReadHandler, eux, horodatent chaque I/O.

ClientHandler.cpp 
int ClientHandler::handle_input() {
    _lastActivity = time(NULL);
    // ... recv + parsing ...
}

int ClientHandler::handle_output() {
    _lastActivity = time(NULL);
    // ... send ...
}
CGIReadHandler.hpp 
// le CGI participe au timeout de la
// boucle : un script qui ne répond
// jamais est tué au bout de
// CLIENT_TIMEOUT au lieu de pendre à vie
time_t getLastActivity() const {
    return _lastActivity;
}
💡
Un seul mécanisme pour tous. Pas de timer par connexion, pas de timerfd, pas de file de priorité : juste un time_t mis à jour à chaque I/O réussie. La précision (±500 ms, le timeout d'epoll_wait) est largement suffisante pour un timeout de 30 secondes — et le coût est nul.
03
checkTimeOut()
La chasse aux inactifs, après chaque epoll_wait
srcs/core/EventLoop.cpp — checkTimeOut()
Notre code 
static const int CLIENT_TIMEOUT = 30; // secondes

// Collecte les handlers inactifs depuis plus de CLIENT_TIMEOUT secondes et
// les ferme, pour ne pas garder indéfiniment un fd enregistré si le client
// ne renvoie plus jamais rien.
void EventLoop::checkTimeOut() {
    time_t now = time(NULL);

    // On collecte d'abord dans un vecteur separé car remove_handler() appelle
    // _table.erase(), ce qui invaliderait les index si on supprimait en plein scan.
    std::vector<IEventHandler*> timedOut;
    for (std::map<IEventHandler*, HandlerEntry*>::iterator it = _table.begin();
            it != _table.end(); ++it) {
        time_t last = it->first->getLastActivity();
        if (last != 0 && (now - last) > CLIENT_TIMEOUT)   // 0 = immortel (ServerHandler)
            timedOut.push_back(it->first);
    }

    for (size_t i = 0; i < timedOut.size(); i++) {
        // avant de supprimer le client, il faut verifier si il nya pas un cgi en cours (erreur 504)
        if (timedOut[i]->handle_timeout())
            continue;   // sursis : le client va livrer son 504 avant de mourir
        // ensuite on peut le supprimer.
        remove_handler(timedOut[i]);
        delete timedOut[i];
    }
}
Pourquoi collecter puis détruire ?

Supprimer un élément d'une std::map pendant qu'on l'itère invalide l'itérateur courant. La passe en deux temps (collecte dans un vector, puis destruction) évite ce piège classique de C++98 — le même raisonnement qui fait qu'on ne delete jamais un handler au milieu du for d'événements sans passer par ret == -1.

Hors du if (n > 0) — volontairement

checkTimeOut() est appelé à chaque tour de boucle, même quand epoll_wait() expire sans événement (retour 0). C'est crucial : les timeouts doivent tomber précisément quand le serveur est inactif — un client silencieux ne génère, par définition, aucun événement. C'est le timeout de 500 ms d'epoll_wait qui garantit ce battement de cœur.

⏰
Le sursis 504. handle_timeout() retourne 1 seulement si un CGI est en cours (_cgiRead != NULL) : le client prépare alors sa réponse 504 Gateway Timeout, passe en WRITE_EVENT avec _keepAlive = false, et sera détruit naturellement après l'envoi. Les détails côté CGI sont dans la fiche CGI dans l'event loop.
04
recv() == 0, EPOLLHUP, send() <= 0
Les trois visages d'une déconnexion
Notre matrice de détection
Client ferme proprement (FIN)epoll signale EPOLLIN → recv() retourne 0 → handle_input() retourne -1 → destruction.
Client disparaît brutalement (RST)epoll signale EPOLLHUP/EPOLLERR → routés vers handle_input() (le recv() échoue → -1), ou fermeture forcée si aucun EPOLLIN/EPOLLOUT n'accompagne l'erreur.
Client coupe pendant notre réponsesend(…, MSG_NOSIGNAL) retourne ≤ 0 (sans SIGPIPE) → handle_output() retourne -1 → destruction.
Client coupe pendant un CGImasque epoll vide mais EPOLLHUP toujours signalé → destruction du client → detachSink() prévient le CGIReadHandler.
srcs/handlers/ClientHandler.cpp — les deux points de détection
Notre code 
int ClientHandler::handle_input() {
    _lastActivity = time(NULL);

    char buf[4096];
    ssize_t n = recv(_fd, buf, sizeof(buf), 0);
    if (n <= 0)
        return -1;   // 0 = fermeture propre, <0 = erreur : même traitement, zéro errno
    // ...
}

int ClientHandler::handle_output() {
    // ...
    ssize_t n = send(_fd, data, left, MSG_NOSIGNAL);
    if (n <= 0)
        return -1;   // client parti : on abandonne la réponse, destruction propre
    // ...
}
⚠️
Jamais de errno, même ici. On ne distingue pas « erreur réseau » de « buffer plein » en lisant errno (interdit par le sujet). La décision repose uniquement sur la valeur de retour — et comme on n'appelle recv/send qu'après notification epoll, un retour ≤ 0 signifie forcément un vrai problème, pas un EAGAIN déguisé.
05
La vie des fd
Chaque fd a exactement un propriétaire qui le ferme
La règle : le destructeur ferme, personne d'autre
Socket d'écoute~ServerHandler() → close(_fd)
Socket client~ClientHandler() → detachSink() éventuel + delete _rh + close(_fd)
Pipe stdin CGI (écriture)~CGIWriteHandler() → close(_fd) — c'est l'EOF du script
Pipe stdout CGI (lecture)~CGIReadHandler() → close(_fd) + kill/waitpid si besoin + 504 si client jamais servi
fd epoll~EventLoop() → delete de tous les handlers restants, puis close(_epfd)
srcs/core/EventLoop.cpp — le destructeur, dernier filet
Notre code 
EventLoop::~EventLoop() {
    for (std::map<IEventHandler*, HandlerEntry*>::iterator it = _table.begin();
            it != _table.end(); ++it) {
        delete it->first;    // chaque destructeur ferme son fd, tue son CGI...
        delete it->second;   // ...et on libère l'entrée de la table
    }
    close(_epfd);
    _instance = NULL;
    std::cout << "\nServer closing..." << std::endl;
    std::cout << "cya babyyyy" << std::endl;
}
💡
Pourquoi ça suffit contre les leaks. Tout objet vivant est dans _table (il s'y enregistre à sa création), et il n'en sort que par remove_handler + delete. Au Ctrl-C, le destructeur balaie ce qui reste. Vérifiable avec valgrind --track-fds=yes ./webserv : zéro fd ouvert, zéro fuite mémoire à la sortie — même avec des clients et des CGI en vol au moment de l'arrêt.
Résumé — rien ne pend, rien ne fuit
30 secondes d'inactivité max, une seule convention de destruction
⌚
Un time_t par handler
_lastActivity mis à jour à chaque I/O ; getLastActivity() == 0 = immortel (socket d'écoute). Le CGI joue avec les mêmes règles que les clients.
🔍
checkTimeOut à chaque tour
Grâce au timeout 500ms d'epoll_wait, la chasse tourne même sans trafic. Collecte puis destruction en deux passes (map C++98 oblige). handle_timeout() offre un sursis aux clients qui doivent livrer un 504.
📣
Déconnexion = valeur de retour
recv() ≤ 0 ou send() ≤ 0 → return -1, sans jamais lire errno. MSG_NOSIGNAL + SIG_IGN neutralisent SIGPIPE. EPOLLHUP/EPOLLERR isolés sont fermés d'office (anti busy-loop).
🔑
Un fd = un propriétaire
Chaque destructeur ferme son propre fd, et uniquement le sien. ~EventLoop() delete tout ce qui reste au shutdown : valgrind-clean même en plein trafic.
