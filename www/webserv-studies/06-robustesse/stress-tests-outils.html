Robustesse — La preuve par le feu

Stress tests & outils
— prouver que ça tient

« Resilience is key. » Un serveur robuste sur le papier ne vaut rien : il faut le bombarder. Voici l'arsenal utilisé sur ce projet — siege, telnet, curl, le tester officiel de 42, notre suite de tests HTML maison — et ce que chaque garantie du code (timeouts, limites, sacrifice de handler) donne concrètement sous charge.

01
L'exigence
Disponible à 100%, même sous le feu

« Stress test your server to ensure it remains available at all times. Resilience is key. »

— webserv.pdf, IV.1 Requirements
Ce que le stress test doit prouver, et quel mécanisme du code répond
Des milliers de connexions simultanéesepoll O(1) + un seul thread : la _table de l'EventLoop grossit, la boucle ne ralentit pas.
Déconnexions brutales en pleine requêterecv()/send() ≤ 0 → return -1, EPOLLHUP géré, destructeurs qui ferment. Pas de fd fantôme.
Requêtes malformées en rafaleLe parser flague 400/413/431, jamais d'exception non attrapée — et le try/catch de la boucle couvre le reste.
Clients lents / silencieux (slowloris)checkTimeOut() : 30 secondes et dehors.
Mémoire stable sur la duréeBuffers bornés (32 Ko headers, body par location), _reset() après chaque requête keep-alive, zéro leak valgrind.
02
siege, curl, telnet, python
L'arsenal, du plus brutal au plus chirurgical
Stress — siege / ab sur notre config (port 8080)
Terminal 
# 255 clients concurrents, 1 minute, en keep-alive
siege -b -c 255 -t 1M http://localhost:8080/

# Apache Bench : 10 000 requêtes, 100 en parallèle
ab -n 10000 -c 100 http://localhost:8080/index.html

# siege sur le CGI (fork massif → surveiller les zombies avec ps)
siege -b -c 50 -t 30S http://localhost:8080/cgi-bin/hello.py

# pendant le siege : vérifier fds et zombies
ls /proc/$(pgrep webserv)/fd | wc -l
ps aux | grep defunct
Chirurgical — telnet / curl, requête par requête
Terminal 
# requête brute au clavier : tester le parsing fragmenté
telnet localhost 8080
GET /index.html HTTP/1.1
Host: localhost
                          ← ligne vide = fin des headers

# chunked : notre readChunked() doit dé-chunker avant le CGI
curl -H "Transfer-Encoding: chunked" -d "hello" http://localhost:8080/cgi-bin/post_echo.py

# 413 : /post_body est limité à 100 octets dans default.conf
curl -X POST --data "$(python3 -c 'print("x"*200)')" -i http://localhost:8080/post_body

# 504 : slow.py dort plus longtemps que CLIENT_TIMEOUT (30s)
curl -i http://localhost:8080/cgi-bin/slow.py

# keep-alive + pipelining : deux requêtes dans le même paquet
printf 'GET / HTTP/1.1\r\nHost: a\r\n\r\nGET / HTTP/1.1\r\nHost: a\r\nConnection: close\r\n\r\n' | nc localhost 8080
💡
Le test pipelining n'est pas gratuit. Il valide un vrai détail de notre ClientHandler::handle_output() : après une réponse keep-alive, le reliquat du buffer est re-parsé immédiatement (_parser.runParsing(empty)) — car si la 2e requête est déjà entière dans le buffer, aucun octet ne viendra re-déclencher handle_input(). Sans ça, la 2e requête attendrait le timeout.
03
./tester + suite maison
Les deux batteries de tests du projet
Le tester binaire officiel

./tester http://localhost:8080 exige une config précise — c'est exactement ce que couvre notre conf/default.conf :

  • location / : GET uniquement
  • location /post_body : POST, body max 100 octets
  • location /directory : racine YoupiBanane (symlink www/directory), index youpi.bad_extension, POST sur *.bla → cgi_test

C'est lui qui a imposé PATH_INFO/REQUEST_URI dans notre buildEnvp(), et le client_max_body_size 1g sur /directory.

La suite de tests HTML maison

http://localhost:8080/default/ — une page par domaine : get.html, post.html, delete.html, upload.html (multipart), cgi.html (tous les scripts de www/cgi-bin/), errors.html (chaque code d'erreur), stress.html (rafales de fetch() depuis le navigateur) et bonus.html (cookies & sessions). Le tout testé depuis un vrai navigateur, comme l'exige le sujet.

04
NGINX comme oracle
Comparer nos réponses à la référence

« NGINX may be used to compare headers and answer behaviours (pay attention to differences between HTTP versions). »

— webserv.pdf, IV.1 Requirements
Comparaison côte à côte
Terminal 
# même requête, deux serveurs, comparer les headers
curl -i http://localhost:8080/index.html      # webserv
curl -i http://localhost:80/index.html        # nginx

# les comportements qu'on a alignés sur nginx :
#  - GET /files (dossier sans slash final) → 301 vers /files/
#  - HTTP/1.1 sans header Connection → keep-alive par défaut
#  - HTTP/1.0 sans header Connection → close par défaut
#  - Content-Length exact + Connection: close sur les erreurs
#  - listen identique en double dans la config → refus au démarrage
💡
La différence de versions HTTP compte. Notre _handleComplete() implémente exactement la règle nginx : Connection explicite s'il existe, sinon keep-alive pour HTTP/1.1 et close pour HTTP/1.0. C'est ce que le sujet appelle « pay attention to differences between HTTP versions ».
05
-std=c++98 -Werror
Les contraintes de compilation, et notre Makefile
Makefile — le nôtre, en entier
Notre code 
NAME = webserv
CXX = c++
CXXFLAGS = -Wall -Wextra -Werror -std=c++98
INCLUDES = -I includes

SRCS = $(shell find . -name "*.cpp" ! -path "./test/*")
OBJS = $(patsubst ./%.cpp, obj/%.o, $(SRCS))

all: $(NAME)

$(NAME): $(OBJS)
	$(CXX) $(CXXFLAGS) -o $(NAME) $(OBJS)

# compilation objet par objet → pas de relink inutile :
# make ne rebuild que les .o dont le .cpp a changé
obj/%.o: ./%.cpp
	@mkdir -p $(dir $@)
	$(CXX) $(CXXFLAGS) $(INCLUDES) -c $< -o $@

clean:
	rm -rf obj

fclean: clean
	rm -f $(NAME)

re: fclean all

.PHONY: all clean fclean re
Ce que C++98 nous a coûté
  • Pas de std::to_string → nos itos() / toString() via stringstream
  • Pas de auto, pas de range-for → itérateurs explicites partout (voir checkTimeOut)
  • Pas de nullptr → NULL
  • Pas de lambdas ni std::function → le polymorphisme classique (IEventHandler) fait le travail
  • Pas de smart pointers → ownership documenté : l'EventLoop détruit les handlers, chaque destructeur ferme son fd
Fonctions externes : la liste blanche

Tout passe par les fonctions autorisées du sujet : execve, pipe, fork, socketpair, htons/htonl/ntohs/ntohl, select/poll/epoll (create1/ctl/wait), kqueue, socket, accept, listen, send, recv, chdir, bind, connect, getaddrinfo, freeaddrinfo, setsockopt, getsockname, getprotobyname, fcntl, close, read, write, waitpid, kill, signal, access, stat, open, opendir, readdir, closedir.

Noté : inet_ntoa n'y figure pas — d'où notre _buildPeerStr() qui reconstruit l'IP octet par octet.
