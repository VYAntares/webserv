Le serveur ne crash jamais
— sacrifier le handler, pas la boucle

Le sujet est brutal : un serveur qui se termine de manière inattendue = 0. Notre stratégie tient en une phrase : toute erreur est locale. Un handler qui explose est sacrifié par le try/catch de l'EventLoop, un syscall qui échoue nettoie derrière lui, un signal lève un simple drapeau — et la boucle continue de tourner.

01
La règle absolue
Ce que le sujet exige, et comment on le découpe

« Your program must not crash under any circumstances (even if it runs out of memory) or terminate unexpectedly. »

— webserv.pdf, General rules
Notre découpage du problème
Erreur de config (au démarrage)throw → attrapé dans main() → message clair + exit propre. Refuser de démarrer n'est pas crasher.
Erreur d'un handler (en vol)try/catch dans EventLoop::handle_events() → on détruit ce handler, les autres continuent.
Erreur d'un syscall (socket, pipe, fork…)chaque chemin d'échec ferme ses fd / tue son enfant avant de throw ou de retourner.
Erreur du client (requête malformée)jamais une exception : un status code (400, 413, 431…) via ErrorHandler.
Signal (Ctrl-C, SIGPIPE…)drapeau atomique ou ignorance pure — jamais de logique dans le handler de signal.
02
try/catch dans la boucle
Une exception ne doit jamais sortir de handle_events()
srcs/core/EventLoop.cpp — handle_events(), le dispatch protégé
Notre code 
// Une exception levée par un handler ne doit jamais sortir de la
// boucle : le sujet impose que le serveur ne s'arrête sous aucune
// circonstance. On sacrifie seulement le handler fautif.
try {
    if (entry->type == ACCEPT_EVENT)
        ret = h->handle_accept();
    else {
        if (events[i].events & (EPOLLIN | EPOLLHUP | EPOLLERR))
            ret = h->handle_input();
        if (ret != -1 && events[i].events & EPOLLOUT)
            ret = h->handle_output();
        // EPOLLERR/EPOLLHUP sans EPOLLIN ni EPOLLOUT : aucun des
        // deux handlers ci-dessus n'a pu constater l'erreur — on
        // ferme, sinon epoll re-signale en boucle (busy loop).
        if (ret != -1 && (events[i].events & (EPOLLHUP | EPOLLERR))
            && !(events[i].events & (EPOLLIN | EPOLLOUT)))
            ret = -1;
    }
} catch (std::exception& e) {
    std::cerr << "[handler error] fd=" << h->getFd() << ": " << e.what() << "\n";
    ret = -1;
} catch (...) {
    std::cerr << "[handler error] fd=" << h->getFd() << ": unknown exception\n";
    ret = -1;
}

if (ret == -1) {
    remove_handler(h);   // retiré d'epoll + de la table
    delete h;            // destructeur → close(fd), détachements, 504 éventuel
}
Pourquoi ça couvre même le bad_alloc

« Even if it runs out of memory » : un new ou un std::string::append qui échoue lève std::bad_alloc — qui hérite de std::exception. Il est donc attrapé ici comme n'importe quelle erreur : le client en cours est sacrifié, sa mémoire libérée (ce qui détend justement la pression mémoire), et le serveur continue.

La convention return -1

Tous nos handlers parlent le même langage : 0 = « je continue », -1 = « retire-moi et détruis-moi ». La destruction passe toujours par le même chemin (remove_handler + delete), et chaque destructeur fait son ménage : close(fd), detachSink(), kill+waitpid pour les CGI. Un seul endroit détruit, zéro double-free.

⚠️
Le cas sournois : EPOLLERR/EPOLLHUP « nus ». Si epoll signale une erreur sans EPOLLIN ni EPOLLOUT, aucun de nos handle_* ne serait appelé pour la constater — et epoll re-signalerait l'événement à chaque tour : busy loop, 100% CPU. D'où le troisième if qui force ret = -1 dans ce cas précis.
03
signal()
SIGINT = drapeau, SIGPIPE = ignoré, arrêt = propre
srcs/core/EventLoop.cpp — les signaux
Notre code 
// Drapeau d'arrêt levé par le signal handler.
// volatile : force la relecture depuis la mémoire à chaque tour (pas de cache registre).
// sig_atomic_t : lecture/écriture atomique, sans risque de valeur à moitié écrite par un signal.
static volatile sig_atomic_t g_stop = 0;

// Handler SIGINT/SIGTERM : lève g_stop pour que handle_events() sorte proprement.
// Seules les fonctions async-signal-safe sont autorisées ici — donc pas de malloc/cout.
static void handle_sigint(int) { g_stop = 1; }

void EventLoop::handle_events() {
    signal(SIGINT, handle_sigint);
    signal(SIGTERM, handle_sigint);
    signal(SIGQUIT, handle_sigint);
    signal(SIGPIPE, SIG_IGN);   // un client qui coupe en plein send ne doit pas nous tuer

    while (!g_stop) {
        int n = epoll_wait(_epfd, events, MAX_EVENTS, 500);   // 500ms → g_stop re-vérifié régulièrement
        // ...
    }
}
Pourquoi un simple drapeau ?

Un handler de signal peut interrompre le programme n'importe où. Y faire un delete, un cout ou toucher une map serait de la roulette russe (seules les fonctions async-signal-safe sont légales). On se contente donc d'écrire un sig_atomic_t. Le vrai nettoyage a lieu dans le flot normal : la boucle sort, EventLoop::destroy() appelle le destructeur qui delete tous les handlers restants (donc ferme tous les fd, tue les CGI en cours) et ferme _epfd.

SIGPIPE, la double ceinture

Écrire dans un socket fermé côté client lève SIGPIPE — qui par défaut termine le processus. Notre protection est double : signal(SIGPIPE, SIG_IGN) au démarrage, et send(_fd, data, left, MSG_NOSIGNAL) dans ClientHandler::handle_output(). Le send échoue alors avec n <= 0 → return -1 → déconnexion propre du client.

💡
Le timeout 500 ms n'est pas décoratif. Sans lui, un serveur au repos dormirait dans epoll_wait(-1) pour toujours : Ctrl-C interromprait bien l'appel (EINTR), mais surtout checkTimeOut() ne tournerait jamais sans trafic. Avec 500 ms : n == 0 (timeout) → on saute le for (events[] n'est pas rempli), on re-vérifie g_stop, on chasse les clients inactifs, et on se rendort.
04
Chaque échec nettoie
Constructeurs qui throw, rollback, et accept insubmersible
srcs/handlers/ServerHandler.cpp — constructeur avec rollback
Notre code 
ServerHandler::ServerHandler(addrport addrs, const Server& server) : _server(server), _fd(-1) {
    try {
        _fd = createSocket();
        bindAddress(_fd, addrs);
        if (listen(_fd, SOMAXCONN) == -1)
            throw std::runtime_error("listen() failed:" + std::string(strerror(errno)));
    } catch (...) {
        close(_fd);   // RAII : jamais de fd orphelin, même si bind/listen échoue
        throw;        // re-throw : main() loggue et passe au listen suivant
    }
    EventLoop::instance()->register_handler(this, ACCEPT_EVENT);
}
srcs/handlers/ServerHandler.cpp — handle_accept(), jamais -1
Notre code 
// Retourne toujours 0 : un échec d'accept() (connexion annulée, plus de fd
// disponibles...) est transitoire et ne doit surtout pas détruire le socket
// d'écoute — retourner -1 ici ferait supprimer ce ServerHandler par EventLoop
// et le port ne répondrait plus jamais.
int ServerHandler::handle_accept() {
    int client_fd = accept(_fd, (struct sockaddr*)&client_addr, &len);
    if (client_fd == -1)
        return 0;   // erreur transitoire : on réessaiera au prochain événement

    if (fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK) == -1) {
        close(client_fd);
        return 0;
    }
    fcntl(client_fd, F_SETFD, FD_CLOEXEC);

    try {
        new ClientHandler(client_fd, _server, client_addr);
    } catch (std::exception& e) {
        std::cerr << "accept: " << e.what() << std::endl;
        close(client_fd);   // le ClientHandler n'existe pas → à nous de fermer
    }
    return 0;
}
Le même réflexe partout dans le code
  • EventLoop::register_handler() — si epoll_ctl(ADD) échoue : l'entrée est retirée de la table et delete avant le throw (pas d'entrée fantôme).
  • CGIProcess — pipe 2 raté : fermer le pipe 1. fork raté : fermer les 4 fd. fcntl raté : fermer les fd + kill(SIGKILL) + waitpid.
  • modify_handler / remove_handler — un échec d'epoll_ctl est loggué, pas throw : on est appelé depuis un handler en plein traitement, une erreur sur un fd ne doit pas faire tomber la boucle.
  • main() — chaque new ServerHandler est dans son propre try/catch : un port occupé n'empêche pas les autres de démarrer.
05
ErrorHandler
Une erreur = une réponse HTTP, jamais un silence
Le circuit d'une erreur

Quand le Router détecte un problème (404, 403, 405, 501…) ou que le parser a flaggé la requête (400, 413, 431), il retourne un ErrorHandler(bloc, code, body). Celui-ci hérite de ARequestHandler comme les autres : même buildResponse(), même circuit d'envoi. La différence : son body vient soit de la error_page configurée, soit d'une page générée.

srcs/http/ARequestHandler.cpp — getErrorPage(), le double filet
Notre code 
void ARequestHandler::getErrorPage() {
    _type = getType(_errorpage);

    std::ifstream file(_errorpage.c_str());
    // page d'erreur configurée mais introuvable → retomber sur la page
    // par défaut plutôt que de renvoyer un body vide
    if (!file.is_open()) {
        _type = getType(".html");
        _body = "<html><body><h1>" + itos(_ncode) + " " + getReason(_ncode) + "</h1></body></html>";
        return;
    }

    std::stringstream ss;
    ss << file.rdbuf();
    _body = ss.str();
    file.close();
}
Trois niveaux de fallback
  • 1. error_page 404 /.../404.html; configurée et lisible → on la sert.
  • 2. Configurée mais introuvable → page générée <h1>404 Not Found</h1>.
  • 3. Pas configurée du tout → même page générée, directement dans buildResponse().

Le client reçoit toujours un body et un status cohérents.

Nos codes en service

getReason() (utils.cpp) couvre : 200, 201, 204, 301, 302, 400, 403, 404, 405, 413, 431, 500, 501, 502, 504. Notre default.conf mappe chacun des codes d'erreur vers une page personnalisée dans www/default/errors/.

Résumé — l'erreur est locale
Le serveur entier ne paie jamais pour la faute d'un seul fd
🛡️
try/catch autour du dispatch
Toute exception (y compris bad_alloc) sacrifie le handler fautif : remove_handler + delete. La boucle epoll ne s'arrête jamais.
🛑
Signaux minimalistes
SIGINT/SIGTERM/SIGQUIT → g_stop (volatile sig_atomic_t). SIGPIPE → SIG_IGN + MSG_NOSIGNAL. Le nettoyage complet a lieu dans le destructeur d'EventLoop, pas dans le handler de signal.
🧹
Chaque échec nettoie derrière lui
Constructeurs avec rollback (close avant throw), register_handler qui annule son insertion, handle_accept qui ne retourne jamais -1, CGIProcess qui tue son enfant si fcntl rate.
📄
Toujours une réponse
error_page configurée → page perso ; introuvable → page générée ; rien de configuré → page générée. Une erreur est un status code, jamais une connexion abandonnée sans réponse.
