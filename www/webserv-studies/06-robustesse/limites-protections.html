Robustesse — Les garde-fous, dans notre code

Limites & protections
— mémoire bornée, racine verrouillée

Un serveur exposé au réseau reçoit du n'importe quoi : headers sans fin, bodies géants, URI en /../../etc/passwd, filenames d'upload piégés. Chaque entrée du client est bornée ou normalisée avant d'être utilisée : MAX_HEADER_SIZE, client_max_body_size par location, normalizePath(), et la sanitization des filenames multipart.

01
MAX_HEADER_SIZE → 431
Des headers sans fin ne doivent pas manger la RAM
L'attaque

Le parser accumule dans _buffer jusqu'à trouver \r\n\r\n. Un client malveillant qui envoie des headers sans jamais envoyer la ligne vide ferait grossir ce buffer à l'infini — un déni de service par la mémoire. La parade : un plafond de 32 Ko (#define MAX_HEADER_SIZE 32768 dans HttpParser.hpp) vérifié à chaque tentative.

srcs/http/HttpParser.cpp — treatHeader()
Notre code 
void HttpParser::treatHeader() {
    _req.isMultipart = false;
    size_t pos = _buffer.find("\r\n\r\n");
    if (pos == std::string::npos) {
        // garde-fou : des headers sans fin ne doivent pas faire grossir
        // le buffer indéfiniment (résilience exigée par le sujet)
        if (_buffer.size() > MAX_HEADER_SIZE)
            setError(431);   // Request Header Fields Too Large
        return;
    }
    // ...
}
💡
Testé par la suite maison. La page /default/stress.html envoie une requête avec un header monstrueux et vérifie que le serveur répond 431 avec la page www/default/errors/431.html configurée dans default.conf — sans consommer plus de 32 Ko.
02
client_max_body_size → 413
Limite par location, résolue dès les headers, body jeté pas stocké
Trois subtilités dans notre implémentation
  • La limite dépend de la location : /post_body accepte 100 octets, /directory 1 Go. On résout donc la bonne limite dès les headers, avant de lire le moindre octet de body, via findLocation() — la même fonction de longest-prefix que le Router.
  • On continue de lire… en jetant : répondre 413 immédiatement et fermer couperait le client en plein upload (il verrait une erreur réseau, pas notre 413). On draine le body sans le stocker, puis on répond.
  • Le chunked compte aussi : readChunked() incrémente _bodyReceived chunk par chunk et flague 413 dès dépassement.
srcs/http/HttpParser.cpp — headerParser() puis treatBody()
Notre code 
// --- dans headerParser() : résoudre la limite AVANT le body ---
// la limite de body dépend de la location matchée (client_max_body_size
// par location) : on la résout dès les headers, avant de lire le body
const Location* loc = findLocation(_req.uri.substr(0, _req.uri.find('?')), *_server);
if (loc)
    _maxBodySize = loc->max_body_client;

findHeaders();
if (_bodyExcepted > _maxBodySize)
    return setError(413);   // Content-Length annonce déjà trop → flag immédiat


// --- dans treatBody() : drainer sans stocker ---
if (_errorCode == 413) {
    // corps trop gros : jeter les données au lieu de les
    // accumuler, pour ne pas exploser la mémoire
    _buffer.erase(0, acceptable);
} else {
    _body.append(_buffer, 0, acceptable);
    _buffer.erase(0, acceptable);
}
srcs/http/HttpParser.cpp — readChunked(), le comptage exact
Bug corrigé — le faux 413 
size_t needed = pos + 2 + size + 2;
if (_buffer.size() < needed)
    return;

// compter le chunk UNIQUEMENT quand il est entier dans le buffer :
// avant, l'incrément se faisait avant le check de complétude, donc un
// chunk arrivant en plusieurs recv() était re-compté à chaque appel
// (jusqu'à ~16x avec des chunks de 64k et un buffer de 4k) → faux 413
_bodyReceived += size;
if (_bodyReceived > _maxBodySize) {
    setError(413);
}
⚠️
setError(413) ne bascule pas l'état tout de suite. Regarde setError() : pour un 413 en mode normal, l'état reste R_BODY — on continue de consommer (et jeter) jusqu'à _bodyReceived == _bodyExcepted, et c'est seulement là que _state = ERROR déclenche la réponse. Le client peut finir son envoi puis lire notre 413 proprement.
03
normalizePath() → 403
GET /../../etc/passwd ne sortira pas du root
L'algorithme : résoudre puis comparer

Un chemin comme www/../../etc/passwd doit être résolu avant d'être comparé à la racine autorisée. letNormalize() découpe le chemin en segments : . et les segments vides sont ignorés, .. dépile le segment précédent. Puis normalizePath() vérifie que le résultat commence toujours par le root normalisé — sinon chaîne vide, que le Router transforme en 403.

srcs/utils/utils.cpp — normalizePath()
Notre code 
std::string normalizePath(const std::string& p, const std::string& root) {
    std::string nroot = letNormalize(root);
    std::string res = letNormalize(p);

    if (res.compare(0, nroot.size(), nroot) != 0)
        return "";   // le chemin résolu sort du root → refusé

    // "wwwevil" commence bien par "www" : vérifier que le caractère
    // suivant est un '/' pour ne pas accepter un préfixe partiel
    if (res.size() > nroot.size() && res[nroot.size()] != '/')
        return "";
    return res;
}
srcs/http/Router.cpp — la sanction
Notre code 
std::string path = resolvePath(loc, uriPath);

Et l'encodage %2e%2e ? Pas d'échappatoire : checkFirstLine() décode les %XX de l'URI (decodeHexa()) avant le routing. Un /%2e%2e/%2e%2e/etc/passwd devient /../../etc/passwd… et se fait attraper par normalizePath comme les autres.
04
MultipartHandler
Le filename vient du client — donc il ment peut-être
L'attaque de l'upload

Dans un POST multipart, le nom du fichier est fourni par le client dans Content-Disposition: form-data; name="file"; filename="...". Rien ne l'empêche d'envoyer filename="../../../home/user/.bashrc" : concaténé naïvement à upload_store, il écrirait hors du dossier d'upload. La parade : ne garder que le basename.

srcs/http/MultipartHandler.cpp
Notre code 
for (i = _req->mp.uploadedFiles.begin(); i != _req->mp.uploadedFiles.end(); i++) {
    // ne garder que le nom de fichier : un filename client contenant
    // "../../x" permettait d'écrire hors du dossier d'upload
    std::string fname = i->second.filename;
    size_t slash = fname.rfind('/');
    if (slash != std::string::npos)
        fname = fname.substr(slash + 1);   // "../../x" → "x"
    if (fname.empty())
        continue;

    std::string   path  = _path + "/" + fname;
    std::ofstream file(path.c_str(), std::ios::binary);
    if (!file.is_open()) {
        _ncode = 500;                          // dossier absent ou non inscriptible
        body += "<li>" + fname + " failed</li>";
        continue;
    }
    // ...écriture binaire + 201 si création...
}
upload_store, pas root + URI

Exigence du sujet : la config décide où vont les uploads. Dans Router::route(), un POST sur une location avec upload_store voit son chemin cible remplacé par ce dossier (./www/upload chez nous). Le Validator vérifie d'ailleurs à froid que upload_store s'accompagne bien de POST dans allowed_methods.

Écriture binaire, taille exacte

file.write(data.c_str(), size) en mode std::ios::binary : une image PNG uploadée ressort octet pour octet identique. Le MultipartParser a découpé les parts sur le boundary exact, y compris pour plusieurs fichiers partageant le même nom de champ (d'où le std::multimap dans la struct Multipart).

05
access() → 403 / 404
Vérifier les permissions avant d'agir — le bon mode pour la bonne méthode
srcs/http/Router.cpp — forbiddenAccess()
Notre code 
int Router::forbiddenAccess(const std::string& uri, const std::string& method) {
    int         mode = 0;
    std::string pathtoaccess = uri;

    if (method == "DELETE")
        mode = W_OK;                              // supprimer = pouvoir écrire
    else if (method == "GET")
        mode = R_OK;                              // lire = pouvoir lire
    else if (method == "POST")
        pathtoaccess = getParentDirectory(uri);   // créer = le DOSSIER doit exister/être accessible
    if (access(pathtoaccess.c_str(), mode) == -1)
        return 1;
    return 0;
}

Seulement maintenant : CGI, multipart ou statique. Le filesystem n'est touché qu'après tous les refus « gratuits ».
💡
Fichiers réguliers : exemptés d'epoll, pas de précautions. Le sujet exempte les fichiers disque de la règle epoll — c'est pourquoi StaticHandler lit avec un simple std::ifstream, écrit avec std::ofstream, liste avec opendir/readdir/closedir et supprime avec unlink(). Mais « exempté d'epoll » ne veut pas dire « exempté de checks » : stat, access et is_open() gardent chaque opération.
Résumé — ne jamais faire confiance au client
Chaque entrée est bornée, normalisée ou vérifiée avant usage
📏
Mémoire bornée
Headers plafonnés à 32 Ko (431). Body plafonné par location (413), résolu dès les headers, drainé sans stockage en cas de dépassement — chunked compris.
🔒
Racine verrouillée
URI décodée (%XX) puis normalisée segment par segment ; tout chemin résolu qui sort du root → 403. Le filename multipart est réduit à son basename avant écriture.
🛡️
Permissions vérifiées par méthode
GET → R_OK, DELETE → W_OK, POST → accès au dossier parent. Ordre du Router : erreurs du parser → 404/501/405 → redirect → 403 traversal → 404 → 403 access → traitement.
