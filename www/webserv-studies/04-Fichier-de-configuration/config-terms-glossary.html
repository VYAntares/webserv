Glossaire des termes à maîtriser

36|

37| Chaque directive expliquée avec ce qu'elle fait réellement à runtime — pas juste sa définition. 38| Les directives core NGINX (server, listen, location, root…) 39| sont alignées sur le module HTTP core officiel. Les directives custom webserv (allowed_methods, 40| upload_store, cgi_pass) sont expliquées avec leur effet attendu dans ton implémentation. 41|

68|    listen               8080;
69|    server_name          localhost;
70|    client_max_body_size 10m;
71|
72|    error_page 404 /errors/404.html;
73|    error_page 500 /errors/500.html;
74|
75|    location / {
76|        root            /var/www/html;
77|        index           index.html;
78|        allowed_methods GET;
79|        autoindex       off;
80|    }
81|
82|    location /api {
83|        root            /var/www/api;
84|        allowed_methods GET POST DELETE;
85|        autoindex       off;
86|    }
87|
88|    location /upload {
89|        root            /var/www;

102|    }

109|    }

Structure globale
129|

130| Deux blocs server = deux serveurs virtuels distincts. 131| Chacun écoute un port différent (8080 et 8081). 132| Dans chaque serveur, les blocs location définissent les règles par préfixe d'URI. 133|

134|

135| Webserv lit ce fichier une seule fois au démarrage, construit ses structures en mémoire, 136| puis les applique à chaque requête sans jamais relire le fichier. 137|

Core NGINX vs directives custom
141|

142| server, listen, server_name, location, 143| root, index, autoindex, error_page, 144| client_max_body_size, return → directives du module HTTP core NGINX. 145|

146|

147| allowed_methods, upload_store, cgi_pass → directives 148| custom webserv, inspirées du sujet mais non standard NGINX. 149|

directives core
164|
Ce que chaque mot-clé fait exactement
165|
166|
167| 168|
169|
170|
server { ... }
171|

Définit un serveur virtuel HTTP. Le bloc rassemble toutes les règles d'un "site" logique : ports d'écoute, pages d'erreur, limite de body, routes.

172|
Contexte: http • Ce fichier: 2 serveurs (8080, 8081) dans un même processus
173|
174| 175|
176|
listen address:port;
177|

Définit le port (et optionnellement l'adresse IP) sur lequel ce serveur accepte les connexions. Chaque listen distinct = un socket serveur ouvert par webserv.

178|
Contexte: server • Exemple: listen 8080; — listen 8081;
179|
180| 181|
182|
server_name name ...;
183|

Nomme le serveur virtuel. Utile pour le virtual hosting : si deux server partagent le même port, le header Host de la requête est comparé aux server_name pour choisir le bon bloc. Optionnel dans webserv.

184|
Contexte: server • Exemple: server_name localhost;
185|
186| 187|
188|
client_max_body_size size;
189|

Taille maximale autorisée pour le corps d'une requête. Le check se fait tôt, sur le header Content-Length, avant de lire le body. Si dépassé : réponse 413 Request Entity Too Large immédiate.

190|
Contexte: http, server, location • Exemples: 10m (serveur 8080), 1m (serveur 8081)
191|
192| 193|
194|
error_page code ... uri;
195|

Associe un code d'erreur HTTP à une page personnalisée. Si le serveur produit ce code, il tente de servir la page. Si la page elle-même est introuvable, il faut un fallback HTML inline — jamais de réponse vide.

196|
Contexte: http, server, location • Exemples: 404 /errors/404.html — 500 /errors/500.html
197|
198| 199|
200|
location [prefix] uri { ... }
201|

Applique une configuration en fonction de l'URI de la requête. Le matching se fait par plus long préfixe : /api/users matche /api plutôt que /. Pas de regex — préfixe uniquement.

202|
Contexte: server, location • Exemples: /, /api, /upload, /files, /old-page, /cgi-bin
203|
204| 205|
206|
root path;
207|

Répertoire racine pour résoudre le chemin physique d'un fichier. La règle du sujet (Cas 2) : chemin = root + (URI − préfixe location). Exemple : location /api + root /var/www/api + URI /api/users.json → fichier à /var/www/api/users.json.

208|
Contexte: http, server, location • Exemple: root /var/www/html;
209|
210| 211|
212|
index file ...;
213|

Fichier servi automatiquement quand l'URI pointe sur un répertoire. Le serveur vérifie l'index avant d'envisager autoindex. Si l'index est absent et autoindex off : 403 Forbidden (répertoire inaccessible). Si le chemin lui-même n'existe pas : 404 Not Found.

214|
Contexte: http, server, location • Exemple: index index.html;
215|
216| 217|
218|
autoindex on | off;
219|

Active le listing du contenu d'un répertoire quand aucun fichier index n'est trouvé. Génère une réponse HTML avec les entrées du dossier. Implémenté avec opendir / readdir.

220|
Contexte: http, server, location • Exemple: /files = on, toutes les autres locations = off
221|
222| 223|
224|
return code [URL|URI];
225|

Renvoie immédiatement un statut HTTP sans résoudre de fichier. Court-circuite tout le reste du pipeline (root, index, CGI, upload). Utilisé pour les redirections permanentes (301) ou temporaires (302).

226|
Contexte: server, location • Exemple: return 301 /new-page;

Analogie — le répertoire téléphonique
234|

235| Le fichier de config, c'est un répertoire téléphonique structuré. 236| Chaque bloc server est une entreprise (son numéro = le port). 237| Les blocs location sont les services internes de cette entreprise 238| ("/accueil", "/comptabilité", "/archives"). 239|

240| Quand un appel arrive, on regarde d'abord au bon numéro de téléphone (port), 241| puis on lit l'annuaire interne pour trouver le service le plus précis. 242| return = "le standard te redirige directement, tu n'attends pas". 243|

244|
245|
246| 247|
248|
🔑client_max_body_size : le check se fait sur Content-Length avant de lire le body. Ne pas attendre la fin de la lecture pour rejeter.
249|
📐Résolution de chemin : root + (URI − préfixe_location). C'est le Cas 2 du sujet — toujours retirer le préfixe de la location avant de concaténer avec root.
250|
⚡return court-circuite tout. Aucune résolution de fichier, aucun CGI, aucun upload. La réponse est envoyée dès que la directive est trouvée.

Analogie — le centre de tri postal
277|

278| Imagine un centre de tri qui reçoit des colis. 279| Le code postal (port) détermine d'abord à quelle agence le colis est affecté (bloc server). 280|

281| Dans l'agence, le début de l'adresse (préfixe d'URI) détermine le service : 282| un colis pour "/api/..." va au guichet "/api", pas au guichet généraliste "/". 283| On choisit toujours le guichet dont l'adresse est la plus précise (plus long préfixe). 284|

285| Ensuite, l'adresse physique finale est calculée : root + le reste de l'adresse sans le préfixe 286| → c'est là où le fichier est lu sur le disque. 287|

Les 3 étapes du routage
294|

295| 1. Sélection du serveur : le port d'arrivée de la connexion détermine quel bloc server répond. 296| Si plusieurs serveurs partagent le même port, on compare le header Host avec server_name. 297|

298|

299| 2. Sélection de la location : on compare l'URI à tous les préfixes déclarés. 300| La location dont le préfixe est le plus long et qui matche l'URI est retenue. 301|

302|

303| 3. Résolution du chemin : root + (URI − préfixe_location). 304| Le préfixe est retiré de l'URI avant de concaténer avec root. 305|

306|
307|
308|
Exemples de résolution complète
309|

310| GET /api/users.json
311| → location choisie : /api (plus long que /)
312| → root /var/www/api + /users.json
313| → fichier : /var/www/api/users.json 314|

315|

316| GET /files/doc.pdf
317| → location choisie : /files
318| → root /var/www/shared + /doc.pdf
319| → fichier : /var/www/shared/doc.pdf — puis autoindex on si dossier 320|

321|

322| GET /old-page
323| → location choisie : /old-page
324| → return 301 /new-page → réponse immédiate, pas de fichier 325|

GET /api/users HTTP/1.1
338|  → port 8080 → server A
339|  → plus long préfixe : location /api
340|  → root /var/www/api + /users → /var/www/api/users
341|
342|GET /files/ HTTP/1.1
343|  → port 8080 → server A
344|  → plus long préfixe : location /files
345|  → root /var/www/shared + / → tente index → autoindex on → listing
346|
347|GET /old-page HTTP/1.1
348|  → port 8080 → server A
349|  → plus long préfixe : location /old-page
350|  → return 301 /new-page → réponse immédiate
351|
352|POST /upload HTTP/1.1  (body: 15 Mo)
353|  → port 8080 → server A
354|  → Content-Length 15m > client_max_body_size 10m → 413 immédiat
355|
356|POST /cgi-bin/script.py HTTP/1.1
357|  → port 8080 → server A

📏Le matching est toujours par plus long préfixe. /api/v2/users matche /api/v2 si elle existe, sinon /api, jamais les deux.
364|
➖Formule path = root + (URI − préfixe). Webserv retire le préfixe de la location avant de coller à root — c'est l'exemple exact du sujet (/kapouet/pouic/toto → /tmp/www/pouic/toto).
365|
🚦Ordre de priorité dans la location : return en premier → puis path resolution → puis CGI si extension matche → sinon fichier statique.

Ce que chaque directive change dans la réponse HTTP
380|
381|
382| 383|
384|
385|
error_page 404 /errors/404.html
386|

Quand le serveur ne trouve pas la ressource demandée, il tente de servir cette page d'erreur dédiée. Si elle-même est absente : réponse 404 avec un body HTML minimal inline.

387|
388| 389|
390|
client_max_body_size 10m
391|

Sur le serveur 8080, tout body déclaré ≥ 10 Mo est rejeté en 413 dès la lecture du header Content-Length, sans toucher au body. Sur 8081 : limite de 1 Mo.

392|
393| 394|
395|
root + URI (path resolution)
396|

397| location /api + root /var/www/api + URI /api/data.json
398| → retire le préfixe /api
399| → ajoute le suffixe /data.json à root
400| → lit /var/www/api/data.json sur le disque. 401|

402|
403| 404|
405|
index index.html
406|

Requête sur GET / → URI est un répertoire → le serveur tente /var/www/html/index.html. Si absent et autoindex off : 403 (répertoire existe mais inaccessible). Si le répertoire lui-même n'existe pas : 404.

407|
408| 409|
410|
autoindex on (location /files)
411|

Requête sur GET /files/ → si aucun index trouvé → le serveur génère une page HTML listant le contenu du dossier via opendir + readdir. Réponse 200 avec ce listing.

412|
413| 414|
415|
return 301 /new-page
416|

Requête sur GET /old-page → réponse immédiate : 301 Moved Permanently avec header Location: /new-page. Zéro accès disque, aucun CGI, aucun traitement body.

417|
418|
419| 420|
421|
🗂️Ordre de résolution : chemin absent → 404. Chemin est un fichier → 200. Chemin est un répertoire : index présent → 200 ; index absent + autoindex on → listing 200 ; index absent + autoindex off → 403. Le 403 ne s'applique qu'aux répertoires qui existent — jamais à un chemin manquant.
422|
🔁La page error_page doit être résolvable par tes propres règles root/location. Une page d'erreur introuvable → fallback body inline obligatoire (jamais de réponse vide).
423|
🧮Toutes ces décisions — 413, 301, listing, index, 403 — sont prises à runtime sur chaque requête, à partir des structures C++ chargées une fois au démarrage.

Directives non standard NGINX
443|

444| allowed_methods, upload_store et cgi_pass ne sont pas dans le module core officiel NGINX. 445| Elles sont conçues pour répondre explicitement aux exigences du sujet webserv. 446|

447|
448| 449|
450|
451|
allowed_methods GET POST DELETE
452|

453| Liste les méthodes HTTP autorisées pour cette location. 454| Si la méthode de la requête n'est pas dans la liste : 455| réponse 405 Method Not Allowed avec le header Allow: GET POST DELETE 456| listant les méthodes acceptées. 457|

458|

459| La vérification se fait tôt dans le pipeline, avant la résolution de chemin ou le CGI. 460|

461|
462|
463|
upload_store /tmp/uploads
464|

465| Dossier de destination pour les fichiers uploadés via multipart/form-data. 466| Le serveur décode le body (boundary splitting), extrait le fichier, et écrit le contenu binaire 467| dans ce dossier. 468|

469|

470| Le dossier doit exister et être writable. À vérifier à la validation de la config, 471| pas à runtime. Nettoyer le nom de fichier : path traversal (..), caractères interdits, 472| collisions. 473|

474|
475|
476| 477|
478|
cgi_pass .py /usr/bin/python3 + .php /usr/bin/php-cgi
479|

480| Associe une extension de fichier à un interpréteur CGI. Quand l'URI correspond à un fichier 481| avec cette extension, le serveur l'exécute via fork + execve. 482|

483|

484| Le processus enfant reçoit les variables CGI en environnement (REQUEST_METHOD, 485| QUERY_STRING, CONTENT_TYPE, CONTENT_LENGTH, etc.), 486| le body via stdin, et retourne sa réponse sur stdout. 487| Le serveur fait chdir dans le répertoire du script avant execve 488| pour que les chemins relatifs fonctionnent. 489|

490|

491| Règle du sujet : fork est réservé exclusivement aux CGI. 492| Le process enfant doit être attendu avec waitpid. 493| Si le CGI ne retourne pas de Content-Length, EOF sur stdout marque la fin. 494|

495|
496| 497|
498| ⚠️ 499|
fork est une règle dure : utiliser fork en dehors du CGI = note 0 immédiate selon le sujet. La seule exception autorisée est l'exécution d'un script CGI via fork + execve.
500|
501| 502|
503|
🔱fork réservé au CGI uniquement. Le processus enfant hérite des pipes (stdin/stdout), des variables d'environnement CGI, et doit être waitpid-é.
504|
🔒upload_store : valider l'existence et les droits du dossier au démarrage. Sanitizer le nom de fichier (retirer .., espaces, caractères interdits) avant l'écriture.
505|
🚫allowed_methods → 405 + header Allow: obligatoire. La vérification se fait avant la résolution de chemin — ne pas chercher le fichier si la méthode est interdite.

checklist d'implémentation
519|
Ce que ton code doit prouver pour chaque terme
520|
521|
522| 523|
524|
🏗️Chaque server { } correspond à une structure Server en mémoire. Chaque location { } à une Location. Chargées au démarrage, immutables ensuite.
525|
🔌Chaque port distinct dans les blocs listen = un socket serveur ouvert. Tous ces sockets sont enregistrés dans poll() / epoll dès le démarrage.
526|
📏Le matching location est déterministe : itérer tous les préfixes, retenir le plus long qui matche l'URI. Implémenter un test avec plusieurs locations chevauchantes.
527|
📁La résolution de chemin suit root + (URI − préfixe_location). Vérifier avec l'exemple du sujet : /kapouet/pouic/toto + root /tmp/www → /tmp/www/pouic/toto.
528|
📊Les codes de statut doivent être précis : 200, 301, 403, 404, 405, 413, 500. Tester chaque cas avec curl ou le navigateur.
529|
🧪Fournir un fichier de config de test qui couvre tous les cas : statique, redirection, 404 custom, 413, autoindex, upload, CGI. Le sujet demande explicitement ces fichiers pour la soutenance.
