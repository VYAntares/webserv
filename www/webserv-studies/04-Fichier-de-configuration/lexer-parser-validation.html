Le problème concret

Ton serveur lit un texte libre avec des accolades, points-virgules, commentaires, directives, etc.

Si tu tentes de tout parser "à la main" en un seul bloc, tu te retrouves vite avec un code fragile, des erreurs difficiles à localiser, et une couverture incomplète des cas d'erreur.

La séparation en 3 phases rend le système stable et lisible :

  • Lexer : découpe le texte en briques atomiques (tokens).
  • Parser : comprend la grammaire et remplit tes structures C++.
  • Validation : vérifie la cohérence avant de lancer le serveur.
🏗️
Analogie Lego

Lexer = tu sors toutes les pièces Lego de la boîte et tu les tries par forme.

Parser = tu assembles selon le plan de montage, pièce par pièce.

Validation = tu vérifies que la maison tient debout avant d'y habiter.

💡
Point clé soutenance : ces 3 phases se font au startup uniquement, pas à chaque requête HTTP. Le runtime utilise simplement les structures déjà construites et validées. Echec au startup = message clair. Echec au runtime = catastrophe silencieuse.
① startup vs runtime : tout le travail de parsing est fait une seule fois au démarrage.
② séparation des responsabilités : chaque phase n'a qu'une seule mission — plus facile à tester et déboguer.
③ fail fast : une config invalide doit faire exit(1) avec un message explicite avant tout bind().
02
lexer()
C'est quoi un lexer, exactement ?
Définition claire

Le lexer (tokenizer) lit le fichier caractère par caractère et produit une suite de tokens.

Un token = une "brique atomique" utile au parser : mot, {, }, ;, fin de fichier.

Il ignore les espaces et les commentaires (#). Il peut aussi stocker le numéro de ligne pour afficher des erreurs précises ("syntax error line 12").

Le lexer ne comprend pas la grammaire. Il ne sait pas si listen est bien placé — c'est le parser qui décide.

✂️
Analogie scanner de caisse

Le lexer lit le texte comme un scanner lit des codes-barres : il identifie chaque article (token) un par un, sans savoir ce que tu vas cuisiner.

C'est le caissier (parser) qui vérifie que tu as bien un plat et ses ingrédients.

enum TokenType {
	TOK_WORD,	// listen, GET, 8080, tout ce qui n'est pas un symbole
	TOK_LBRACE,	// {
	TOK_RBRACE,	// }
	TOK_SEMI,	// ;
	TOK_EOF		// évite de checker _pos < _token.size()
};

struct Token {
	TokenType	type;
	std::string	value;
	size_t		line;
	size_t		col;	// débogage
};

Notre code 
// prend l'input et retourne un vecteur de tokens.
std::vector<Token> Lexer::tokenize() {
	std::vector<Token> token;
	while (!atEnd()) {
		skipWhitespacesAndComments();	// espace, tab, \\r, \\n + commentaires #
		if (atEnd())
			break;
		char c = current();
		if (c == '{') {
			token.push_back(makeToken(TOK_LBRACE, "{"));
			advance();
		} else if (c == '}') {
			token.push_back(makeToken(TOK_RBRACE, "}"));
			advance();
		} else if (c == ';') {
			token.push_back(makeToken(TOK_SEMI, ";"));
			advance();
		} else if (c == '"' || c == '\'')
			token.push_back(readQuote(c));	// gestion quotes simples et doubles
		else
			token.push_back(readWord());
	}
	token.push_back(makeToken(TOK_EOF, ""));
	return token;
}
⚠️
Le lexer ne "comprend" pas la config. Il découpe seulement. Si un token listen apparaît dans un bloc location, c'est le parser (puis la validation) qui doit le refuser — pas le lexer.
① 5 types de tokens suffisent pour le format nginx-like : TOK_WORD, TOK_LBRACE, TOK_RBRACE, TOK_SEMI, TOK_EOF.
② stocker le numéro de ligne ET de colonne dans chaque token te permet d'afficher "erreur line 12 col 5" exacte lors du parsing.
③ ignorer # en sautant tout jusqu'au \n suivant — via skipWhitespacesAndComments(), les commentaires disparaissent avant le parser.
03
parser()
Que fait le parser ?
Son rôle exact

Le parser lit la séquence de tokens et applique les règles de grammaire : ce qui est autorisé, où, et dans quel ordre.

Règles de base :

  • Un bloc server doit ouvrir avec { et fermer avec }.
  • Une directive simple se termine par ;.
  • location /path { ... } est un sous-bloc dans server.
  • Certaines directives acceptent plusieurs arguments (allowed_methods GET POST;).

Le parser remplit ensuite les structures Server / Location (définies dans ConfigStruct.hpp).

🧠
Analogie grammaire

Le lexer reconnaît les mots. Le parser construit des phrases grammaticalement correctes.

"listen 8080 ;" → sujet + complément + point final = phrase valide.
"8080 listen ;" → même mots, grammaire incorrecte → rejeté.

// parse un bloc server : appelle le parser de location ou parse une directive
void	Parser::parseServerBlock() {
	Server s;

	consume();				// mange le token "server"
	expect(TOK_LBRACE);

	while (current().type != TOK_RBRACE) {
		if (current().value == "location")
			parseLocationBlock(s);
		else if (current().type == TOK_WORD)
			parseDirective(s);
		else if (current().type == TOK_EOF)
			throw std::runtime_error("unclosed server block");
		else
			throw std::runtime_error("unexpected token");
	}
	expect(TOK_RBRACE);

	// max_body_client à 1MB par défaut si non défini
	if (s.max_body_client == SIZE_MAX)
		s.max_body_client = 1024 * 1024;
	_c.cfg.push_back(s);
}
🧩
Résumé parser : il transforme une suite de tokens en structures métier utilisables au runtime. Sans parser, tu as des tokens. Avec parser, tu as des Server et Location (définis dans ConfigStruct.hpp) prêts à être exploités et validés.
① recursive descent : parseServer() appelle parseLocation() — structure naturelle qui suit la hiérarchie de la config.
② allowed_methods est multi-valeurs : boucle sur les WORD jusqu'au SEMICOLON — ne pas oublier ce cas particulier.
③ erreur inconnue immédiate : directive non reconnue → throw avec le numéro de ligne. Ne jamais silencieusement ignorer un token inconnu.
04
validate()
La validation, c'est quoi ?
La différence avec le parser

Parser = validité syntaxique ("est-ce que le fichier est bien écrit ?").

Validation = validité fonctionnelle ("est-ce que ça a du sens pour lancer un serveur HTTP ?").

Exemples de checks de validation spécifiques à notre webserv :

  • Pas de directive listen dans un serveur → erreur.
  • Aucun bloc location dans un server → erreur.
  • Duplicat de listen entre deux serveurs → refusé.
  • Méthode inconnue ou dupliquée dans allowed_methods (seuls GET/POST/DELETE valides).
  • root OU return obligatoire dans chaque location.
  • error_page path doit être absolu (commence par /).
  • upload_store exige POST dans allowed_methods.
  • Extension cgi_pass doit commencer par ., interpréteur par /.
  • Propagation server→locations : root, index, max_body, autoindex, return, error_page si absents dans la location.
🔍
Analogie inspection de chantier

Le parser vérifie que les plans respectent le format standard.

La validation est l'inspecteur qui va sur place vérifier que les fondations existent, que les portes s'ouvrent, que les câbles sont branchés.

Une belle maison sur des plans incorrects → bloquée au parser.
Des plans corrects mais sans fondations → bloquée à la validation.

srcs/config/Validator.cpp — validate()
Notre code 
Config Validator::validate() {
	if (_c.cfg.empty())
		throw std::runtime_error("Config is empty.");

	for (size_t i = 0; i < _c.cfg.size(); i++) {
		Server& s = _c.cfg[i];
		if (s.listen.empty())
			throw std::runtime_error("No 'listen' directive in server");
		if (s.locations.empty())
			throw std::runtime_error("No location block in Server " + toString(i+1));
		checkDuplicateIntraServer(s, i);
		propogateServerToLocations(s);
		for (size_t j = 0; j < s.locations.size(); j++)
			checkLocationConfig(s.locations[j]);
	}
	return _c;
}

// propagation des directives du server vers ses locations
void Validator::propogateServerToLocations(Server& s) {
	for (size_t i = 0; i < s.locations.size(); i++) {
		Location& l = s.locations[i];
		if (l.root.empty())              l.root = s.root;
		if (l.index.empty())             l.index = s.index;
		if (l.max_body_client == SIZE_MAX)  l.max_body_client = s.max_body_client;
		if (l.autoindex == -1)             l.autoindex = s.autoindex;
		if (l.return_path.first == -1)     l.return_path = s.return_path;
		// propagation des error_page : le server définit, la location hérite si absent
		for (std::map<int, str>::iterator it = s.error_page.begin();
			it != s.error_page.end(); ++it) {
				if (l.error_page.find(it->first) == l.error_page.end())
					l.error_page[it->first] = it->second;
			}
	}
}
🚫
Ne lance jamais le serveur avec une config invalide. Mieux vaut exit(1) au startup avec un message clair que planter en plein traitement de requête. Un serveur qui démarre avec une config incohérente est un bug caché garanti.
① parser ≠ validation : syntaxe correcte ne signifie pas config fonctionnelle — les deux phases sont indispensables.
② propagation automatique : propogateServerToLocations() hérite root, index, max_body, autoindex, return, error_page du server vers chaque location si absent.
③ messages d'erreur précis : chaque throw indique exactement quelle directive est invalide et pourquoi — le constructeur ConfigLoader ajoute le nom du fichier en prefix.

Notre code 
// ConfigLoader vérifie le bon lancement du programme, lit le fichier,
// puis lance le parsing : lexer → parser → validator.
ConfigLoader::ConfigLoader(int argc, char **argv) {
	if (argc > 2)
		throw std::runtime_error("Too many arguments, usage: ./webserv [configFile path]");

	std::string path = (argc == 2) ? std::string(argv[1]) : DEFAULT_CONFIG_PATH;
	if (argc == 1)
		std::cerr << "No config file provided, using default: " << DEFAULT_CONFIG_PATH;

	// vérification de l'extension .conf
	if (path.find(".conf") == std::string::npos)
		throw std::runtime_error("Config file must have: .conf extension.");

	// stat() vérifie que le path est un fichier régulier (S_ISREG)
	struct stat st;
	if (stat(path.c_str(), &st) != 0 || !S_ISREG(st.st_mode))
		throw std::runtime_error("Not a regular file: " + path);

	// lecture du contenu via ifstream → ostringstream
	std::ifstream file(path.c_str());
	if (!file.is_open()) throw std::runtime_error("Can not open config file: " + path);

	std::ostringstream ss;
	ss << file.rdbuf();
	_input = ss.str();
	if (_input.empty()) throw std::runtime_error("Config file is empty.");

	try {
		startLexer();		// phase 1 : Lexer
		startParser();		// phase 2 : Parser
		startValidator();	// phase 3 : Validator
	} catch (std::exception &e) {
		throw std::runtime_error(std::string(e.what())
			+ "\nparser: configuration file " + path + " failed.");
	}
}
Lecture runtime

Une fois ce pipeline terminé, chaque requête HTTP fait seulement :

  • Choix du Server par port (et Host: header si plusieurs server_name).
  • Choix de la Location (préfixe le plus long via findLocation()).
  • Application des règles déjà validées — aucun accès fichier de config en runtime.

Le travail "lourd" est déplacé au startup pour garder le runtime propre et robuste.

Que se passe-t-il en cas d'erreur ?
  • Fichier config introuvable → readFile() lance une exception → exit(1).
  • Syntaxe cassée (accolade manquante) → lexer/parser lèvent une exception → exit(1).
  • Config incohérente (port invalide, CGI manquant) → validation lève une exception → exit(1).

Dans tous les cas, le serveur ne démarre pas et affiche le message précis sur stderr.

✅
Réponse directe soutenance : lexer découpe, parser structure, validation sécurise. Les trois phases se font au startup et sont obligatoires pour un webserv stable. Le runtime n'accède jamais au fichier de config.
① try/catch global dans main() : toute exception du pipeline devient un exit(1) propre avec message stderr.
② l'ordre des 3 phases est fixe : lex → parse → validate. Impossible d'inverser — le parser a besoin des tokens, la validation a besoin des structs.
③ runtime = lecture seule des structs Server / Location. Aucun re-parsing pendant le traitement des requêtes.
