Ce que fait un fichier de config dans webserv
59|59|

Le problème réel
65|65|

66|66| Sans config, ton serveur est figé : un seul port, un seul dossier, une seule politique HTTP. 67|67| Avec un fichier de configuration, tu externalises ces décisions : ports d'écoute, routes, 68|68| méthodes autorisées, erreurs, CGI, upload, etc. 69|69|

70|70|

71|71| Et surtout : un seul binaire peut gérer plusieurs serveurs logiques. 72|72| Chaque bloc server décrit un comportement distinct (port, routes, limites), 73|73| mais tout tourne dans le même programme webserv. 74|74|

Pipeline conceptuel
79|79|
    80|80|
  • Lire le fichier brut.
    • 81|81|
  • Tokeniser (WORD, {, }, ;, commentaires).
    • 82|82|
  • Parser en structures Server / Location.
    • 83|83|
  • Valider toute la config au startup.
    • 84|84|
  • Appliquer la config à chaque requête entrante.

Analogie — le manuel d'exploitation
93|93|

94|94| Le fichier de config, c'est le manuel d'exploitation laissé dans la salle serveur. 95|95| Le serveur le lit une seule fois au démarrage, charge toutes les règles en mémoire, 96|96| puis les applique à chaque requête sans jamais re-lire le fichier. 97|97|

98|98| Tu ne changes pas le comportement en modifiant le binaire — tu mets à jour le manuel. 99|99| Et si le manuel est mal écrit (config invalide), le serveur refuse de démarrer 100|100| plutôt que de fonctionner avec des règles incohérentes. 101|101|

165|165|    server {
166|166|      listen 8081;
167|167|      client_max_body_size 1m;
168|168|
169|169|      location / {
170|170|        root            /var/www/site2;
171|171|        index           index.html;
172|172|        allowed_methods GET;
173|173|        autoindex       off;
174|174|      }
175|175|    }
176|176|
177|177| 178|178|
179|179| 📌 180|180|
Point clé (souvent flou) : les deux blocs server sont chargés en mémoire au startup. 181|181| Ensuite, une requête entrante est d'abord routée par son port local (8080 ou 8081), 182|182| puis seulement par les location du serveur sélectionné.
183|183|
184|184| 185|185|
186|186|
🔄Le pipeline config est séquentiel : lire → tokeniser → parser → valider → appliquer. Chaque étape doit se terminer sans erreur avant la suivante.
187|187|
💾Toute la config est en mémoire dès le démarrage. Pas de relecture à chaque requête. Les structures Server / Location sont immutables après validation.
188|188|
🌐Un seul binaire = plusieurs serveurs logiques. Chaque bloc server est un "site" distinct, distingué par son port (et optionnellement son server_name).

Le bloc server
205|205|

206|206| Un bloc server { ... } décrit une "personnalité" complète du serveur : ports d'écoute, 207|207| règles de routes, limites de body, pages d'erreur, CGI, etc. 208|208| Si tu écris plusieurs blocs server, webserv n'ouvre pas plusieurs programmes : 209|209| il ouvre plusieurs configurations et choisit la bonne à chaque connexion. 210|210|

listen en pratique
236|236|
    237|237|
  • Port valide : 1..65535.
    • 238|238|
  • Port < 1024 : droits élevés souvent requis en Linux.
    • 239|239|
  • Chaque listen implique un socket serveur distinct.
    • 240|240|
  • Plusieurs blocs server peuvent coexister dans un seul fichier et un seul process.
    • 241|241|
  • Avec epoll, chaque socket d'écoute est enregistré en EPOLLIN.
    • 242|242|

server_name (optionnel webserv)
246|246|

247|247| server_name permet de faire des virtual hosts par header Host. Le sujet webserv ne l'exige 248|248| pas strictement. Tu peux l'ignorer au début, puis l'ajouter si tu veux un routing plus proche de NGINX. 249|249|

250|250|
251|251|
Algorithme si implémenté
252|252|

1) filtrer par port d'arrivée, 2) comparer Host avec les server_name, 3) fallback sur serveur par défaut du port.

Analogie — les guichets d'une poste
295|295|

296|296| Chaque bloc server est un guichet distinct. 297|297| Port 8080 = guichet A, port 8081 = guichet B. 298|298|

299|299| Quand une requête arrive, le serveur regarde d'abord par quelle porte elle est entrée (le port), 300|300| puis dirige vers le bon guichet. Le guichet a ses propres règles (ses location). 301|301|

302|302| Si deux guichets partagent la même porte (même port), on lit le header Host 303|303| pour choisir lequel — c'est le principe du server_name. 304|304|

305|305|
306|306|
307|307| 308|308|
309|309| ⚠️ 310|310|
Important : si tu n'implémentes pas server_name, plusieurs serveurs sur le même port deviennent ambigus. 311|311| Dans ce cas, définis clairement une règle (ex: "premier server du port") et documente-la.
312|312|
313|313| 314|314|
315|315|
🔌Chaque listen unique = un socket serveur distinct. Webserv doit ouvrir autant de sockets d'écoute qu'il y a de ports distincts dans la config.
316|316|
🔢Port valide : 1–65535. Ports < 1024 nécessitent des droits élevés sur Linux. À vérifier à la validation, pas à runtime.
317|317|
🏷️server_name est optionnel selon le sujet. Si absent : sélection par port uniquement. Si présent : matching sur le header Host après filtrage par port.

client_max_body_size
351|351|

352|352| Limite la taille maximale du body client. C'est crucial pour éviter les abus mémoire/disque. 353|353| En réception HTTP, tu compares tôt le Content-Length à cette limite. 354|354| Si dépassement, réponse 413 sans continuer à lire inutilement. 355|355|

356|356|
357|357|
Unités
358|358|

k, m, g (insensible à la casse). Sans suffixe : octets bruts.

359|359|
360|360|
361|361| 362|362|
363|363|
error_page
364|364|

365|365| Associe un code HTTP à un fichier de réponse custom. Quand le serveur produit l'erreur, 366|366| il essaie d'abord de servir cette page personnalisée. Sinon fallback sur un body minimal. 367|367|

368|368|
369|369| ⚠️ 370|370|
Piège fréquent : la page d'erreur doit exister dans un chemin résolvable avec tes règles de root/location.
371|371|
372|372|
373|373|
374|374| 375|375|
376|376|
⛔client_max_body_size : compare le Content-Length reçu tôt dans le parsing de la requête. Réponse 413 immédiate, ne pas continuer à lire le body.
377|377|
🗂️error_page : le chemin de la page custom doit être résolvable via tes propres règles root/location. Une erreur qui pointe vers une page inexistante génère une double erreur.
378|378|
🔁Fallback obligatoire : si la page error_page n'est pas trouvée, le serveur doit quand même répondre avec un body minimal (HTML inline). Ne jamais renvoyer de réponse vide.

size_t parseSize(const std::string& s) {
390|390|  if (s.empty()) throw std::runtime_error("empty size");
391|391|
392|392|  char suffix = std::tolower(s.back());
393|393|  std::string number = s;
394|394|  size_t factor = 1;
395|395|
396|396|  if (suffix == 'k' || suffix == 'm' || suffix == 'g') {
397|397|    number = s.substr(0, s.size() - 1);
398|398|    if (suffix == 'k') factor = 1024;
399|399|    if (suffix == 'm') factor = 1024 * 1024;
400|400|    if (suffix == 'g') factor = 1024 * 1024 * 1024;
401|401|  }
402|402|
403|403|  size_t value = static_cast<size_t>(std::strtoull(number.c_str(), NULL, 10));
404|404|  return value * factor;

allowed_methods
454|454|

455|455| Directive custom webserv : contrôle des verbes autorisés par route. 456|456| Si méthode non autorisée, réponse 405 Method Not Allowed + header 457|457| Allow: listant les méthodes valides. 458|458|

459|459|
460|460|
461|461|
return
462|462|

463|463| Court-circuite toute la suite. Quand return est défini, on ne cherche pas de fichier, 464|464| on répond immédiatement avec le code (et Location si redirection). 465|465|

466|466|
467|467|
468|468| 469|469|
470|470|
🚦allowed_methods : si la méthode n'est pas dans la liste → 405 Method Not Allowed + header Allow: GET, POST, DELETE. Ce header est obligatoire avec le 405.
471|471|
↩️return court-circuite tout le reste. Si défini dans la location matchée, on répond immédiatement — pas de lookup fichier, pas de CGI, pas de root.
472|472|
📐Matching par préfixe uniquement (plus long préfixe = winner). Voir la page dédiée Sélection de route : préfixe le plus long pour l'algo complet.

const Location* findLocation(const Server& srv, const std::string& uri) {
484|484|  const Location* best = NULL;
485|485|  size_t bestLen = 0;
486|486|
487|487|  for (std::vector<Location>::const_iterator it = srv.locations.begin(); it != srv.locations.end(); ++it) {

547|547|

548|548| Quand activé, tu génères une page HTML listant les entrées du dossier via opendir/readdir. 549|549| Chaque entrée doit devenir un lien cliquable. C'est utile en debug/partage, à manier prudemment en prod. 550|550|
