04 — Configuration — Routing

Sélection de route : préfixe le plus long

L'URI vient du client. Le serveur la compare à toutes ses location et retient celle dont le préfixe est le plus précis. Pas de regex — juste du matching de préfixe avec une règle de frontière de segment.

01
d'où vient l'URI ?
Extraite de la première ligne de la requête HTTP
Ce que le client envoie

Quand un client se connecte, la première ligne de la requête HTTP suit le format : MÉTHODE URI VERSION.

C'est ton parser HTTP qui lit cette ligne et extrait l'URI. Elle n'est pas inventée par le serveur — elle vient du client. Tu la stockes dans ta structure de requête : req.uri.

L'URI peut contenir un chemin (/get/files/fd), une query string (/search?q=foo), ou les deux. Pour le routing, seul le chemin est comparé aux locations.

Ce que le serveur en fait

Une fois l'URI extraite, le serveur suit toujours le même pipeline :

1. Sélectionner le bloc server par port d'arrivée.
2. Comparer l'URI aux préfixes des location du serveur.
3. Retenir la location avec le préfixe le plus long.
4. Appliquer ses directives (méthodes, root, CGI, upload…).

Si aucune location ne matche : 404 Not Found — même / n'existant pas.

requête brute reçue sur socket
http request line 
GET /get/files/fd HTTP/1.1      ← URI = /get/files/fd
Host: localhost:8080
User-Agent: curl/8.7.1
Connection: keep-alive

[body optionnel]
💡
L'URI n'est pas inventée par le serveur. Elle est fournie par le client dans la requête. Ton parser HTTP la lit, la décode (URL-encoding si besoin), et la stocke avant toute décision de routing.
02
préfixe le plus long
Comment choisir entre plusieurs locations qui matchent
🚉
Analogie — le panneau d'affichage de gare

Imagine un panneau d'itinéraire en gare. Plusieurs lignes affichent des destinations : Paris, Paris-Nord, Paris-Nord-Hall-B.

Si tu demandes à aller à "Paris-Nord-Hall-B", tu peux prendre le couloir général "Paris", le couloir "Paris-Nord"… ou le couloir exact "Paris-Nord-Hall-B". On te dirige toujours vers le chemin le plus précis qui correspond à ta destination.

C'est exactement ça le matching par préfixe le plus long : la location la plus spécifique qui commence comme ton URI est celle retenue.

config d'exemple
3 locations, 3 niveaux de précision 
location / {
    allowed_methods GET;
}

location /get/files {
    allowed_methods GET POST;
}

location /get/files/fd {
    allowed_methods GET;
}
URI reçueLocations qui matchentGagnante (la plus longue)
/get/files/fd / · /get/files · /get/files/fd /get/files/fd
/get/files/report.txt / · /get/files /get/files
/get/other / /
/get/filesX / (frontière invalide pour /get/files) / (pas /get/files !)
💡
Notre approche : réduction progressive, pas de boundary check. Au lieu de vérifier les frontières de segment, on réduit l'URI segment par segment (rfind('/') + erase) jusqu'à trouver un match EXACT avec un location.path. /get/filesX ne matchera jamais /get/files car l'égalité est stricte. C'est plus simple et tout aussi robuste.
📏On compare l'URI à tous les préfixes, on retient le plus long. Pas de court-circuit à la première correspondance.
🚫Pas de regex dans webserv. Le sujet dit explicitement que les regex ne sont pas requises. Préfixe uniquement.
🔄Si aucune location ne matche (même / absente) : 404 Not Found immédiat.
03
findLocation()
Implémentation C++98 correcte avec règle de frontière
srcs/utils/utils.cpp — findLocation()
Notre code 
// srcs/utils/utils.cpp — réduction progressive de l'URI
// jusqu'à trouver un match EXACT avec une location.path
const Location* findLocation(const std::string& uri, const Server& server)
{
    std::string shorturi = uri;
    const Location *loc = NULL;
    int len = -1;
    if (uri.empty() || uri[0] != '/')
        return NULL;

    while (true) {
        for (size_t i = 0; i < server.locations.size(); i++) {
            if (server.locations[i].path == shorturi &&
                len < (int)(server.locations[i].path.length())) {
                loc = &server.locations[i];
                len = (int)(server.locations[i].path.length());
            }
        }
        if (len != -1) break;           // trouvé !
        if (shorturi.empty()) break;           // rien trouvé → NULL
        size_t i = shorturi.rfind('/');       // coupe au dernier '/'
        shorturi.erase(i);
        if (shorturi.empty()) shorturi = "/";  // retombe sur /
    }
    return loc;  // NULL si aucune location ne matche → 404
}
Les 3 étapes de l'algo (réduction progressive)

① Boucle externe : l'URI est raccourcie progressivement — on cherche un match EXACT de shorturi parmi les location.path.

② Découpage : à chaque échec, rfind('/') + erase() retire le dernier segment. L'URI /a/b/c devient /a/b, puis /a, puis /.

③ Le plus long gagne : on garde la location dont le path est le plus long (variable len). On ne s'arrête pas au premier match.

Urgent : l'URI passée à findLocation() ne contient JAMAIS la query string. Le Router (Router::route()) et HttpParser::headerParser() retirent ?xxx via uri.substr(0, uri.find('?')) — bug corrigé : avant, /upload?x=1 ne matchait pas /upload.

Ce que retourne NULL

Si la fonction retourne NULL, aucune location ne matche l'URI. Le comportement attendu : 404 Not Found.

Même si tu as une location /, elle ne matche que les URI qui commencent par / — ce qui est presque toujours vrai. Mais si la config n'a pas de /, les URI non couvertes produisent un NULL.

Toujours vérifier le retour de findLocation() avant d'utiliser le pointeur.

🔍L'algo est O(n) par réduction : on essaie l'URI complète, puis segment par segment via rfind('/') + erase(). Au pire, on arrive à /.
⚠️L'URI sans query string est essentiel. findLocation() est appelée par le Router ET par HttpParser::headerParser() — dans les deux cas avec uri.substr(0, uri.find('?')).
🔒Vérifier NULL avant de déréférencer. Un NULL retourné = 404, jamais un crash. Le Router vérifie avant de continuer.
04
flow complet
De l'URI reçue au fichier servi — ordre exact
Pipeline de traitement (Router::route)

1. Si req.error != 200 → ErrorHandler immédiat.

2. Extraire le chemin de l'URI (sans query string).

3. findLocation() → location gagnante ou NULL (404).

4. methodImplemented() (GET/POST/DELETE/HEAD) sinon 501.

5. methodAllowed() → 405 si méthode interdite.

6. Si return_path présent → redirect immédiat (avant tout accès disque).

7. resolvePath() → chemin disque (root + URI, normalizePath).

8. Si extension CGI → CGIHandler (return NULL, asynchrone). Sinon → StaticHandler.

Résolution du chemin disque (étape 6)

Après avoir trouvé la location, le chemin physique du fichier est :

chemin = root + (URI − préfixe_location)

Exemple : location /get/files avec root /var/www, URI /get/files/report.txt :

→ retire le préfixe /get/files
→ reste /report.txt
→ fichier : /var/www/report.txt

C'est le Cas 2 du sujet (strip prefix), confirmé par l'exemple officiel : /kapouet/pouic/toto/pouet → /tmp/www/pouic/toto/pouet.

routing-trace.txt
trace complète de 3 requêtes 
GET /files/monfichier.txt HTTP/1.1
  → port 8080  → server sélectionné
  → findLocation("/files/monfichier.txt") → match EXACT avec /files
  → autoindex on ✓  → throwList() (listing du répertoire)

GET /cgi-bin/hello.py HTTP/1.1
  → findLocation("/cgi-bin/hello.py") → /cgi-bin
  → cgi_pass .py /usr/bin/python3 ✓
  → extension .py trouvée dans cgi_pass → CGIHandler → NULL (asynchrone)

POST /upload/photo.jpg HTTP/1.1
  → findLocation("/upload/photo.jpg") → /upload
  → allowed_methods GET POST DELETE ✓
  → upload_store ./www/upload → fichier sauvegardé

DELETE /upload/vieux.txt HTTP/1.1
  → findLocation("/upload/vieux.txt") → /upload
  → allowed_methods GET POST DELETE → DELETE ✓
  → unlink() → 204 No Content

GET /private/secrets.html HTTP/1.1
  → findLocation("/private/secrets.html") → /private
  → allowed_methods GET ✓
  → forbiddenAccess() → R_OK manquant → 403 Forbidden
1️⃣L'ordre est fixe et séquentiel. Une étape échoue → réponse immédiate, on ne continue pas.
➖Formule de résolution : root + (URI − préfixe). Retirer le préfixe de la location AVANT de coller à root.
⚡return court-circuite tout — étape 5 avant la résolution disque. Si présent, aucun accès fichier n'a lieu.

Le routing n'est pas une option décorative. C'est la décision qui pilote tout le reste — root, méthodes, CGI, upload.

— Toute erreur dans findLocation() propage des bugs dans chaque étape suivante.

curl -i http://localhost:8080/not-covered
Ce que tu dois observer

Pour chaque requête, le serveur doit loguer (ou afficher) la location sélectionnée. Si la sélection n'est pas celle attendue, le bug est dans ton findLocation() — souvent la règle de frontière de segment.

Le test ④ (/get/filesX) est le plus important : c'est le piège classique que beaucoup ratent. Si ce test échoue, tu as une fausse correspondance.

Checklist de validation

✅ Match exact → bonne location
✅ Sous-chemin → location parente
✅ Fallback → location /
✅ Frontière invalide → pas de match
✅ Méthode interdite → 405 + Allow:
✅ NULL retourné → 404, jamais un crash

🧪Le test ④ (/get/filesX ne matche pas /get/files) doit être dans tes tests unitaires. C'est le bug de frontière le plus fréquent.
🔎Si tu as du mal à déboguer, log la location sélectionnée à chaque requête (std::cerr << "location: " << loc->path). Tu verras immédiatement si le routing est bon.
🗣️À la soutenance : être capable d'expliquer ces 6 cas à l'oral prouve que tu maîtrises entièrement le routing.
