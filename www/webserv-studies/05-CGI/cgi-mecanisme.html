CGI — Le mécanisme, dans notre code

fork() + execve() + pipe()
— la classe CGIProcess

Le seul endroit du projet où fork() est autorisé. Dans notre webserv, tout le mécanisme bas-niveau vit dans une seule classe : CGIProcess (srcs/cgi/CGIProcess.cpp). Elle crée les deux pipes, fork, branche stdin/stdout de l'enfant, chdir dans le dossier du script et execve l'interpréteur.

01
Router::isCgi()
Comment une requête devient une requête CGI
Extension → interpréteur, via la config

Tout part du fichier de configuration : chaque location peut déclarer des directives cgi_pass .py /usr/bin/python3;. Le parser les range dans Location::cgi_pass, une std::map<extension, interpréteur>. Au moment du routing, Router::isCgi() extrait l'extension de l'URI et regarde si la location matched la connaît.

srcs/http/Router.cpp
Notre code 
std::string Router::isCgi(const std::string& uri, const Location* loc) {
    size_t pos = uri.rfind('.');
    std::string ext;
    if (pos == std::string::npos)
        return "";
    else
        ext = uri.substr(pos);
    if (loc->cgi_pass.find(ext) != loc->cgi_pass.end())
        return loc->cgi_pass.find(ext)->second;
    return "";
}
srcs/http/Router.cpp — dans Router::route()
Le déclencheur 
std::string interpreter = isCgi(uriPath, loc);
if (!interpreter.empty() && (req.method == "GET" || req.method == "POST")) {
    CGIHandler cgi(req, path, interpreter, peerAddr, loc, sink);
    return NULL;   // ← pas de handler : la réponse arrivera plus tard, en asynchrone
}
💡
return NULL est le signal asynchrone. Pour une page statique, Router::route() retourne un ARequestHandler* et la réponse est construite immédiatement. Pour un CGI, il retourne NULL : ClientHandler comprend alors que la réponse viendra plus tard, via le callback onCgiDone() (voir la fiche Intégrer le CGI dans l'event loop).
⚠️
GET et POST uniquement. Chez nous, un DELETE sur un fichier .py ne lance pas l'interpréteur : il tombe dans le circuit statique classique. Seuls GET et POST déclenchent l'exécution CGI.

Qui fait quoi entre le Router et le script Python
Quatre classes, quatre responsabilités

srcs/handlers/CGIHandler.cpp
L'orchestrateur complet 
CGIHandler::CGIHandler(const HttpRequest& req, std::string& path, std::string& interpreter,
                        const std::string& peerAddr, const Location* loc, IResponseSink* sink) {
    // Processus CGI (fork + exec + pipes)
    _process = new CGIProcess(req, path, interpreter, peerAddr);

    // brancher les pipes
    CGIWriteHandler* wr = new CGIWriteHandler(_process->getWriteFd(), req.body);
    EventLoop::instance()->register_handler(wr, WRITE_EVENT);

    CGIReadHandler* rd = new CGIReadHandler(_process->getReadFd(), _process->getPid(), loc, sink);
    EventLoop::instance()->register_handler(rd, READ_EVENT);
    // le client doit connaître son CGIReadHandler pour pouvoir se détacher
    // si l'un des deux meurt avant l'autre (voir IResponseSink)
    if (sink)
        sink->onCgiStart(rd);
}
💡
Le script CGI ne comprend rien à HTTP. C'est un programme à part (Python, bash…) qui lit du texte sur son entrée standard et écrit du texte sur sa sortie standard, comme dans un terminal. « Mettre le body en stdin » veut dire : ce qu'on écrit dans le pipe arrive à l'enfant exactement comme s'il l'avait lu au clavier. Ce que l'enfant print, on le récupère par l'autre pipe — et ça devient le corps de la réponse HTTP.
03
pipe() × 2
Quatre file descriptors pour un aller-retour
Un pipe = un sens unique

Un pipe() crée 2 fd : [0] pour lire, [1] pour écrire. Comme il nous faut les deux sens (envoyer le body au script, lire sa réponse), on en crée deux :

  • pipe_stdin[0] → l'enfant lit (son futur STDIN)
  • pipe_stdin[1] → le parent écrit le body dedans
  • pipe_stdout[0] → le parent lit la réponse du CGI
  • pipe_stdout[1] → l'enfant écrit (son futur STDOUT)
srcs/cgi/CGIProcess.cpp — constructeur
Notre code 
CGIProcess::CGIProcess(const HttpRequest& req, std::string& path,
                        std::string& interpreter, const std::string& peerAddr)
                        : _req(req), _peerAddr(peerAddr),
                          _pid(-1), _write_fd(-1), _read_fd(-1) {
    int pipe_stdin[2];
    int pipe_stdout[2];
    if (pipe(pipe_stdin) == -1)
        throw std::runtime_error("pipe() stdin failed");
    if (pipe(pipe_stdout) == -1) {
        close(pipe_stdin[0]);   // le 1er pipe a réussi : le fermer avant de throw,
        close(pipe_stdin[1]);   // sinon 2 fds fuient à chaque échec
        throw std::runtime_error("pipe() stdout failed");
    }
    CGIFork(pipe_stdout, pipe_stdin, interpreter, path);
}
⚠️
La règle d'or des pipes. Tant qu'un fd d'écriture traîne ouvert quelque part (même par erreur, dans le mauvais processus), le lecteur en face ne voit jamais l'EOF et peut rester bloqué indéfiniment. C'est pour ça que chaque processus ferme immédiatement les extrémités qui ne le concernent pas — et c'est aussi la raison du FD_CLOEXEC posé plus loin.
04
CGIFork()
Un processus devient deux — qui garde quoi ?
Le partage des 4 fd

fork() duplique le processus courant : les deux copies partagent à cet instant les 4 fd de pipe. Chacune ne garde que ce qui la concerne et ferme le reste :

pipe_stdin[0]—lecture→ enfant (dup2 → STDIN_FILENO)
pipe_stdin[1]—écriture→ parent (_write_fd)
pipe_stdout[0]—lecture→ parent (_read_fd)
pipe_stdout[1]—écriture→ enfant (dup2 → STDOUT_FILENO)
srcs/cgi/CGIProcess.cpp — CGIFork(), le fork et l'erreur
Notre code 
void CGIProcess::CGIFork(int* pipe_stdout, int* pipe_stdin,
                          std::string& interpreter, std::string& path) {
    _pid = fork();
    // erreur, fermer tous les pipes.
    if (_pid == -1) {
        close(pipe_stdout[0]); close(pipe_stdout[1]);
        close(pipe_stdin[0]); close(pipe_stdin[1]);
        throw std::runtime_error("fork() failed");
    }
    // _pid == 0 → bloc enfant  (section 05)
    // _pid  > 0 → bloc parent  (section 06)
}
🧬
fork() = photocopie

Après fork(), deux processus exécutent la même ligne de code. La seule différence : la valeur de retour. 0 chez l'enfant, le PID de l'enfant chez le parent. C'est ce if qui sépare les deux mondes.

🔒
Le seul fork() du projet

Le sujet est formel : « fork() must be used only to run another program (CGI, for example) ». Chez nous il n'existe qu'à un seul endroit : CGIProcess::CGIFork(), immédiatement suivi d'un execve().

05
Bloc enfant
dup2 + chdir + execve — devenir l'interpréteur
srcs/cgi/CGIProcess.cpp — CGIFork(), bloc enfant
Notre code 
if (_pid == 0) {
    dup2(pipe_stdin[0], STDIN_FILENO);      // connecter le stdin au pipe (enfant lit)
    dup2(pipe_stdout[1], STDOUT_FILENO);    // connecter le stdout au pipe (enfant écrit)
    close(pipe_stdin[0]); close(pipe_stdin[1]);
    close(pipe_stdout[0]); close(pipe_stdout[1]);

    // On se place dans le dossier du script (accès aux fichiers relatifs,
    // exigé par le sujet). L'argument passé à l'interpréteur doit alors
    // être le SEUL nom de fichier : passer "www/cgi-bin/hello.py" après
    // un chdir dans "www/cgi-bin" ferait chercher un chemin dédoublé.
    size_t slash = path.rfind('/');
    std::string dir  = (slash == std::string::npos) ? "." : path.substr(0, slash);
    std::string file = (slash == std::string::npos) ? path : path.substr(slash + 1);
    if (chdir(dir.c_str()) == -1)
        exit(1);

    char** envp = buildEnvp(path);
    char* av[] = { (char*)interpreter.c_str(), (char*)file.c_str(), NULL };
    execve(interpreter.c_str(), av, envp);
    exit(1);   // on n'arrive ici QUE si execve a échoué
}
Pourquoi dup2 ?

dup2(pipe_stdin[0], STDIN_FILENO) fait pointer le fd 0 de l'enfant vers le bout de lecture du pipe. Quand le script Python fera sys.stdin.read(), il lira en réalité ce que notre serveur écrit dans _write_fd. Idem pour stdout : chaque print() part dans le pipe que le parent lit. Après le dup2, les 4 fd originaux sont fermés — les copies 0 et 1 suffisent.

Pourquoi chdir + nom de fichier seul ?

Le sujet exige : « The CGI should be run in the correct directory for relative path file access ». On chdir() donc dans le dossier du script (ex. www/cgi-bin), et on passe à l'interpréteur le nom de fichier seul (hello.py) : garder le chemin complet après le chdir ferait chercher www/cgi-bin/www/cgi-bin/hello.py. C'est aussi ce qui permet à session.py d'ouvrir son dossier sessions/ en relatif.

💡
execve() ne revient jamais… sauf en cas d'échec. S'il réussit, tout l'espace mémoire du processus est remplacé par l'interpréteur : le code après execve() n'existe plus. S'il échoue (interpréteur introuvable, pas exécutable…), on exit(1) — et ce code de sortie non-nul sera vu par le parent via waitpid(), qui transformera ça en 500 Internal Server Error (voir la fiche sur les pièges). C'est aussi pour ça que le strdup() de l'envp n'est jamais free : soit execve emporte tout le tas avec lui, soit exit(1) suit immédiatement.
06
Bloc parent
O_NONBLOCK + FD_CLOEXEC — préparer les pipes pour epoll
srcs/cgi/CGIProcess.cpp — CGIFork(), bloc parent
Notre code 
else {
    close(pipe_stdin[0]);
    close(pipe_stdout[1]);
    _write_fd = pipe_stdin[1];   // parent écrit ici → stdin de l'enfant
    _read_fd  = pipe_stdout[0];  // parent lit ici ← stdout de l'enfant

    // Non-bloquant seulement côté parent : chaque extrémité d'un pipe est
    // une open file description à part, donc ça ne touche pas le stdin/
    // stdout (bloquants) de l'enfant. Indispensable avec epoll : sinon un
    // read()/write() peut bloquer tout le event loop si le pipe est plein
    // ou vide au moment de l'appel.
    if (fcntl(_write_fd, F_SETFL, fcntl(_write_fd, F_GETFL) | O_NONBLOCK) == -1
        || fcntl(_read_fd, F_SETFL, fcntl(_read_fd, F_GETFL) | O_NONBLOCK) == -1) {
        // nettoyer avant de throw : sinon fds et process enfant fuient
        close(_write_fd);
        close(_read_fd);
        kill(_pid, SIGKILL);
        waitpid(_pid, NULL, 0);
        throw std::runtime_error("fcntl() failed: " + std::string(strerror(errno)));
    }

    // FD_CLOEXEC : si un AUTRE CGI est forké pendant que celui-ci tourne,
    // l'enfant du second hériterait de nos deux fds de pipe — le premier
    // CGI ne verrait alors jamais l'EOF sur son stdin (fd d'écriture
    // encore ouvert ailleurs) et resterait bloqué.
    fcntl(_write_fd, F_SETFD, FD_CLOEXEC);
    fcntl(_read_fd, F_SETFD, FD_CLOEXEC);
}
O_NONBLOCK côté parent seulement

Subtilité importante : le flag est posé sur _write_fd et _read_fd après le fork, donc uniquement sur les open file descriptions du parent. Le stdin/stdout de l'enfant restent bloquants — c'est ce qu'un script Python attend : sys.stdin.read() doit dormir jusqu'à l'EOF, pas retourner immédiatement.

Le scénario du double CGI

Sans FD_CLOEXEC : le CGI n°1 tourne, ses pipes sont ouverts dans le serveur. Un client lance le CGI n°2 → fork() duplique tous les fd du serveur, y compris _write_fd du CGI n°1. Même si notre CGIWriteHandler ferme son fd, la copie héritée par l'enfant n°2 reste ouverte → le CGI n°1 ne reçoit jamais son EOF et pènd à vie. FD_CLOEXEC fait fermer ces copies automatiquement pendant l'execve().

Résumé — CGIProcess en 4 mouvements
pipe ×2 → fork → (enfant : dup2 + chdir + execve) / (parent : O_NONBLOCK + FD_CLOEXEC)
🔌
Deux pipes, quatre fd
pipe_stdin transporte le body vers le script, pipe_stdout ramène sa sortie. Chaque processus ferme les extrémités qui ne le concernent pas.
🧬
L'enfant devient l'interpréteur
dup2 branche stdin/stdout sur les pipes, chdir se place dans le dossier du script, execve(interpreter, [interpreter, file], envp) remplace le processus. exit(1) si échec → 500 côté serveur.
⚡
Le parent prépare epoll
O_NONBLOCK sur les deux fd (jamais de blocage de l'event loop), FD_CLOEXEC pour que les futurs CGI n'héritent pas des pipes des autres.
🛡️
Chaque échec nettoie derrière lui
pipe raté → fermer le pipe déjà ouvert. fork raté → fermer les 4 fd. fcntl raté → fermer les fd + kill(SIGKILL) + waitpid. Jamais de fd ni de zombie qui fuit.
