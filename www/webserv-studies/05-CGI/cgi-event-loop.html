CGI — L'intégration Reactor, dans notre code

Le CGI dans l'event loop
— l'interface IResponseSink

Un script peut mettre 3 secondes à répondre. Pendant ce temps, le serveur doit continuer à servir tout le monde. Dans notre webserv, les pipes CGI sont des handlers comme les autres dans l'EventLoop (pattern Reactor), et le client reçoit sa réponse plus tard via un callback : IResponseSink::onCgiDone() (pattern Observer).

01
Le problème
Une réponse qui n'existe pas encore
Requête statique — synchrone

Router::route() retourne un StaticHandler, la réponse est construite immédiatement (lire un fichier disque est exempt de la règle epoll), et ClientHandler bascule en WRITE_EVENT dans la foulée. Tout se passe dans le même tour de boucle.

Requête CGI — asynchrone

Le script tourne dans un autre processus, à son rythme. Attendre sa sortie avec un read() bloquant gèlerait tout le serveur (interdit par le sujet). Router::route() retourne donc NULL, et la réponse arrivera… quand epoll dira que le pipe a parlé.

srcs/handlers/ClientHandler.cpp — _handleComplete()
Notre code — le client se met en attente 
_rh = Router::route(req, _server, _peerAddr, this);
if (!_rh) {
    // CGI -> async : la réponse arrivera via onCgiDone()
    // La réponse CGI est construite sans connaître notre keep-alive,
    // elle annonce toujours "Connection: close" → on ferme après envoi.
    _keepAlive = false;
    // Plus rien à lire tant que le CGI tourne : on ne garde que la
    // détection de déconnexion (EPOLLHUP/EPOLLERR restent signalés).
    EventLoop::instance()->modify_handler(this, static_cast<EventType>(0));
    return;
}
💡
modify_handler(this, 0) = mode silence radio. Le client reste enregistré dans epoll mais sans EPOLLIN ni EPOLLOUT : on ne veut ni lire une nouvelle requête au milieu du CGI, ni être notifié « prêt à écrire » alors qu'on n'a encore rien à écrire. EPOLLHUP/EPOLLERR restent signalés par le kernel même avec un masque vide : si le client raccroche pendant le CGI, on le voit quand même.
02
IResponseSink
L'interface qui découple le CGI du client (pattern Observer)
includes/core/IResponseSink.hpp
Notre code — le contrat complet 
class CGIReadHandler;

// Interface (pattern Observer) : CGIReadHandler ne connait que ce contrat,
// pas ClientHandler directement, ce qui evite un #include circulaire.
class IResponseSink {
    public:
        virtual void onCgiDone(const std::string& rawHttpResp) = 0;
        virtual void onCgiStart(CGIReadHandler* rd) { (void)rd; }
        virtual ~IResponseSink() {}
};
Pourquoi une interface plutôt qu'un ClientHandler* ?

CGIReadHandler n'a pas besoin de savoir ce qu'est un client : il a juste besoin d'un endroit où déposer la réponse finie. ClientHandler hérite donc de deux interfaces : IEventHandler (pour epoll) et IResponseSink (pour le CGI). Le this passé à Router::route(req, server, peerAddr, this) voyage jusqu'au CGIReadHandler sous forme d'IResponseSink*. Bénéfices : pas d'include circulaire, et un couplage minimal entre les deux mondes.

srcs/handlers/ClientHandler.cpp — les deux callbacks
Notre code 
void ClientHandler::onCgiStart(CGIReadHandler* rd) {
    _cgiRead = rd;      // mémoriser le CGI en cours, pour le timeout et le détachement
}

void ClientHandler::onCgiDone(const std::string& rawHttpResp) {
    _response = rawHttpResp;
    _cgiRead  = NULL;   // le CGIReadHandler se détruit juste après, ne plus y toucher
    _sent     = 0;
    EventLoop::instance()->modify_handler(this, WRITE_EVENT);   // réveil : à envoyer !
}
03
Le cycle complet
Trois handlers dans la même boucle epoll
Une requête GET /cgi-bin/hello.py, vue par l'EventLoop
1
EPOLLIN sur le fd client → ClientHandler::handle_input() → requête complète → Router::route() détecte .py → CGIHandler fork le script et enregistre deux nouveaux handlers dans l'EventLoop. Le client passe en masque vide (silence radio).
↓
2
EPOLLOUT sur le pipe stdin → CGIWriteHandler::handle_output() écrit le body (en une ou plusieurs notifications). Terminé → return -1 → l'EventLoop le détruit → son destructeur close(_fd) → EOF sur le stdin du script.
↓ (pendant ce temps, les autres clients sont servis normalement)
3
EPOLLIN sur le pipe stdout → CGIReadHandler::handle_input() accumule la sortie dans _out, notification après notification, jusqu'à read() == 0 (EOF).
↓
4
EOF → reapChild() + analyse du status + parseCgi() → _sink->onCgiDone(raw) → le ClientHandler stocke la réponse et repasse en WRITE_EVENT. Le CGIReadHandler retourne -1 et disparaît.
↓
5
EPOLLOUT sur le fd client → ClientHandler::handle_output() envoie la réponse par send() successifs, puis ferme (la réponse CGI est toujours Connection: close).
💡
Pourquoi ça ne bloque jamais. Chaque flèche ci-dessus est un événement epoll distinct. Entre deux flèches, la boucle traite ce qu'elle veut : d'autres clients, d'autres CGI, des timeouts. Les pipes sont non-bloquants (O_NONBLOCK posé dans CGIProcess), et chaque handler fait un seul read/write par notification. Le sujet est respecté à la lettre : même les I/O des pipes CGI passent par epoll_wait().
⚠️
Qui possède quoi ? CGIHandler est un objet local dans Router::route() : il meurt à la fin de la fonction, après avoir enregistré les deux handlers de pipe. Ce sont eux (et l'EventLoop qui les détruit sur return -1) qui portent le cycle de vie : CGIWriteHandler ferme le stdin, CGIReadHandler ferme le stdout, tue et reap l'enfant.
04
getLastActivity()
Le CGI participe au timeout global de la boucle
Pas de timer dédié : le même mécanisme que les clients

EventLoop::checkTimeOut() tourne après chaque epoll_wait() (dont le timeout de 500 ms garantit un passage régulier même sans trafic). Tout handler dont getLastActivity() date de plus de CLIENT_TIMEOUT (30 s) est détruit. CGIReadHandler expose son _lastActivity (mis à jour à chaque lecture) : un script muet est donc tué au bout de 30 s, et son destructeur envoie le 504 au client.

srcs/handlers/ClientHandler.cpp — handle_timeout()
Notre code — le 504 côté client 
// Dernière chance de répondre avant que checkTimeOut() ne nous détruise.
// On ne fait AUCUN send() ici : on prépare juste la réponse et on redemande
// à epoll de nous notifier en écriture, comme le fait déjà onCgiDone().
int ClientHandler::handle_timeout() {
    if (!_cgiRead)
        return 0;      // pas de CGI en cours, rien a faire, comportement inchangé

    ErrorHandler err(_server, 504, "");
    _response = err.buildResponse();
    _sent = 0;
    _keepAlive = false;
    EventLoop::instance()->modify_handler(this, WRITE_EVENT);
    return 1;          // "ne me détruis pas tout de suite : j'ai un 504 à livrer"
}
srcs/core/EventLoop.cpp — checkTimeOut()
Notre code — le sursis 
for (size_t i = 0; i < timedOut.size(); i++) {
    // avant de supprimer le client, il faut verifier si il nya pas un cgi en cours (erreur 504)
    if (timedOut[i]->handle_timeout())
        continue;                        // sursis accordé : il enverra son 504
    // ensuite on peut le supprimer.
    remove_handler(timedOut[i]);
    delete timedOut[i];
}
💡
Deux chemins vers le 504, un seul gagnant. Client et CGIReadHandler timeoutent tous les deux à 30 s, mais l'ordre importe peu : si le CGIReadHandler meurt en premier, son destructeur appelle onCgiDone(504). Si le client passe par handle_timeout() d'abord, il prépare lui-même son 504 et gagne un sursis pour l'envoyer. Dans les deux cas : « a request to your server should never hang indefinitely ». Testé en live avec www/cgi-bin/slow.py.
05
detachSink()
Quand l'un des deux meurt avant l'autre — zéro use-after-free
Le scénario dangereux

Le client lance un CGI puis ferme son navigateur. L'EventLoop détruit le ClientHandler (EPOLLHUP)… mais le CGIReadHandler détient toujours un _sink pointant vers l'objet détruit. À l'EOF du script, il appellerait onCgiDone() sur de la mémoire libérée — use-after-free, crash. La solution : les deux objets se connaissent (via onCgiStart) et se préviennent mutuellement à leur destruction.

~ClientHandler() 
ClientHandler::~ClientHandler() {
    if (_cgiRead)
        _cgiRead->detachSink();
    delete _rh;
    close(_fd);
}
CGIReadHandler.hpp 
// appelé par ~ClientHandler si le
// client meurt avant le CGI : on ne
// doit plus jamais toucher au sink
void detachSink() { _sink = NULL; }
La symétrie complète
Client meurt d'abord~ClientHandler appelle _cgiRead->detachSink() → le CGI finit dans le vide, sans jamais toucher le sink (chaque accès est gardé par if (_sink))
CGI meurt d'abord (cas normal)onCgiDone() remet _cgiRead = NULL côté client → le destructeur du client ne touchera pas un CGIReadHandler déjà détruit
Résumé — le CGI asynchrone
Deux pipes = deux handlers. Un callback pour livrer. Un timeout pour ne jamais pendre.
🎲
Router retourne NULL
Le signal asynchrone : pas de handler = la réponse viendra plus tard. Le client passe en masque epoll vide et attend.
🔌
Les pipes sont des citoyens de l'EventLoop
CGIWriteHandler (WRITE_EVENT) pousse le body, CGIReadHandler (READ_EVENT) collecte la sortie. Même interface IEventHandler, même règle « un I/O par notification », même epoll que les sockets.
🔔
IResponseSink = Observer
Le CGI ne connaît pas le client, juste le contrat onCgiDone(). onCgiStart() crée le lien inversé pour pouvoir se détacher proprement (detachSink) si l'un meurt avant l'autre.
⏰
30 secondes, pas une de plus
getLastActivity() intègre le CGI au checkTimeOut() global. Script muet → SIGKILL + waitpid + 504 livré au client via handle_timeout() ou le destructeur.
