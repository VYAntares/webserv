CGI — Le protocole RFC 3875, dans notre code

Les variables d'environnement
— buildEnvp()

Le CGI ne reçoit pas la requête HTTP brute : il reçoit le body sur stdin et tout le reste en variables d'environnement. Dans notre webserv, c'est CGIProcess::buildEnvp() et ses quatre helpers (addUri, addHost, addHeaders, addRemoteAddr) qui traduisent la HttpRequest en environnement conforme à la RFC 3875.

01
Le contrat CGI/1.1
Body sur stdin, métadonnées dans l'environnement

« Have a careful look at the environment variables involved in the web server-CGI communication. The full request and arguments provided by the client must be available to the CGI. »

— webserv.pdf, IV.3
Ce qui passe par stdin

Le body de la requête, et rien d'autre. Chez nous il est écrit dans le pipe par CGIWriteHandler, morceau par morceau, au rythme d'epoll. Le script sait combien lire grâce à CONTENT_LENGTH — ou lit jusqu'à l'EOF (déclenché quand CGIWriteHandler ferme le fd d'écriture).

Ce qui passe par l'environnement

Tout le reste : méthode, URI, query string, version HTTP, taille et type du body, identité du client, et chaque header de la requête. Le script les lit avec os.environ["..."] en Python ou $VARIABLE en bash.

02
buildEnvp()
Le chef d'orchestre — accumule dans un vector<string>
Pourquoi un vector<string> intermédiaire ?

execve() attend un char** terminé par NULL, mais on ne connaît pas à l'avance le nombre de variables (ça dépend du nombre de headers de la requête). On accumule donc d'abord dans un std::vector<std::string> (taille dynamique), puis on convertit une seule fois à la fin.

srcs/cgi/CGIProcess.cpp
Notre code 
char** CGIProcess::buildEnvp(std::string& path) {
    std::vector<std::string> envp;

    std::ostringstream oss;
    oss << _req.body.size();
    envp.push_back("CONTENT_LENGTH=" + oss.str());

    envp.push_back("REQUEST_METHOD=" + _req.method);
    envp.push_back("SCRIPT_FILENAME=" + path);
    envp.push_back("SERVER_PROTOCOL=" + _req.version);
    envp.push_back("GATEWAY_INTERFACE=CGI/1.1");
    envp.push_back("SERVER_SOFTWARE=webserv/1.0");

    addUri(&envp);         // SCRIPT_NAME, QUERY_STRING, PATH_INFO, REQUEST_URI
    addHost(&envp);        // SERVER_NAME, SERVER_PORT
    addHeaders(&envp);     // CONTENT_TYPE + tous les HTTP_*
    addRemoteAddr(&envp);  // REMOTE_ADDR, REMOTE_PORT

    return convertToCharStarStarBabyyy(&envp);
}
💡
CONTENT_LENGTH vient de _req.body.size(), pas du header. Nuance cruciale : si la requête était chunked, le header Content-Length n'existe pas. Comme notre HttpParser a déjà dé-chunké le body avant le routing, la taille réelle du body reconstitué est toujours la bonne valeur à transmettre. C'est exactement ce que demande le sujet : le CGI ne voit jamais de chunks.
03
addUri() & addHost()
Découper l'URI — et satisfaire le cgi_test de 42
srcs/cgi/CGIProcess.cpp — addUri()
Notre code 
// SCRIPT_NAME = chemin avant le '?', QUERY_STRING = ce qui suit.
// Exemple : "/cgi-bin/multiply.py?a=4&b=7"
//   → SCRIPT_NAME=/cgi-bin/multiply.py
//   → QUERY_STRING=a=4&b=7
void CGIProcess::addUri(std::vector<std::string>* envp) {
    size_t pos = _req.uri.find('?');
    std::string scriptPath = (pos != std::string::npos) ? _req.uri.substr(0, pos) : _req.uri;
    envp->push_back("SCRIPT_NAME=" + scriptPath);
    envp->push_back("QUERY_STRING=" + (pos != std::string::npos ? _req.uri.substr(pos + 1) : ""));
    // PATH_INFO et REQUEST_URI obligatoires pour cgi_test (42) : il vérifie
    // que PATH_INFO existe ET correspond à REQUEST_URI, sinon 500
    // ("PATH_INFO not found" / "PATH_INFO incorrect").
    envp->push_back("PATH_INFO=" + scriptPath);
    envp->push_back("REQUEST_URI=" + _req.uri);
}
srcs/cgi/CGIProcess.cpp — addHost()
Notre code 
// SERVER_NAME/SERVER_PORT extraits du header "Host: nom[:port]".
// Port 80 par défaut si absent du header.
void CGIProcess::addHost(std::vector<std::string>* envp) {
    std::map<std::string, std::string>::const_iterator it;

    it = _req.headers.find("Host");
    std::string host = (it != _req.headers.end() ? it->second : "");
    std::string serverPort = "80";
    size_t hostSep = host.find(':');
    if (hostSep != std::string::npos) {
        envp->push_back("SERVER_NAME=" + host.substr(0, hostSep));
        envp->push_back("SERVER_PORT=" + host.substr(hostSep + 1));
    } else {
        envp->push_back("SERVER_NAME=" + host);
        envp->push_back("SERVER_PORT=" + serverPort);
    }
}
⚠️
Le piège du tester officiel. Le binaire cgi_test de 42 (monté sur location /directory avec cgi_pass .bla dans notre default.conf) refuse de répondre si PATH_INFO est absent ou ne correspond pas à REQUEST_URI. D'où les deux lignes surlignées : sans elles, POST sur *.bla renvoyait un 500 mystérieux.
04
addHeaders() & addRemoteAddr()
User-Agent → HTTP_USER_AGENT — la convention RFC 3875
srcs/cgi/CGIProcess.cpp — addHeaders()
Notre code 
// Convertit chaque header restant en variable HTTP_<NOM_EN_MAJUSCULES>,
// tirets remplacés par underscores (convention CGI/1.1, RFC 3875) :
// ex. "User-Agent: curl/8.0"  →  HTTP_USER_AGENT=curl/8.0
void CGIProcess::addHeaders(std::vector<std::string>* envp) {
    std::map<std::string, std::string>::const_iterator it;

    it = _req.headers.find("Content-Type");
    std::string contentType = (it != _req.headers.end() ? it->second : "");
    envp->push_back("CONTENT_TYPE=" + contentType);

    for (it = _req.headers.begin(); it != _req.headers.end(); ++it) {
        const std::string& key = it->first;
        // Content-Type et Content-Length ont déjà leurs variables dédiées,
        // pas de préfixe HTTP_ pour eux
        if (key == "Content-Type" || key == "Content-Length")
            continue;

        std::string name = "HTTP_";
        for (size_t i = 0; i < key.size(); i++) {
            if (key[i] == '-')
                name += '_';
            else
                name += std::toupper(static_cast<unsigned char>(key[i]));
        }
        envp->push_back(name + "=" + it->second);
    }
}
srcs/cgi/CGIProcess.cpp — addRemoteAddr()
Notre code 
// REMOTE_ADDR/REMOTE_PORT : adresse du client qui a fait la requête HTTP.
// _peerAddr arrive déjà formaté "ip:port" depuis ClientHandler::_buildPeerStr().
void CGIProcess::addRemoteAddr(std::vector<std::string>* envp) {
    size_t sep = _peerAddr.find(':');
    if (sep != std::string::npos) {
        envp->push_back("REMOTE_ADDR=" + _peerAddr.substr(0, sep));
        envp->push_back("REMOTE_PORT=" + _peerAddr.substr(sep + 1));
    } else {
        envp->push_back("REMOTE_ADDR=" + _peerAddr);
    }
}
Récapitulatif — tout ce que notre serveur transmet
REQUEST_METHODGET / POST (depuis _req.method)
CONTENT_LENGTHtaille réelle du body déjà dé-chunké (_req.body.size())
CONTENT_TYPEheader Content-Type, chaîne vide si absent
SCRIPT_NAME / SCRIPT_FILENAMEURI sans query / chemin disque complet du script
QUERY_STRINGtout ce qui suit le ? de l'URI
PATH_INFO / REQUEST_URIexigés par cgi_test — URI sans query / URI complète
SERVER_NAME / SERVER_PORTdécoupés depuis le header Host
REMOTE_ADDR / REMOTE_PORTIP:port du client (depuis accept())
SERVER_PROTOCOLHTTP/1.1 ou HTTP/1.0 (_req.version)
GATEWAY_INTERFACE / SERVER_SOFTWARECGI/1.1 / webserv/1.0
HTTP_*chaque autre header, majuscules, tirets → underscores
05
convertToCharStarStarBabyyy()
Du C++ au C — le format qu'execve() exige
srcs/cgi/CGIProcess.cpp
Notre code 
// Convertit le vector<string> en char** NULL-terminated attendu par execve().
// strdup() alloue chaque chaîne séparément ; ces allocations ne sont jamais
// libérées explicitement, mais ça n'a pas d'importance ici : soit execve()
// réussit et remplace tout l'espace mémoire du processus (le tas actuel
// disparaît avec), soit elle échoue et exit(1) suit juste après.
char** CGIProcess::convertToCharStarStarBabyyy(std::vector<std::string>* envp) {
    char** env;

    env = (char**)malloc((envp->size() + 1) * sizeof(char*));
    if (!env)
        throw std::runtime_error("malloc() on env failed");

    for (size_t i = 0; i < envp->size(); i++) {
        std::string& s = (*envp)[i];
        env[i] = strdup(s.c_str());
    }
    env[envp->size()] = NULL;   // ← le NULL final, obligatoire pour execve

    return env;
}
💡
Pourquoi strdup et pas juste c_str() ? Les pointeurs retournés par c_str() appartiennent aux std::string du vector : si le vector est détruit ou réalloue, ils deviennent invalides. strdup() fait des copies indépendantes qui survivent jusqu'à l'execve(). Et comme cette fonction n'est appelée que dans l'enfant, entre fork et execve, la « fuite » n'en est pas une : la mémoire disparaît avec l'execve.

Résumé — l'environnement CGI
La HttpRequest entière, traduite en variables pour un processus qui ne parle pas HTTP
📦
Deux canaux, un contrat
Body → stdin (via CGIWriteHandler). Métadonnées → environnement (via buildEnvp). Sortie du script → stdout (via CGIReadHandler).
🔧
vector<string> puis char**
Nombre de variables inconnu d'avance → accumulation dynamique, conversion unique en char** NULL-terminated juste avant execve.
🎯
PATH_INFO = REQUEST_URI, sinon 500
Le cgi_test officiel de 42 vérifie ces deux variables. CONTENT_LENGTH vient du body déjà dé-chunké, jamais du header brut.
