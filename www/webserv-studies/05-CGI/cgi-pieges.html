CGI — Les pièges, dans notre code

EOF, chunked & exit codes
— la classe CGIReadHandler

Trois vérités désagréables : le CGI attend un flux terminé par EOF (pas des chunks), sa sortie n'est pas une réponse HTTP (il faut la parser), et un script peut crasher, boucler ou ne jamais répondre. Dans notre webserv, tout ça est géré par CGIReadHandler : parseCgi(), reapChild() et le trio d'erreurs 500 / 502 / 504.

01
Dé-chunker avant d'envoyer
Le CGI ne connaît pas le protocole chunked

« For chunked requests, your server needs to un-chunk them, the CGI will expect EOF as the end of the body. »

— webserv.pdf, IV.3
Comment c'est réglé chez nous : en amont, dans le parser

Ce piège est désamorcé avant même que le CGI existe. Quand une requête arrive en Transfer-Encoding: chunked, notre HttpParser passe en état R_CHUNKED et readChunked() reconstitue le body brut : il lit chaque taille en hexa, extrait le contenu du chunk, et accumule dans _body. Au moment où Router::route() s'exécute, req.body est déjà un bloc continu — CGIWriteHandler n'écrit jamais de chunks dans le pipe, et CONTENT_LENGTH reflète la taille reconstituée.

srcs/handlers/CGIWriteHandler.cpp — le body part d'un bloc
Notre code 
int CGIWriteHandler::handle_output() {
    if (_sent >= _body.size())          // body vide : rien à envoyer
        return -1;

    ssize_t n = write(_fd, _body.data() + _sent, _body.size() - _sent);
    if (n <= 0)                          // le child a fermé son stdin → abandon
        return -1;

    _sent += static_cast<size_t>(n);
    return (_sent >= _body.size()) ? -1 : 0;   // -1 → EventLoop nous détruit
}

CGIWriteHandler::~CGIWriteHandler() {
    close(_fd);   // ← fermer notre bout d'écriture = EOF sur le stdin du script
}
💡
Le return -1 qui vaut un EOF. Quand tout le body est parti, handle_output() retourne -1 : l'EventLoop retire le handler et le delete. Son destructeur ferme _fd — le dernier fd d'écriture sur ce pipe. C'est exactement ça, « le CGI attend EOF » : le sys.stdin.read() du script se termine à cet instant précis. Un body vide (GET) suit le même chemin : premier passage → -1 immédiat → EOF.
02
read() == 0
L'EOF du pipe de sortie = « le script a fini de parler »
Pas de Content-Length côté CGI ? Pas grave.

Un script CGI n'annonce presque jamais la taille de sa sortie. Le seul signal fiable de fin, c'est l'EOF sur le pipe stdout : quand le script se termine (ou ferme sa sortie), read() retourne 0. Notre CGIReadHandler::handle_input() accumule donc dans _out tant que n > 0, et ne construit la réponse qu'au moment du n == 0.

srcs/handlers/CGIReadHandler.cpp — handle_input()
Notre code (essentiel) 
int CGIReadHandler::handle_input() {
    _lastActivity = time(NULL);

    char buf[4096];
    ssize_t n = read(_fd, buf, sizeof(buf));

    if (n > 0) {
        _out.append(buf, n);
        return 0;             // pas fini : epoll nous rappellera
    }
    if (n < 0) {
        if (_sink) {
            // erreur réelle (epoll a dit "lisible")
            ErrorHandler err(*_loc, 502, "");
            _sink->onCgiDone(err.buildResponse());
            _responded = true;
        }
        return -1;
    }

    // n == 0 : EOF — le CGI a fermé son stdout, _out est complet
    int status = 0;
    reapChild(status);
    // ... analyse du status + parseCgi() → sections suivantes ...
    return -1;                    // EventLoop nous retire et nous delete
}
⚠️
Un read() par événement, jamais de boucle. Même règle que pour les sockets clients : le pipe est non-bloquant, on lit 4096 octets max par notification epoll, et on laisse epoll re-notifier (level-triggered) tant qu'il reste des données. Zero errno, zero drain-loop.
03
parseCgi()
La sortie CGI n'est PAS une réponse HTTP
Le format RFC 3875 : headers CGI + ligne vide + body

Un script bien élevé sort quelque chose comme Status: 404\r\nContent-Type: text/html\r\n\r\n<h1>...</h1>. Ce ne sont pas des headers HTTP : Status: n'existe pas en HTTP, et il manque la request line, Content-Length, Connection… Avant ce parsing, notre serveur collait la sortie brute dans le body — et les headers du script s'affichaient en haut de la page. parseCgi() sépare donc headers CGI et body, puis reconstruit une vraie réponse HTTP via StaticHandler::buildResponse().

srcs/handlers/CGIReadHandler.cpp — parseCgi()
Notre code 
std::string CGIReadHandler::parseCgi() {
    int         code = 200;
    std::string type;
    std::string body;
    std::string cookie;

    size_t sep = _out.find("\r\n\r\n");   // séparateur headers/body...
    size_t s = 4;
    if (sep == std::string::npos) {
        sep = _out.find("\n\n");            // ...tolérance : LF seul (script unix)
        s   = 2;
    }
    if (sep == std::string::npos)
        body = _out;                        // aucun header : tout est body
    else {
        std::string headers = _out.substr(0, sep);
        body                = _out.substr(sep + s);

        std::istringstream  iss(headers);
        std::string         line;
        while (std::getline(iss, line)) {
            if (!line.empty() && line[line.size() - 1] == '\r')
                line.erase(line.size() - 1);
            if (line.compare(0, 7, "Status:") == 0)
                code = std::atoi(line.c_str() + 7);       // Status: 404 → code 404
            else if (line.compare(0, 13, "Content-Type:") == 0) {
                size_t v = 13;
                while (v < line.size() && line[v] == ' ')
                    v++;
                type = line.substr(v);
            }
            else if (line.compare(0, 11, "Set-Cookie:") == 0) {
                size_t v = 11;
                while (v < line.size() && line[v] == ' ')
                    v++;
                cookie = line.substr(v);                   // bonus cookies/sessions
            }
        }
    }
    if (code <= 0)
        code = 200;

    StaticHandler st(*_loc, code, body);   // reconstruit une VRAIE réponse HTTP
    if (!type.empty())
        st.setType(type);
    if (!cookie.empty())
        st.setCookie(cookie);

    return st.buildResponse();
}
Trois headers CGI reconnus
  • Status: → devient le status code HTTP (200 par défaut)
  • Content-Type: → recopié dans la réponse via setType()
  • Set-Cookie: → recopié via setCookie() (bonus sessions)
Tolérance CRLF / LF

La RFC veut du \r\n, mais un script Python écrit naturellement \n. On cherche d'abord \r\n\r\n, puis on retombe sur \n\n. Nos scripts test_crlf.py et test_lf.py valident les deux chemins, et test_no_headers.py le cas « aucun header : tout est body ».

04
reapChild()
waitpid(WNOHANG) — récupérer l'enfant sans geler le serveur
srcs/handlers/CGIReadHandler.cpp — reapChild()
Notre code 
// Récupère le statut de l'enfant sans jamais bloquer la boucle : si le CGI a
// fermé son stdout mais tourne encore (ex: boucle infinie après le print),
// l'ancien waitpid bloquant gelait TOUT le serveur. On le tue proprement et
// on sert quand même la sortie déjà reçue.
void CGIReadHandler::reapChild(int& status) {
    // WNOHANG dit au noyau : "regarde si le process _pid est déjà mort.
    // S'il l'est, rends-moi son status.
    // S'il ne l'est PAS, ne m'endors pas, rends-moi 0 tout de suite."
    pid_t r = waitpid(_pid, &status, WNOHANG);
    if (r == 0) {
        kill(_pid, SIGKILL);          // encore vivant → on le tue
        waitpid(_pid, &status, 0);   // reap immédiat : il vient d'être SIGKILLé
        status = 0;
    }
    _reaped = true;
}
💡
Pourquoi pas de handler SIGCHLD ? On n'en a pas besoin : le cycle de vie de l'enfant est entièrement piloté par les événements du pipe. EOF sur stdout → on reap. Timeout 30s → le destructeur reap. Un signal handler global apporterait de la concurrence (interruption au milieu de n'importe quel code) pour zéro bénéfice. Aucun zombie possible : chaque chemin de sortie passe par un waitpid().
05
WIFEXITED / WIFSIGNALED
Traduire la mort du script en status code HTTP
srcs/handlers/CGIReadHandler.cpp — suite du handle_input(), après l'EOF
Notre code 
std::string raw;
// Retourne vrai si le processus a été tué par un signal qu'il n'a pas géré —
// par exemple un segfault (SIGSEGV), un SIGKILL, un SIGTERM, etc.
// Dans ce cas il n'y a pas de "code de sortie" au sens normal.
if (WIFSIGNALED(status)) {
    ErrorHandler err(*_loc, 502, "");   // crash (SIGSEGV...) → 502 Bad Gateway
    raw = err.buildResponse();
}
// WIFEXITED → terminé normalement (return / exit).
// WEXITSTATUS → le code passé à exit(), l'équivalent du $? du shell.
else if (WIFEXITED(status) && WEXITSTATUS(status) != 0) {
    ErrorHandler err(*_loc, 500, _out);  // exit(1) → 500 Internal Server Error
    raw = err.buildResponse();
}
else
    raw = parseCgi();                    // sortie propre → réponse normale

if (_sink) {
    _sink->onCgiDone(raw);               // livrer au client via l'interface
    _responded = true;
}
return -1;
srcs/handlers/CGIReadHandler.cpp — le destructeur, filet de sécurité
504 Gateway Timeout 
// Trois responsabilités à la destruction (EOF normal, timeout, arrêt serveur) :
// 1. fermer notre bout du pipe
// 2. tuer + reaper l'enfant s'il ne l'a pas déjà été (pas de zombie)
// 3. si aucun client n'a encore reçu de réponse (timeout CGI), envoyer 504 —
//    "une requête ne doit jamais rester en attente indéfiniment" (sujet)
CGIReadHandler::~CGIReadHandler() {
    close(_fd);
    if (!_reaped) {
        kill(_pid, SIGKILL);
        waitpid(_pid, NULL, 0);
    }
    // destructeur appelé sans passer par handle_input() (ex: timeout après
    // 30s d'inactivité dans EventLoop::checkTimeOut) -> client jamais servi
    if (_sink && !_responded) {
        ErrorHandler err(*_loc, 504, "");
        _sink->onCgiDone(err.buildResponse());
    }
}

Résumé — les pièges désamorcés
Chunked réglé en amont, EOF comme signal, sortie parsers, enfant toujours reapé
🔄
Chunked → déjà dé-chunké
HttpParser::readChunked() reconstitue le body avant le routing. Le CGI reçoit un flux continu + un EOF quand CGIWriteHandler ferme le pipe.
📝
Sortie CGI ≠ réponse HTTP
parseCgi() extrait Status: / Content-Type: / Set-Cookie:, puis StaticHandler reconstruit la vraie réponse (status line, Content-Length, Connection).
✂️
Jamais de zombie, jamais de gel
waitpid(WNOHANG) au moment de l'EOF ; si l'enfant vit encore : SIGKILL + reap. Le destructeur refait le même travail pour les chemins timeout/arrêt.
🚨
500 / 502 / 504
exit≠0 → 500. Tué par signal ou pipe en erreur → 502. Muet 30 secondes → 504 depuis le destructeur. Le client reçoit toujours une réponse.
