Bonus — CGI multiples, dans notre code

Plusieurs interpréteurs CGI
— une map extension → binaire

Le bonus demande de supporter plus d'un langage CGI. Chez nous, ce n'est même pas une fonctionnalité à part : le serveur ne connaît aucun langage en dur. Chaque location déclare des paires extension → interpréteur dans la config ; au routing, on lit l'extension de l'URI et on lance le binaire correspondant. Ajouter PHP, Perl ou Ruby, c'est une ligne de config, zéro ligne de C++.

01
cgi_pass → Location::cgi_pass
La config déclare les couples extension / binaire
Une map par location

Tout part du fichier de configuration. Une directive cgi_pass .ext /chemin/interpréteur; associe une extension à un exécutable. Le parser les range dans Location::cgi_pass, une std::map<extension, interpréteur>. Rien ne limite à Python : n'importe quel binaire exécutable fait l'affaire, et on peut en déclarer autant qu'on veut dans la même location.

includes/config/ConfigStruct.hpp — la structure
Notre code 
struct Location : public BaseBlock {
    std::map<str, str>   cgi_pass;        // ".py" → "/usr/bin/python3", ".sh" → "/usr/bin/bash" ...
    std::vector<str>      allowed_methods;
    std::string            upload_store;
    std::string            path;
};
srcs/config/Parser.cpp — lecture de la directive
Notre code 
else if (key == "cgi_pass") {

💡
Le Validator vérifie chaque couple. Avant de démarrer, il contrôle que chaque extension commence par un . et que l'interpréteur pointé est bien exécutable. Une config qui pointe vers un binaire inexistant est refusée au lancement, pas découverte à la première requête.
02
Router::isCgi()
L'extension de l'URI choisit l'interpréteur
Regarder l'URI, interroger la map

Une fois la location matchée, Router::isCgi() extrait l'extension du chemin demandé (tout ce qui suit le dernier .) et regarde si la map cgi_pass de cette location la connaît. Si oui, il renvoie le chemin de l'interpréteur ; sinon une chaîne vide — et la requête repart dans le circuit statique classique.

srcs/http/Router.cpp — isCgi()
Notre code 
std::string Router::isCgi(const std::string& uri, const Location* loc) {
    size_t pos = uri.rfind('.');
    std::string ext;
    if (pos == std::string::npos)
        return "";
    else
        ext = uri.substr(pos);            // ".py", ".sh", ".php" ...
    if (loc->cgi_pass.find(ext) != loc->cgi_pass.end())
        return loc->cgi_pass.find(ext)->second;  // ← l'interpréteur choisi
    return "";
}
srcs/http/Router.cpp — dans route()
Le déclencheur 
std::string interpreter = isCgi(uriPath, loc);
if (!interpreter.empty() && (req.method == "GET" || req.method == "POST")) {
    CGIHandler cgi(req, path, interpreter, peerAddr, loc, sink);
    return NULL;   // asynchrone : réponse plus tard via onCgiDone()
}
💡
La décision tient dans la valeur de retour. isCgi() renvoie soit un chemin d'interpréteur, soit "". Ce simple std::string non vide est ce qui bascule la requête du monde « fichier statique » vers le monde « exécuter un programme ». Pour en savoir plus sur la suite asynchrone, voir la fiche CGIProcess.
03
execve(interpreter, ...)
Le même mécanisme pour tous les langages
Pourquoi ça marche pour n'importe quel langage

Le cœur de l'astuce : le serveur ne sait pas ce qu'est « Python » ou « bash ». Il reçoit une chaîne (le chemin de l'interpréteur) et la passe telle quelle à execve(). Le contrat CGI est universel : l'enfant lit le body sur stdin, écrit sa réponse sur stdout. Que ce soit python3 script.py ou bash script.sh, le serveur s'en moque totalement.

srcs/cgi/CGIProcess.cpp — bloc enfant
Notre code 
// interpreter vient de la config, file est le nom du script.
// execve les lance ensemble : "python3 hello.py", "bash calc.sh", ...
char* av[] = { (char*)interpreter.c_str(), (char*)file.c_str(), NULL };
execve(interpreter.c_str(), av, envp);
exit(1);   // seulement si execve échoue → 500 côté serveur
🔁
Le serveur est agnostique

Aucune condition if (langage == "python") nulle part. Le serveur manipule uniquement des chemins et des flux. C'est la config qui apporte toute la connaissance des langages — le code C++ reste identique quel que soit le CGI.

➕
Ajouter un langage = 1 ligne

Pour supporter PHP, il suffit d'ajouter cgi_pass .php /usr/bin/php-cgi; dans la location. Pas de recompilation, pas de nouveau code : la map grandit, isCgi() trouve la nouvelle extension, execve() lance php-cgi.

04
default.conf
Python et Bash dans la même location
Notre configuration réelle

Dans conf/default.conf, la location /cgi-bin déclare deux interpréteurs côte à côte. Un GET /cgi-bin/hello.py part vers python3, un GET /cgi-bin/sysinfo.sh part vers bash — même serveur, même location, deux langages, choisis uniquement par l'extension.

conf/default.conf
Notre config 
location /cgi-bin {
    root       www;
    cgi_pass   .py /usr/bin/python3;   # *.py  → interpréteur Python
    cgi_pass   .sh /usr/bin/bash;      # *.sh  → interpréteur Bash
}
Le parcours de deux requêtes
GET /cgi-bin/hello.pyext .py → /usr/bin/python3 → execve
GET /cgi-bin/sysinfo.shext .sh → /usr/bin/bash → execve
GET /cgi-bin/page.htmlext .html inconnue → "" → fichier statique
⚠️
L'extension décide, pas le nom. Comme isCgi() se base sur le dernier . de l'URI, un fichier rapport.py.txt serait traité comme .txt (statique), et un archive.tar.py comme .py (CGI). C'est toujours la dernière extension qui compte.
Résumé — plusieurs CGI en 3 maillons
config (map) → routing (extension) → execve (interpréteur générique)
📝
La config porte la connaissance
cgi_pass .ext /bin remplit une map<extension, interpréteur> par location, validée au démarrage.
🔍
Le routing choisit par extension
isCgi() lit la dernière extension de l'URI et renvoie l'interpréteur, ou "" pour rester en statique.
⚙️
L'exécution est agnostique
execve(interpreter, ...) lance n'importe quel binaire ; le contrat stdin/stdout est le même pour tous. Nouveau langage = 1 ligne de config.
