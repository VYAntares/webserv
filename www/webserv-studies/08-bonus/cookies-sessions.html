Bonus — Cookies & Sessions, dans notre code

Cookies & Sessions
— un aller-retour sans machinerie dédiée

L'élégance de notre implémentation : un cookie n'est rien de spécial. À l'aller, il voyage comme n'importe quel header dans la std::map de la requête ; au retour, il ressort par le champ _cookie générique de buildResponse(). L'état de session, lui, vit côté serveur dans un fichier JSON identifié par un cookie sid.

01
HttpParser → _req.headers
Le cookie arrive comme un header ordinaire
Aucun traitement particulier

Quand un navigateur a déjà un cookie, il l'envoie dans un header Cookie: sid=8f3a.... Chez nous il n'y a aucun code spécifique aux cookies à l'entrée : le parser HTTP range chaque header dans la même std::map<std::string, std::string> — Host, User-Agent, Cookie… tous logent au même endroit. Le cookie est déjà là, prêt à être lu, dès la fin du parsing.

srcs/http/HttpParser.cpp — parsing des headers
Notre code 
// une seule ligne stocke TOUS les headers, cookie compris :
// "Cookie: sid=8f3a...\r\n"  →  headers["Cookie"] = "sid=8f3a..."
_req.headers[line.substr(0, sep)] = line.substr(sep + 2);
💡
Un cookie = une chaîne de texte. Côté HTTP, un cookie n'est qu'une paire clé=valeur transportée dans un header. Toute la logique (« qui est ce client ? », « quel est son panier ? ») est décidée ailleurs — ici, par le script CGI de session. Le serveur ne fait que transporter la chaîne.
02
CGIProcess::addHeaders()
Cookie → HTTP_COOKIE dans l'environnement
La convention CGI/1.1 (RFC 3875)

Le script de session ne lit pas la requête HTTP : il lit des variables d'environnement. Avant l'execve(), addHeaders() transforme chaque header en HTTP_<NOM> (majuscules, tirets → underscores). Le header Cookie devient donc HTTP_COOKIE — encore une fois, aucune ligne dédiée aux cookies : ils profitent de la règle générale.

srcs/cgi/CGIProcess.cpp — addHeaders()
Notre code 
for (it = _req.headers.begin(); it != _req.headers.end(); ++it) {
    const std::string& key = it->first;
    if (key == "Content-Type" || key == "Content-Length")
        continue;                       // eux ont déjà leur variable dédiée

    std::string name = "HTTP_";
    for (size_t i = 0; i < key.size(); i++) {
        if (key[i] == '-') name += '_';
        else              name += std::toupper((unsigned char)key[i]);
    }
    envp->push_back(name + "=" + it->second);  // Cookie → HTTP_COOKIE=sid=8f3a...
}
www/cgi-bin/session.py — côté script
Le CGI lit l'env 
# le script récupère le cookie dans l'environnement, pas dans la requête HTTP

03
Session = fichier serveur
Le cookie identifie, le serveur se souvient
Cookie ≠ Session

Un cookie est côté client : une petite chaîne que le navigateur renvoie à chaque requête. Une session est côté serveur : l'état réel (visites, préférences…), trop gros ou trop sensible pour vivre dans le cookie. Le cookie ne contient qu'un identifiant (sid) ; le serveur s'en sert pour retrouver le bon fichier.

1
Pas de sid valide ? On en génère un (uuid4().hex, 32 hex) et on marque la session comme neuve.
↓
2
On charge sessions/<sid>.json (ou on crée un état vide) — chemin relatif possible grâce au chdir() du serveur dans le dossier du script.
↓
3
On applique l'action (visit incrémente le compteur, set stocke une clé, logout détruit tout), puis on réécrit le JSON.
www/cgi-bin/session.py — identification + état
Notre CGI de session 
# sid invalide ou absent → nouvelle session
if not re.match(r"^[0-9a-f]{32}$", sid):
    sid = ""
if not sid:
    sid = uuid.uuid4().hex          # identifiant unique
    new_session = True

# l'état vit dans un fichier serveur, indexé par le sid
session = load(sid) or {"created_at": now, "visits": 0}

if action == "visit":
    session["visits"] += 1
    save(sid, session)          # → sessions/<sid>.json
⚠️
Nettoyage automatique (TTL 1h). À chaque appel, cleanup() supprime les fichiers de session dont la dernière modification remonte à plus de SESS_TTL = 3600 s. Sans ça, le dossier sessions/ gonflerait indéfiniment à chaque nouveau visiteur.
04
buildResponse() → Set-Cookie
Le cookie repart par le champ générique de la réponse
Un seul champ, un seul point d'insertion

Côté réponse, même philosophie qu'à l'aller : pas de sous-système cookie. La classe de base ARequestHandler possède un champ _cookie et un setter setCookie(). buildResponse() — la seule fonction qui sérialise la réponse HTTP — insère la ligne Set-Cookie: si ce champ est rempli. Tout handler (statique, erreur, CGI) en hérite gratuitement.

srcs/http/ARequestHandler.cpp — buildResponse()
Notre code 
oss << "HTTP/1.1 " << _ncode << " " << getReason(_ncode) << "\r\n";
if (!_location.empty())
    oss << "Location: " << _location << "\r\n";
if (!_type.empty())
    oss << "Content-Type: " << _type << "\r\n";
if (!_cookie.empty())                              // ← le seul endroit
    oss << "Set-Cookie: " << _cookie << "\r\n";
Le cas CGI : relayer le Set-Cookie du script

Quand la réponse vient d'un CGI, le script émet lui-même un header Set-Cookie:. CGIReadHandler::parseCgi() lit la sortie du script, repère cette ligne, et la réinjecte dans le champ générique via setCookie() — on retombe exactement sur le même chemin que ci-dessus.

srcs/handlers/CGIReadHandler.cpp — parseCgi()
Notre code 
else if (line.compare(0, 11, "Set-Cookie:") == 0) {
    size_t v = 11;
    while (v < line.size() && line[v] == ' ') v++;

}
// ... plus bas, on le passe au StaticHandler qui bâtira la réponse :

www/cgi-bin/session.py — émission du cookie
Le CGI pose le cookie 
if new_session:
    set_cookie = "sid=%s; Path=/" % sid
# ... logout expire le cookie : "sid=deleted; Expires=Thu, 01 Jan 1970 ..."

sys.stdout.write("Content-Type: application/json\r\n")
if set_cookie:
    sys.stdout.write("Set-Cookie: %s\r\n" % set_cookie)
sys.stdout.write("\r\n")
Résumé — le cycle d'un cookie en 4 temps
header entrant → HTTP_COOKIE → fichier de session → Set-Cookie sortant
📥
Aller — le cookie est un header comme les autres
Le parser range Cookie dans _req.headers, puis addHeaders() le passe au CGI en HTTP_COOKIE. Zéro code dédié aux cookies.
💾
État — la session vit côté serveur
Le cookie ne porte qu'un sid ; l'état réel est dans sessions/<sid>.json, nettoyé après 1 h.
📤
Retour — un seul point d'insertion
setCookie() remplit _cookie, buildResponse() émet Set-Cookie. Le CGI relaie son propre header par le même chemin.
