03 — Le protocole HTTP

Status Codes & Headers
La réponse HTTP

Le serveur reçoit une requête, la traite, et renvoie une réponse. Cette réponse a une structure précise : une status line, des headers, et un body. Chaque status code a une sémantique — ton serveur ne peut pas répondre n'importe quoi.

« Your HTTP response status codes must be accurate. » — webserv.pdf, IV.1 Requirements
« The status-code element is a three-digit integer code giving the result of the attempt to understand and satisfy the request. » — RFC 7231, §6
01
buildResponse()
Anatomie d'une réponse HTTP

Une réponse HTTP suit exactement le même format que la requête : une status line (au lieu d'une request line), des headers, puis optionnellement un body. Chaque ligne se termine par \r\n.

Réponse HTTP complète annotée — 200 OK
HTTP/1.1 200 OK\r\n← Status Line Content-Type: text/html; charset=UTF-8\r\n← MIME type Content-Length: 1024\r\n← Taille du body en octets Connection: keep-alive\r\n← Connexion persistante Server: webserv/1.0\r\n \r\n← Fin des headers <!DOCTYPE html><html>...</html>← Body (1024 octets)

La status line se décompose en trois parties : la version du protocole, le status code (un nombre à 3 chiffres), et la reason phrase (un texte lisible pour les humains). Ton serveur doit envoyer les trois.

La status line — décomposition
HTTP/1.1
Version
200
Status Code
OK
Reason Phrase

Dans ton code C++, construire une réponse c'est concaténer des strings : "HTTP/1.1 200 OK\r\n" + headers + "\r\n" + body. Le \r\n final vide sépare les headers du body — c'est obligatoire, même si le body est vide.

02
setHeaders()
Les headers de réponse essentiels

Tu n'as pas besoin d'envoyer 50 headers. Concentre-toi sur ceux qui sont nécessaires pour que le navigateur comprenne ta réponse. Le reste est optionnel.

Header Rôle Exemple
Content-Type Le type MIME du body. Sans ça, le navigateur ne sait pas comment afficher le contenu. text/html; charset=UTF-8
Content-Length La taille exacte du body en octets. Permet au client de savoir quand il a tout reçu. 1024
Connection keep-alive pour réutiliser la connexion TCP, close pour la fermer après. keep-alive
Server Identification du serveur. Optionnel mais classique. webserv/1.0
Location URL de destination pour les redirections (301, 302). Obligatoire avec 3xx. /new-page
Allow Liste des méthodes autorisées. Obligatoire avec une réponse 405. GET, POST
Retry-After Indique au client combien de temps attendre avant de re-tenter. Utilisé avec 413 et 503. 120
Transfer-Encoding Si tu envoies en chunked, pas besoin de Content-Length. chunked
⚠️
Content-Type est critique. Si tu envoies du HTML sans Content-Type: text/html, le navigateur l'affichera comme du texte brut. Les types courants : text/html, text/css, application/javascript, image/png, image/jpeg, application/octet-stream (binaire par défaut).
⚠️
Content-Length doit être exact. Si la taille ne correspond pas au body réel : trop court → le navigateur coupe le contenu ; trop long → il attend indéfiniment des octets qui n'arrivent pas. Toujours utiliser body.size() ou strlen() pour calculer.
03
2xx
Succès — la requête a été traitée

« The 2xx (Successful) class of status code indicates that the client's request was successfully received, understood, and accepted. »

— RFC 7231, §6.3

Les codes 2xx signifient que la requête a été reçue, comprise et acceptée par le serveur. Le contenu exact du payload varie selon la méthode utilisée et le code spécifique.

200 OK Le classique universel

« The 200 (OK) status code indicates that the request has succeeded. The payload sent in a 200 response depends on the request method. »

— RFC 7231, §6.3.1

La requête a été reçue, comprise, et traitée avec succès. Le body de la réponse contient le résultat demandé. C'est le code par défaut pour tout ce qui fonctionne. Selon le RFC, le contenu du payload dépend de la méthode : GET → la ressource demandée ; POST → le résultat de l'action ; DELETE → une description du statut. Ce code est cacheable par défaut.

Scénario GET : le navigateur demande GET /index.html → le fichier existe et est lisible. Tu le lis, tu l'envoies → 200 + body HTML.

Scénario POST : un script CGI traite un formulaire et retourne une page de confirmation → 200 + body de la réponse CGI.

HTTP/1.1 200 OK\r\n
Content-Type: text/html\r\n
Content-Length: 1024\r\n
\r\n
[body HTML ici]
GET POST DELETE
201 Created Ressource créée

« The 201 (Created) status code indicates that the request has been fulfilled and has resulted in one or more new resources being created. The primary resource created by the request is identified by either a Location header field in the response, or, if no Location field is received, by the effective request URI. »

— RFC 7231, §6.3.2

La requête a abouti à la création d'une nouvelle ressource. Utilisé principalement avec POST pour les uploads. Le header Location doit indiquer l'URL de la ressource créée. Le payload du 201 décrit et référence la ou les ressources créées.

Scénario : le client envoie POST /uploads avec un fichier. Tu sauvegardes le fichier → tu réponds 201 avec Location: /uploads/photo.jpg.

HTTP/1.1 201 Created\r\n
Location: /uploads/photo.jpg\r\n
Content-Length: 0\r\n
\r\n
POST
204 No Content Succès sans body Attention

« The 204 (No Content) status code indicates that the server has successfully fulfilled the request and that there is no additional content to send in the response payload body. Metadata in the response header fields refer to the target resource and its selected representation after the requested action was applied. »

— RFC 7231, §6.3.5

La requête a réussi mais il n'y a rien à renvoyer dans le body. Utilisé pour DELETE principalement. Ne jamais envoyer de body avec un 204. Les headers de métadonnées (ETag, Last-Modified, etc.) se réfèrent à la ressource cible après l'application de l'action demandée.

Scénario : le client envoie DELETE /uploads/photo.jpg. Tu fais unlink() avec succès → 204, aucun body. Le navigateur comprend : "c'est fait, rien à afficher".

HTTP/1.1 204 No Content\r\n
Connection: close\r\n
\r\n
[aucun body — ni une seule lettre]
DELETE POST
04
3xx
Redirections — la ressource est ailleurs

« The 3xx (Redirection) class of status code indicates that further action needs to be taken by the user agent in order to fulfill the request. »

— RFC 7231, §6.4

Les codes 3xx indiquent que le client doit effectuer une action supplémentaire pour compléter la requête. Si un header Location est fourni, le user-agent peut suivre automatiquement la redirection. Dans webserv, tu gères les redirections via la directive return de ta config.

301 Moved Permanently Redirection permanente

« The 301 (Moved Permanently) status code indicates that the target resource has been assigned a new permanent URI and any future references to this resource ought to use one of the enclosed URIs. »

— RFC 7231, §6.4.2

La ressource a définitivement changé d'URL. Le header Location indique la nouvelle URL. Le navigateur mémorise ce changement et suit automatiquement. Ce code est cacheable par défaut — le navigateur n'a pas besoin de redemander cette URL.

Config : tu as une directive return 301 /new-page sur une route. Quand le client frappe l'ancienne URL → tu réponds 301 avec le nouveau chemin dans Location.

Dossier sans slash : si le client demande GET /folder (sans slash final) mais que folder est un répertoire → tu réponds 301 vers /folder/. NGINX fait ça automatiquement.

HTTP/1.1 301 Moved Permanently\r\n
Location: /new-page\r\n
Content-Length: 0\r\n
\r\n
GET
302 Found Redirection temporaire

« The 302 (Found) status code indicates that the target resource resides temporarily under a different URI. Since the redirection might be altered on occasion, the client ought to continue to use the effective request URI for future requests. »

— RFC 7231, §6.4.3

Similaire au 301 mais temporaire. Le navigateur ne mémorise pas. Utile pour le pattern POST-Redirect-GET : après un POST réussi, tu rediriges vers une page de confirmation avec un 302 pour éviter la re-soumission si l'utilisateur recharge la page.

Note historique : en pratique, la plupart des user-agents changent la méthode en GET lors du suivi d'un 302, même si ce n'est pas ce que le RFC prescrit.

Cas : le client envoie POST /login avec les identifiants. L'authentification réussit → tu réponds 302 vers /dashboard. Le navigateur fait un GET sur /dashboard. Si l'utilisateur recharge, il recharge le GET, pas le POST.

HTTP/1.1 302 Found\r\n
Location: /dashboard\r\n
Content-Length: 0\r\n
\r\n
GET POST
05
4xx
Erreurs client — c'est lui qui s'est trompé

« The 4xx (Client Error) class of status code indicates that the client seems to have erred. Except when responding to a HEAD request, the server SHOULD send a representation containing an explanation of the error situation, and whether it is a temporary or permanent condition. »

— RFC 7231, §6.5

Les codes 4xx signifient que le client semble avoir fait une erreur. Sauf pour une requête HEAD, le serveur devrait envoyer un body expliquant la situation d'erreur, et si c'est une condition temporaire ou permanente. Ces codes s'appliquent à toute méthode.

400 Bad Request Requête malformée

« The 400 (Bad Request) status code indicates that the server cannot or will not process the request due to something that is perceived to be a client error (e.g., malformed request syntax, invalid request message framing, or deceptive request routing). »

— RFC 7231, §6.5.1

La requête envoyée par le client est syntaxiquement incorrecte ou ne respecte pas le protocole HTTP. Le serveur ne peut pas l'interpréter. C'est une erreur de parsing.

Cas 1 : la Request Line est invalide — ex: GETGET /index HTTP/1.1 sans espace correct, ou une version HTTP inconnue. Cas 2 : en HTTP/1.1, le header Host est obligatoire. S'il est absent → 400 immédiatement. Cas 3 : un header malformé — ex: Content-Length: abc (valeur non numérique). Cas 4 : l'URI contient des caractères illégaux non encodés.

HTTP/1.1 400 Bad Request\r\n
Content-Type: text/html\r\n
Content-Length: [taille page erreur]\r\n
Connection: close\r\n
\r\n
[ta page d'erreur 400 par défaut]

403 Forbidden Accès refusé

« The 403 (Forbidden) status code indicates that the server understood the request but refuses to authorize it. If authentication credentials were provided in the request, the server considers them insufficient to grant access. »

— RFC 7231, §6.5.3

Le serveur a compris la requête mais refuse de l'exécuter. Ce n'est pas un problème d'authentification — c'est une règle de configuration ou de droits fichier.

Droits Unix : GET /secret.txt → le fichier existe mais les permissions ne permettent pas la lecture (access() retourne -1). Directory listing off : GET /uploads/ → c'est un répertoire, pas de fichier index, et le directory listing est désactivé dans la config → 403.

🔒
Astuce sécurité RFC : si le serveur ne veut pas révéler qu'une ressource existe, il peut répondre 404 au lieu de 403. Cela évite de confirmer l'existence de fichiers sensibles à un attaquant. Le RFC dit explicitement : « If the server does not wish to make this reason public, the status code 404 (Not Found) can be used instead. »
HTTP/1.1 403 Forbidden\r\n
Content-Type: text/html\r\n
Content-Length: [taille]\r\n
\r\n

404 Not Found Introuvable

« The 404 (Not Found) status code indicates that the origin server did not find a current representation for the target resource or is not willing to disclose that one exists. A 404 status code does not indicate whether this lack of representation is temporary or permanent. »

— RFC 7231, §6.5.4

La ressource demandée n'existe pas à l'URL indiquée. C'est le code d'erreur le plus connu sur le web. Cacheable par défaut selon le RFC — le navigateur peut mémoriser qu'une URL retourne 404 sans redemander à chaque fois.

Fichier absent : GET /page-qui-nexiste-pas.html → stat() retourne -1 avec errno ENOENT. Déjà supprimé : DELETE /uploads/fichier-deja-supprime.jpg → le fichier n'existe plus. Aucun match : aucun bloc location de la config ne matche l'URI demandée.

HTTP/1.1 404 Not Found\r\n
Content-Type: text/html\r\n
Content-Length: [taille]\r\n
\r\n
[ta page d'erreur 404 — obligatoire dans webserv]
Toute méthode
405 Method Not Allowed Méthode interdite sur cette route

« The 405 (Method Not Allowed) status code indicates that the method received in the request-line is known by the origin server but not supported by the target resource. The origin server MUST generate an Allow header field in a 405 response containing a list of the target resource's currently supported methods. »

— RFC 7231, §6.5.5

La ressource existe, mais la méthode HTTP utilisée n'est pas autorisée sur elle. La réponse DOIT inclure un header Allow listant les méthodes acceptées. Ce code est cacheable par défaut.

Config : la config autorise uniquement GET sur /static. Le client envoie POST /static/page.html → 405.

HTTP/1.1 405 Method Not Allowed\r\n
Allow: GET, POST\r\n
Content-Length: [taille]\r\n
\r\n
Toute méthode non listée
413 Content Too Large Body trop grand Webserv critique

« The 413 (Payload Too Large) status code indicates that the server is refusing to process a request because the request payload is larger than the server is willing or able to process. The server MAY close the connection to prevent the client from continuing the request. If the condition is temporary, the server SHOULD generate a Retry-After header field. »

— RFC 7231, §6.5.11

Le body de la requête dépasse la taille maximale configurée (client_max_body_size dans ta config). Le sujet demande explicitement de l'implémenter. Tu dois stopper la lecture et répondre immédiatement.

Content-Length : ta config dit client_max_body_size 1m. Le client envoie un POST avec Content-Length: 50000000 (50 Mo). Dès que tu lis le header et que tu vois que la taille dépasse → 413 sans attendre de lire le body.

Chunked : si le client envoie les données en chunked, tu dois surveiller la taille au fur et à mesure que tu déchunkes. Si ça dépasse pendant le décodage → 413 et fermer la connexion.

⚠️
RFC : le serveur PEUT fermer la connexion pour empêcher le client de continuer à envoyer des données. En pratique dans webserv, tu envoies le 413 avec Connection: close puis tu fermes le socket. Si la condition est temporaire, le RFC recommande d'ajouter un header Retry-After.
HTTP/1.1 413 Content Too Large\r\n
Connection: close\r\n
Content-Length: [taille]\r\n
\r\n
POST
06
5xx
Erreurs serveur — c'est ton code qui a un problème

« The 5xx (Server Error) class of status code indicates that the server is aware that it has erred or is incapable of performing the requested method. Except when responding to a HEAD request, the server SHOULD send a representation containing an explanation of the error situation. »

— RFC 7231, §6.6

Les codes 5xx signifient que le serveur sait qu'il a commis une erreur ou qu'il est incapable d'effectuer la méthode demandée. Le serveur devrait envoyer un body expliquant la situation d'erreur. Ces erreurs s'appliquent à toute méthode.

500 Internal Server Error Erreur interne

« The 500 (Internal Server Error) status code indicates that the server encountered an unexpected condition that prevented it from fulfilling the request. »

— RFC 7231, §6.6.1

Quelque chose a mal tourné côté serveur — une erreur inattendue que le serveur ne sait pas gérer autrement. C'est le "catch-all" des erreurs serveur. Dans webserv, ça arrive souvent avec le CGI.

fork()/execve() : le lancement du script CGI échoue → 500. CGI crash : le script plante (segfault, exception) et retourne un exit code non nul → 500. I/O error : open() sur le fichier à servir échoue pour une raison inattendue (disque plein, etc.) → 500. Mémoire : erreur d'allocation mémoire dans ton serveur C++. Le sujet dit que ton serveur ne doit JAMAIS crash — tu dois gérer ça proprement et retourner 500 plutôt que de planter.

HTTP/1.1 500 Internal Server Error\r\n
Content-Type: text/html\r\n
Content-Length: [taille]\r\n
\r\n
[ta page d'erreur 500 par défaut]

501 Not Implemented Méthode non implémentée

« The 501 (Not Implemented) status code indicates that the server does not support the functionality required to fulfill the request. This is the appropriate response when the server does not recognize the request method and is not capable of supporting it for any resource. »

— RFC 7231, §6.6.2

Le serveur ne supporte pas la méthode HTTP demandée. Différent du 405 : ici la méthode est inconnue du serveur, pas juste interdite sur une route. Ce code est cacheable par défaut.

Cas : le client envoie PATCH /file HTTP/1.1 ou PUT /file HTTP/1.1. Ton serveur n'implémente que GET, POST, DELETE → 501.

HTTP/1.1 501 Not Implemented\r\n
Content-Type: text/html\r\n
Content-Length: [taille]\r\n
Connection: close\r\n
\r\n
[ta page d'erreur 501 — "méthode PATCH non supportée"]
Toute méthode inconnue
502 Bad Gateway Réponse invalide du CGI

« The 502 (Bad Gateway) status code indicates that the server, while acting as a gateway or proxy, received an invalid response from an inbound server it accessed while attempting to fulfill the request. »

— RFC 7231, §6.6.3

Le serveur agissait comme passerelle et a reçu une réponse invalide du service en amont. Dans webserv, ça s'applique quand le CGI retourne une réponse malformée.

Cas : le script Python/PHP que tu as lancé via execve() retourne une sortie qui n'est pas du HTTP valide — il a oublié ses headers CGI, ou a écrit des données corrompues sur stdout → 502.

HTTP/1.1 502 Bad Gateway\r\n
Content-Type: text/html\r\n
Content-Length: [taille]\r\n
Connection: close\r\n
\r\n
[ta page d'erreur 502 — "le script CGI a retourné une réponse invalide"]

504 Gateway Timeout CGI trop lent Webserv critique

« The 504 (Gateway Timeout) status code indicates that the server, while acting as a gateway or proxy, did not receive a timely response from an upstream server it needed to access in order to complete the request. »

— RFC 7231, §6.6.5

Le serveur a attendu une réponse du CGI mais le délai imparti a expiré. Le sujet dit qu'une requête ne doit jamais bloquer indéfiniment.

Cas : tu lances un script Python via fork() + execve(). Le script entre dans une boucle infinie. Après X secondes de timeout, tu kill() le processus fils et tu réponds 504 au client.

Implémentation : dans ta boucle epoll, tu vérifies le timestamp de lancement de chaque CGI. Si now - start > timeout → kill + 504.

HTTP/1.1 504 Gateway Timeout\r\n
Content-Type: text/html\r\n
Content-Length: [taille]\r\n
Connection: close\r\n
\r\n
[ta page d'erreur 504 — "le script CGI n'a pas répondu à temps"]

📦
Codes cacheable par défaut (RFC 7231, §6.1) : les réponses avec ces codes peuvent être mises en cache par le navigateur sans directive explicite : 200, 203, 204, 206, 300, 301, 404, 405, 410, 414, 501. Tous les autres codes ne sont cacheable que si des headers de cache explicites sont présents.

L'ordre dans lequel tu vérifies les erreurs détermine quel code tu renvoies. Une requête peut déclencher plusieurs erreurs — tu dois renvoyer la première qui matche.

ordre de vérification
Logic 
// Ordre que ton handler doit respecter :

 1. Requête malformée ?            → 400
 2. Méthode inconnue ?              → 501
 3. Aucune route ne matche ?        → 404
 4. Body trop grand (POST) ?        → 413
 5. Méthode non autorisée / route ? → 405
 6. Redirection configurée ?        → 301
 7. Fichier introuvable ?           → 404
 8. Pas les droits ?                → 403
 9. Erreur interne / CGI crash ?    → 500
10. CGI timeout ?                   → 504
11. Tout OK                         → 200 / 201 / 204
⚠️
Règle d'or webserv : chaque réponse d'erreur doit avoir un body — ta page d'erreur HTML par défaut. Le sujet dit explicitement "your server must have default error pages". Si une page d'erreur personnalisée est configurée pour ce code (error_page 404 /custom_404.html) → tu la sers. Sinon → tu génères une page minimale de ton cru.

Distinction 405 vs 501 : 501 = "je ne connais pas cette méthode du tout" (PATCH, PUT, TOTO). 405 = "je la connais mais elle n'est pas permise sur cette route". La vérification 501 doit venir avant 405 dans l'ordre.
