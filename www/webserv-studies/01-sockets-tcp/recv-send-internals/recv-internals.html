recv() internals

Que se passe-t-il vraiment quand tu appelles recv() ? Le trajet des données depuis la carte réseau jusqu'à ton buffer utilisateur, en passant par le kernel, les sk_buff, et le socket receive buffer.

01
Le syscall recv()
De l'userspace au kernel
Ce qui se passe quand tu appelles recv()

recv(fd, buf, len, flags) n'est pas une simple lecture mémoire. C'est un appel système (syscall __NR_recvfrom sur Linux) qui traverse la frontière userspace → kernel. Le CPU bascule en mode superviseur via une interruption logicielle (syscall instruction sur x86-64).

Dans le kernel, le parcours est : sys_recvfrom() → sock_recvmsg() → tcp_recvmsg(). C'est tcp_recvmsg() qui fait le vrai travail : il copie les données depuis le socket receive buffer du kernel vers ton buffer en userspace.

🏦
Analogie

Tu vas au guichet de la banque (syscall). Tu donnes ton numéro de compte (fd) et un sac pour les billets (buffer). Le caissier (kernel) va chercher dans le coffre (receive buffer) et te donne ce qu'il y a. Si le coffre est vide, soit tu attends (bloquant), soit on te dit de revenir plus tard (EAGAIN).

//    e. Met à jour la fenêtre TCP (pour dire au client "j'ai de la place")
//    f. Retourne le nombre d'octets copiés
02
Les buffers kernel
Où s'accumulent les données reçues
Le socket receive buffer (sk_rcvbuf)

Chaque socket TCP a un buffer de réception dans le kernel : sk->sk_rcvbuf. C'est une file d'attente de structures sk_buff (la structure fondamentale pour les paquets réseau dans Linux).

Quand des données arrivent du réseau, le kernel les stocke dans ce buffer avant même que tu appelles recv(). Les données s'accumulent. Quand tu appelles recv(), le kernel copie depuis ce buffer vers ton buffer userspace.

La taille par défaut est configurable : net.core.rmem_default (souvent 212 992 octets ~ 208 KB). Maximum : net.core.rmem_max.

struct sk_buff — le conteneur de données

Chaque paquet réseau reçu est encapsulé dans un sk_buff. C'est la structure la plus importante du networking Linux. Elle contient :

  • Les données — le payload du paquet TCP
  • Les métadonnées — pointeurs vers les headers (Ethernet, IP, TCP)
  • Les pointeurs de navigation — head, data, tail, end
  • Le chaînage — liste doublement chaînée pour la file d'attente

Quand recv() copie les données, les sk_buff correspondants sont libérés de la receive queue et le kernel notifie le pair TCP qu'il a de la place (mise à jour de la fenêtre de réception).

Userspace — ton programme
char buffer[4096]
Ton buffer à toi. Les données arrivent ici après recv(). Tu en fais ce que tu veux.
↑ copy_to_user() — copie kernel → userspace
Kernelspace — struct sock
sk->sk_receive_queue
File d'attente de sk_buff. Les données s'accumulent ici automatiquement quand les paquets TCP arrivent du réseau. recv() les consomme.
sk->sk_rcvbuf (taille max)
Limite du buffer de réception. Si plein, le kernel dit au client "stop, je n'ai plus de place" (fenêtre TCP = 0). Le client arrête d'envoyer.
↑ tcp_v4_rcv() — traitement du paquet TCP
Carte réseau (NIC) + driver
Ring buffer (DMA)
La carte réseau reçoit les trames Ethernet et les place en mémoire via DMA (Direct Memory Access). Le driver crée un sk_buff par paquet et le passe à la stack réseau.
⚠️
Buffer plein = données perdues ? Non. Si le receive buffer kernel est plein, la fenêtre TCP tombe à 0. Le client le voit dans les ACK et arrête d'envoyer. C'est le mécanisme de flow control de TCP. Aucune donnée n'est perdue — le client attend que tu appelles recv() pour libérer de la place.
03
Le parcours complet
De la carte réseau à ton buffer
Le trajet d'un paquet TCP reçu — étape par étape
  • 1. NIC reçoit la trame Ethernet — la carte réseau détecte une trame, vérifie le CRC, la place en mémoire via DMA dans le ring buffer
  • 2. Interruption matérielle (IRQ) — la NIC déclenche une interruption. Le handler du driver la convertit en softirq (NAPI) pour éviter de bloquer le CPU
  • 3. NAPI poll — le kernel traite les paquets par batch. Chaque paquet brut devient un sk_buff
  • 4. Stack IP — ip_rcv() vérifie les headers IP, le checksum, le routage
  • 5. Stack TCP — tcp_v4_rcv() identifie le socket de destination via le 4-tuple (src_ip, src_port, dst_ip, dst_port), vérifie le séquencement
  • 6. Ajout à la receive queue — le sk_buff est ajouté à sk->sk_receive_queue. Si un processus dort dans recv(), il est réveillé
  • 7. recv() → copy_to_user() — quand tu appelles recv(), tcp_recvmsg() copie les données des sk_buff vers ton buffer, libère les sk_buff, et met à jour la fenêtre TCP

⏱️ Les données arrivent AVANT recv(). Le kernel accumule les paquets dès qu'ils arrivent. recv() ne fait que copier ce qui est déjà là.
🔄 epoll sait avant toi. Quand un sk_buff arrive dans la receive queue, le kernel réveille epoll_wait() avec EPOLLIN. C'est pour ça qu'on attend epoll.
📦 Plusieurs sk_buff pour une requête. Une requête HTTP peut arriver en plusieurs paquets TCP. Chaque paquet = un sk_buff. recv() peut les consommer d'un coup ou partiellement.
04
Les flags de recv()
Modifier le comportement de la lecture
Le 4e paramètre de recv()

recv(fd, buf, len, flags) — le dernier paramètre est un bitmask de flags qui modifient le comportement de la lecture. En général on passe 0, mais certains flags sont utiles à connaître.

FlagEffet dans le kernelUsage
0 Comportement par défaut — copie et consomme les données 99% des cas. Ce qu'on utilise dans webserv
MSG_PEEK Copie les données sans les retirer de la receive queue. Les sk_buff restent en place Inspecter les données avant de décider quoi en faire. Utile pour sniffer les premiers octets
MSG_WAITALL Bloque jusqu'à ce que exactement len octets soient disponibles (ou erreur/déconnexion) Dangereux en non-bloquant. Ne pas utiliser dans webserv
MSG_DONTWAIT Force le mode non-bloquant pour cet appel uniquement, même si le socket est bloquant Alternative ponctuelle à O_NONBLOCK
MSG_NOSIGNAL Empêche l'envoi de SIGPIPE si le pair a fermé la connexion Pertinent pour send(), pas recv()
recv_flags.cpp
Exemples 
// Usage standard — flags = 0
ssize_t n = recv(fd, buf, sizeof(buf), 0);

// MSG_PEEK — regarder sans consommer
// Les données restent dans le buffer kernel
ssize_t peeked = recv(fd, buf, 4, MSG_PEEK);
// buf contient les 4 premiers octets, mais ils sont encore dans sk_receive_queue
// Un prochain recv() les retournera à nouveau

// MSG_DONTWAIT — non-bloquant ponctuel
ssize_t n = recv(fd, buf, sizeof(buf), MSG_DONTWAIT);
// Retourne EAGAIN immédiatement si rien dispo, même si le socket est bloquant
💡
Pour webserv : on utilise recv(fd, buf, sizeof(buf), 0). Le socket est déjà en mode O_NONBLOCK via fcntl(), donc pas besoin de MSG_DONTWAIT. Et on ne fait jamais MSG_WAITALL car on est event-driven — on lit ce qui est disponible et on attend le prochain EPOLLIN.
05
Mode non-bloquant
Ce qui change dans le kernel
recv() bloquant (par défaut)

Si la sk_receive_queue est vide, le kernel met ton processus en état TASK_INTERRUPTIBLE (sleeping) sur la wait queue du socket. Le processus ne consomme aucun CPU. Il dort.

Quand un paquet arrive et que tcp_v4_rcv() ajoute un sk_buff à la queue, le kernel appelle wake_up_interruptible() pour réveiller ton processus. recv() reprend et copie les données.

recv() non-bloquant (O_NONBLOCK)

Si la sk_receive_queue est vide, le kernel ne met pas ton processus en sommeil. tcp_recvmsg() vérifie le flag O_NONBLOCK, voit que la queue est vide, et retourne immédiatement -1 avec errno = EAGAIN.

C'est pour ça qu'on a besoin d'epoll : en non-bloquant, recv() ne te dit pas QUAND les données arriveront. C'est epoll_wait() qui dort à ta place et te réveille avec EPOLLIN quand la receive queue n'est plus vide.

tcp_recvmsg_simplified.c
Logique kernel simplifiée 
// Pseudo-code simplifié de tcp_recvmsg() dans le kernel

int tcp_recvmsg(struct sock *sk, struct msghdr *msg, size_t len, int flags) {

    lock_sock(sk);

    while (skb_queue_empty(&sk->sk_receive_queue)) {
        // La queue est vide — pas de données

        if (sk->sk_socket->file->f_flags & O_NONBLOCK) {
            return -EAGAIN;  // Non-bloquant → retourne immédiatement
        }

        // Bloquant → dormir jusqu'à ce qu'un paquet arrive
        sk_wait_data(sk, &timeo);
    }

    // Des données sont disponibles — les copier vers l'userspace
    int copied = 0;
    while (copied < len && !skb_queue_empty(&sk->sk_receive_queue)) {
        struct sk_buff *skb = skb_peek(&sk->sk_receive_queue);

        copy_to_user(msg->msg_iov, skb->data, chunk);  // kernel → userspace
        copied += chunk;

        if (!(flags & MSG_PEEK))
            skb_unlink(skb);  // retirer le sk_buff de la queue
    }

    // Mettre à jour la fenêtre TCP
    tcp_cleanup_rbuf(sk, copied);  // → envoie un ACK avec la nouvelle fenêtre

    release_sock(sk);
    return copied;
}
🔑 copy_to_user() est la vraie copie. C'est le moment où les données passent du kernel à ton buffer. C'est aussi le goulot d'étranglement potentiel pour les gros transferts.
📡 tcp_cleanup_rbuf() met à jour la fenêtre TCP. En vidant le buffer, tu dis au client "j'ai de la place, continue d'envoyer". Le flow control est automatique.
🔒 lock_sock() verrouille le socket. Deux threads ne peuvent pas appeler recv() sur le même socket en même temps. Dans webserv, un seul thread = pas de problème.
Résumé — recv() internals
Le trajet des données du réseau jusqu'à ton buffer.
📡
NIC → sk_buff
La carte réseau reçoit le paquet via DMA. Le driver crée un sk_buff et le passe à la stack TCP.
📦
sk_receive_queue
Les sk_buff s'accumulent dans la receive queue du socket. Les données attendent que recv() les consomme.
📋
copy_to_user()
recv() copie les données du kernel vers ton buffer. Les sk_buff sont libérés. La fenêtre TCP est mise à jour.
🔔
epoll + EPOLLIN
Quand un sk_buff arrive dans la queue, epoll_wait() est réveillé avec EPOLLIN. C'est ton signal pour appeler recv().
