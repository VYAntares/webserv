Multi-port — Config

Écouter sur plusieurs ports

Un port = une porte d'entrée. Pourquoi il faut autant de sockets que de ports, comment la config pilote la création des sockets, et comment epoll_data permet de lier chaque client à son bloc serveur.

01
Port = quai d'entrée
Un port, un socket, une porte
Ce qu'est un port

Un port est un numéro de quai que le kernel utilise pour router les paquets TCP vers le bon destinataire. Quand un paquet arrive sur la machine, le kernel lit le port de destination et le redirige vers le processus qui écoute sur ce numéro.

Il n'existe aucun raccourci kernel pour "un socket sur plusieurs ports". Écouter sur deux ports = créer deux sockets distincts, deux bind(), deux listen().

🚉
Analogie

Une gare avec plusieurs quais. Chaque quai (port) reçoit un train différent. Un contrôleur sur le quai 8080 ne peut pas aussi surveiller le quai 8081 — il faut un contrôleur distinct par quai.

« Your server must be able to listen to multiple ports to deliver different content. »

— webserv.pdf, IV.1 Requirements

02
config → init
La config détermine le nombre de sockets
Pourquoi des ports différents ?

Dans un fichier de configuration nginx-style, chaque bloc server déclare son propre listen PORT et son propre root. Ce sont des services logiquement séparés.

Le root est le répertoire de fichiers servi, pas une IP. Un seul processus, une seule machine — mais deux sites distincts accessibles sur deux ports différents.

Pourquoi aussi virtual hosting ?

En production (nginx réel), on sépare souvent par nom de domaine via le header Host: sur le même port 80.

Pour Webserv, la séparation par port est beaucoup plus simple à implémenter — un port différent = un socket différent = identification triviale.

main.cpp — notre code réel
Init 
// Pour chaque bloc server, pour chaque directive listen :
// un ServerHandler = un socket d'écoute qui s'auto-enregistre dans l'EventLoop.
for (size_t i = 0; i < c.cfg.size(); i++) {
    for (size_t j = 0; j < c.cfg[i].listen.size(); j++) {
        try {
            new ServerHandler(c.cfg[i].listen[j], c.cfg[i]);   // ← (addr, port) + config du bloc
        } catch (std::exception &e) {
            std::cerr << e.what() << std::endl;   // un port occupé ≠ tout s'arrête
        }
    }
}

// Et dans le constructeur de ServerHandler :
//   socket() + SO_REUSEADDR + O_NONBLOCK + FD_CLOEXEC
//   bind(addr résolue par la config) + listen(SOMAXCONN)
//   EventLoop::instance()->register_handler(this, ACCEPT_EVENT);  ← même epoll pour tous
💡 La config pilote tout. Notre default.conf déclare deux serveurs (8080 pour le site principal + tester, 8079 pour site2) → deux ServerHandler dans la même _table de l'EventLoop. Et comme listen accepte plusieurs valeurs (listen 8080 8081;), un même bloc server peut ouvrir plusieurs ports — d'où la double boucle.
03
epoll_wait()
Identifier quel server_fd a déclenché
La question clé

Quand epoll_wait() se réveille et retourne un événement, ce fd peut être n'importe lequel des server_fds ou des client_fds enregistrés. Il faut savoir ce qu'on reçoit pour savoir quoi faire.

Si c'est un server_fd → une nouvelle connexion est en attente → appeler accept().
Si c'est un client_fd → des données à lire → appeler recv().

srcs/core/EventLoop.cpp — notre dispatch
Boucle principale 
while (!g_stop) {
    int n = epoll_wait(_epfd, events, MAX_EVENTS, 500);

    for (int i = 0; i < n; i++) {
        HandlerEntry* entry = static_cast<HandlerEntry*>(events[i].data.ptr);
        IEventHandler* h    = entry->handler;

        if (entry->type == ACCEPT_EVENT)   // ← pas de is_server_fd() : le TYPE est dans l'entrée
            h->handle_accept();          // ServerHandler du port 8080 OU 8079, peu importe
        else {
            // Client ou pipe CGI : le polymorphisme appelle le bon code
            if (events[i].events & EPOLLIN)  h->handle_input();
            if (events[i].events & EPOLLOUT) h->handle_output();
        }
    }
    checkTimeOut();
}

// Pas de recherche linéaire "à qui est ce fd ?" : le HandlerEntry stocké
// dans ev.data.ptr transporte à la fois QUI (handler) et QUOI (type).
Événement sur un server_fd

Le kernel a complété un handshake TCP et mis une connexion dans la accept queue. Il faut appeler accept() pour extraire le client et obtenir son fd. Le nouveau client_fd doit ensuite être enregistré dans epoll.

Événement sur un client_fd

Des données sont disponibles dans le buffer de réception du kernel. recv() peut être appelé sans bloquer. Après lecture, on construit la réponse HTTP et on envoie avec send().

04
epoll_data_t
Associer chaque client à son bloc serveur
Le problème

Quand on accepte un client sur server_fd_a (port 8080), on sait qu'il appartient au bloc serveur A. Mais après, son client_fd est enregistré dans epoll. Comment se souvenir de son bloc serveur d'origine quand epoll_wait() remonte cet événement plus tard ?

struct epoll_event — stocker un pointeur dans l'événement
struct epoll_event
events EPOLLIN
data.fd int (le fd)
data.ptr void* (pointeur libre)
data.u32 uint32_t (index)
data.u64 uint64_t
data.ptr — c'est le champ le plus puissant. On peut y stocker un pointeur vers une structure qui contient à la fois le fd et la config du bloc serveur associé.
data.u32 — alternative simple : stocker l'index du bloc serveur dans le tableau de config. Plus léger qu'un pointeur.
data.fd — utilisation la plus courante mais la moins riche : on ne stocke que le fd, sans métadonnées supplémentaires.
epoll_data_usage.cpp
data.ptr 
// Chez nous, la "structure qui porte les infos" existe à deux niveaux :

// 1. Le HandlerEntry, stocké dans ev.data.ptr (EventLoop.hpp) :
struct HandlerEntry {
    IEventHandler* handler;   // QUI est prêt (ServerHandler, ClientHandler, CGI*...)
    EventType      type;      // QUOI faire (ACCEPT_EVENT, READ_EVENT, WRITE_EVENT)
};

// 2. La config du bloc serveur, portée par les handlers eux-mêmes :
class ServerHandler { Server _server; /* ... */ };
class ClientHandler { Server _server; /* ... */ };

// L'héritage du contexte se fait à l'accept :
int ServerHandler::handle_accept() {
    // ...
    new ClientHandler(client_fd, _server, client_addr);   // ← le client reçoit LA config
    //                              ↑ du port par lequel il est entré (8080 ou 8079)
}

// Plus tard, quand le client route sa requête :
//   Router::route(req, _server, ...) → les locations du BON bloc serveur.
💡 Héritage du contexte serveur. Quand handle_accept() crée le ClientHandler, il lui passe son _server — le client "hérite" du bloc serveur de la porte par laquelle il est entré. C'est comme ça qu'un client du port 8079 est servi par www/default/site2 et un client du 8080 par www : deux contenus différents, exigence du sujet.

Squelette multi-port complet
Notre assemblage réel — main.cpp + ServerHandler + EventLoop
Notre code 
// ── main.cpp : un ServerHandler par listen de la config ──────────────
ConfigLoader cl(argc, argv);
Config c = cl.getConfig();
for (size_t i = 0; i < c.cfg.size(); i++)
    for (size_t j = 0; j < c.cfg[i].listen.size(); j++)
        new ServerHandler(c.cfg[i].listen[j], c.cfg[i]);   // try/catch par port dans le vrai code

EventLoop::instance()->handle_events();

// ── ServerHandler : chaque instance fait socket+bind+listen puis ─────
//    s'enregistre en ACCEPT_EVENT dans le MÊME epoll (singleton EventLoop).
//    À l'accept, le ClientHandler hérite du _server (config du bloc) :
new ClientHandler(client_fd, _server, client_addr);

// ── EventLoop : dispatch par HandlerEntry, aucune connaissance des ports ──
HandlerEntry* entry = static_cast<HandlerEntry*>(events[i].data.ptr);
if (entry->type == ACCEPT_EVENT)
    entry->handler->handle_accept();

// ── Nettoyage : return -1 → remove_handler (DEL + delete entry) puis ──
//    delete handler (close du fd dans le destructeur). Aucun leak :
//    ~EventLoop() delete tout ce qui reste au shutdown.
⚠️ Fuite mémoire à éviter. Chaque new HandlerEntry (dans register_handler) a son delete dans remove_handler, et chaque handler est détruit par l'EventLoop sur return -1 — jamais ailleurs. C'est cette convention unique qui évite double-free et fd fantômes, vérifiée avec valgrind --track-fds=yes.
En résumé
Les quatre mécanismes du multi-port dans Webserv
🚪
N ports = N sockets d'écoute
Chaque bloc server dans la config crée un socket distinct avec son propre bind() + listen(). Pas de raccourci kernel possible.
📡
Un seul epoll pour tous
Tous les server_fds et tous les client_fds sont enregistrés dans le même epoll. Un seul epoll_wait() gère toutes les I/O.
🔍
is_listener → accept() ou recv()
À chaque événement, on détermine si c'est un server_fd (nouvelle connexion) ou un client_fd (données à lire) pour appeler le bon syscall.
🧬
epoll_data.ptr = contexte hérité
Le client hérite du pointeur vers le bloc serveur du listener par lequel il est entré. La config est toujours accessible sans recherche.
server_fd
Socket d'écoute lié à un port précis. Créé une fois au démarrage par bloc server dans la config. Jamais utilisé pour recv() — uniquement pour accept().
client_fd
Socket retourné par accept(). Représente une connexion TCP établie avec un client. C'est sur lui qu'on appelle recv() et send().
epoll_data.ptr
Champ union dans struct epoll_event. Permet de stocker un pointeur arbitraire attaché à un fd enregistré dans epoll. Retourné intact par epoll_wait().
virtual hosting
Servir plusieurs sites depuis une même machine. Par port (approche Webserv) ou par header Host: (approche production). Un seul processus, N services distincts.
héritage de contexte
Mécanisme par lequel un client_fd "hérite" du pointeur vers le bloc serveur du server_fd qui l'a accepté. Évite toute recherche en O(n) dans la boucle principale.
