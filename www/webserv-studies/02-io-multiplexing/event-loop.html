Concept critique — I/O sans errno

La boucle événementielle
sans jamais checker errno

Le sujet interdit de vérifier errno après read/write. Pourtant, le pattern classique edge-triggered repose entièrement sur EAGAIN. Voici comment architecturer un serveur event-driven qui respecte cette contrainte — en laissant epoll orchestrer tout le flux.

01
La contrainte
Ce que le sujet interdit exactement

« Interdiction de vérifier errno pour ajuster le comportement du serveur après une opération read ou write. »

— webserv subject
Ce qui est interdit

Utiliser la valeur de errno pour décider quoi faire après un recv() ou send(). Le cas typique :

  • Tester errno == EAGAIN pour savoir si le buffer est vide/plein
  • Tester errno == EWOULDBLOCK pour re-arm un fd
  • Boucler « tant que EAGAIN » pour drainer un buffer

Tout pattern qui utilise errno comme signal de contrôle pour piloter la machine à états du serveur est interdit.

Ce qui est autorisé

Le retour de la fonction lui-même (n) reste utilisable :

  • n > 0 — des octets ont été lus/écrits
  • n == 0 — le client a fermé la connexion (recv)
  • n <= 0 — erreur réelle → fermer proprement

On décide uniquement sur la valeur de retour, jamais sur errno. C'est epoll qui nous dit quand un fd est prêt — pas errno.

02
Le piège classique
Pourquoi la plupart des tutos ne marchent pas pour nous
Le problème

Tous les tutos epoll montrent le même pattern edge-triggered : boucler sur recv/send jusqu'à EAGAIN. C'est le pattern « drain loop ». Il fonctionne parfaitement — mais il viole la règle du sujet.

Pattern classique ET
Interdit par le sujet 
// ❌ Le "drain loop" classique — INTERDIT
// On boucle jusqu'à EAGAIN pour vider le buffer kernel

void handle_read_INTERDIT(int fd, int epfd) {
    while (true) {
        char buf[4096];
        ssize_t n = recv(fd, buf, sizeof(buf), 0);
        if (n < 0) {
            if (errno == EAGAIN || errno == EWOULDBLOCK)  // ← INTERDIT
                break;   // "buffer vide, on sort"
            close(fd); return;
        }
        if (n == 0) { close(fd); return; }
        request += buf;
    }
}

void handle_write_INTERDIT(int fd, int epfd) {
    while (sent < total) {
        ssize_t n = send(fd, data + sent, total - sent, 0);
        if (n < 0) {
            if (errno == EAGAIN || errno == EWOULDBLOCK)  // ← INTERDIT
                break;   // "buffer plein, on attend"
            close(fd); return;
        }
        sent += n;
    }
}
⚠️
Pourquoi c'est tent&ant ? En Edge Triggered classique, epoll ne te notifie qu'une fois par changement d'état. Donc le réflexe est de boucler jusqu'à EAGAIN pour tout drainer. Mais le sujet dit non. On doit trouver une autre architecture.
03
L'architecture correcte
Laisser epoll orchestrer — un seul appel par événement
Le principe fondamental

Au lieu de boucler jusqu'à EAGAIN, on fait un seul recv() ou send() par événement epoll. On met à jour l'état de la connexion. Si ce n'est pas fini, on ne boucle pas — on laisse epoll nous re-notifier quand le fd est à nouveau prêt.

C'est epoll qui pilote le flux, pas nous. On ne décide jamais « est-ce que le buffer est vide ? » (ce serait checker errno). On dit simplement : « epoll, rappelle-moi quand il y a quelque chose ».

Avant — le drain loop
Qui décide ?errno — je boucle tant que pas EAGAIN
Combien d'appels ?Autant que nécessaire en une seule passe
Quand je sors ?Quand errno == EAGAIN
Après — epoll orchestre
Qui décide ?epoll_wait() — il me réveille quand c'est prêt
Combien d'appels ?Un seul recv()/send() par événement
Quand je sors ?Immédiatement après le recv()/send()
État par connexion — chez nous : ClientHandler

Pour que ça marche, chaque connexion doit garder son état en mémoire entre deux événements. Dans notre webserv, cet objet est le ClientHandler : il porte le parseur (qui accumule), la réponse en cours et l'offset d'envoi. Et l'événement epoll ne transporte pas le handler directement, mais un HandlerEntry qui garde aussi le type d'enregistrement.

Notre état par connexion 
// L'état qui survit entre deux événements epoll (extrait) :
class ClientHandler : public IEventHandler, public IResponseSink {
    int               _fd;
    size_t            _sent;        // combien d'octets déjà envoyés
    HttpParser        _parser;      // accumule la requête entre les recv()
    std::string       _response;    // la réponse HTTP à envoyer
    bool              _keepAlive;
    time_t            _lastActivity;
    // ...
};

// Ce que epoll nous rend dans ev.data.ptr :
struct HandlerEntry {
    IEventHandler*  handler;
    EventType       type;    // ACCEPT_EVENT, READ_EVENT, WRITE_EVENT...
};
srcs/core/EventLoop.cpp — le dispatch réel
Notre code 
while (!g_stop) {
    int n = epoll_wait(_epfd, events, MAX_EVENTS, 500);

    for (int i = 0; i < n; i++) {
        HandlerEntry* entry = static_cast<HandlerEntry*>(events[i].data.ptr);
        IEventHandler* h    = entry->handler;   // ← on retrouve le handler
        int ret = 0;

        try {
            if (entry->type == ACCEPT_EVENT)
                ret = h->handle_accept();
            else {
                if (events[i].events & (EPOLLIN | EPOLLHUP | EPOLLERR))
                    ret = h->handle_input();    // ← UN seul recv()/read()
                if (ret != -1 && events[i].events & EPOLLOUT)
                    ret = h->handle_output();   // ← UN seul send()/write()
                if (ret != -1 && (events[i].events & (EPOLLHUP | EPOLLERR))
                    && !(events[i].events & (EPOLLIN | EPOLLOUT)))
                    ret = -1;                   // erreur "nue" → fermer (anti busy-loop)
            }
        } catch (std::exception& e) { ret = -1; } catch (...) { ret = -1; }

        if (ret == -1) {
            remove_handler(h);   // epoll_ctl(DEL) + table
            delete h;            // le destructeur ferme le fd
        }
    }
    checkTimeOut();              // chasse les inactifs > 30s
}
💡
ev.data.ptr au lieu de ev.data.fd. On stocke un pointeur vers le HandlerEntry directement dans l'événement epoll. Quand epoll_wait() nous réveille, on retrouve immédiatement le handler et son type d'enregistrement, sans chercher dans une map. Le polymorphisme (IEventHandler) fait le reste : le même handle_input() appelle ClientHandler::handle_input pour un socket ou CGIReadHandler::handle_input pour un pipe CGI — l'EventLoop ne connaît aucun type concret.
04
handle_read()
Lire sans jamais checker errno
Le principe

epoll_wait() nous a dit EPOLLIN sur ce fd → il y a des données. On fait un seul recv(). Trois cas possibles, tous décidés sur la valeur de retour n :

  • n > 0 — on a reçu des données → on les donne au HttpParser, qui accumule
  • n == 0 — le client a fermé la connexion (EOF) → return -1
  • n < 0 — erreur réelle → return -1

Si la requête n'est pas complète, on ne boucle pas. On sort avec return 0. epoll nous rappellera quand il y aura plus de données à lire.

srcs/handlers/ClientHandler.cpp — handle_input()
Notre code — zéro errno 
int ClientHandler::handle_input() {
    _lastActivity = time(NULL);   // pour le timeout 30s de checkTimeOut()

    char buf[4096];
    // UN SEUL recv() — epoll nous a dit que le fd est prêt
    ssize_t n = recv(_fd, buf, sizeof(buf), 0);
    if (n <= 0)
        return -1;   // déconnexion ou erreur → l'EventLoop nous détruit
    std::string data(buf, n);

    // Le parseur accumule et avance sa machine à états
    // (R_HEADERS → R_BODY / R_CHUNKED → COMPLETE / ERROR)
    _parser.runParsing(data);
    if (_parser.getState() == HttpParser::COMPLETE)
        _handleComplete();   // → Router → buildResponse → modify_handler(WRITE_EVENT)
    else if (_parser.getState() == HttpParser::ERROR)
        _handleError();

    return 0;   // requête pas finie : epoll re-notifiera EPOLLIN
}
💡
Pas de boucle, pas d'errno. Si le client envoie une grosse requête en plusieurs paquets TCP, on reçoit un EPOLLIN par paquet. Chaque appel fait un seul recv() et nourrit le HttpParser — c'est lui qui détecte le \r\n\r\n, le Content-Length atteint ou le dernier chunk, pas un simple find(). C'est epoll qui « boucle » pour nous en nous rappelant. Et le passage en WRITE_EVENT se fait via EventLoop::modify_handler(), qui encapsule le epoll_ctl(EPOLL_CTL_MOD).
05
handle_write()
Écrire sans jamais checker errno
Le principe

epoll_wait() nous a dit EPOLLOUT → le buffer kernel a de la place. On fait un seul send(). Il envoie ce qu'il peut (peut-être tout, peut-être une partie).

  • n > 0 — on met à jour _sent += n
  • Si _sent >= _response.size() — tout est parti → _reset() + retour en READ_EVENT (keep-alive) ou return -1 (close)
  • Si _sent < total — pas tout parti → ne rien faire. epoll nous rappellera avec EPOLLOUT
  • n <= 0 — erreur réelle → return -1
srcs/handlers/ClientHandler.cpp — handle_output()
Notre code — zéro errno 
int ClientHandler::handle_output() {
    _lastActivity = time(NULL);

    const char*  data = _response.c_str() + _sent;
    size_t       left = _response.size()  - _sent;

    // epoll nous a dit que le fd est prêt → on envoie ce qu'on peut
    // MSG_NOSIGNAL : pas de SIGPIPE si le client a coupé entre-temps
    ssize_t n = send(_fd, data, left, MSG_NOSIGNAL);
    if (n <= 0)
        return -1;   // erreur réelle → l'EventLoop nous détruit

    _sent += static_cast<size_t>(n);
    if (_sent >= _response.size()) {
        if (_keepAlive) {
            _reset();   // delete _rh, clear response, parser.reset()
            EventLoop::instance()->modify_handler(this, READ_EVENT);
            // Pipelining : une requête entière peut déjà être dans le buffer
            // du parseur — aucun octet n'arrivera pour re-déclencher
            // handle_input, il faut re-parser le reliquat maintenant.
            std::string empty;
            _parser.runParsing(empty);
            if (_parser.getState() == HttpParser::COMPLETE)
                _handleComplete();
            else if (_parser.getState() == HttpParser::ERROR)
                _handleError();
            return 0;
        }
        return -1;   // Connection: close → EventLoop détruira ce handler
    }

    // Si _sent < total : on ne fait RIEN. On ne boucle pas. Pas d'errno.
    // epoll nous rappellera avec EPOLLOUT quand le buffer kernel aura de la place.
    return 0;
}
⚠️
EPOLL_CTL_MOD, pas ADD ! Le client_fd est déjà dans epoll (ajouté lors de l'accept). On utilise MOD pour changer les événements surveillés (de EPOLLIN à EPOLLOUT). EPOLL_CTL_ADD sur un fd déjà enregistré retournerait EEXIST.
06
La mécanique derrière
Pourquoi ça fonctionne sans errno
Le buffer kernel — la clé de tout

Chaque socket a deux buffers dans le kernel : un pour la lecture (recv), un pour l'écriture (send). epoll surveille l'état de ces buffers et te prévient :

  • EPOLLIN — le buffer de lecture contient des données → recv() ne bloquera pas
  • EPOLLOUT — le buffer d'écriture a de la place → send() ne bloquera pas

Quand on fait un recv()/send() après une notification epoll, on a la garantie que l'appel va fonctionner — au moins partiellement. Pas besoin de errno pour vérifier.

Le cycle de vie d'une requête — en détail
1
Client se connecte → ACCEPT_EVENT sur le ServerHandler → handle_accept() fait accept() → new ClientHandler qui s'enregistre en READ_EVENT
↓
2
Client envoie des données → EPOLLIN sur son fd → un recv() dans handle_input() → le HttpParser accumule
↓ (répété si la requête arrive en plusieurs paquets)
3
Parser COMPLETE → _handleComplete() route et construit _response → modify_handler(this, WRITE_EVENT)
↓
4
Buffer d'écriture a de la place → EPOLLOUT → un send() dans handle_output() → met à jour _sent
↓ (répété si la réponse est grande)
5
_sent >= _response.size() → keep-alive : _reset() + retour READ_EVENT, sinon return -1 → l'EventLoop détruit le handler (le destructeur ferme le fd)
📦
Analogie — le facteur

Imagine que tu envoies un colis trop gros pour la boîte aux lettres. Au lieu de forcer (« boucler »), tu mets ce qui rentre, et tu laisses le facteur te prévenir quand la boîte est vidée. Le facteur, c'est epoll. Tu ne regardes jamais toi-même si la boîte est pleine (errno).

🔄
Analogie — LT re-notifications

En Level Triggered (pas de EPOLLET), epoll te rappelle tant que la condition est vraie. S'il reste des données à lire ou de la place pour écrire, tu seras re-notifié au prochain epoll_wait(). C'est ça qui remplace la boucle EAGAIN.

💡
Pourquoi Level Triggered est notre allié. En Level Triggered (le mode par défaut, sans EPOLLET), epoll re-notifie tant que des données sont disponibles ou que le buffer a de la place. C'est exactement ce qu'il nous faut : un seul recv()/send() par événement, et epoll fait le reste. En Edge Triggered, on serait obligé de drainer jusqu'à EAGAIN — ce qui est interdit. LT + un appel par événement = la solution parfaite.

Code final complet 
// Chez nous, le "serveur complet" n'est pas un fichier unique : c'est
// l'assemblage EventLoop + handlers. Voici le vrai main.cpp, en entier :

int main(int argc, char **argv) {
    try {
        ConfigLoader cl(argc, argv);            // lexer → parser → validator
        Config c = cl.getConfig();
        for (size_t i = 0; i < c.cfg.size(); i++) {
            for (size_t j = 0; j < c.cfg[i].listen.size(); j++) {
                try {
                    new ServerHandler(c.cfg[i].listen[j], c.cfg[i]);   // s'auto-enregistre en ACCEPT_EVENT
                } catch (std::exception &e) {
                    std::cerr << e.what() << std::endl;   // un port raté ≠ tout s'arrête
                }
            }
        }
        EventLoop::instance()->handle_events();   // la boucle Reactor — un seul epoll
        EventLoop::destroy();
    } catch (std::exception &e) {
        std::cerr << e.what() << std::endl;
        return 1;
    }
    return 0;
}

✅ Zéro errno. Aucun errno n'est testé. Toute la logique repose sur la valeur de retour de recv()/send() et les notifications epoll.
🔄 Un appel par événement. Un seul recv() par EPOLLIN, un seul send() par EPOLLOUT. C'est epoll qui « boucle » pour nous via ses re-notifications LT.
📋 ClientHandler = l'état qui survit. Le HttpParser accumule, _response attend, _sent progresse — entre deux événements epoll.
💤 Quasi 0% CPU au repos. epoll_wait(…, 500) dort par tranches de 500 ms — juste le temps de re-vérifier g_stop et les timeouts. Pas de busy-wait, pas de boucle EAGAIN.

Ni errno, ni boucle EAGAIN. epoll orchestre tout.
