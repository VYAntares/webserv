04 — Fichier de configuration

Glossaire des termes à maîtriser

Cette page explique précisément les directives utilisées dans ton exemple de configuration. Les définitions server, listen, server_name, location, root, index, client_max_body_size et error_page sont alignées avec la documentation officielle du module HTTP core de NGINX.

01
config sample
Le fichier qu’on utilise comme référence
webserv.conf
Extrait complet 
# Site principal sur le port 8080
server {
    listen 8080;
    server_name localhost;
    client_max_body_size 10m;

    error_page 404 /errors/404.html;
    error_page 500 /errors/500.html;

    location / {
        root            /var/www/html;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
    }

    location /api {
        root            /var/www/api;
        allowed_methods GET POST DELETE;
        autoindex       off;
    }

    location /upload {
        root            /var/www;
        allowed_methods POST;
        upload_store    /tmp/uploads;
    }

    location /files {
        root            /var/www/shared;
        allowed_methods GET;
        autoindex       on;
    }

    location /old-page {
        return 301 /new-page;
    }

    location /cgi-bin {
        root            /var/www/cgi-bin;
        allowed_methods GET POST;
        cgi_pass        .py /usr/bin/python3;
        cgi_pass        .php /usr/bin/php-cgi;
    }
}

# Deuxième serveur sur port 8081
server {
    listen 8081;
    client_max_body_size 1m;

    location / {
        root            /var/www/site2;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
    }
}
Comment lire ce fichier

Tu as 2 blocs server (deux serveurs virtuels), chacun écoute un port différent (8080 et 8081). Dans chaque serveur, des blocs location définissent les règles par préfixe d’URI.

Les directives dites “core NGINX” viennent du module HTTP core. D’autres directives ici (allowed_methods, upload_store, cgi_pass) sont des choix de conception côté webserv, inspirés du sujet, pas des directives standard du module core officiel.

02
directives core
Définitions exactes (manuel NGINX)
server { ... }

Définit un serveur virtuel HTTP. Le bloc rassemble les règles d’un “site” logique.

Contexte: http • Dans ton fichier: 2 serveurs (8080, 8081)
listen address:port;

Définit l’adresse/port sur lequel ce serveur accepte les connexions.

Contexte: server • Exemple: listen 8080; puis listen 8081;
server_name name ...;

Nomme le serveur virtuel pour le routage par en-tête Host quand plusieurs serveurs partagent un même port.

Contexte: server • Exemple: server_name localhost;
client_max_body_size size;

Taille maximale autorisée pour le corps de requête. Au-delà: réponse 413.

Contexte: http, server, location • Exemples: 10m et 1m
error_page code ... uri;

Associe un code d’erreur à une URI interne ou à une redirection dédiée.

Contexte: http, server, location • Exemples: 404 et 500
location [modificateur] uri { ... }

Applique une configuration en fonction de l’URI. Le matching se fait d’abord sur les préfixes, puis regex si présentes.

Contexte: server, location • Exemples: /, /api, /upload, /files, /old-page, /cgi-bin
root path;

Répertoire racine utilisé pour construire le chemin local du fichier à servir.

Contexte: http, server, location • Exemple: /var/www/html
index file ...;

Fichier par défaut d’un répertoire (module index), typiquement index.html.

Contexte: http, server, location • Exemple: index index.html;
autoindex on | off;

Active ou non le listing d’un dossier lorsqu’aucun index n’est trouvé (module autoindex).

Contexte: http, server, location • Exemple: /files = on
return code [text|URL|URI];

Renvoie immédiatement un statut HTTP (souvent pour redirection ou sortie rapide).

Contexte: server, location, if • Exemple: return 301 /new-page;

« Sets configuration depending on a request URI. » et « Sets the root directory for requests. »

— ngx_http_core_module: location, root
03
matching
Comment une requête choisit son bloc
Ordre concret sur ton config

1) Le port d’arrivée sélectionne d’abord le bloc server (8080 ou 8081).
2) Ensuite, dans ce serveur, la requête est comparée aux location par préfixe.
3) La route retenue applique ses directives: fichiers statiques, redirection, upload, CGI.

GET /files/report.txt → location /files POST /upload → location /upload GET /old-page → location /old-page
routing-trace.txt
Exemples rapides 
GET /api/users HTTP/1.1   -> match location /api
                           -> root /var/www/api

GET /files/ HTTP/1.1      -> match location /files
                           -> autoindex on
                           -> listing de dossier si pas d'index

GET /old-page HTTP/1.1    -> match location /old-page
                           -> return 301 /new-page

POST /upload HTTP/1.1     -> match location /upload
                           -> limite body (server)
                           -> stockage upload /tmp/uploads
04
effets
Ce que chaque terme change réellement
error_page 404 /errors/404.html;

Quand la ressource est introuvable, ton serveur doit produire cette page d’erreur dédiée au lieu d’un message brut.

client_max_body_size 10m;

Sur le serveur 8080, un upload de 15 Mo doit être rejeté en 413 avant traitement applicatif.

root + URI

Exemple: location /api + root /var/www/api + URI /api/users.json donne un chemin local résolu selon ta logique de mapping.

index index.html;

Sur un répertoire, le serveur tente d’abord index.html avant d’envisager un listing ou un 404.

autoindex on;

Pour /files, le serveur renvoie la liste des entrées du dossier si aucun index n’est présent.

return 301 /new-page;

Réponse immédiate de redirection permanente. Aucune résolution de fichier derrière cette route.

05
webserv-only
Termes custom de ton projet
Directives non core NGINX

allowed_methods, upload_store et cgi_pass ne sont pas des directives du module core officiel NGINX. Elles sont utilisées ici pour répondre explicitement au sujet webserv.

En pratique: allowed_methods pilote 405, upload_store choisit la destination disque, cgi_pass associe une extension à un interpréteur.

allowed_methods GET POST DELETE upload_store /tmp/uploads cgi_pass .py /usr/bin/python3
Point important: si tu veux un mode “strictement NGINX”, l’équivalent de limitation de méthodes est classiquement géré avec limit_except. Mais pour webserv, ton format custom est totalement cohérent si le parser et l’exécution respectent bien les règles du sujet.
06
ready-to-implement
Checklist de validation du vocabulaire
Ce que ton implémentation doit prouver

- Chaque terme du fichier a une structure C++ claire (server, location, erreurs, limites body, redirections).
- Le matching server puis location est déterministe.
- Les effets runtime sont visibles: 301, 404 custom, 413, autoindex, CGI, upload.
- Les directives custom webserv sont validées tôt, avec messages d’erreur explicites si syntaxe invalide.

Référence principale utilisée: ngx_http_core_module (server, listen, server_name, location, root, error_page, client_max_body_size).

— https://nginx.org/en/docs/http/ngx_http_core_module.html
