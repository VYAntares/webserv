04 — Fichier de configuration

Glossaire des termes à maîtriser

Chaque directive expliquée avec ce qu'elle fait réellement à runtime — pas juste sa définition. Les directives core NGINX (server, listen, location, root…) sont alignées sur le module HTTP core officiel. Les directives custom webserv (allowed_methods, upload_store, cgi_pass) sont expliquées avec leur effet attendu dans ton implémentation.

01
config sample
Le fichier de référence qu'on va décortiquer
webserv.conf
Fichier de référence complet 
# Serveur principal — port 8080
server {
    listen               8080;
    server_name          localhost;
    client_max_body_size 10m;

    error_page 404 /errors/404.html;
    error_page 500 /errors/500.html;

    location / {
        root            /var/www/html;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
    }

    location /api {
        root            /var/www/api;
        allowed_methods GET POST DELETE;
        autoindex       off;
    }

    location /upload {
        root            /var/www;
        allowed_methods POST;
        upload_store    /tmp/uploads;
    }

    location /files {
        root            /var/www/shared;
        allowed_methods GET;
        autoindex       on;
    }

    location /old-page {
        return 301 /new-page;
    }

    location /cgi-bin {
        root            /var/www/cgi-bin;
        allowed_methods GET POST;
        cgi_pass        .py  /usr/bin/python3;
        cgi_pass        .php /usr/bin/php-cgi;
    }
}

# Deuxième serveur — port 8081
server {
    listen               8081;
    client_max_body_size 1m;

    location / {
        root            /var/www/site2;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
    }
}
Structure globale

Deux blocs server = deux serveurs virtuels distincts. Chacun écoute un port différent (8080 et 8081). Dans chaque serveur, les blocs location définissent les règles par préfixe d'URI.

Webserv lit ce fichier une seule fois au démarrage, construit ses structures en mémoire, puis les applique à chaque requête sans jamais relire le fichier.

Core NGINX vs directives custom

server, listen, server_name, location, root, index, autoindex, error_page, client_max_body_size, return → directives du module HTTP core NGINX.

allowed_methods, upload_store, cgi_pass → directives custom webserv, inspirées du sujet mais non standard NGINX.

02
directives core
Ce que chaque mot-clé fait exactement
server { ... }

Définit un serveur virtuel HTTP. Le bloc rassemble toutes les règles d'un "site" logique : ports d'écoute, pages d'erreur, limite de body, routes.

Contexte: http • Ce fichier: 2 serveurs (8080, 8081) dans un même processus
listen address:port;

Définit le port (et optionnellement l'adresse IP) sur lequel ce serveur accepte les connexions. Chaque listen distinct = un socket serveur ouvert par webserv.

Contexte: server • Exemple: listen 8080; — listen 8081;
server_name name ...;

Nomme le serveur virtuel. Utile pour le virtual hosting : si deux server partagent le même port, le header Host de la requête est comparé aux server_name pour choisir le bon bloc. Optionnel dans webserv.

Contexte: server • Exemple: server_name localhost;
client_max_body_size size;

Taille maximale autorisée pour le corps d'une requête. Le check se fait tôt, sur le header Content-Length, avant de lire le body. Si dépassé : réponse 413 Request Entity Too Large immédiate.

Contexte: http, server, location • Exemples: 10m (serveur 8080), 1m (serveur 8081)
error_page code ... uri;

Associe un code d'erreur HTTP à une page personnalisée. Si le serveur produit ce code, il tente de servir la page. Si la page elle-même est introuvable, il faut un fallback HTML inline — jamais de réponse vide.

Contexte: http, server, location • Exemples: 404 /errors/404.html — 500 /errors/500.html
location [prefix] uri { ... }

Applique une configuration en fonction de l'URI de la requête. Le matching se fait par plus long préfixe : /api/users matche /api plutôt que /. Pas de regex dans webserv.

Contexte: server, location • Exemples: /, /api, /upload, /files, /old-page, /cgi-bin
root path;

Répertoire racine pour résoudre le chemin physique d'un fichier. La règle du sujet (Cas 2) : chemin = root + (URI − préfixe location). Exemple : location /api + root /var/www/api + URI /api/users.json → fichier à /var/www/api/users.json.

Contexte: http, server, location • Exemple: root /var/www/html;
index file ...;

Fichier servi automatiquement quand l'URI pointe sur un répertoire. Le serveur vérifie l'index avant d'envisager autoindex. Si l'index n'existe pas et autoindex off : 403 Forbidden.

Contexte: http, server, location • Exemple: index index.html;
autoindex on | off;

Active le listing du contenu d'un répertoire quand aucun fichier index n'est trouvé. Génère une réponse HTML avec les entrées du dossier. Implémenté avec opendir / readdir.

Contexte: http, server, location • Exemple: /files = on, toutes les autres locations = off
return code [URL|URI];

Renvoie immédiatement un statut HTTP sans résoudre de fichier. Court-circuite tout le reste du pipeline (root, index, CGI, upload). Utilisé pour les redirections permanentes (301) ou temporaires (302).

Contexte: server, location • Exemple: return 301 /new-page;
📚
Analogie — le répertoire téléphonique

Le fichier de config, c'est un répertoire téléphonique structuré. Chaque bloc server est une entreprise (son numéro = le port). Les blocs location sont les services internes de cette entreprise ("/accueil", "/comptabilité", "/archives").

Quand un appel arrive, on regarde d'abord au bon numéro de téléphone (port), puis on lit l'annuaire interne pour trouver le service le plus précis. return = "le standard te redirige directement, tu n'attends pas".

🔑client_max_body_size : le check se fait sur Content-Length avant de lire le body. Ne pas attendre la fin de la lecture pour rejeter.
📐Résolution de chemin : root + (URI − préfixe_location). C'est le Cas 2 du sujet — toujours retirer le préfixe de la location avant de concaténer avec root.
⚡return court-circuite tout. Aucune résolution de fichier, aucun CGI, aucun upload. La réponse est envoyée dès que la directive est trouvée.

« Sets configuration depending on a request URI. » et « Sets the root directory for requests. »

— ngx_http_core_module: location, root
03
matching
Comment une requête choisit son bloc — et quel fichier elle lit
📬
Analogie — le centre de tri postal

Imagine un centre de tri qui reçoit des colis. Le code postal (port) détermine d'abord à quelle agence le colis est affecté (bloc server).

Dans l'agence, le début de l'adresse (préfixe d'URI) détermine le service : un colis pour "/api/..." va au guichet "/api", pas au guichet généraliste "/". On choisit toujours le guichet dont l'adresse est la plus précise (plus long préfixe).

Ensuite, l'adresse physique finale est calculée : root + le reste de l'adresse sans le préfixe → c'est là où le fichier est lu sur le disque.

Les 3 étapes du routage

1. Sélection du serveur : le port d'arrivée de la connexion détermine quel bloc server répond. Si plusieurs serveurs partagent le même port, on compare le header Host avec server_name.

2. Sélection de la location : on compare l'URI à tous les préfixes déclarés. La location dont le préfixe est le plus long et qui matche l'URI est retenue.

3. Résolution du chemin : root + (URI − préfixe_location). Le préfixe est retiré de l'URI avant de concaténer avec root.

Exemples de résolution complète

GET /api/users.json
→ location choisie : /api (plus long que /)
→ root /var/www/api + /users.json
→ fichier : /var/www/api/users.json

GET /files/doc.pdf
→ location choisie : /files
→ root /var/www/shared + /doc.pdf
→ fichier : /var/www/shared/doc.pdf — puis autoindex on si dossier

GET /old-page
→ location choisie : /old-page
→ return 301 /new-page → réponse immédiate, pas de fichier

routing-trace.txt
Trace complète de chaque requête 
GET /api/users HTTP/1.1
  → port 8080 → server A
  → plus long préfixe : location /api
  → root /var/www/api + /users → /var/www/api/users

GET /files/ HTTP/1.1
  → port 8080 → server A
  → plus long préfixe : location /files
  → root /var/www/shared + / → tente index → autoindex on → listing

GET /old-page HTTP/1.1
  → port 8080 → server A
  → plus long préfixe : location /old-page
  → return 301 /new-page → réponse immédiate

POST /upload HTTP/1.1  (body: 15 Mo)
  → port 8080 → server A
  → Content-Length 15m > client_max_body_size 10m → 413 immédiat

POST /cgi-bin/script.py HTTP/1.1
  → port 8080 → server A
  → plus long préfixe : location /cgi-bin
  → extension .py → cgi_pass /usr/bin/python3 → fork + execve
📏Le matching est toujours par plus long préfixe. /api/v2/users matche /api/v2 si elle existe, sinon /api, jamais les deux.
➖Formule path = root + (URI − préfixe). Webserv retire le préfixe de la location avant de coller à root — c'est l'exemple exact du sujet (/kapouet/pouic/toto → /tmp/www/pouic/toto).
🚦Ordre de priorité dans la location : return en premier → puis path resolution → puis CGI si extension matche → sinon fichier statique.
04
effets runtime
Ce que chaque directive change dans la réponse HTTP
error_page 404 /errors/404.html

Quand le serveur ne trouve pas la ressource demandée, il tente de servir cette page d'erreur dédiée. Si elle-même est absente : réponse 404 avec un body HTML minimal inline.

client_max_body_size 10m

Sur le serveur 8080, tout body déclaré ≥ 10 Mo est rejeté en 413 dès la lecture du header Content-Length, sans toucher au body. Sur 8081 : limite de 1 Mo.

root + URI (path resolution)

location /api + root /var/www/api + URI /api/data.json
→ retire le préfixe /api
→ ajoute le suffixe /data.json à root
→ lit /var/www/api/data.json sur le disque.

index index.html

Requête sur GET / → URI est un répertoire → le serveur tente /var/www/html/index.html. Si absent et autoindex off : 403 Forbidden.

autoindex on (location /files)

Requête sur GET /files/ → si aucun index trouvé → le serveur génère une page HTML listant le contenu du dossier via opendir + readdir. Réponse 200 avec ce listing.

return 301 /new-page

Requête sur GET /old-page → réponse immédiate : 301 Moved Permanently avec header Location: /new-page. Zéro accès disque, aucun CGI, aucun traitement body.

🗂️Ordre de résolution d'un répertoire : 1) tenter l'index → 2) si autoindex on générer le listing → 3) sinon 403. Ne jamais envoyer un 404 pour un dossier qui existe.
🔁La page error_page doit être résolvable par tes propres règles root/location. Une page d'erreur introuvable → fallback body inline obligatoire (jamais de réponse vide).
🧮Toutes ces décisions — 413, 301, listing, index, 403 — sont prises à runtime sur chaque requête, à partir des structures C++ chargées une fois au démarrage.
05
webserv-only
Les directives propres à ton projet
Directives non standard NGINX

allowed_methods, upload_store et cgi_pass ne sont pas dans le module core officiel NGINX. Elles sont conçues pour répondre explicitement aux exigences du sujet webserv.

allowed_methods GET POST DELETE

Liste les méthodes HTTP autorisées pour cette location. Si la méthode de la requête n'est pas dans la liste : réponse 405 Method Not Allowed avec le header Allow: GET POST DELETE listant les méthodes acceptées.

La vérification se fait tôt dans le pipeline, avant la résolution de chemin ou le CGI.

upload_store /tmp/uploads

Dossier de destination pour les fichiers uploadés via multipart/form-data. Le serveur décode le body (boundary splitting), extrait le fichier, et écrit le contenu binaire dans ce dossier.

Le dossier doit exister et être writable. À vérifier à la validation de la config, pas à runtime. Nettoyer le nom de fichier : path traversal (..), caractères interdits, collisions.

cgi_pass .py /usr/bin/python3 + .php /usr/bin/php-cgi

Associe une extension de fichier à un interpréteur CGI. Quand l'URI correspond à un fichier avec cette extension, le serveur l'exécute via fork + execve.

Le processus enfant reçoit les variables CGI en environnement (REQUEST_METHOD, QUERY_STRING, CONTENT_TYPE, CONTENT_LENGTH, etc.), le body via stdin, et retourne sa réponse sur stdout. Le serveur fait chdir dans le répertoire du script avant execve pour que les chemins relatifs fonctionnent.

Règle du sujet : fork est réservé exclusivement aux CGI. Le process enfant doit être attendu avec waitpid. Si le CGI ne retourne pas de Content-Length, EOF sur stdout marque la fin.

⚠️
fork est une règle dure : utiliser fork en dehors du CGI = note 0 immédiate selon le sujet. La seule exception autorisée est l'exécution d'un script CGI via fork + execve.
🔱fork réservé au CGI uniquement. Le processus enfant hérite des pipes (stdin/stdout), des variables d'environnement CGI, et doit être waitpid-é.
🔒upload_store : valider l'existence et les droits du dossier au démarrage. Sanitizer le nom de fichier (retirer .., espaces, caractères interdits) avant l'écriture.
🚫allowed_methods → 405 + header Allow: obligatoire. La vérification se fait avant la résolution de chemin — ne pas chercher le fichier si la méthode est interdite.
06
checklist d'implémentation
Ce que ton code doit prouver pour chaque terme
🏗️Chaque server { } correspond à une structure ServerConfig en mémoire. Chaque location { } à une LocationConfig. Chargées au démarrage, immutables ensuite.
🔌Chaque port distinct dans les blocs listen = un socket serveur ouvert. Tous ces sockets sont enregistrés dans poll() / epoll dès le démarrage.
📏Le matching location est déterministe : itérer tous les préfixes, retenir le plus long qui matche l'URI. Implémenter un test avec plusieurs locations chevauchantes.
📁La résolution de chemin suit root + (URI − préfixe_location). Vérifier avec l'exemple du sujet : /kapouet/pouic/toto + root /tmp/www → /tmp/www/pouic/toto.
📊Les codes de statut doivent être précis : 200, 301, 403, 404, 405, 413, 500. Tester chaque cas avec curl ou le navigateur.
🧪Fournir un fichier de config de test qui couvre tous les cas : statique, redirection, 404 custom, 413, autoindex, upload, CGI. Le sujet demande explicitement ces fichiers pour la soutenance.
