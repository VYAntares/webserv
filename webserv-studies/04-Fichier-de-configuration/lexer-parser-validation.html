Le problème concret

Ton serveur lit un texte libre avec des accolades, points-virgules, commentaires, directives, etc.

Si tu veux parser ce texte directement “à la main” dans tous les sens, tu te retrouves vite avec du code fragile et des erreurs difficiles à diagnostiquer.

La séparation en 3 phases rend le système stable et lisible :

  • Lexer : découpe le texte en briques propres (tokens).
  • Parser : comprend la grammaire et remplit tes structures C++.
  • Validation : vérifie que la config est cohérente avant de lancer le serveur.
🏗️
Analogie simple

Lexer = tu coupes les pièces Lego. Parser = tu assembles selon le plan. Validation = tu vérifies que la maison tient debout avant d’y habiter.

💡
Point clé soutenance : ces 3 phases se font au startup (au lancement), pas à chaque requête HTTP. Les requêtes runtime utilisent simplement les structures déjà construites.
02
lexer()
C’est quoi un lexer, exactement ?
Définition claire

Le lexer (tokenizer) lit le fichier caractère par caractère et produit une suite de tokens.

Un token = une “brique atomique” utile au parser : mot, {, }, ;, fin de fichier.

Il ignore les espaces et les commentaires. Il peut aussi stocker la ligne pour afficher des erreurs précises.

Token TypeExempleRôle
WORDserver, listen, 8080directive, argument, chemin, valeur
OPEN_BRACE{ouverture de bloc
CLOSE_BRACE}fermeture de bloc
SEMICOLON;fin de directive
END_OF_FILEEOFfin de flux
lexer.cpp
tokenisation
enum TokenType {
  WORD, OPEN_BRACE, CLOSE_BRACE, SEMICOLON, END_OF_FILE
};

struct Token {
  TokenType type;
  std::string value;
  int line;
};

std::vector<Token> lex(const std::string& content) {
  std::vector<Token> tokens;
  size_t i = 0;
  int line = 1;

  while (i < content.size()) {
    char c = content[i];

    if (isspace(c)) { if (c == '\n') line++; i++; continue; }
    if (c == '#') { while(i < content.size() && content[i] != '\n') i++; continue; }

    if (c == '{') { tokens.push_back((Token){OPEN_BRACE, "{", line}); i++; continue; }
    if (c == '}') { tokens.push_back((Token){CLOSE_BRACE, "}", line}); i++; continue; }
    if (c == ';') { tokens.push_back((Token){SEMICOLON, ";", line}); i++; continue; }

    std::string word;
    while (i < content.size() && !isspace(content[i])
           && content[i] != '{' && content[i] != '}'
           && content[i] != ';' && content[i] != '#') {
      word += content[i++];
    }
    tokens.push_back((Token){WORD, word, line});
  }

  tokens.push_back((Token){END_OF_FILE, "", line});
  return tokens;
}
⚠️
Le lexer ne “comprend” pas la config. Il découpe seulement. Si un token listen apparaît dans un mauvais contexte, c’est le parser (puis la validation) qui doit le refuser.
03
parser()
Que fait le parser ?
Son rôle exact

Le parser lit la séquence de tokens et applique les règles de grammaire : ce qui est autorisé, où, et dans quel ordre.

Exemple de règle :

  • Un bloc server doit ouvrir avec { et fermer avec }.
  • Une directive simple se termine par ;.
  • location /path { ... } est un sous-bloc dans server.

Le parser remplit ensuite tes structures ServerConfig / LocationConfig.

🧠
Analogie

Le lexer lit des mots séparés. Le parser reconstruit des phrases qui ont du sens grammatical. Même mots, mais pas du tout la même responsabilité.

parser.cpp
recursive descent
class ConfigParser {
  std::vector<Token> _t;
  size_t _pos;

  Token& cur() { return _t[_pos]; }
  Token eat() { return _t[_pos++]; }

  void expect(TokenType tt) {
    if (cur().type != tt) throw std::runtime_error("syntax error line " + toString(cur().line));
    eat();
  }

  LocationConfig parseLocation() {
    LocationConfig loc;
    loc.path = eat().value; // ex: /api
    expect(OPEN_BRACE);

    while (cur().type != CLOSE_BRACE) {
      std::string d = eat().value;
      if (d == "root") { loc.root = eat().value; expect(SEMICOLON); }
      else if (d == "index") { loc.index = eat().value; expect(SEMICOLON); }
      else { throw std::runtime_error("unknown directive in location: " + d); }
    }

    expect(CLOSE_BRACE);
    return loc;
  }
};
🧩
Résumé parser : il transforme une suite de tokens en structures métier utilisables au runtime. Sans parser, tu as des tokens; avec parser, tu as des ServerConfig prêts à être exploités.
04
validate()
La validation, c’est quoi ?
La différence avec le parser

Parser = validité syntaxique (“est-ce que le fichier est bien écrit ?”).

Validation = validité fonctionnelle (“est-ce que ça a du sens pour lancer le serveur ?”).

Exemples d’erreurs de validation (pas forcément de syntaxe) :

  • Port hors plage 1..65535.
  • Méthode inconnue PUT si ton projet n’accepte que GET/POST/DELETE.
  • Chemin root inexistant.
  • Code de redirection invalide.
validation.cpp
startup checks
void validateConfig(const std::vector<ServerConfig>& servers) {
  for (size_t i = 0; i < servers.size(); ++i) {
    const ServerConfig& s = servers[i];

    if (s.port < 1 || s.port > 65535)
      throw std::runtime_error("invalid port");

    if (s.locations.empty())
      throw std::runtime_error("server has no location");

    for (size_t j = 0; j < s.locations.size(); ++j) {
      const LocationConfig& loc = s.locations[j];

      for (size_t k = 0; k < loc.allowedMethods.size(); ++k) {
        std::string m = loc.allowedMethods[k];
        if (m != "GET" && m != "POST" && m != "DELETE")
          throw std::runtime_error("invalid method: " + m);
      }
    }
  }
}
🚫
Ne lance jamais le serveur avec une config invalide. Mieux vaut échouer au startup avec un message clair que planter plus tard en plein traitement de requêtes.
05
pipelineComplet()
Du fichier .conf au runtime HTTP
main.cpp
startup orchestration
int main(int argc, char**argv) {
  std::string content = readFile(argv[1]);

  std::vector<Token> tokens = lex(content);
  std::vector<ServerConfig> servers = parse(tokens);
  validateConfig(servers);

  initListeningSockets(servers);
  runEventLoop(servers);
  return 0;
}
Lecture runtime

Une fois ce pipeline terminé, chaque requête HTTP fait seulement :

  • Choix du serveur par port (et host si implémenté).
  • Choix de la location (préfixe le plus long).
  • Application des règles déjà validées.

Donc le travail “complexe” est déplacé au startup pour garder le runtime propre et robuste.

✅
Réponse directe à ta question : lexer découpe, parser structure, validation sécurise. Les trois sont obligatoires pour un webserv stable.
