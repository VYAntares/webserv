Le problème concret

Ton serveur lit un texte libre avec des accolades, points-virgules, commentaires, directives, etc.

Si tu tentes de tout parser "à la main" en un seul bloc, tu te retrouves vite avec un code fragile, des erreurs difficiles à localiser, et une couverture incomplète des cas d'erreur.

La séparation en 3 phases rend le système stable et lisible :

  • Lexer : découpe le texte en briques atomiques (tokens).
  • Parser : comprend la grammaire et remplit tes structures C++.
  • Validation : vérifie la cohérence avant de lancer le serveur.
🏗️
Analogie Lego

Lexer = tu sors toutes les pièces Lego de la boîte et tu les tries par forme.

Parser = tu assembles selon le plan de montage, pièce par pièce.

Validation = tu vérifies que la maison tient debout avant d'y habiter.

💡
Point clé soutenance : ces 3 phases se font au startup uniquement, pas à chaque requête HTTP. Le runtime utilise simplement les structures déjà construites et validées. Echec au startup = message clair. Echec au runtime = catastrophe silencieuse.
① startup vs runtime : tout le travail de parsing est fait une seule fois au démarrage.
② séparation des responsabilités : chaque phase n'a qu'une seule mission — plus facile à tester et déboguer.
③ fail fast : une config invalide doit faire exit(1) avec un message explicite avant tout bind().
02
lexer()
C'est quoi un lexer, exactement ?
Définition claire

Le lexer (tokenizer) lit le fichier caractère par caractère et produit une suite de tokens.

Un token = une "brique atomique" utile au parser : mot, {, }, ;, fin de fichier.

Il ignore les espaces et les commentaires (#). Il peut aussi stocker le numéro de ligne pour afficher des erreurs précises ("syntax error line 12").

Le lexer ne comprend pas la grammaire. Il ne sait pas si listen est bien placé — c'est le parser qui décide.

✂️
Analogie scanner de caisse

Le lexer lit le texte comme un scanner lit des codes-barres : il identifie chaque article (token) un par un, sans savoir ce que tu vas cuisiner.

C'est le caissier (parser) qui vérifie que tu as bien un plat et ses ingrédients.

lexer.cpp
tokenisation
enum TokenType {
  WORD, OPEN_BRACE, CLOSE_BRACE, SEMICOLON, END_OF_FILE
};

struct Token {
  TokenType type;
  std::string value;
  int line;   // pour les messages d'erreur
};

std::vector<Token> lex(const std::string& content) {
  std::vector<Token> tokens;
  size_t i = 0;
  int line = 1;

  while (i < content.size()) {
    char c = content[i];

    if (isspace(c)) { if (c == '\n') line++; i++; continue; }  // skip whitespace
    if (c == '#')  { while(i < content.size() && content[i] != '\n') i++; continue; }  // skip comments

    if (c == '{') { tokens.push_back({OPEN_BRACE,  "{", line}); i++; continue; }
    if (c == '}') { tokens.push_back({CLOSE_BRACE, "}", line}); i++; continue; }
    if (c == ';') { tokens.push_back({SEMICOLON,   ";", line}); i++; continue; }

    // WORD : tout le reste jusqu'au prochain délimiteur
    std::string word;
    while (i < content.size()
           && !isspace(content[i])
           && content[i] != '{' && content[i] != '}'
           && content[i] != ';' && content[i] != '#')
      word += content[i++];
    tokens.push_back({WORD, word, line});
  }

  tokens.push_back({END_OF_FILE, "", line});
  return tokens;
}
⚠️
Le lexer ne "comprend" pas la config. Il découpe seulement. Si un token listen apparaît dans un bloc location, c'est le parser (puis la validation) qui doit le refuser — pas le lexer.
① 5 types de tokens suffisent pour le format nginx-like : WORD, OPEN_BRACE, CLOSE_BRACE, SEMICOLON, END_OF_FILE.
② stocker le numéro de ligne dans chaque token te permet d'afficher "erreur ligne 12" exacte lors du parsing.
③ ignorer # en sautant tout jusqu'au \n suivant — les commentaires doivent disparaître avant le parser.
03
parser()
Que fait le parser ?
Son rôle exact

Le parser lit la séquence de tokens et applique les règles de grammaire : ce qui est autorisé, où, et dans quel ordre.

Règles de base :

  • Un bloc server doit ouvrir avec { et fermer avec }.
  • Une directive simple se termine par ;.
  • location /path { ... } est un sous-bloc dans server.
  • Certaines directives acceptent plusieurs arguments (allowed_methods GET POST;).

Le parser remplit ensuite tes structures ServerConfig / LocationConfig.

🧠
Analogie grammaire

Le lexer reconnaît les mots. Le parser construit des phrases grammaticalement correctes.

"listen 8080 ;" → sujet + complément + point final = phrase valide.
"8080 listen ;" → même mots, grammaire incorrecte → rejeté.

parser.cpp
recursive descent
class ConfigParser {
  std::vector<Token> _t;
  size_t             _pos;

  Token& cur() { return _t[_pos]; }
  Token  eat() { return _t[_pos++]; }

  void expect(TokenType tt) {
    if (cur().type != tt) throw std::runtime_error("syntax error line " + toString(cur().line));
    eat();
  }

  LocationConfig parseLocation() {
    LocationConfig loc;
    loc.path = eat().value;   // ex: /api
    expect(OPEN_BRACE);

    while (cur().type != CLOSE_BRACE) {
      std::string d = eat().value;
      if      (d == "root")    { loc.root      = eat().value;  expect(SEMICOLON); }
      else if (d == "index")   { loc.index     = eat().value;  expect(SEMICOLON); }
      else if (d == "autoindex"){ loc.autoindex = (eat().value == "on"); expect(SEMICOLON); }
      else if (d == "allowed_methods") {   // multi-valeurs jusqu'au ;
        while (cur().type == WORD)
          loc.allowedMethods.push_back(eat().value);
        expect(SEMICOLON);
      }
      else if (d == "return")  { loc.retCode = atoi(eat().value.c_str()); loc.retUrl = eat().value; expect(SEMICOLON); }
      else if (d == "upload_store"){ loc.uploadStore = eat().value; expect(SEMICOLON); }
      else if (d == "cgi_pass")   { loc.cgiExt     = eat().value; loc.cgiPath = eat().value; expect(SEMICOLON); }
      else { throw std::runtime_error("unknown directive: " + d + " line " + toString(cur().line)); }
    }

    expect(CLOSE_BRACE);
    return loc;
  }
};
🧩
Résumé parser : il transforme une suite de tokens en structures métier utilisables au runtime. Sans parser, tu as des tokens. Avec parser, tu as des ServerConfig et LocationConfig prêts à être exploités et validés.
① recursive descent : parseServer() appelle parseLocation() — structure naturelle qui suit la hiérarchie de la config.
② allowed_methods est multi-valeurs : boucle sur les WORD jusqu'au SEMICOLON — ne pas oublier ce cas particulier.
③ erreur inconnue immédiate : directive non reconnue → throw avec le numéro de ligne. Ne jamais silencieusement ignorer un token inconnu.
04
validate()
La validation, c'est quoi ?
La différence avec le parser

Parser = validité syntaxique ("est-ce que le fichier est bien écrit ?").

Validation = validité fonctionnelle ("est-ce que ça a du sens pour lancer un serveur HTTP ?").

Exemples de checks de validation spécifiques à webserv :

  • Port hors plage 1..65535.
  • Méthode inconnue dans allowed_methods (seuls GET/POST/DELETE sont valides).
  • Code de redirection invalide (doit être 3xx).
  • Chemin upload_store inexistant ou non accessible en écriture.
  • Interpréteur cgi_pass inexistant ou non exécutable.
  • Deux blocs server avec le même listen port et le même server_name.
  • Aucun bloc location dans un server.
🔍
Analogie inspection de chantier

Le parser vérifie que les plans respectent le format standard.

La validation est l'inspecteur qui va sur place vérifier que les fondations existent, que les portes s'ouvrent, que les câbles sont branchés.

Une belle maison sur des plans incorrects → bloquée au parser.
Des plans corrects mais sans fondations → bloquée à la validation.

validation.cpp
startup checks
void validateConfig(const std::vector<ServerConfig>& servers) {
  for (size_t i = 0; i < servers.size(); ++i) {
    const ServerConfig& s = servers[i];

    if (s.port < 1 || s.port > 65535)
      throw std::runtime_error("invalid port: " + toString(s.port));

    if (s.locations.empty())
      throw std::runtime_error("server on port " + toString(s.port) + " has no location");

    for (size_t j = 0; j < s.locations.size(); ++j) {
      const LocationConfig& loc = s.locations[j];

      for (size_t k = 0; k < loc.allowedMethods.size(); ++k) {
        std::string m = loc.allowedMethods[k];
        if (m != "GET" && m != "POST" && m != "DELETE")
          throw std::runtime_error("invalid method: " + m);
      }

      if (!loc.uploadStore.empty() && !directoryExists(loc.uploadStore))
        throw std::runtime_error("upload_store does not exist: " + loc.uploadStore);

      if (!loc.cgiPath.empty() && access(loc.cgiPath.c_str(), X_OK) != 0)
        throw std::runtime_error("cgi_pass not executable: " + loc.cgiPath);

      if (!loc.retUrl.empty()) {
        int code = loc.retCode;
        if (code < 300 || code > 399)
          throw std::runtime_error("return code must be 3xx: " + toString(code));
      }
    }
  }
}
🚫
Ne lance jamais le serveur avec une config invalide. Mieux vaut exit(1) au startup avec un message clair que planter en plein traitement de requête. Un serveur qui démarre avec une config incohérente est un bug caché garanti.
① parser ≠ validation : syntaxe correcte ne signifie pas config fonctionnelle — les deux phases sont indispensables.
② access(path, X_OK) pour vérifier que l'interpréteur CGI est exécutable avant de lancer le serveur.
③ messages d'erreur précis : chaque throw doit indiquer exactement quelle directive est invalide et pourquoi — essentiel pour la soutenance.
05
pipelineComplet()
Du fichier .conf au runtime HTTP
main.cpp
startup orchestration
int main(int argc, char** argv) {
  if (argc != 2) {
    std::cerr << "usage: ./webserv <config>\n";
    return 1;
  }

  try {
    std::string content = readFile(argv[1]);

    std::vector<Token>       tokens  = lex(content);       // phase 1 : lexer
    std::vector<ServerConfig> servers = parse(tokens);      // phase 2 : parser
    validateConfig(servers);                               // phase 3 : validation

    initListeningSockets(servers);   // bind() / listen() sur chaque port
    runEventLoop(servers);           // epoll_wait() → dispatch requêtes
  }
  catch (const std::exception& e) {
    std::cerr << "[webserv] fatal: " << e.what() << "\n";
    return 1;
  }
  return 0;
}
Lecture runtime

Une fois ce pipeline terminé, chaque requête HTTP fait seulement :

  • Choix du ServerConfig par port (et Host: header si plusieurs server_name).
  • Choix de la LocationConfig (préfixe le plus long).
  • Application des règles déjà validées — aucun accès fichier de config en runtime.

Le travail "lourd" est déplacé au startup pour garder le runtime propre et robuste.

Que se passe-t-il en cas d'erreur ?
  • Fichier config introuvable → readFile() lance une exception → exit(1).
  • Syntaxe cassée (accolade manquante) → lexer/parser lèvent une exception → exit(1).
  • Config incohérente (port invalide, CGI manquant) → validation lève une exception → exit(1).

Dans tous les cas, le serveur ne démarre pas et affiche le message précis sur stderr.

✅
Réponse directe soutenance : lexer découpe, parser structure, validation sécurise. Les trois phases se font au startup et sont obligatoires pour un webserv stable. Le runtime n'accède jamais au fichier de config.
① try/catch global dans main() : toute exception du pipeline devient un exit(1) propre avec message stderr.
② l'ordre des 3 phases est fixe : lex → parse → validate. Impossible d'inverser — le parser a besoin des tokens, la validation a besoin des structs.
③ runtime = lecture seule des structs ServerConfig / LocationConfig. Aucun re-parsing pendant le traitement des requêtes.
