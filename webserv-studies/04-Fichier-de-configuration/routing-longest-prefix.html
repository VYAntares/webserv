04 — Configuration — Routing

Sélection de route : préfixe le plus long

Oui, c’est bien au moment où on reçoit la requête HTTP : on extrait l’URI depuis la première ligne, puis on choisit la location la plus spécifique qui matche ce préfixe.

01
dOuVientURI()
L’URI vient directement de la requête HTTP reçue
Réponse à ta question

Oui, c’est exactement quand le serveur reçoit la requête HTTP.

Le client (navigateur, curl, Postman…) envoie une première ligne: METHODE URI VERSION.

Exemple: GET /get/files/fd HTTP/1.1. L’URI ici est /get/files/fd.

requête brute reçue sur socket
http request line 
GET /get/files/fd HTTP/1.1      ← l’URI est /get/files/fd
Host: localhost:8080
User-Agent: curl/8.7.1

[body optionnel]
💡
Donc l’URI n’est pas inventée par le serveur. Elle est fournie par le client dans la requête. Ton parser HTTP la lit et la stocke (ex: req.uri).
02
prefixMatching()
Comment choisir entre plusieurs location qui matchent
config d’exemple
server 
location / {
  allowed_methods GET;
}

location /get/files {
  allowed_methods GET POST;
}

location /get/files/fd {
  allowed_methods GET;
}
URI reçuelocations qui matchentgagnante
/get/files/fd /, /get/files, /get/files/fd /get/files/fd
/get/files/report.txt /, /get/files /get/files
/unknown / /
Ce que ça veut dire “préfixe le plus long”

Tu compares l’URI avec chaque chemin de location qui commence pareil.

S’il y a plusieurs candidats, tu gardes celui qui a le chemin le plus long, car il est le plus précis.

Donc pour ta question : oui, si tu as /get/files et /get/files/fd, alors l’URI /get/files/fd choisit /get/files/fd.

⚠️
Piège important : il faut une frontière de segment. /get/files ne doit pas matcher /get/filesX. Sinon le routing devient faux.
03
findLocation()
Algorithme minimal, clair, et correct
routing.cpp
c++98 
const LocationConfig* findLocation(const ServerConfig& srv, const std::string& uri) {
  const LocationConfig* best = NULL;
  size_t bestLen = 0;

  for (size_t i = 0; i < srv.locations.size(); ++i) {
    const std::string& p = srv.locations[i].path;
    if (uri.compare(0, p.size(), p) != 0) continue;

    bool boundaryOk =
      (uri.size() == p.size()) ||
      (p[p.size() - 1] == '/') ||
      (uri[p.size()] == '/');

    if (!boundaryOk) continue;

    if (p.size() > bestLen) {
      best = &srv.locations[i];
      bestLen = p.size();
    }
  }
  return best;
}
Version ultra simple
  • Tu pars de req.uri (extraite de la requête HTTP).
  • Tu testes toutes les location.path du serveur choisi.
  • Tu gardes le match avec la longueur la plus grande.
  • Cette location devient la règle active pour la requête.
04
requestFlow()
Ordre réel d’exécution à retenir
Ce qu’il faut faire
  • 1) Choisir le server par port (et Host si implémenté).
  • 2) Choisir la location par préfixe le plus long.
  • 3) Vérifier allowed_methods (sinon 405).
  • 4) Si return existe, répondre immédiatement.
  • 5) Résoudre le chemin disque et servir CGI/statique.
Pourquoi cet ordre
  • Sans route gagnante, tu ne sais pas quelle politique appliquer.
  • Tu peux servir le mauvais fichier avec le mauvais root.
  • Tu peux accepter une méthode qui devrait être interdite.
  • Tu peux ignorer une redirection return qui devait court-circuiter.

En résumé: le routing n’est pas une option décorative. C’est la décision qui pilote tout le reste.

sanityChecks()
Exemples concrets à tester (débutant)
requêtes simples
curl 
# URI = /get/files/fd  → doit choisir location /get/files/fd
curl -i http://localhost:8080/get/files/fd

# URI = /get/files/report.txt → doit choisir /get/files
curl -i http://localhost:8080/get/files/report.txt

# URI = /unknown → fallback sur /
curl -i http://localhost:8080/unknown
Ce que tu dois observer

Le serveur logge (ou trace) la location sélectionnée à chaque requête.

Si la sélection n’est pas celle attendue, alors le bug est dans ton match de préfixe (souvent la frontière de segment).

Quand tu arrives à expliquer ces 3 requêtes à l’oral, ce point est maîtrisé.
