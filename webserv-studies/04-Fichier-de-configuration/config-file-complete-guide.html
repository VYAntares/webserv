04 — Fichier de configuration

Pourquoi un fichier de configuration ?

Cette page est volontairement longue et complète : on va du rôle du fichier de config jusqu'à l'implémentation webserv directive par directive, avec des exemples concrets de config, des snippets d'algorithme, et des checklists de validation. Objectif : zéro zone floue avant de coder.

01
vision globale
Ce que fait un fichier de config dans webserv
Le problème réel

Sans config, ton serveur est figé : un seul port, un seul dossier, une seule politique HTTP. Avec un fichier de configuration, tu externalises ces décisions : ports d'écoute, routes, méthodes autorisées, erreurs, CGI, upload, etc.

Et surtout : un seul binaire peut gérer plusieurs serveurs logiques. Chaque bloc server décrit un comportement distinct (port, routes, limites), mais tout tourne dans le même programme webserv.

Pipeline conceptuel
  • Lire le fichier brut.
  • Tokeniser (WORD, {, }, ;, commentaires).
  • Parser en structures ServerConfig / LocationConfig.
  • Valider toute la config au startup.
  • Appliquer la config à chaque requête entrante.
📖
Analogie — le manuel d'exploitation

Le fichier de config, c'est le manuel d'exploitation laissé dans la salle serveur. Le serveur le lit une seule fois au démarrage, charge toutes les règles en mémoire, puis les applique à chaque requête sans jamais re-lire le fichier.

Tu ne changes pas le comportement en modifiant le binaire — tu mets à jour le manuel. Et si le manuel est mal écrit (config invalide), le serveur refuse de démarrer plutôt que de fonctionner avec des règles incohérentes.

« You can take inspiration from the 'server' section of the NGINX configuration file. »

— webserv.pdf, IV.3
webserv.conf
Exemple de référence (multi-serveurs) 
# Site principal sur le port 8080
    server {
      listen 8080;
      server_name localhost;
      client_max_body_size 10m;

      error_page 404 /errors/404.html;
      error_page 500 /errors/500.html;

      location / {
        root            /var/www/html;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
      }

      location /api {
        root            /var/www/api;
        allowed_methods GET POST DELETE;
        autoindex       off;
      }

      location /upload {
        root            /var/www;
        allowed_methods POST;
        upload_store    /tmp/uploads;
      }

      location /files {
        root            /var/www/shared;
        allowed_methods GET;
        autoindex       on;
      }

      location /old-page {
        return 301 /new-page;
      }

      location /cgi-bin {
        root            /var/www/cgi-bin;
        allowed_methods GET POST;
        cgi_pass        .py /usr/bin/python3;
        cgi_pass        .php /usr/bin/php-cgi;
      }
    }

    # Deuxième serveur sur port 8081
    server {
      listen 8081;
      client_max_body_size 1m;

      location / {
        root            /var/www/site2;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
      }
    }
📌
Point clé (souvent flou) : les deux blocs server sont chargés en mémoire au startup. Ensuite, une requête entrante est d'abord routée par son port local (8080 ou 8081), puis seulement par les location du serveur sélectionné.
🔄Le pipeline config est séquentiel : lire → tokeniser → parser → valider → appliquer. Chaque étape doit se terminer sans erreur avant la suivante.
💾Toute la config est en mémoire dès le démarrage. Pas de relecture à chaque requête. Les structures ServerConfig / LocationConfig sont immutables après validation.
🌐Un seul binaire = plusieurs serveurs logiques. Chaque bloc server est un "site" distinct, distingué par son port (et optionnellement son server_name).
02
server + listen + server_name
Le conteneur principal d'un site
Le bloc server

Un bloc server { ... } décrit une "personnalité" complète du serveur : ports d'écoute, règles de routes, limites de body, pages d'erreur, CGI, etc. Si tu écris plusieurs blocs server, webserv n'ouvre pas plusieurs programmes : il ouvre plusieurs configurations et choisit la bonne à chaque connexion.

listen-examples.conf
listen 
server {
  listen 8080;                # serveur logique A
}

server {
  listen 8081;                # serveur logique B
}

# Même binaire webserv, deux comportements différents selon le port
# req sur :8080 -> server A ; req sur :8081 -> server B
listen en pratique
  • Port valide : 1..65535.
  • Port < 1024 : droits élevés souvent requis en Linux.
  • Chaque listen implique un socket serveur distinct.
  • Plusieurs blocs server peuvent coexister dans un seul fichier et un seul process.
  • Avec epoll, chaque socket d'écoute est enregistré en EPOLLIN.
server_name (optionnel webserv)

server_name permet de faire des virtual hosts par header Host. Le sujet webserv ne l'exige pas strictement. Tu peux l'ignorer au début, puis l'ajouter si tu veux un routing plus proche de NGINX.

Algorithme si implémenté

1) filtrer par port d'arrivée, 2) comparer Host avec les server_name, 3) fallback sur serveur par défaut du port.

same-port-virtual-hosts.conf
Même port, sites différents 
# 2 serveurs logiques sur le même port 8080
server {
  listen 8080;
  server_name site-a.com www.site-a.com;
  location / { root /var/www/site-a; }
}

server {
  listen 8080;
  server_name site-b.com;
  location / { root /var/www/site-b; }
}

# Sélection à runtime (idée) :
# req sur :8080 + Host: site-a.com -> server A
# req sur :8080 + Host: site-b.com -> server B
# Host inconnu -> server par défaut du port (souvent le 1er déclaré)
💡
Réponse à ta question : oui, tu peux avoir 4-5 blocs server, dont 2-3 sur le même port. Ça sert justement à héberger plusieurs sites sur une même machine (virtual hosting), en les distinguant via le header Host + server_name.
🏢
Analogie — les guichets d'une poste

Chaque bloc server est un guichet distinct. Port 8080 = guichet A, port 8081 = guichet B.

Quand une requête arrive, le serveur regarde d'abord par quelle porte elle est entrée (le port), puis dirige vers le bon guichet. Le guichet a ses propres règles (ses location).

Si deux guichets partagent la même porte (même port), on lit le header Host pour choisir lequel — c'est le principe du server_name.

⚠️
Important : si tu n'implémentes pas server_name, plusieurs serveurs sur le même port deviennent ambigus. Dans ce cas, définis clairement une règle (ex: "premier server du port") et documente-la.
🔌Chaque listen unique = un socket serveur distinct. Webserv doit ouvrir autant de sockets d'écoute qu'il y a de ports distincts dans la config.
🔢Port valide : 1–65535. Ports < 1024 nécessitent des droits élevés sur Linux. À vérifier à la validation, pas à runtime.
🏷️server_name est optionnel selon le sujet. Si absent : sélection par port uniquement. Si présent : matching sur le header Host après filtrage par port.
03
client_max_body_size + error_page
Politiques globales de sécurité et d'UX
core-directives.conf
Directives clés 
server {
  client_max_body_size 10m;

  error_page 404 /errors/404.html;
  error_page 500 502 503 504 /errors/50x.html;
}
client_max_body_size

Limite la taille maximale du body client. C'est crucial pour éviter les abus mémoire/disque. En réception HTTP, tu compares tôt le Content-Length à cette limite. Si dépassement, réponse 413 sans continuer à lire inutilement.

Unités

k, m, g (insensible à la casse). Sans suffixe : octets bruts.

error_page

Associe un code HTTP à un fichier de réponse custom. Quand le serveur produit l'erreur, il essaie d'abord de servir cette page personnalisée. Sinon fallback sur un body minimal.

⚠️
Piège fréquent : la page d'erreur doit exister dans un chemin résolvable avec tes règles de root/location.
⛔client_max_body_size : compare le Content-Length reçu tôt dans le parsing de la requête. Réponse 413 immédiate, ne pas continuer à lire le body.
🗂️error_page : le chemin de la page custom doit être résolvable via tes propres règles root/location. Une erreur qui pointe vers une page inexistante génère une double erreur.
🔁Fallback obligatoire : si la page error_page n'est pas trouvée, le serveur doit quand même répondre avec un body minimal (HTML inline). Ne jamais renvoyer de réponse vide.
size-parse.cpp
Exemple C++ 
size_t parseSize(const std::string& s) {
  if (s.empty()) throw std::runtime_error("empty size");

  char suffix = std::tolower(s.back());
  std::string number = s;
  size_t factor = 1;

  if (suffix == 'k' || suffix == 'm' || suffix == 'g') {
    number = s.substr(0, s.size() - 1);
    if (suffix == 'k') factor = 1024;
    if (suffix == 'm') factor = 1024 * 1024;
    if (suffix == 'g') factor = 1024 * 1024 * 1024;
  }

  size_t value = static_cast<size_t>(std::strtoull(number.c_str(), NULL, 10));
  return value * factor;
}
04
location + allowed_methods + return
Le cœur du routing HTTP
Règle de matching

En webserv (sans regex), tu fais un matching par préfixe. Parmi les location compatibles, la plus longue gagne. Cette règle suffit pour couvrir l'énoncé et reproduire le comportement attendu.

location-routing.conf
Exemple de routes 
location / {
  allowed_methods GET;
  root /var/www/html;
}

location /api {
  allowed_methods GET POST DELETE;
  root /var/www/api;
}

location /old-page {
  return 301 /new-page;
}
allowed_methods

Directive custom webserv : contrôle des verbes autorisés par route. Si méthode non autorisée, réponse 405 Method Not Allowed + header Allow: listant les méthodes valides.

return

Court-circuite toute la suite. Quand return est défini, on ne cherche pas de fichier, on répond immédiatement avec le code (et Location si redirection).

🚦allowed_methods : si la méthode n'est pas dans la liste → 405 Method Not Allowed + header Allow: GET, POST, DELETE. Ce header est obligatoire avec le 405.
↩️return court-circuite tout le reste. Si défini dans la location matchée, on répond immédiatement — pas de lookup fichier, pas de CGI, pas de root.
📐Matching par préfixe uniquement (plus long préfixe = winner). Voir la page dédiée Sélection de route : préfixe le plus long pour l'algo complet.
longest-prefix.cpp
Algo de sélection 
const LocationConfig* findLocation(const ServerConfig& srv, const std::string& uri) {
  const LocationConfig* best = NULL;
  size_t bestLen = 0;

  for (std::vector<LocationConfig>::const_iterator it = srv.locations.begin(); it != srv.locations.end(); ++it) {
    const std::string& p = it->path;
    if (uri.compare(0, p.size(), p) == 0 &&
        (uri.size() == p.size() || uri[p.size()] == '/')) {
      if (p.size() > bestLen) {
        best = &(*it);
        bestLen = p.size();
      }
    }
  }
  return best;
}
05
root + index + autoindex
Résolution du chemin et réponse fichier
Point critique root

La seule formule correcte (confirmée par l'exemple du sujet) : root + URI résiduelle — on retire le préfixe de la location gagnante, puis on colle le résidu au root. Résultat : /kapouet/pouic/toto/pouet avec location /kapouet et root /tmp/www → /tmp/www/pouic/toto/pouet. Le préfixe /kapouet disparaît.

path-resolution.txt
Exemple concret 
# Seule formule correcte — root + URI résiduelle (préfixe retiré)
URI      = /kapouet/pouic/toto/pouet
location = /kapouet        # préfixe → retiré
résidu   = /pouic/toto/pouet
root     = /tmp/www
résultat = /tmp/www/pouic/toto/pouet
index

Si la cible est un dossier, tu testes les index dans l'ordre (index.html, puis index.htm, etc.). Si aucun index trouvé : soit autoindex on (listing), soit refus (403 en pratique).

autoindex

Quand activé, tu génères une page HTML listant les entrées du dossier via opendir/readdir. Chaque entrée doit devenir un lien cliquable. C'est utile en debug/partage, à manier prudemment en prod.

📌Le sujet donne l'exemple Cas 2 : /kapouet/pouic/toto/pouet avec root /tmp/www → /tmp/www/pouic/toto/pouet. Le préfixe /kapouet est retiré avant la concat.
🗂️index : tester les fichiers dans l'ordre de la liste. Premier trouvé = servi. Si aucun et autoindex off → 403.
📋autoindex on : générer une page HTML avec opendir/readdir. Chaque entrée = lien cliquable. Utilise les fonctions autorisées dans le sujet.

Arbre de décision 
if (!exists(path)) {

upload_store

Pour un multipart/form-data, tu découpes les parts avec le boundary, extrais le fichier, puis écris le contenu binaire dans le dossier cible. Le dossier doit exister et être writable.

🔒
Point sécurité : nettoyer le nom de fichier (path traversal, caractères interdits, collisions).
cgi_pass

Si l'extension matche, tu exécutes l'interpréteur via fork + execve. Tu injectes les variables CGI en environnement, le body en stdin, puis tu lis stdout.

Variables incontournables

REQUEST_METHOD, QUERY_STRING, CONTENT_LENGTH, CONTENT_TYPE, SCRIPT_NAME, PATH_INFO.

cgi-launch.cpp
Flux CGI simplifié 
pid_t pid = fork();
if (pid == 0) {
  // child
  dup2(inPipe[0], STDIN_FILENO);
  dup2(outPipe[1], STDOUT_FILENO);
  chdir(scriptDirectory.c_str());
  execve(interpreter.c_str(), argv, envp);
  _exit(1);
}

// parent
write(inPipe[1], requestBody.data(), requestBody.size());
close(inPipe[1]);
std::string cgiOutput = readAll(outPipe[0]);
waitpid(pid, NULL, 0);
💡
Chunked + CGI : pour les requêtes chunked, il faut reconstituer le body avant de le donner au CGI. Le script CGI attend un flux body normal terminé par EOF.
🔱fork est autorisé uniquement pour le CGI. C'est une règle dure du sujet — l'utiliser ailleurs annule la note.
📂Le CGI doit être exécuté dans le bon répertoire de travail (chdir avant execve). Les scripts qui utilisent des chemins relatifs en dépendent.
📦Upload : le dossier upload_store doit exister et être writable. Nettoyer le nom de fichier (path traversal, .., caractères interdits).
🔄Si le CGI ne retourne pas de Content-Length, le serveur utilise EOF pour détecter la fin de la réponse — lire stdout jusqu'à fermeture du pipe.
07
lexer + parser + validation
Comment l'implémenter proprement en C++
Lexer

Le lexer transforme un texte brut en tokens structurés. Cette étape simplifie radicalement le parser et améliore les erreurs de diagnostic (ligne/colonne).

  • Ignorer espaces et commentaires # ....
  • Sortir les séparateurs : {, }, ;.
  • Capturer les mots/directives/valeurs en WORD.
Parser

Le parser lit la suite de tokens et remplit des structures C++. Le plus simple : parser récursif-descendant avec expect() et exceptions explicites.

  • server ouvre un ServerConfig.
  • location ouvre un LocationConfig.
  • Chaque directive remplit un champ strictement typé.
config-parser.hpp
Structures recommandées 
struct LocationConfig {
  std::string path;
  std::string root;
  std::vector<std::string> indexFiles;
  std::vector<std::string> allowedMethods;
  bool autoindex;
  std::map<std::string, std::string> cgiPass;
  std::string uploadStore;
  int returnCode;
  std::string returnTarget;
};

struct ServerConfig {
  std::vector<std::pair<std::string, int> > listens;
  std::vector<std::string> serverNames;
  size_t clientMaxBodySize;
  std::map<int, std::string> errorPages;
  std::vector<LocationConfig> locations;
};
✂️Le lexer supprime commentaires et espaces. Il produit une suite plate de tokens : WORD, {, }, ;. Le parser ne voit jamais de whitespace.
🌲Le parser lit les tokens et remplit les structs. Approche recommandée : récursif-descendant avec expect(token) — si le token attendu n'est pas là, lever une exception claire.
🛡️La validation s'exécute après le parsing, avant d'ouvrir tout socket. Elle vérifie les ports valides, les locations non-vides, les chemins accessibles, les méthodes connues.
config-validation.cpp
Validation startup 
void validate(const std::vector<ServerConfig>& servers) {
  for (size_t i = 0; i < servers.size(); ++i) {
    const ServerConfig& s = servers[i];

    if (s.listens.empty()) throw std::runtime_error("server without listen");
    if (s.locations.empty()) throw std::runtime_error("server without location");

    for (size_t j = 0; j < s.listens.size(); ++j) {
      if (s.listens[j].second < 1 || s.listens[j].second > 65535)
        throw std::runtime_error("invalid listen port");
    }
  }
}
08
ordre de priorité runtime
Ce que fait le serveur quand une requête arrive
Algorithme complet conseillé
  • Sélectionner le server par port (et éventuellement Host).
  • Sélectionner la location par plus long préfixe.
  • Vérifier allowed_methods (sinon 405).
  • Si return existe : répondre immédiatement.
  • Construire le chemin fichier (root + stratégie URI).
  • Si extension CGI match : exécuter CGI.
  • Sinon servir fichier / index / autoindex / erreur.
request-dispatch.cpp
Pseudo-code d'orchestration 
Response dispatch(const Request& req) {
  const ServerConfig* srv = selectServer(req.localPort, req.host);
  const LocationConfig* loc = selectLocation(*srv, req.uri);

  if (!methodAllowed(*loc, req.method)) {
    return make405(loc->allowedMethods);
  }

  if (loc->returnCode != 0) {
    return makeReturn(loc->returnCode, loc->returnTarget);
  }

  std::string path = resolvePath(*loc, req.uri);

  if (isCgiTarget(*loc, path)) {
    return executeCgi(req, *loc, path);
  }

  return serveStatic(req, *loc, path, srv->errorPages);
}
⚠️
Robustesse attendue : toute erreur de route/fichier/CGI doit renvoyer une réponse HTTP propre, jamais faire planter la boucle événementielle.
💡
Évaluation : prépare plusieurs fichiers de config + fichiers web par défaut pour démontrer toutes les features (méthodes, redirection, erreurs, listing, upload, CGI).
1️⃣L'ordre de priorité est fixe : server → location → methods → return → path → CGI → static. Une étape qui échoue renvoie une réponse HTTP, elle ne saute pas à l'étape suivante.
🔒Chaque branche de l'arbre de décision doit produire une réponse. Pas de chemin sans réponse, pas de réponse vide. Le serveur ne doit jamais crasher quelle que soit la requête.
🧪Le sujet demande explicitement de fournir des fichiers de config de test. Prépare un fichier qui couvre tous les cas : statique, redirection, 404, upload, CGI, autoindex.

my-webserv.conf
Configuration complète

    listen 8080;
    server_name localhost;
    client_max_body_size 10m;

        allowed_methods POST;

        root            /var/www/shared;
        allowed_methods GET;
        autoindex       on;
    }

    # Redirection permanente
    location /old-page {
        return 301 /new-page;
    }

    # CGI Python
    location /cgi-bin {
        root            /var/www/cgi-bin;
        allowed_methods GET POST;
        cgi_pass        .py /usr/bin/python3;
        cgi_pass        .php /usr/bin/php-cgi;

}

# Deuxième serveur sur port 8081

    listen 8081;
    client_max_body_size 1m;

    location / {
        root            /var/www/site2;
        index           index.html;
        allowed_methods GET;

Ce que ce fichier implique au démarrage
  • Création de 2 contextes ServerConfig.
  • Ouverture de 2 sockets d'écoute : 8080 et 8081.
  • Construction de la table de routes pour chaque serveur.
  • Préchargement/validation des directives globales (taille body, pages d'erreur).
  • Préparation des maps CGI (extension → interpréteur).
Exemples de requêtes et résultat attendu
  • GET / sur port 8080 → route /, fichier index.html.
  • GET /api/users sur 8080 → route /api, méthodes GET/POST/DELETE.
  • POST /upload sur 8080 → route /upload, stockage /tmp/uploads.
  • GET /old-page sur 8080 → redirection 301 Location: /new-page.
  • GET /foo.py sous /cgi-bin → exécution via /usr/bin/python3.
  • GET / sur 8081 → racine /var/www/site2.

10
glossaire exhaustif
Tous les termes importants (définitions + implications code)
Termes de structure
  • directive : instruction de config terminée par ;.
  • block directive : directive avec bloc { ... }.
  • context : zone de validité d'une directive.
  • server block : site virtuel complet (ports + routes).
  • location block : règles applicables à un préfixe d'URI.
  • token : unité lexicale produite par ton lexer.
  • AST / config model : représentation interne parsée.
Termes réseau/HTTP
  • interface : IP sur laquelle le socket écoute (ex: 127.0.0.1).
  • port : numéro de service TCP (ex: 8080).
  • Host header : nom d'hôte demandé par le client HTTP.
  • method : verbe HTTP (GET/POST/DELETE...).
  • body : données transportées par la requête.
  • multipart/form-data : format d'upload de fichiers.
  • Content-Length : taille annoncée du body.
Termes filesystem/routing
  • root : ancrage disque pour résoudre les fichiers.
  • index : fichier par défaut d'un dossier.
  • autoindex : listing auto d'un répertoire.
  • longest prefix match : location la plus spécifique.
  • error_page : page custom pour un code HTTP.
  • redirect : réponse 3xx + header Location.
  • fallback : comportement de secours si règle absente.
Termes CGI
  • CGI : exécution d'un script externe pour générer la réponse.
  • interpreter : binaire appelé via execve (python/php).
  • env vars : infos requête passées au script CGI.
  • stdin : body envoyé au script.
  • stdout : réponse CGI récupérée par le serveur.
  • EOF : fin de flux, surtout critique si pas de Content-Length CGI.
  • working directory : dossier d'exécution du script.
