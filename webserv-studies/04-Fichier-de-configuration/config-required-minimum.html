04 — Sujet IV.3

Ce que la config doit absolument contenir

Tout ce que le sujet IV.3 exige, organisé par scope : ce qui se configure au niveau server (global au site) et ce qui se configure au niveau location (par route). Dix directives. Zéro oubli.

02
niveau server
listen · error_page · client_max_body_size
webserv.conf — server level
3 directives requises 
# ① listen — chaque port distinct = un socket serveur ouvert
server {
    listen               8080;       # socket A
    server_name          localhost;  # optionnel (virtual host)

    # ② error_page — pages d'erreur personnalisées
    error_page 404 /errors/404.html;
    error_page 500 /errors/500.html;

    # ③ client_max_body_size — limite body (check sur Content-Length)
    client_max_body_size 10m;
}

server {
    listen               8081;       # socket B
    client_max_body_size 1m;
}
REQUIS listen

Ouvre un socket TCP sur ce port au démarrage. Plusieurs blocs server peuvent partager le même port — la sélection se fait ensuite par server_name. Chaque port distinct = un socket séparé enregistré dans poll().

REQUIS error_page

Associe un code HTTP à une page custom. Le serveur la sert à la place de la réponse d'erreur générique. Si la page est elle-même introuvable : fallback body HTML inline obligatoire. Ne jamais envoyer de réponse vide.

REQUIS client_max_body_size

Vérifie Content-Length en début de parsing — avant de lire le body. Si la valeur dépasse la limite : 413 Request Entity Too Large immédiat. Ne pas attendre la fin de la lecture pour rejeter.

03
niveau location
Les 7 directives par route — toutes requises
webserv.conf — location level
7 directives requises 
# ① allowed_methods  ② root  ③ index  ④ autoindex
location / {
    root            /var/www/html;
    index           index.html;
    allowed_methods GET;
    autoindex       off;
}

# ⑤ return (redirection HTTP)
location /old-page {
    return 301 /new-page;
}

# ⑥ upload_store (upload de fichiers)
location /upload {
    root            /var/www;
    allowed_methods POST;
    upload_store    /tmp/uploads;
}

# ⑦ cgi_pass (CGI par extension de fichier)
location /cgi-bin {
    root            /var/www/cgi-bin;
    allowed_methods GET POST;
    cgi_pass        .py  /usr/bin/python3;
    cgi_pass        .php /usr/bin/php-cgi;
}
① REQUIS allowed_methods

Méthodes HTTP acceptées pour cette route. Si la requête utilise une méthode absente de la liste : 405 Method Not Allowed + header Allow: listant les méthodes valides. Vérifié avant toute résolution de fichier.

② REQUIS return

Redirection HTTP immédiate. Court-circuite tout le reste : aucun accès disque, aucun CGI. Typiquement 301 (permanente) ou 302 (temporaire). Doit produire le header Location: dans la réponse.

③ REQUIS root

Répertoire racine. Résolution : root + (URI − préfixe_location). C'est le Cas 2 du sujet : location /api + root /var/www/api + URI /api/data.json → fichier /var/www/api/data.json.

④ REQUIS autoindex

Active le listing du contenu d'un dossier quand aucun index n'est trouvé. off = 403 Forbidden si pas d'index. Implémenté avec opendir / readdir pour générer le HTML du listing.

⑤ REQUIS index

Fichier servi automatiquement quand la requête pointe sur un répertoire. Le serveur tente root + "/" + index. Si absent et autoindex off : 403. Si absent et autoindex on : listing du dossier.

⑥ REQUIS upload_store

Dossier de destination des fichiers uploadés. Le serveur décode le body multipart/form-data, extrait le fichier, et l'écrit dans ce dossier. Le dossier doit exister et être writable. Nettoyer le nom de fichier (path traversal, ..).

⑦ REQUIS cgi_pass

Associe une extension à un interpréteur. Si l'URI correspond à cette extension : fork + execve de l'interpréteur avec les variables CGI en env, le body en stdin, réponse sur stdout. fork est réservé exclusivement au CGI.

🔢Ordre d'exécution dans une location : allowed_methods → return (si présent) → path resolution → CGI (si extension matche) → static file / autoindex.
⚡return court-circuite tout. Si présent dans la location, les autres directives (root, index, cgi_pass) sont ignorées.
🔱cgi_pass implique fork + execve. C'est le seul endroit du code où fork est autorisé — règle dure du sujet.
04
config de test complète
Un seul fichier qui démontre les 10 directives requises
webserv-eval.conf
toutes les 10 directives IV.3 
# ═══ SERVEUR A — port 8080 ═══════════════════════════════════
server {
    listen               8080;           # ① listen
    server_name          localhost;
    client_max_body_size 10m;            # ③ client_max_body_size

    error_page 404 /errors/404.html;     # ② error_page
    error_page 500 /errors/500.html;

    # Statique — méthodes + root + index + autoindex
    location / {
        root            /var/www/html;  # ③ root
        index           index.html;     # ⑤ index
        allowed_methods GET;            # ① allowed_methods
        autoindex       off;            # ④ autoindex
    }

    # Listing de dossier
    location /files {
        root            /var/www/shared;
        allowed_methods GET;
        autoindex       on;             # ④ autoindex on
    }

    # Redirection
    location /old-page {
        return 301 /new-page;           # ② return
    }

    # Upload
    location /upload {
        root            /var/www;
        allowed_methods POST;
        upload_store    /tmp/uploads;   # ⑥ upload_store
    }

    # CGI
    location /cgi-bin {
        root            /var/www/cgi-bin;
        allowed_methods GET POST;
        cgi_pass        .py  /usr/bin/python3;   # ⑦ cgi_pass
        cgi_pass        .php /usr/bin/php-cgi;
    }
}

# ═══ SERVEUR B — port 8081 ═══════════════════════════════════
server {
    listen               8081;           # ① listen (second port)
    client_max_body_size 1m;

    location / {
        root            /var/www/site2;
        index           index.html;
        allowed_methods GET;
        autoindex       off;
    }
}
💡
Conseil soutenance : garde ce fichier tel quel dans ton repo. Pour chaque directive, tu dois pouvoir montrer le comportement en live — une requête curl qui déclenche un 301, un 413, un 405, un listing autoindex, un upload, un CGI.
🧪Test listen : deux ports distincts, vérifier que les deux répondent. Test multi-port = obligatoire.
🗂️Test error_page : requête sur une URL inexistante → réponse 404 avec ta page custom, pas le body générique.
⛔Test client_max_body_size : curl -X POST --data-binary @bigfile.bin avec un fichier > limite → 413 immédiat.
🚫Test allowed_methods : DELETE / → 405 + header Allow: GET. La méthode interdite ne doit jamais atteindre la résolution de fichier.
