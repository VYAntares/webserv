04 — Fichier de configuration

C'est quoi NGINX ?

On part d'exemples concrets du Beginner's Guide : structure server/location, mapping URI → fichiers, et commandes nginx -s. Ensuite, on relie ces exemples à ce que webserv doit implémenter en C++.

01
nginx
Un serveur web orienté configuration
nginx.conf
Exemple minimal 
# Structure de base vue dans le guide officiel
events {
}

http {
  server {
    location / {
      root /data/www;
    }
  }
}
Définition claire

NGINX est un serveur web et un proxy HTTP dont le comportement est piloté par des directives déclaratives dans nginx.conf. Tu ne codes pas les routes en dur : tu décris ce qui doit se passer, et NGINX l'applique à runtime.

Le guide officiel insiste sur une idée simple : NGINX est modulaire, et ses modules sont contrôlés par la configuration. Le fichier de config est donc le centre de gravité du serveur.

Ce que ça t'apprend pour webserv
  • Ton binaire doit démarrer à partir d'un fichier de config, pas d'un comportement hardcodé.
  • Une directive = un paramètre de comportement exploité pendant le traitement des requêtes.
  • Les règles de routage se décident via les blocs et contextes, pas via des if dispersés.
🏨
Analogie — c'est quoi NGINX concrètement ?

Imagine un hôtel avec un réceptionniste et un manuel de procédures. Chaque client (requête HTTP) arrive à l'accueil.

Le réceptionniste ne décide jamais à l'instinct — il consulte le manuel (nginx.conf) : « si le client demande /images/, envoie-le dans le couloir /data/images/ ».

Tu n'as pas besoin de recoder le réceptionniste pour changer les règles — tu mets à jour le manuel. C'est exactement le modèle que ton webserv doit reproduire.

« The way nginx and its modules work is determined in the configuration file. »

— NGINX Beginner's Guide
📋Le comportement d'NGINX est piloté par la config, pas par le code. Webserv doit fonctionner pareil : le binaire est stable, c'est le fichier de config qui définit tout.
🚫Pas de comportement hardcodé. Si le port, la route, la page d'erreur sont écrits en dur dans le C++ → c'est faux. Tout doit venir du fichier de config au démarrage.
⚙️Conséquence directe : webserv démarre en lisant et validant le fichier. Si la config est invalide → erreur propre et exit. Jamais de démarrage partiel silencieux.
02
master + workers
Qui fait quoi dans NGINX
Architecture runtime

NGINX exécute un master process et plusieurs worker processes. Le master lit/valide la config et gère le cycle de vie des workers. Les workers traitent les connexions et les requêtes selon un modèle événementiel.

master = contrôle worker = I/O modèle event-driven
Parallèle webserv

Webserv n'exige pas de reproduire le multi-process de NGINX. En revanche, il faut reproduire la logique : une phase lecture/validation de config au démarrage, puis une phase boucle événementielle stable qui applique ces règles à chaque requête.

👨‍🍳
Analogie

Master = chef de cuisine qui lit la recette (nginx.conf) et organise les brigades. Workers = cuisiniers qui exécutent les commandes (requêtes).

Sur nginx -s reload, le chef forme une nouvelle brigade avec la nouvelle recette, attend que les anciens terminent leurs plats en cours, puis les remplace. Le service ne s'interrompt jamais.

Pour webserv, pas besoin de multi-process — mais la même logique en deux temps s'applique : phase init, puis boucle de traitement.

« nginx has one master process and several worker processes. »

— NGINX Beginner's Guide
📖Phase 1 — init : lire et valider le fichier de config. Si erreur → exit propre. Pas de socket ouvert, pas de port écouté.
🔄Phase 2 — boucle événementielle : appliquer les règles lues à chaque requête qui arrive. La config ne change plus pendant cette phase.
🛡️Webserv ne doit jamais crasher. Cette séparation init/runtime rend ça possible : les erreurs de config sont attrapées avant que le service démarre.
03
configuration syntax
Directives, blocs, contextes
syntax-demo.conf
Directive simple vs bloc 
# simple directive: finit par ;
listen 8080;

# block directive: contient { ... }
server {
  location /images/ {
    root /data;
  }
}

# commentaire: tout ce qui suit # est ignoré
Structure officielle

Une directive simple se termine par ;. Une directive bloc utilise des accolades { }. Quand un bloc peut contenir d'autres directives, on parle de contexte (ex: events, http, server, location).

Hiérarchie standard : directives globales en main, puis http, puis server, puis location. Les commentaires commencent par #.

simple directive
Nom + paramètres + ;. Exemple conceptuel: listen, root, index.
block directive
Nom + paramètres + bloc { ... } contenant d'autres directives.
context
Zone de portée d'une directive. location s'applique dans server, etc.
comment
Tout ce qui suit # sur une ligne est ignoré par le parser.
📌Toute directive simple finit par ;. Si ton parser trouve un token sans ; → erreur de syntaxe à rejeter proprement, pas à ignorer silencieusement.
🪆Les blocs s'imbriquent : server dans http, location dans server. Ton parser doit gérer les accolades récursives et maintenir un contexte courant.
💬Les commentaires (#) doivent être supprimés au niveau du lexer — ils ne font pas partie de la grammaire et ne doivent jamais atteindre le parser.
🌳La hiérarchie main → server → location définit les contextes de portée. Une directive dans location ne s'applique qu'à ce bloc, pas au serveur entier.
04
location + root
Servir du contenu statique
server-static.conf
Exemple officiel statique 
server {
  location / {
    root /data/www;
  }

  location /images/ {
    root /data;
  }
}

# /images/example.png → /data/images/example.png
# /some/example.html → /data/www/some/example.html
🗂️
Analogie — le tri postal

Imagine un centre de tri postal. L'URI est l'adresse sur l'enveloppe. Tu as deux règles : une pour / (toutes les lettres) et une pour /images/ (lettres spéciales).

Une enveloppe adressée à /images/logo.png correspond aux deux — mais le tri retient la règle la plus précise : /images/ gagne.

Ensuite, l'adresse physique de livraison est calculée :
root /data + /images/logo.png → fichier à /data/images/logo.png.

Logique de matching

NGINX teste l'URI contre les location. Entre plusieurs préfixes compatibles, la règle est : le préfixe le plus long gagne. C'est la base de l'aiguillage des requêtes statiques.

Logique de résolution

Une fois la route choisie, NGINX compose le chemin local via root + URI pour trouver le fichier. S'il n'existe pas : réponse d'erreur (souvent 404). C'est exactement le comportement décrit dans ton sujet webserv IV.3.

« If there are several matching location blocks nginx selects the one with the longest prefix. »

— NGINX Beginner's Guide, Serving Static Content
🏆Longest prefix wins — pas le premier match. Entre / et /images/, une URI /images/logo.png matche /images/. Webserv doit implémenter exactement cette logique.
📁Résolution : root + URI résiduelle (URI moins le préfixe de la location). Exemple du sujet : route /kapouet + root /tmp/www → /kapouet/pouic/toto/pouet devient /tmp/www/pouic/toto/pouet — le préfixe /kapouet disparaît.
❓Si le fichier résolu n'existe pas → 404. Si c'est un répertoire sans index → 403 ou listing selon autoindex on/off. Ces trois cas doivent tous être gérés.
05
nginx -s
Start, stop, reload sans downtime brutal
terminal
Commandes utiles 
# démarrer nginx
nginx

# recharger la config après modification
nginx -s reload

# arrêt propre (graceful)
nginx -s quit

# arrêt rapide
nginx -s stop
Signaux de contrôle

Le master reçoit des signaux : stop (arrêt rapide), quit (graceful), reload (relecture de config), reopen (reopen logs).

Sur reload, NGINX valide d'abord la nouvelle config. Si elle est correcte, il démarre de nouveaux workers et demande aux anciens de finir leurs requêtes en cours. Si elle est invalide, rollback automatique vers l'ancienne config.

nginx -s reload validation avant application graceful shutdown des anciens workers
✅reload valide d'abord la nouvelle config. Si invalide → rollback automatique, les anciens workers continuent. Aucun downtime, aucune perte de requête.
🚦Pour webserv, le reload dynamique n'est pas requis. Mais le principe s'applique au démarrage : valider la config avant d'ouvrir le moindre socket. Config invalide → exit propre.
🔌nginx -s quit = graceful (finit les requêtes en cours). nginx -s stop = kill brutal. Webserv doit au minimum gérer CTRL+C proprement sans laisser des sockets ouverts.
06
webserv IV.3
Ce que tu dois en implémenter
Checklist directe du sujet
  • Paires interface:port (plusieurs serveurs/sites).
  • error_page par défaut.
  • client_max_body_size.
  • Par route : méthodes, redirection, root, autoindex, index, upload, CGI par extension.
Traduction implémentation
  • Un lexer + parser robuste pour blocs imbriqués et directives.
  • Validation stricte au démarrage avant d'ouvrir les sockets.
  • Matching des location sans regex (préfixe).
  • Moteur runtime stable : pas de crash, serveur résilient.

« You can take inspiration from the 'server' section of the NGINX configuration file. »

— webserv.pdf, IV.3 Configuration file
🔌Plusieurs paires interface:port = plusieurs server blocks. Le serveur écoute sur plusieurs ports simultanément — un seul poll() surveille tous les fds.
🔀CGI par extension : si l'URI finit en .php ou .py, webserv doit exécuter un CGI externe (via fork + execve). Le seul endroit où fork est autorisé.
📐Le sujet dit explicitement : pas de regex pour les routes. Le matching est uniquement par préfixe — ce que tu viens d'apprendre dans la section 04.
⚖️En cas de doute sur un comportement, comparer avec NGINX. C'est la référence officielle du sujet pour tous les cas ambigus (headers, codes de statut, comportement 404/403).
