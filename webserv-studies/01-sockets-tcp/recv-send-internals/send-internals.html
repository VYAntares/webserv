send() internals

Que se passe-t-il vraiment quand tu appelles send() ? Le trajet de tes données depuis ton buffer jusqu'à la carte réseau, en passant par le send buffer kernel, la segmentation TCP, et les retransmissions.

01
Le syscall send()
De ton buffer au kernel
Ce qui se passe quand tu appelles send()

send(fd, buf, len, flags) est un syscall (__NR_sendto) qui copie tes données depuis l'userspace vers le socket send buffer du kernel. send() ne transmet rien sur le réseau directement.

Le parcours kernel est : sys_sendto() → sock_sendmsg() → tcp_sendmsg(). C'est tcp_sendmsg() qui copie tes données dans des sk_buff, les ajoute à la write queue, et déclenche tcp_push() pour commencer l'envoi.

send() retourne dès que les données sont dans le buffer kernel — pas quand elles sont reçues par le client. L'envoi réel est asynchrone.

📮
Analogie

Tu déposes tes lettres au bureau de poste (send buffer). Le postier (kernel TCP) les prend et les met dans des camions (segments TCP). Tu repars immédiatement — tu n'attends pas que le destinataire les reçoive. Si le bureau est plein (EAGAIN), le postier te dit de revenir plus tard.

02
Le send buffer kernel
Où tes données attendent d'être envoyées
sk_write_queue et sk_wmem_queued

Chaque socket TCP a un buffer d'envoi dans le kernel. sk->sk_write_queue est la file d'attente des sk_buff à envoyer. sk->sk_wmem_queued est la mémoire totale utilisée par ces sk_buff.

La limite est sk->sk_sndbuf (configurable via net.core.wmem_default, souvent 212 992 octets ~ 208 KB). Si la write queue est pleine, send() ne peut plus accepter de données.

Pourquoi le buffer se remplit

Le send buffer ne se vide pas quand les données sont envoyées sur le réseau. Il se vide quand le client confirme la réception (ACK TCP). Tant que l'ACK n'est pas reçu, les données restent dans le buffer au cas où il faudrait retransmettre.

Si le client est lent à confirmer (réseau lent, client occupé, congestion), le send buffer se remplit progressivement. Quand il est plein → send() retourne EAGAIN (non-bloquant) ou bloque.

Kernelspace — struct sock
sk->sk_write_queue
File d'attente de sk_buff prêts à être envoyés. send() retourne dès que les données sont ici. L'envoi réseau est asynchrone.
TCP segmentation + retransmission
Le kernel découpe les données en segments TCP (MSS ~ 1460 octets), ajoute les headers TCP/IP, et gère les retransmissions. Les sk_buff restent en queue jusqu'à réception de l'ACK.
↓ dev_queue_xmit() → driver NIC
Carte réseau (NIC) + driver
TX ring buffer (DMA)
Le driver place les paquets dans le ring buffer de transmission. La NIC les envoie sur le fil via DMA. Une interruption confirme l'envoi.
💡
send() ne garantit pas la livraison. Quand send() retourne 1024, ça veut dire que 1024 octets sont dans le buffer kernel, pas chez le client. C'est TCP qui garantit la livraison via ACK + retransmission. Si le client crashe avant de recevoir, les données dans le send buffer sont perdues silencieusement.

📦 MSS ~ 1460 octets. Le Maximum Segment Size est la taille max du payload TCP. 50 KB = ~35 segments. Chaque segment voyage indépendamment sur le réseau.
🔄 ACK = libération du buffer. Les sk_buff restent dans la write queue tant que le client n'a pas ACK. La retransmission utilise ces mêmes sk_buff.
🔔 EPOLLOUT. Quand un ACK libère de la place dans le send buffer, le kernel signale EPOLLOUT. C'est le moment de reprendre l'envoi si tu avais eu un envoi partiel.
04
L'envoi partiel
Pourquoi send() n'envoie pas tout
Le mécanisme

Tu appelles send(fd, buf, 50000, 0). Mais le send buffer kernel n'a que 16 KB de libre (le reste est occupé par des données en attente d'ACK).

tcp_sendmsg() copie seulement 16 384 octets et retourne 16384. Les 33 616 octets restants ne sont pas copiés. C'est à toi de les renvoyer plus tard.

En mode non-bloquant, si le buffer est complètement plein (0 octets de libre), send() retourne -1 avec EAGAIN.

La solution avec epoll

Après un envoi partiel :

  • Sauvegarder l'offset — tu sais que 16 384 octets ont été envoyés, il reste 33 616
  • S'enregistrer en EPOLLOUT — epoll_ctl(MOD, fd, EPOLLOUT)
  • Attendre le signal — quand le client ACK et que le buffer se libère, epoll_wait() retourne EPOLLOUT
  • Reprendre l'envoi — send(fd, buf + 16384, 33616, 0)
  • Répéter jusqu'à ce que tout soit envoyé
tcp_sendmsg_simplified.c
Logique kernel simplifiée 
// Pseudo-code simplifié de tcp_sendmsg()

int tcp_sendmsg(struct sock *sk, struct msghdr *msg, size_t size) {

    lock_sock(sk);
    int copied = 0;

    while (copied < size) {

        // Vérifier l'espace disponible dans le send buffer
        int available = sk->sk_sndbuf - sk->sk_wmem_queued;

        if (available <= 0) {
            if (sk->sk_socket->file->f_flags & O_NONBLOCK) {
                if (copied == 0) return -EAGAIN;  // rien copié → EAGAIN
                break;                              // déjà copié partiellement → retourner
            }
            // Mode bloquant : dormir jusqu'à ce que de la place se libère (ACK)
            sk_stream_wait_memory(sk);
            continue;
        }

        // Copier les données de l'userspace vers un sk_buff
        int chunk = min(available, size - copied);
        copy_from_user(skb->data, msg->msg_iov, chunk);  // userspace → kernel
        copied += chunk;
    }

    // Déclencher l'envoi TCP
    tcp_push(sk, flags);

    release_sock(sk);
    return copied;  // ← peut être MOINS que size !
}
⚠️
Le piège classique : ne pas vérifier la valeur retournée par send(). Si tu envoies 50 KB et que send() retourne 16 384, 34 616 octets n'ont pas été envoyés. Ils sont toujours dans ton buffer. Tu dois les renvoyer après EPOLLOUT. Ignorer cette valeur = réponses HTTP tronquées, fichiers incomplets, clients qui timeout.
05
SIGPIPE et EPIPE
Quand le client est déjà parti
Ce qui se passe dans le kernel

Le client ferme sa connexion → le serveur reçoit un TCP FIN (ou RST). Le socket passe en état CLOSE_WAIT. Tu ne le sais peut-être pas encore.

Tu appelles send() sur ce socket mort. tcp_sendmsg() détecte que le socket n'est plus connecté et fait deux choses :

  • Envoie un signal SIGPIPE à ton processus (comportement par défaut : terminer le programme)
  • Retourne -1 avec errno = EPIPE
Comment se protéger

Solution 1 : ignorer SIGPIPE globalement au démarrage du serveur :

signal(SIGPIPE, SIG_IGN);

Solution 2 : utiliser le flag MSG_NOSIGNAL sur chaque appel send :

send(fd, buf, len, MSG_NOSIGNAL);

Dans les deux cas, send() retourne -1 avec EPIPE sans tuer ton serveur. Tu fermes proprement le fd et tu passes au client suivant.

FlagEffet dans le kernelUsage dans webserv
0 Comportement par défaut. SIGPIPE envoyé si le pair est déconnecté Dangereux sans signal(SIGPIPE, SIG_IGN)
MSG_NOSIGNAL Supprime SIGPIPE pour cet appel. Retourne EPIPE sans signal Sûr — alternative à l'ignorer globalement
MSG_DONTWAIT Non-bloquant ponctuel. Inutile si le socket est déjà O_NONBLOCK Pas nécessaire (nos sockets sont déjà non-bloquants)
MSG_MORE Indique au kernel que d'autres données suivront. Retarde l'envoi (Nagle-like) Peut optimiser l'envoi de headers + body séparément
send_safe.cpp
Protection SIGPIPE 
// Au démarrage du serveur — ignorer SIGPIPE globalement
signal(SIGPIPE, SIG_IGN);

// Maintenant send() sur un socket fermé retourne EPIPE sans crash
ssize_t n = send(fd, buf, len, 0);

if (n == -1) {
    if (errno == EPIPE) {
        // Client déconnecté — nettoyer proprement
        epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);
        close(fd);
    } else if (errno == EAGAIN) {
        // Buffer plein — attendre EPOLLOUT
    }
}
💡
Avec epoll, on peut éviter le problème : si le client se déconnecte, epoll_wait() retourne EPOLLHUP ou EPOLLERR sur ce fd. Si tu gères ces événements correctement (fermer le fd), tu n'appelleras jamais send() sur un socket mort. SIGPIPE/EPIPE ne se produit que si tu envoies avant d'avoir traité l'événement epoll.

Le trajet de tes données depuis ton buffer jusqu'au client.

copy_from_user()

sk_write_queue
Les sk_buff attendent dans la write queue. Ils sont libérés quand le client confirme avec un ACK TCP.
