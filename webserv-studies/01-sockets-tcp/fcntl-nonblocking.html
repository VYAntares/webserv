Blocking vs Non-blocking

Pourquoi un socket bloque le programme, ce que fait vraiment fcntl(), comment EAGAIN s'inscrit dans le multiplex, et pourquoi fcntl reste obligatoire même avec epoll.

01
accept() / recv()
Le problème du blocking
Par défaut

Un socket est bloquant à sa création. Cela signifie que certains appels système suspendent le processus entier jusqu'à obtenir une réponse, peu importe le temps que ça prend.

Les deux appels concernés dans un serveur HTTP :

🧍
Analogie

Un caissier qui fixe la porte d'entrée sans bouger, refusant de servir quiconque jusqu'à ce qu'un nouveau client entre. Les clients déjà dans le magasin attendent indéfiniment.

blocking_problem.cpp
Problème 
// Ces deux appels GÈLENT le programme jusqu'à réponse

accept(server_fd, ...);   // ← suspend jusqu'à ce qu'un client arrive
recv(client_fd, buf, size, 0);  // ← suspend jusqu'à recevoir des données
Ce qui se passe au niveau du kernel

Quand recv() est appelé sur un socket bloquant sans données disponibles :

💤 Le kernel met le processus en état SLEEPING
✅ Le processus ne consomme plus de CPU
❌ Mais il ne fait rien d'autre non plus
🔔 Le kernel le réveille quand des données arrivent
⚠️ À l'échelle : avec 1000 clients simultanés en mode bloquant → il faudrait 1000 threads, un par client. C'est lourd, consomme de la mémoire, et ne scale pas.
02
fcntl() + O_NONBLOCK
La solution partielle
Ce que fait fcntl littéralement

fcntl signifie "file control". C'est une fonction générique qui manipule un file descriptor déjà ouvert. Elle fait des choses différentes selon le cmd passé.

Ce que fait O_NONBLOCK exactement

fcntl ne fait qu'écrire un seul bit dans une structure interne du kernel associée au fd. Ce bit change le comportement de tous les appels suivants sur ce fd.

cmd Rôle Exemple d'utilisation
F_GETFL Lire les flags de statut actuels du fd Toujours faire ça avant F_SETFL pour préserver les autres flags
F_SETFL Écrire de nouveaux flags de statut Activer O_NONBLOCK sur le fd
F_SETFD Écrire les flags du fd lui-même Utiliser FD_CLOEXEC pour fermer auto à execve()
set_nonblocking.cpp
Bonne pratique 
// Étape 1 : lire les flags actuels (IMPORTANT : préserve les autres flags)
int flags = fcntl(fd, F_GETFL);

// Étape 2 : ajouter O_NONBLOCK avec un OR bitwise
fcntl(fd, F_SETFL, flags | O_NONBLOCK);  // ← LE bit qui change tout
💡 Pourquoi le F_GETFL d'abord ? Si on fait juste fcntl(fd, F_SETFL, O_NONBLOCK), on écrase TOUS les flags existants (lecture, écriture...). Le | ajoute O_NONBLOCK sans toucher au reste.

EAGAIN : la réponse non-bloquante

Quand il n'y a rien à lire

En mode non-bloquant, au lieu de geler le programme, le kernel retourne immédiatement -1 avec errno positionné à EAGAIN.

EAGAIN signifie littéralement "Try Again" — "rien pour l'instant, réessaie plus tard". C'est normal, pas une erreur.

handle_eagain.cpp 
int n = recv(client_fd, buf, size, 0);
if (n == -1 && errno == EAGAIN) {
    // Pas de données disponibles MAINTENANT
    // C'est NORMAL — pas une erreur
    // On reviendra plus tard
    return;
}

Piège classique : chaque fd individuellement

server_setup.cpp
Erreur fréquente 
// ✅ Au démarrage : server_fd non-bloquant
int server_fd = socket(AF_INET, SOCK_STREAM, 0);
fcntl(server_fd, F_SETFL, fcntl(server_fd, F_GETFL) | O_NONBLOCK);

// Dans la boucle principale...
int client_fd = accept(server_fd, ...);

// ⚠️ ATTENTION : client_fd est BLOQUANT par défaut !
// accept() crée un NOUVEAU fd — il ne hérite PAS du mode non-bloquant
fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK);  // ← CRUCIAL
⚠️ L'erreur la plus fréquente : oublier de mettre le client_fd en non-bloquant après accept(). Le server_fd en non-bloquant ne suffit pas — chaque nouveau fd issu de accept() commence en mode bloquant.
03
Blocking → Non-blocking → epoll
Les trois états comparés
État 1 — Bloquant
Sans fcntl — comportement par défaut
✅ CPU = 0% en attente
❌ 1 client à la fois
❌ Programme suspendu indéfiniment
État 2 — Non-bloquant seul
fcntl sans epoll — busy-wait
✅ Ne gèle jamais
✅ Peut gérer N clients
❌ CPU = 100% en permanence
État 3 — fcntl + epoll
La solution finale
✅ Ne gèle jamais (fcntl)
✅ CPU ≈ 0% en attente (epoll)
✅ N clients simultanément
states_comparison.cpp
Visualisation 
// ─── État 1 : Bloquant ───────────────────────────────────────────────
while(1) {
    accept()   ← PROGRAMME GELÉ (CPU=0%, mais rien d'autre ne tourne)
                  ... client arrive ...
    recv()     ← PROGRAMME GELÉ À NOUVEAU
                  ... données arrivent ...
    send()
}

// ─── État 2 : Non-bloquant seul (busy-wait) ──────────────────────────
while(1) {
    accept()  → EAGAIN → continue   ← retourne immédiatement
    accept()  → EAGAIN → continue
    accept()  → client dispo !
    recv()    → EAGAIN → continue   ← boucle tourne en permanence
}                                   ← CPU = 100% même sans trafic

// ─── État 3 : fcntl + epoll ──────────────────────────────────────────
while(1) {
    epoll_wait()  ← DORT jusqu'à ce qu'un fd soit prêt (CPU≈0%)
                      fd prêt → on se réveille
    accept() ou recv()  ← garanti d'avoir quelque chose, EAGAIN rare
    send()
}
04
fcntl + epoll_wait
Complémentaires, pas alternatifs
Outil Problème résolu Sans lui
fcntl + O_NONBLOCK Empêche le gel du programme recv() suspend le programme indéfiniment
epoll_wait() Empêche le gaspillage CPU Boucle tourne pour rien (busy-wait)
💡 epoll dit quand agir. fcntl garantit que quand on agit, ça ne bloque jamais.
epoll_wait() te dit "ce fd est prêt" — mais si le fd est encore bloquant, recv() peut quand même bloquer dans des cas limites (données fragmentées, conditions de course). fcntl est le filet de sécurité absolu.
epoll_pattern.cpp
Pattern correct 
// Pattern correct : fcntl TOUJOURS, même avec epoll
int client_fd = accept(server_fd, ...);

fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK);  // ← toujours

struct epoll_event ev;
ev.events  = EPOLLIN;
ev.data.fd = client_fd;
epoll_ctl(epoll_fd, EPOLL_CTL_ADD, client_fd, &ev);

Version minimale avec fcntl seul

Code de base transformé avec fcntl — modifications minimales commentées. Note : ce code fait du busy-waiting. La prochaine étape = remplacer par epoll_wait().

minimal_nonblocking.cpp
Étape intermédiaire 
#include <fcntl.h>   // pour fcntl()
#include <errno.h>   // pour EAGAIN

// Au démarrage : server_fd non-bloquant
fcntl(server_fd, F_SETFL, fcntl(server_fd, F_GETFL) | O_NONBLOCK);

while (1) {
    int client_fd = accept(server_fd, ...);

    if (client_fd == -1) {
        if (errno == EAGAIN)
            continue;  // pas de client pour l'instant → reboucle
        break;         // vraie erreur
    }

    // CRUCIAL : client_fd est bloquant par défaut après accept()
    fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK);

    int n = recv(client_fd, buf, sizeof(buf) - 1, 0);
    if (n == -1 && errno == EAGAIN) {
        // Pas de données encore → dans un vrai serveur,
        // on mettrait ce client en attente via epoll
        close(client_fd);
        continue;
    }
    // traiter la requête...
}
// Note : ce code fait du busy-waiting (CPU 100%)
// La prochaine étape = remplacer par epoll_wait()
05
F_SETFD + FD_CLOEXEC
Le bonus pour le CGI
Le problème sans FD_CLOEXEC

Pour le CGI, on fait fork() + execve(). Sans FD_CLOEXEC, tous les file descriptors ouverts (sockets clients, pipes...) sont hérités par le processus enfant.

Résultat : fuite garantie sur tous tes sockets ouverts. Le processus CGI garde des références à des connexions qu'il n'a pas à toucher.

La solution

FD_CLOEXEC marque un fd pour qu'il soit fermé automatiquement lors d'un execve(). Il reste ouvert dans le processus parent, mais le fils n'en hérite pas.

Note : c'est F_SETFD (flag du fd lui-même) et non F_SETFL (flag de statut d'accès).

cgi_pipes.cpp
CGI 
// Sur les pipes du CGI — fermeture automatique lors de execve()
fcntl(pipe_fd, F_SETFD, FD_CLOEXEC);
// Quand execve() tourne dans le fils, ce fd se ferme automatiquement
// Le processus CGI ne voit jamais ce fd

Résumé des appels fcntl dans Webserv

fcntl_cheatsheet.cpp
Référence rapide 
// 1. Rendre un socket non-bloquant (server_fd ET chaque client_fd)

server.cpp — avant / après fcntl
Le code de socket-lifecycle, transformé

Voici le code complet issu de socket-lifecycle — serveur séquentiel de base — auquel on ajoute les appels fcntl() nécessaires pour le rendre non-bloquant. Le code original est intact ; les lignes vertes sont les ajouts.

1 — Création du socket → rendre non-bloquant immédiatement

server.cpp — create_server_socket()
+fcntl
code original
ajout fcntl
#include <sys/socket.h>
#include <arpa/inet.h>
#include <unistd.h>
#include <fcntl.h>    // ← nouveau : pour fcntl() et O_NONBLOCK
#include <cerrno>     // ← nouveau : pour errno et EAGAIN
#include <iostream>

static const int PORT    = 8080;
static const int BACKLOG = 10;

// ── Étape 1 : crée le socket TCP/IPv4 ──────────────────────────
int create_server_socket() {
    int fd = socket(AF_INET, SOCK_STREAM, 0);
    if (fd == -1) { perror("socket"); return -1; }
    int opt = 1;
    setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
    fcntl(fd, F_SETFL, fcntl(fd, F_GETFL) | O_NONBLOCK);  // ← server_fd non-bloquant
    return fd;
}

2 — accept() → rendre le client_fd non-bloquant

server.cpp — accept_next_client()
+fcntl
code original
ajout fcntl
// ── Étape 4 : accepte UN client de la file d'attente ──────────
int accept_next_client(int server_fd) {
    struct sockaddr_in client_addr;
    socklen_t len = sizeof(client_addr);
    int client_fd = accept(server_fd,
                           (struct sockaddr*)&client_addr, &len);
    if (client_fd == -1) {
        if (errno == EAGAIN) return -2;  // ← nouveau : pas d'erreur, juste rien à accepter
        perror("accept"); return -1;
    }
    std::cout << "[+] client fd=" << client_fd
              << " from " << inet_ntoa(client_addr.sin_addr) << "\n";

    // ← nouveau : client_fd ne hérite PAS du mode non-bloquant du server_fd
    fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK);
    return client_fd;
}

3 — handle_client() → gérer EAGAIN sur recv()

server.cpp — handle_client()
+EAGAIN
code original
ajout fcntl
// ── Étape 5 : lit la requête, envoie la réponse ───────────────
void handle_client(int client_fd) {
    char buf[4096];
    int  n;
    do {
        n = recv(client_fd, buf, sizeof(buf) - 1, 0);
    } while (n == -1 && errno == EAGAIN);  // ← retry jusqu'à avoir quelque chose
    if (n <= 0) { close(client_fd); return; }
    buf[n] = '\0';
    std::cout << buf;

    const char* response =
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: text/html\r\n"
        "Content-Length: 22\r\n"
        "Connection: close\r\n"
        "\r\n"
        "<h1>Hello World!</h1>\n";

    send(client_fd, response, strlen(response), 0);
    close(client_fd);
}

4 — main() → boucle non-bloquante avec gestion EAGAIN

server.cpp — main()
+fcntl / busy-wait
code original
ajout fcntl
// ── main : assemble les étapes ────────────────────────────────
int main() {
    int server_fd = create_server_socket();  // fcntl déjà appelé à l'intérieur
    if (server_fd == -1) return 1;

    if (bind_address(server_fd, PORT) == -1) return 1;
    if (listen(server_fd, BACKLOG) == -1) { perror("listen"); return 1; }

    std::cout << "Serveur sur http://localhost:" << PORT << "\n";

    // Boucle principale — non-bloquante, mais toujours séquentielle (busy-wait)
    // → accept() et recv() retournent EAGAIN au lieu de bloquer
    // → CPU = 100% même sans trafic (busy-wait) — prochaine étape : epoll_wait()
    while (true) {
        int client_fd = accept_next_client(server_fd);
        if (client_fd == -2) continue;  // ← EAGAIN — pas de client pour l'instant
        if (client_fd == -1) continue;
        handle_client(client_fd);
    }
    return 0;
}
⚠️ Ce code est une étape intermédiaire. Il ne gèle plus (fcntl ✅) mais tourne en permanence même sans clients (busy-wait ❌). La vraie solution finale remplace la boucle while(true) par epoll_wait() qui dort intelligemment jusqu'à ce qu'un fd soit prêt.
08
Termes & Définitions
Glossaire rapide
fd
Entier retourné par le kernel pour identifier une ressource ouverte (socket, fichier, pipe…). Toutes les opérations I/O passent par lui.
bloquant
L'appel suspend le processus jusqu'à obtenir une réponse. Mode par défaut de tous les sockets et fichiers.
non-bloquant
L'appel retourne immédiatement, avec EAGAIN si rien n'est disponible. Activé via fcntl(fd, F_SETFL, O_NONBLOCK).
EAGAIN
Code d'erreur signifiant "pas de données maintenant, réessaie plus tard". C'est normal — pas une vraie erreur, juste un état temporaire.
busy-wait
Boucle qui tourne sans s'arrêter pour vérifier en permanence si quelque chose est prêt. Gaspille 100% du CPU même sans trafic.
epoll
Mécanisme Linux qui dort jusqu'à ce qu'un fd soit prêt à lire/écrire. Remplace le busy-wait et utilise le CPU seulement quand nécessaire.
multiplexing
Gérer plusieurs connexions simultanées dans un seul thread. Possible grâce à la combinaison fcntl + epoll.
O_NONBLOCK
Flag de statut qui active le mode non-bloquant sur un fd. Un seul bit dans la structure interne du kernel.
F_GETFL / F_SETFL
Commandes fcntl pour lire/écrire les flags de statut d'accès d'un fd (O_RDONLY, O_WRONLY, O_NONBLOCK…).
FD_CLOEXEC
Flag qui ferme automatiquement un fd lors d'un execve(). Essentiel pour éviter les fuites de fd dans les processus CGI enfants.
En résumé
Les deux appels fcntl indispensables dans Webserv
🔓
fcntl(fd, F_SETFL, flags | O_NONBLOCK)
Rend un socket non-bloquant. À appliquer sur le server_fd ET sur chaque client_fd après accept(). Garantit que le programme ne gèle jamais.
🔒
fcntl(fd, F_SETFD, FD_CLOEXEC)
Protège les pipes CGI contre les fuites de fd. Le fd se ferme automatiquement dans le processus fils lors de execve().
⚡
EAGAIN = état normal
En mode non-bloquant, EAGAIN signifie "rien maintenant". Ce n'est pas une erreur — c'est le signal pour passer au client suivant ou attendre epoll.
🔁
fcntl + epoll = duo obligatoire
epoll dit quand agir (efficacité CPU). fcntl garantit que l'action ne bloque jamais (sécurité). Les deux sont nécessaires, ni l'un ni l'autre seul ne suffit.
