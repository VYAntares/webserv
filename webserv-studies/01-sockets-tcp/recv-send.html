Comment lire et écrire sur un socket TCP. La théorie d'abord — sans epoll — pour comprendre ce que ces fonctions font réellement. Ensuite, pourquoi dans Webserv on ne les appelle jamais sans passer par epoll_wait() avant.

01
recv()
Recevoir des données
Ce que ça fait

recv() lit des données arrivées sur un socket connecté et les copie dans ton buffer. C'est l'équivalent réseau de read() — en fait sur Linux, read(fd, buf, n) sur un socket fonctionne exactement pareil, mais recv() ajoute un paramètre flags qui permet des comportements avancés (MSG_PEEK, MSG_WAITALL, etc.).

Signature : recv(fd, buffer, taille_max, flags)
Pour un usage standard : flags = 0.

📬
Analogie

recv() c'est aller vider ta boîte aux lettres. Tu dis combien de lettres tu peux porter (taille_max). Si la boîte est vide, tu attends devant (mode bloquant), ou tu rentres bredouille immédiatement (mode non-bloquant).

recv_basic.cpp
Théorie — sans epoll 
char    buffer[4096];
ssize_t n = recv(client_fd, buffer, sizeof(buffer), 0);

if (n > 0) {
    // n octets reçus — buffer[0..n-1] contient les données
    process(buffer, n);
}
else if (n == 0) {
    // Le client a fermé la connexion proprement (TCP FIN reçu)
    close(client_fd);
}
else { // n == -1
    if (errno == EAGAIN || errno == EWOULDBLOCK) {
        // Socket non-bloquant : pas de données disponibles, réessayer plus tard
    } else {
        // Vraie erreur réseau — fermer le fd
        close(client_fd);
    }
}
Valeur retournéeSignificationAction
> 0 Nombre d'octets effectivement reçus Traiter les données dans buffer[0..n-1]
== 0 Le client a fermé la connexion (TCP FIN) Fermer le fd, libérer les ressources
-1 + EAGAIN Pas de données disponibles (socket non-bloquant) Rien à faire — epoll_wait() préviendra quand ça reviendra
-1 + autre errno Erreur réseau réelle (connexion reset, etc.) Fermer le fd, logger l'erreur
📏recv() ne garantit pas de lire exactement n octets. Il peut en lire moins — c'est le piège principal. On détaille ça en section 3.
🔗read(fd, buf, n) et recv(fd, buf, n, 0) sont équivalents sur un socket Linux. recv() est préféré par convention réseau.
🚩MSG_PEEK : lit sans consommer les données (elles restent dans le buffer kernel). Utile pour inspecter avant de décider.

02
send()
Envoyer des données
Ce que ça fait

send() copie tes données dans le buffer d'envoi du kernel et retourne immédiatement — il ne attend pas que les données arrivent chez le destinataire. Le kernel gère l'envoi TCP en arrière-plan (segmentation, retransmission, etc.).

Signature : send(fd, buffer, taille, flags)
Pour un usage standard : flags = 0.

Le problème : send() peut envoyer moins que demandé si le buffer kernel est plein. Il retourne le nombre d'octets réellement acceptés — pas forcément tous.

📮
Analogie

Tu déposes une lettre à la poste. Dès que c'est déposé, send() retourne. La lettre n'est pas encore chez le destinataire — elle est dans le camion postal (buffer kernel). Si la poste est pleine (EAGAIN), elle te dit de revenir plus tard.

send_loop.cpp
Envoi complet 
// MAUVAIS — envoie peut-être seulement une partie
send(client_fd, response, response_len, 0);

// BON — boucle jusqu'à tout avoir envoyé
size_t  total = 0;
while (total < response_len) {
    ssize_t sent = send(client_fd, response + total, response_len - total, 0);
    if (sent == -1) {
        if (errno == EAGAIN) break; // buffer plein — reprendre plus tard via epoll EPOLLOUT
        break; // vraie erreur
    }
    total += sent;
}
Valeur retournéeSignificationAction
> 0 Nombre d'octets acceptés dans le buffer kernel Peut être inférieur à ce qu'on voulait — continuer la boucle
-1 + EAGAIN Buffer d'envoi kernel plein (socket non-bloquant) Attendre EPOLLOUT via epoll_wait() avant de réessayer
-1 + EPIPE / autre Le client a fermé sa réception, ou erreur réseau Fermer le fd — ne pas oublier de gérer SIGPIPE
⚠️
SIGPIPE — le crash silencieux : si tu fais send() sur un socket dont l'autre côté est fermé, le kernel envoie le signal SIGPIPE à ton processus — qui le tue par défaut. Solution : ignorer ce signal globalement avec signal(SIGPIPE, SIG_IGN) au démarrage, et gérer l'erreur EPIPE retournée par send().

03
Le vrai piège
TCP ne livre pas des messages, il livre des octets
Le problème fondamental

TCP est un protocole de flux d'octets — pas de messages. Quand le client envoie "GET / HTTP/1.1\r\nHost: ...\r\n\r\n", ton serveur peut recevoir ça en plusieurs morceaux séparés dans le temps, ou au contraire plusieurs requêtes collées ensemble dans un seul recv().

Le kernel accumule les octets entrants dans un buffer. recv() te donne ce qui est disponible à l'instant — jamais plus que taille_max, parfois beaucoup moins. Il n'y a aucune garantie de frontière entre requêtes.

🚰
Analogie

TCP c'est un tuyau d'eau, pas un service de colis. Tu verses 500 octets d'un côté — de l'autre, tu peux recevoir 100 octets, puis 200, puis 200. Ou tout d'un coup. Ou mélangés avec les 500 octets du message suivant. À toi de reconstituer.

read_loop.cpp
Boucle d'accumulation 
// Accumulate until we have a complete HTTP request (\r\n\r\n)
std::string raw_request;
char        buf[4096];

while (true) {
    ssize_t n = recv(client_fd, buf, sizeof(buf), 0);
    if (n <= 0) break;

    raw_request.append(buf, n);  // accumuler dans un string/buffer

    // Chercher la fin des headers HTTP
    if (raw_request.find("\r\n\r\n") != std::string::npos)
        break;  // on a une requête complète
}
// Ensuite parser raw_request — headers + body séparément
⚠️
Le cas du body HTTP : les headers se terminent par \r\n\r\n, mais le body suit immédiatement. Sa taille est dans l'header Content-Length. Il faut donc deux phases : lire jusqu'à \r\n\r\n, parser Content-Length, puis continuer à lire exactement Content-Length octets de body. Si Content-Length est absent sur un POST : body vide.
📦Buffer par client. Chaque connexion a son propre buffer d'accumulation. Dans Webserv, c'est typiquement un std::string ou un vecteur associé au fd client.
🔢Taille du buffer. 4096 ou 8192 octets par appel recv() est standard. Plus grand = moins d'appels syscall, mais plus de mémoire stack. À garder en heap pour de gros transferts.
🔄Pipelining HTTP. Un client peut envoyer plusieurs requêtes sans attendre les réponses. Ton buffer peut contenir le début d'une 2e requête après la fin de la 1ère.
📏Chunked Transfer Encoding. Si pas de Content-Length, le body peut être en chunks. Chaque chunk précède sa taille en hexa. Cas plus complexe à parser.
✅
La bonne architecture Webserv : chaque client a un objet/struct avec son fd, son buffer d'entrée (string qui s'accumule), et son buffer de sortie (réponse à envoyer). epoll_wait() signale quand lire (EPOLLIN) ou quand écrire (EPOLLOUT). On ne fait jamais qu'un seul recv() par événement epoll — on accumule et on traite quand la requête est complète.
04
close() / shutdown()
Fermer une connexion proprement
close(fd)

Ferme le file descriptor et libère toutes les ressources associées. Le kernel envoie un TCP FIN au client — signalant la fin de la connexion.

Si le fd est dupliqué (via dup() ou fork()), la connexion TCP reste ouverte tant qu'il reste au moins un fd pointant dessus. close() ne ferme la connexion que quand le dernier fd est fermé.

Dans Webserv : appeler close(client_fd) puis retirer ce fd de epoll via epoll_ctl(epfd, EPOLL_CTL_DEL, client_fd, NULL).

shutdown(fd, how)

Plus fin que close() : permet de fermer un seul sens de la connexion tout en gardant le fd ouvert.

SHUT_RD — plus de lecture (le recv() retournera 0 désormais)
SHUT_WR — plus d'écriture (envoie un TCP FIN au client)
SHUT_RDWR — les deux (équivalent à close() sans libérer le fd)

Cas d'usage Webserv : après avoir envoyé la réponse complète, faire shutdown(fd, SHUT_WR) signale proprement au client que c'est terminé, sans avoir à close() immédiatement.

close_connection.cpp
Pattern Webserv 
void close_client(int epfd, int client_fd) {
    // 1. Retirer du epoll AVANT de fermer (sinon comportement indéfini)
    epoll_ctl(epfd, EPOLL_CTL_DEL, client_fd, NULL);

    // 2. Optionnel : signaler la fin d'envoi proprement
    shutdown(client_fd, SHUT_WR);

    // 3. Libérer le fd
    close(client_fd);
    // 4. Libérer les buffers associés à ce client dans ta structure
}

// Détection de déconnexion côté recv :
ssize_t n = recv(client_fd, buf, sizeof(buf), 0);
if (n == 0)
    close_client(epfd, client_fd);  // client déconnecté
🔢Fuite de file descriptors. Chaque accept() crée un fd. Si tu ne fais jamais close(), le kernel manque de fds (limite ~1024 par défaut). Le serveur plantera en silence.
🏁EPOLLHUP. epoll signale automatiquement EPOLLHUP quand le client se déconnecte. Il faut le détecter et appeler close_client() — même si on n'a pas appelé recv().
⚡TIME_WAIT. Après close(), le port client reste en TIME_WAIT ~60s côté kernel. Pour le serveur ce n'est pas un problème — c'est le port client du navigateur qui attend, pas le tien.
05
La règle absolue
Jamais recv/send sans epoll_wait() avant
Pourquoi c'est une règle du sujet

En théorie, rien n'empêche d'appeler recv() directement. Mais le sujet webserv est catégorique : toute lecture ou écriture sur un socket DOIT être précédée d'une vérification via epoll_wait(). Violer cette règle = note de 0.

La raison technique : si tu appelles recv() sur un socket bloquant sans avoir vérifié qu'il y a des données, ton serveur se fige — et tous les autres clients attendent. Sur un socket non-bloquant, ça retourne EAGAIN, ce qui est une erreur à gérer — du code inutile.

La bonne séquence

1. epoll_wait() bloque jusqu'à ce qu'un fd soit prêt.
2. Pour chaque événement retourné :
  → EPOLLIN sur server_fd → accept()
  → EPOLLIN sur client_fd → recv() garanti non-bloquant
  → EPOLLOUT sur client_fd → send() garanti de la place
  → EPOLLHUP / EPOLLERR → close()

Tu n'appelles recv() que sur un fd dont epoll t'a confirmé qu'il a des données. Aucun risque de blocage. Jamais.

⚠️ Règle du sujet — Note de 0 si non respectée

« You must never do a read or a write operation without going through poll() (or equivalent). Calling read/recv or write/send on these descriptors without prior readiness check will result in a grade of 0. Regular disk files are exempt. »

Traduction pratique : chaque recv() ou send() dans ton code doit être déclenché par un événement epoll_wait() — pas appelé "à la main" dans une boucle simple. Les seules exceptions sont les fichiers disque (read() sur un fd de fichier, pas de socket).

Récap — l'essentiel
Ce que tu dois savoir avant d'écrire la moindre ligne de Webserv
📬
recv(fd, buf, n, 0)
Retourne > 0 (données), 0 (client déconnecté), -1 EAGAIN (rien dispo), -1 erreur. Vérifier chaque cas.
📮
send(fd, buf, n, 0)
Peut envoyer moins que n. Toujours boucler. Gérer EAGAIN (buffer plein) et EPIPE (client fermé). Ignorer SIGPIPE.
🚰
TCP = flux d'octets
Les données arrivent fragmentées. Accumuler dans un buffer par client. Chercher \r\n\r\n puis Content-Length octets de body.
🔒
close() / shutdown()
epoll_ctl DEL → shutdown(SHUT_WR) → close(). Toujours dans cet ordre. Ne jamais laisser un fd fuiter.
