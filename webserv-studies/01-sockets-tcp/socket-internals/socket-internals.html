socket() internals — Kernel deep dive

Sous le capot
de socket()

Ce qui se passe vraiment dans le kernel quand un file descriptor est créé. Des trois couches de structures jusqu'aux outils pour tout observer en temps réel.

01
file descriptor
Juste un entier
Ce que c'est vraiment

Un file descriptor est un simple entier — 3, 4, 5… C'est un index dans une table privée au processus. C'est tout ce que le programme voit.

Quand socket() retourne 3, c'est la même chose que quand open("file.txt") retourne 3.

Unix traite tout de la même façon : fichiers, sockets, pipes, terminaux — même interface, même fd. C'est le principe "everything is a file".

🎫
Analogie

Le fd est un ticket de vestiaire. Le numéro 3 est sur ton ticket. Derrière le comptoir (dans le kernel), il y a le vrai manteau. Toi tu gardes juste le ticket — léger, simple, opaque.

📌Les fd 0, 1, 2 sont toujours stdin, stdout, stderr — réservés dès le démarrage du processus.
🔗Un même fd peut être dupliqué avec dup2() — deux entiers différents pointent vers le même objet kernel.
🔒Le fd ne contient aucune donnée réseau. C'est un passeport, pas la destination.
02
Où se situe le fd ?
Les trois niveaux de structures

Il y a trois couches distinctes, de ton processus jusqu'au kernel. Le fd 3 n'est que la surface visible — l'iceberg va beaucoup plus bas.

USER SPACE · ton processus
Table des fd — privée au processus, en mémoire user
fd → open file description
0 stdin  (/dev/pts/4)
1 stdout (/dev/pts/4)
2 stderr  (/dev/pts/4)
3 open file description #N →
▼
syscall — franchit la frontière kernel
KERNEL SPACE · mémoire protégée ring 0
Table des open file descriptions partageable via dup()
type=SOCK_STREAM
flags=O_RDWR
*inode=→ inode #49051 ↓
▼
VFS · Table des inodes
inode socket [49051]
famille=AF_INET
type=SOCK_STREAM
*sock=→ struct socket ↓
▼
struct socket Linux kernel — RAM ring 0
state=SS_UNCONNECTEDavant bind()
proto=tcp_prot
sk_rcvbuf= buffer réception · vide
sk_sndbuf= buffer envoi    · vide
sock_addr=NULLavant bind()
🧱Le fd 3 est en user space — sur la stack de ta fonction, ou dans un registre CPU.
🔐La struct socket est en kernel space — mémoire protégée ring 0. Ton processus ne peut pas y accéder directement.
🛂Le fd est le passeport pour traverser la frontière user/kernel via un syscall.
🔀La "open file description" est partageable — deux fd (même process ou fork) peuvent pointer vers la même entrée.
03
La vie d'un syscall
Ce qui se passe exactement lors de socket()
kernel — chemin d'exécution
syscall x86_64 #41
// Ton code — user space
int fd = socket(AF_INET, SOCK_STREAM, 0);
         │
         │  instruction CPU : syscall  (numéro 41 sur x86_64)
         │  ↓ bascule en ring 0 — le kernel prend la main
         ▼
kernel:  sys_socket()
             ├─ sock_create()        alloue struct socket en RAM kernel
             ├─ inet_create()        initialise pour AF_INET / TCP
             ├─ tcp_v4_init_sock()   initialise buffers send / recv
             └─ sock_map_fd()        crée l'entrée dans la fd table du processus
                     │
                     │  retour en ring 3 — ton processus reprend
                     ▼
user:    fd = 3   // tu as juste ça. Tout le reste vit dans le kernel.
Avant bind()

La struct socket est créée, les buffers existent, mais sock_addr = NULL.

Personne ne peut envoyer de paquets à ce socket — il n'a pas d'adresse IP ni de port. C'est un point de communication sans localisation.

Coût réel du syscall

Chaque socket() implique un context switch complet ring 3 → ring 0 → ring 3. C'est ~100–300 ns.

Sur un serveur qui crée des milliers de connexions courtes, ce coût s'accumule. C'est une des raisons pour lesquelles les connexions persistantes (HTTP keep-alive) existent.

💡
strace peut tout voir. strace -e trace=socket ./webserv intercepte chaque appel système et affiche ses arguments et sa valeur de retour. Tu verras exactement quand le kernel crée chaque fd.
⚡Le numéro du syscall socket() est 41 sur x86_64 Linux — visible avec strace -e raw=socket.
🔄Le context switch ring 3 → ring 0 sauvegarde tous les registres CPU avant d'exécuter le code kernel.
📦tcp_v4_init_sock() alloue les buffers send/recv — par défaut ~87 KB chacun sous Linux.
04
Observer depuis le terminal
Tout est visible dans /proc

Linux expose l'intégralité de l'état kernel dans le pseudo-filesystem /proc. Tu peux voir la table des fd, les états TCP, les buffers — tout, en temps réel, sans root.

/proc/<PID>/fd/
table des fd du processus
ls -la /proc/5358/fd/

# Résultat :
0 -> /dev/pts/4          (stdin)
1 -> /dev/pts/4          (stdout)
2 -> /dev/pts/4          (stderr)
3 -> socket:[49051]      ← ton socket, inode 49051
🔍
Le fd 3 est un symlink vers socket:[49051]. Le numéro 49051 est l'inode du socket dans le VFS kernel. C'est ce qui relie ta fd table à l'entrée dans /proc/net/tcp.
/proc/net/tcp
état réseau de tous les sockets TCP
cat /proc/net/tcp

#  sl  local_address  rem_address  st  tx_queue rx_queue  inode
   3: 00000000:1F90  00000000:0000  0A  00000000:00000000  49051

─────────── Décodage (tout est en hexadécimal, big-endian inversé) ───────────

local_address : 00000000:1F90
  00000000  →  0.0.0.0   (toutes les interfaces)
  1F90      →  0x1F90 = 8080 en décimal  ← le port

rem_address   : 00000000:0000
  Personne connecté — normal en état LISTEN

st            : 0A
  0A = 10 = TCP_LISTEN  ← socket en écoute

tx_queue:rx_queue : 00000000:00000000
  Buffers send et receive tous les deux vides

inode         : 49051
  Correspond à socket:[49051] dans /proc/<PID>/fd/3

Table complète des états TCP (colonne st)

Hex Nom Signification
01ESTABLISHEDConnexion active — données qui circulent
02SYN_SENTClient a envoyé SYN, attend SYN-ACK
03SYN_RECVServeur a reçu SYN, a envoyé SYN-ACK, attend ACK
04FIN_WAIT1Fermeture initiée, FIN envoyé
05FIN_WAIT2Notre FIN ACKé, attend FIN de l'autre côté
06TIME_WAITAttend expiration 2×MSL avant de libérer le port
07CLOSEFermé
08CLOSE_WAITL'autre a fermé, notre app n'a pas encore appelé close()
09LAST_ACKAttend le dernier ACK après avoir envoyé FIN
0ALISTENEn écoute — c'est l'état après listen()
0BCLOSINGLes deux côtés envoient FIN simultanément
ss — vue synthétique
socket statistics
# Vue lisible des sockets en écoute
ss -tlnp | grep webserv
# LISTEN 0  10  0.0.0.0:8080  0.0.0.0:*  users:(("webserv",pid=5358,fd=3))

# Stats TCP internes : RTT, retransmissions, congestion window
ss -tlnp -e -i | grep -A3 webserv

# Vue haut niveau de tous les fd ouverts (plus lisible que /proc)
lsof -p <PID>
🗂️/proc/<PID>/fd/ : table des fd du processus — chaque socket est un symlink vers son inode.
📊/proc/net/tcp : tout l'état TCP en hex — adresses, ports, états, buffers, inodes.
🔢Les adresses sont en big-endian inversé — 0x1F90 = port 8080, 0x7F000001 = 127.0.0.1.
05
Qu'est-ce qui est dans le fd ?
Le contenu réel de la struct socket
⚠️
Le fd 3 est un symlink. Il ne "contient" rien de lisible. Un cat /proc/5358/fd/3 bloque — cat fait read() sur le socket, et un socket LISTEN n'a rien à lire tant qu'aucun client n'envoie de données.
linux/net/sock.h — struct socket (simplifié)
kernel source
struct socket {
    // État de la connexion — évolue avec chaque syscall
    socket_state   state;     // SS_UNCONNECTED → SS_CONNECTED (après accept)
    struct proto  *proto;     // tcp_prot — pointeur vers l'impl TCP

    // Buffers — le vrai travail se passe ici
    int            sk_rcvbuf;  // buffer réception  (~87 KB par défaut)
    int            sk_sndbuf;  // buffer envoi      (~87 KB par défaut)

    // Adresse — NULL avant bind()
    struct sockaddr sock_addr; // { 0.0.0.0, port 8080 } après bind()

    // File d'attente des connexions — remplie par listen()
    struct request_sock_queue accept_queue;
}
Les buffers en action

Quand un client fait curl http://localhost:8080 mais que ton serveur n'a pas encore appelé recv() :

1. Les données de curl arrivent par le réseau
2. Le kernel les stocke dans sk_rcvbuf
3. tx_queue dans /proc/net/tcp augmente
4. Quand tu fais recv(), le kernel copie de sk_rcvbuf vers ton buffer user-space

L'accept_queue

Après listen(backlog), le kernel maintient une file d'attente de connexions TCP établies mais pas encore récupérées par ton accept().

Si la file est pleine (backlog atteint), le kernel rejette silencieusement les nouvelles connexions avec ECONNREFUSED côté client. Ou les ignore, selon le paramètre tcp_abort_on_overflow.

📥Le sk_rcvbuf est ajustable avec setsockopt(SO_RCVBUF) — utile pour des transferts à haut débit.
🔄recv() ne fait pas de I/O réseau direct — il copie depuis le buffer kernel vers user space.
📋L'accept_queue a deux niveaux : SYN queue (connexions en cours) et accept queue (connexions établies).
06
Observer en temps réel
Voir une connexion naître et mourir

Avec trois terminaux, tu peux observer l'intégralité du cycle de vie d'une connexion TCP en temps réel — depuis le syscall accept() jusqu'à la fermeture du fd client.

Terminal 1 — surveiller les syscalls
strace
strace -p <PID> -e trace=accept,recv,send,read,write,close

─── Quand un client arrive, tu verras :

accept(3, {sa_family=AF_INET, sin_port=htons(54321), sin_addr=inet_addr("127.0.0.1")}, [16]) = 4
recv(4, "GET / HTTP/1.1\r\nHost: localhost...", 1024, 0) = 78
send(4, "HTTP/1.1 200 OK\r\n...", 150, 0)        = 150
close(4)                                            = 0
Terminal 2 — états TCP en direct
watch
watch -n 0.1 'cat /proc/net/tcp'

─── Une connexion passe par :
    SYN_RECV (03)  →  ESTABLISHED (01)  →  TIME_WAIT (06)  →  disparaît

─── Ou avec ss pour plus de lisibilité :
watch -n 0.1 'ss -tnp | grep webserv'
Terminal 3 — envoyer la requête
curl
curl -v http://localhost:8080/

─── -v montre les headers HTTP et le handshake TLS si HTTPS
─── --trace-ascii - montre chaque octet échangé
👁️strace a un coût en perfs — ne jamais laisser activé en prod. En développement c'est indispensable.
⏱️TIME_WAIT dure 2 × MSL (≈ 60 secondes par défaut). C'est pourquoi on utilise SO_REUSEADDR.
🔢Note que accept() retourne 4 — le serveur_fd reste 3, et chaque client obtient un nouveau fd.
07
fd leak
Le bug silencieux qui tue ton serveur
🚨
Piège classique en évaluation Webserv. Chaque accept() crée un nouveau client_fd (4, 5, 6, 7…). Si tu ne fais pas close(client_fd) après avoir traité la requête — fuite de fd. Ton serveur crashe après ~1021 connexions.
diagnostic du fd leak
debug
# Voir la limite par processus
ulimit -n          ← souvent 1024

# Vérifier combien de fd ton serveur a ouverts en ce moment
ls /proc/<PID>/fd/ | wc -l

# Si ce nombre grimpe sans descendre → fd leak confirmé
watch -n 1 'ls /proc/<PID>/fd/ | wc -l'

# Augmenter la limite pour tester sans crasher (session courante)
ulimit -n 65536

# L'erreur que tu verras si la limite est atteinte :
# accept() = -1  EMFILE (Too many open files)
Code qui leak

int client_fd = accept(server_fd, ...);
handle_request(client_fd);
// ← oubli de close(client_fd) !
// fd 4 jamais libéré

Code correct

int client_fd = accept(server_fd, ...);
handle_request(client_fd);
close(client_fd);  ← toujours

🔢Linux alloue les fd en ordre croissant, en réutilisant les plus petits disponibles. Le premier fd libre après 0/1/2 est toujours alloué.
💀EMFILE = Too many open files — le processus a atteint sa limite. ENFILE = la limite système globale est atteinte.
🛡️En évaluation, utilise valgrind --track-fds=yes pour détecter tous les fd non fermés à la fin du programme.
