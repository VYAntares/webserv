accept()
Un appel, un nouveau fd

struct sockaddr_in client_addr;
socklen_t client_len = sizeof(client_addr);

int client_fd = accept(server_fd,
                       (struct sockaddr*)&client_addr,

💡
client_addr est rempli par le kernel avec l'IP et le port du client — sans que tu ne les demandes explicitement. Tu peux passer NULL si tu n'as pas besoin de ces infos, le kernel remplit quand même la connexion.
🔗La connexion TCP est déjà établie quand accept() retourne — le handshake s'est fait pendant listen(), pas maintenant.
📦accept() ne crée pas la connexion — il la récupère dans la file. Le kernel a déjà fait le travail.

✕
recv() sur server_fd → erreur. Le server_fd est en mode LISTEN — il n'a pas de données à lire, il est fait pour accept() uniquement.
✕
Oublier close(client_fd) → fd leak. Le serveur plante après ~1021 clients avec EMFILE: Too many open files.
🏗️server_fd est la porte d'entrée permanente. client_fd est le guichet temporaire ouvert pour chaque client.
🔢Les client_fd sont alloués en ordre croissant — 4, 5, 6… chaque accept() consomme un fd jusqu'à close().
03
Pourquoi accept() bloque
Le kernel endort ton processus
File NON vide — retour immédiat
1
Tu appelles accept(server_fd, ...)
2
Le kernel trouve un client dans la file
3
Retourne client_fd immédiatement — pas de blocage
File VIDE — processus endormi
1
Tu appelles accept(server_fd, ...)
2
Aucun client en attente — le kernel met le processus en état SLEEPING
3
Ton code ne s'exécute plus. Le CPU fait autre chose
4
Un client arrive → kernel réveille le processus → accept() retourne
le vrai problème — serveur séquentiel
deadlock partiel
// Situation avec 3 clients :
// client 1 → attend ta réponse
// client 2 → en train d'envoyer ses données (lentement)
// client 3 → connecté, n'a pas encore envoyé

// Ton code séquentiel :
while (1) {
    client_fd = accept(server_fd, ...);  ← bloqué ici, attend un 4e client
    recv(client_fd, ...);               ← ou bloqué ici si client envoie lentement
    send(client_fd, ...);
    close(client_fd);
}
// clients 1, 2, 3 → personne n'est traité pendant ce temps
⚠️
recv() bloque aussi. Si le client n'a pas encore envoyé ses données, recv() gèle ton processus exactement comme accept(). Chaque appel réseau est potentiellement bloquant. Un serveur séquentiel ne traite qu'un seul client à la fois.
😴Le blocage place le processus en état SLEEPING dans le scheduler Linux — il consomme zéro CPU pendant ce temps.
⚡Le kernel réveille le processus via une interruption quand le paquet réseau arrive — pas de polling actif.
04
La solution : epoll()
Un seul point de blocage, N fd surveillés

Au lieu de bloquer sur chaque appel réseau séparément, epoll_wait() bloque une seule fois et te dit exactement quels fd sont prêts — sans jamais bloquer sur accept() ou recv().

Sans epoll()
accept() → bloque si pas de client
recv() → bloque si pas de données
send() → bloque si buffer plein
blocages aléatoires partout
un seul client traité à la fois
Avec epoll()
epoll_wait(epfd, events, …)
→ bloque ici une seule fois
sur TOUS les fd enregistrés
accept() → jamais bloquant
recv() → jamais bloquant
un seul point de blocage contrôlé
N clients traités simultanément
server.cpp — boucle avec epoll()
non-bloquant
int epfd = epoll_create1(0);          // crée l'instance epoll
epoll_ctl(epfd, EPOLL_CTL_ADD, server_fd, &ev);  // enregistre server_fd

while (1) {
    int n = epoll_wait(epfd, events, MAX_EV, -1);  // seul point de blocage
    for (int i = 0; i < n; i++) {
        if (events[i].data.fd == server_fd)
            accept(server_fd, ...);   // ← jamais bloquant
        else
            recv(events[i].data.fd, ...);  // ← jamais bloquant
    }
}
💡
epoll() sera détaillé dans la cartographie. L'essentiel ici : epoll_wait() est le seul endroit où le programme se met en attente. Tous les accept() et recv() qui suivent opèrent sur des fd déjà prêts — zéro blocage.
🎯epoll_wait() répond à "qui est prêt ?" — accept() et recv() appelés après ne bloquent jamais.
⚡epoll est O(1) — il notifie uniquement les fd actifs, sans scanner toute la liste. Idéal pour N connexions.
Récapitulatif accept()
Ce que chaque concept représente vraiment
📥
accept()
Extrait la première connexion de la file. La connexion est déjà établie — le kernel a fait le handshake.
🚪
server_fd
Porte d'entrée permanente. LISTEN. Uniquement pour accept(). Un seul pour tout le serveur.
🎫
client_fd
Guichet temporaire. ESTABLISHED. Pour recv()/send(). Fermé après la requête — toujours close().
😴
Blocking
File vide → processus SLEEPING jusqu'à l'arrivée d'un client. Zéro CPU consommé.
🎯
epoll_wait()
Seul point de blocage. Retourne la liste des fd prêts — accept() et recv() qui suivent ne bloquent jamais.
⚡
epoll O(1)
Notifie uniquement les fd actifs, sans scanner toute la liste. Détaillé dans la cartographie.
