01
sockaddr_in
16 octets en mémoire — pas plus

Avant d'appeler bind(), tu remplis une struct. C'est juste un bloc de 16 octets contigus en RAM. Pas plus, pas moins.

server.cpp
remplir sockaddr_in
struct sockaddr_in addr;
memset(&addr, 0, sizeof(addr));       // met les 16 octets à zéro
addr.sin_family      = AF_INET;       // protocole IPv4
addr.sin_port        = htons(8080);   // port en network byte order
addr.sin_addr.s_addr = INADDR_ANY;    // 0.0.0.0 — toutes les interfaces
struct sockaddr_in en RAM
16 octets · layout mémoire

⚠️
Pourquoi memset est crucial : sans lui, sin_zero contient ce qui traîne dans la stack — des bytes aléatoires. Le kernel peut rejeter le bind() ou se comporter bizarrement. memset(&addr, 0, sizeof(addr)) garantit que tout est propre avant de remplir les champs. C'est aussi ce qui met INADDR_ANY à zéro automatiquement.
📐16 octets exactement — c'est la taille que le kernel attend. sizeof(addr) doit être passé à bind().
🧹memset avant tout — même si tu penses remplir tous les champs, le padding sin_zero doit être à zéro.
🔄Le cast (struct sockaddr*) dans bind() est nécessaire — l'API accepte différents types de sockaddr via ce cast générique.
02
htons()
Le problème des octets

Ton CPU (x86/x86_64) est little-endian. Le réseau TCP/IP est big-endian (appelé network byte order). Ce sont deux conventions opposées pour stocker un nombre multi-octets en mémoire.

✕  Little-endian — ton CPU x86
Port 8080 = 0x1F90 stocké en mémoire :
0x90
0x1F
octet faible en premier
réseau lit [90][1F] = port 36895 ✕
✓  Big-endian — réseau TCP/IP
Port 8080 = 0x1F90 après htons() :
0x1F
0x90
octet fort en premier
réseau lit [1F][90] = port 8080 ✓
ce que fait htons()
byte swap
htons(8080)
  entrée  :  0x1F90  (little-endian en mémoire : [90][1F])
  swap    :  inverse les 2 octets
  sortie  :  0x901F  (stocké en mémoire : [1F][90])
  lecture réseau (big-endian) : [1F][90] = 0x1F90 = 8080 ✓

── Sur un CPU big-endian, htons() ne fait RIEN ──────────────────
   Les octets sont déjà dans le bon ordre. La fonction s'adapte
   automatiquement à l'architecture. C'est pour ça qu'on l'utilise
   toujours, même si on ne sait pas sur quel CPU ça tourne.
Fonction Direction Taille Usage typique
htons() host → network 16 bits port dans sockaddr_in
htonl() host → network 32 bits adresse IP dans sockaddr_in
ntohs() network → host 16 bits lire un port reçu
ntohl() network → host 32 bits lire une IP reçue
🔍
Vérification dans /proc/net/tcp : le port y est affiché en big-endian brut. 1F90 = 8080. C'est exactement ce que htons() a produit et mis dans sin_port.
🔡host to network short — short = 16 bits = la taille d'un port TCP.
🔄Sur big-endian (SPARC, PowerPC…) htons() est un no-op. Mais on l'écrit toujours pour la portabilité.
🐛Oublier htons() est un bug silencieux — ton serveur bind sur le bon fd mais personne ne peut s'y connecter au bon port.
03
INADDR_ANY
Écouter sur toutes les interfaces

Ta machine a plusieurs interfaces réseau simultanément. Sur quelle IP tu veux écouter ?

IP précise — restrictif

addr.sin_addr.s_addr = inet_addr("127.0.0.1");

→ Ton serveur répond uniquement aux connexions depuis la machine locale.

curl localhost:8080  ✓
curl 192.168.2.2:8080  ✕ connexion refusée

INADDR_ANY — ouvert

addr.sin_addr.s_addr = INADDR_ANY;

→ Ton serveur répond sur toutes les interfaces.

curl localhost:8080  ✓
curl 192.168.2.2:8080  ✓
curl [depuis le réseau]  ✓

💡
INADDR_ANY vaut 0 — c'est pourquoi le memset suffit à le setter. sin_addr.s_addr est déjà à zéro après memset(&addr, 0, sizeof(addr)). La ligne = INADDR_ANY est techniquement redondante, mais elle documente l'intention.
🌐Pour Webserv : utilise INADDR_ANY. Un serveur HTTP doit être accessible depuis le réseau, pas seulement en local.
🔢INADDR_ANY = 0x00000000 — stocké tel quel dans sin_addr.s_addr, pas de htonl() nécessaire car 0 est 0 dans tous les endianness.
🔍Dans /proc/net/tcp, tu verras local_address: 00000000:1F90 — les 8 zéros = INADDR_ANY.
04
Sous le capot du syscall
Les 6 étapes de bind() dans le kernel
server.cpp
bind()
bind(server_fd, (struct sockaddr*)&addr, sizeof(addr));
                      ↑                        ↑
                  pointeur user space       toujours 16
1
syscall — bascule en ring 0
Ton processus émet l'instruction syscall. Le CPU bascule en ring 0. Le kernel reçoit : le fd, un pointeur vers ta struct en user space, et la taille (16).
2
validation du fd
Le kernel vérifie que le fd existe dans la table du processus, qu'il pointe vers un socket, et qu'il n'est pas déjà bindé. Sinon → EINVAL.
3
copy_from_user() — copie sécurisée
Le kernel ne fait jamais confiance à un pointeur user space. Il copie tes 16 octets dans sa propre mémoire protégée via copy_from_user(). Si le pointeur est invalide → EFAULT.
4
inet_bind() — vérifie la disponibilité du port
La fonction spécifique à AF_INET vérifie que le port n'est pas déjà utilisé par un autre processus. C'est ici que EADDRINUSE est déclenché si le port est pris — typiquement à cause de TIME_WAIT.
5
mise à jour de la struct socket
Le kernel met à jour la struct socket en mémoire kernel :
sk_num = 8080  ·  sk_rcv_saddr = 0.0.0.0
state reste SS_UNCONNECTED — c'est listen() qui le changera.
6
enregistrement dans la hash table TCP
Le kernel maintient une table de hashage indexée par port. Désormais, quand un paquet TCP arrive sur le port 8080, le kernel sait immédiatement à quelle struct socket le livrer. C'est ça le vrai effet de bind().
📋copy_from_user() protège le kernel — un pointeur user space malformé ne peut pas corrompre la mémoire kernel.
🗃️La hash table TCP est la raison pour laquelle bind() est "rapide" — c'est une lookup O(1) pour router les paquets entrants.
🔗Après bind(), le socket a un port mais n'accepte pas encore de connexions — il faut listen() ensuite.
05
SO_REUSEADDR
Pourquoi "Address already in use" et comment l'éviter
🔄
Le scénario classique : tu lances webserv → bind sur le port 8080 → OK. Un client se connecte. Tu tues webserv (Ctrl+C). Tu le relances immédiatement → bind() retourne EADDRINUSE. Pourquoi ?
Cycle de vie d'un port après fermeture
ESTABLISHED — connexion active
FIN_WAIT / CLOSE_WAIT — fermeture en cours
TIME_WAIT — ~60 secondes · le port reste réservé
port libéré — nouvelle connexion possible
💡
TCP garantit la livraison — il doit s'assurer qu'aucun paquet retardé encore en transit ne vienne perturber une future connexion sur le même port. TIME_WAIT dure 2 × MSL (Maximum Segment Lifetime ≈ 30s), soit ~60s au total.

Tu peux l'observer : ss -tnp | grep 8080 → TIME-WAIT 0 0 0.0.0.0:8080 ...
server.cpp — ordre correct
SO_REUSEADDR
int server_fd = socket(AF_INET, SOCK_STREAM, 0);  // 1. créer

int opt = 1;
setsockopt(server_fd, SOL_SOCKET,                  // 2. option — AVANT bind()
           SO_REUSEADDR, &opt, sizeof(opt));

struct sockaddr_in addr;
memset(&addr, 0, sizeof(addr));
addr.sin_family      = AF_INET;
addr.sin_port        = htons(8080);
addr.sin_addr.s_addr = INADDR_ANY;

bind(server_fd, (struct sockaddr*)&addr,           // 3. bind
     sizeof(addr));
Ce que fait SO_REUSEADDR

Dit au kernel : "si le port est en TIME_WAIT, laisse-moi quand même le réutiliser".

Ce n'est pas dangereux. Les connexions en TIME_WAIT appartiennent à l'ancienne instance. Elles vont expirer toutes seules. Le nouveau bind() crée une nouvelle entrée dans la hash table TCP — les deux coexistent proprement.

Pourquoi c'est critique en dev

Sans SO_REUSEADDR, tu attends 30 à 60 secondes entre chaque Ctrl+C et relance.

En évaluation Webserv, le correcteur va tuer et relancer ton serveur. Si bind() plante, ton serveur est dysfonctionnel. C'est une faute rédhibitoire.

⚡SO_REUSEADDR doit être posé après socket() et avant bind() — sinon il n'a aucun effet.
🕐TIME_WAIT existe pour garantir la fiabilité TCP — il empêche qu'un vieux paquet retardé perturbe une nouvelle connexion.
🔍ss -tnp | grep 8080 te montre si ton port est en TIME_WAIT et combien de temps il lui reste.
06
Les ports
Pourquoi 0 à 65535 ?

Les ports TCP sont encodés sur 16 bits dans le header TCP. 216 = 65 536 valeurs possibles — ports 0 à 65535. Décision du RFC 793 (1981). Gravée dans le protocole depuis 44 ans.

0 – 1023
Ports système — Well-known
Nécessitent les droits root pour bind(). Réservés aux services standard.
HTTP=80 · HTTPS=443 · SSH=22 · FTP=21 · SMTP=25
1024 – 49151
Ports enregistrés
Conventions IANA, pas de droits root requis. Zone de travail pour Webserv.
8080 (HTTP alt) · 4242 (42 project) · 3306 (MySQL) · 5432 (PostgreSQL)
49152 – 65535
Ports éphémères
Assignés automatiquement aux clients pour le port source d'une connexion sortante. Tu ne bindes jamais ici.
Quand tu fais curl, ton OS choisit un port dans cette plage comme port source.
🔢16 bits dans le header TCP = décision de 1981. La migration IPv6 n'a pas changé la taille des ports.
🔑Ports < 1024 nécessitent CAP_NET_BIND_SERVICE (root ou capability). Un utilisateur normal ne peut pas bind sur le port 80.
🎯Pour Webserv : utilise 8080 ou 4242 — dans la plage enregistrée, pas besoin de sudo.
Récapitulatif bind()
Ce que chaque élément fait vraiment
📐
sockaddr_in
16 octets en RAM — IP + port + famille + padding. Toujours memset avant de remplir.
🧹
memset(&addr, 0, ...)
Garantit que sin_zero est à zéro ET initialise INADDR_ANY automatiquement.
🔄
htons(8080)
Convertit le port en big-endian pour le réseau. Sans ça, le port est silencieusement incorrect.
🌐
INADDR_ANY
0.0.0.0 — écoute sur toutes les interfaces. Vaut 0, déjà setté par memset.
🗃️
bind()
Enregistre port+IP dans la hash table TCP du kernel. Les paquets entrants sont routés vers ton socket.
⚡
SO_REUSEADDR
Permet de rebind un port en TIME_WAIT. Obligatoire en dev, rédhibitoire en éval si absent.
