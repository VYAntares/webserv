01
listen()
Deux effets, un seul appel
server.cpp
listen()
listen(server_fd, 10);
//               ^^
//               backlog — taille de la file d'attente
Effet 1 — Socket passif

Avant listen(), le socket peut théoriquement initier une connexion (mode actif).

Après listen(), il ne peut plus que recevoir des connexions entrantes. Ce changement est irréversible — tu ne peux pas désactiver le mode passif.

C'est ce qui distingue un socket serveur d'un socket client.

Effet 2 — File d'attente kernel

Le kernel va maintenant accepter automatiquement les connexions TCP entrantes et les stocker dans une file interne.

Ton code peut être occupé ailleurs — les clients arrivent quand même, le kernel les met en attente jusqu'à ce que tu appelles accept().

C'est le découplage fondamental entre réseau et application.

🔒Irréversible — un socket en mode listen ne peut pas revenir en mode actif. Si tu veux changer, tu fermes et tu recrées.
⚡Le kernel commence à répondre aux SYN entrants immédiatement après listen() — sans attendre ton accept().

💡
Le backlog n'est pas le nombre maximum de clients simultanés. C'est le nombre de connexions déjà établies qui peuvent attendre en file pendant que ton code est occupé ailleurs — dans un recv() ou un traitement long. Dès que accept() les récupère, un slot se libère.
En pratique pour Webserv

10 ou 128 — peu importe pour un projet d'étude. poll() sera bien plus rapide que la file ne se remplit.

Le noyau Linux ignore les valeurs trop grandes et applique le plafond système /proc/sys/net/core/somaxconn (souvent 128 ou 4096).

Deux queues en réalité

Linux maintient en fait deux files :

SYN queue — connexions dont le handshake est en cours (SYN reçu, SYN-ACK envoyé, ACK pas encore arrivé).

Accept queue — connexions dont le handshake est terminé, prêtes pour accept(). C'est ce que le backlog contrôle.

📋Backlog plein → le kernel répond ECONNREFUSED ou ignore silencieusement le SYN, selon tcp_abort_on_overflow.
🔢Le kernel peut arrondir le backlog à la puissance de 2 supérieure. listen(fd, 10) peut donner une file de 16.
03
TCP handshake
Le kernel s'en occupe seul

Tu n'écris pas une ligne pour le SYN / SYN-ACK / ACK. Tout se passe sans que ton code intervienne, dès que listen() est appelé.

🔑
Ton accept() arrive après le handshake — la connexion est déjà établie quand tu la récupères. Tu reçois un fd vers un socket déjà connecté, prêt pour recv() et send().
🤝Le kernel répond aux SYN en microsecondes. Ton application n'est pas sur le chemin critique du handshake.
🛡️Le kernel gère aussi les retransmissions si un SYN-ACK est perdu — transparent pour ton code.
04
L'état dans le kernel
Avant et après listen()
Avant listen()
state = SS_UNCONNECTED
tcp_state = CLOSE (07)
accept_q = NULL pas de file
SYN entrants = ignorés
Après listen()
state = SS_UNCONNECTED
tcp_state = LISTEN (0A)
accept_q = allouée backlog slots
SYN entrants = traités auto
/proc/net/tcp
observer l'état
cat /proc/net/tcp | grep 1F90

#  local_address   rem_address  st  ...  inode
   00000000:1F90  00000000:0000  0A  ...  49051
#                               ^^
#                               0A = 10 = TCP_LISTEN ← c'est nous

ss -tlnp | grep 8080
