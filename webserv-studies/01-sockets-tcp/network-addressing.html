Adressage réseau
& options de socket

Comment transformer un couple IP:port en quelque chose que le kernel comprend, pourquoi les octets sont dans le mauvais ordre par défaut, et comment configurer un socket pour qu'il se comporte correctement.

01
htons / htonl / ntohs / ntohl
L'ordre des octets en réseau
Le problème

Un nombre multi-octets — disons le port 8080 — peut être stocké en mémoire dans deux ordres différents selon le processeur.

Little-endian (x86, ARM modernes) : l'octet de poids faible en premier.
Big-endian : l'octet de poids fort en premier.

Le réseau utilise toujours le big-endian (appelé "network byte order" dans la RFC 791). Si ton CPU est little-endian et que tu envoies un port sans conversion, la valeur arrive inversée à l'autre bout.

📦
Analogie

C'est comme écrire une date : en France c'est jour/mois/année, aux États-Unis c'est mois/jour/année. Même informations, ordre différent. Si les deux côtés ne parlent pas le même ordre, la date est mal comprise. htons() est le traducteur.

Fonction Signification Usage
htons() Host TO Network Short (16 bits) Convertir un port avant de le mettre dans sockaddr_in.sin_port
htonl() Host TO Network Long (32 bits) Convertir une adresse IPv4 (INADDR_ANY, INADDR_LOOPBACK, ...)
ntohs() Network TO Host Short (16 bits) Lire un port reçu du réseau pour l'afficher / l'utiliser
ntohl() Network TO Host Long (32 bits) Lire une adresse IPv4 reçue du réseau
bind_example.cpp
Usage concret 
struct sockaddr_in addr;
memset(&addr, 0, sizeof(addr));

addr.sin_family      = AF_INET;
addr.sin_port        = htons(8080);   // ← TOUJOURS htons() pour le port
addr.sin_addr.s_addr = htonl(INADDR_ANY); // ← htonl() pour l'IP (0.0.0.0)

// Lire le port d'un client entrant :
struct sockaddr_in client;
// ... après accept() ...
int port = ntohs(client.sin_port); // ← ntohs() pour récupérer le port lisible
🔁Sur un CPU big-endian (rare), ces fonctions ne font rien — c'est du no-op. Mais les appeler reste obligatoire pour la portabilité.
🔢Ports = 16 bits → htons/ntohs. Adresses IPv4 = 32 bits → htonl/ntohl. Mnémonique : s = short, l = long.
🌐Hors webserv, tu retrouves ces fonctions partout : parseurs de protocoles binaires, jeux réseau, bases de données distribuées — partout où des nombres transitent sur le réseau.
02
getaddrinfo() / freeaddrinfo()
Résoudre une adresse proprement
Ce que ça fait

getaddrinfo() prend un nom d'hôte (ex: "localhost", "google.com", ou NULL) et un service (ex: "8080" ou "http") et retourne une liste chaînée de structures addrinfo prêtes à être utilisées directement.

Elle gère IPv4 et IPv6 de façon transparente, fait la résolution DNS si nécessaire, et remplace les anciennes fonctions gethostbyname() (obsolète, non thread-safe, IPv4 seulement).

Pour Webserv : utile pour résoudre les adresses des blocs server de la config.

📖
Analogie

C'est un annuaire universel. Tu donnes un nom et un type de service, il te rend une fiche complète avec l'adresse exacte à utiliser — déjà formatée, déjà dans le bon byte order. Tu n'as plus qu'à la copier dans ta struct sockaddr.

resolve.cpp
Usage complet 
struct addrinfo  hints, *res;
memset(&hints, 0, sizeof(hints));

hints.ai_family   = AF_UNSPEC;    // accepte IPv4 et IPv6
hints.ai_socktype = SOCK_STREAM;  // on veut du TCP
hints.ai_flags    = AI_PASSIVE;   // NULL hostname → INADDR_ANY (toutes les interfaces)

int status = getaddrinfo(NULL, "8080", &hints, &res);
if (status != 0) {
    std::cerr << gai_strerror(status) << "\n";
    return 1;
}

// res est une liste chaînée — prendre le premier qui marche
int fd = socket(res->ai_family, res->ai_socktype, res->ai_protocol);
bind(fd, res->ai_addr, res->ai_addrlen);

freeaddrinfo(res);  // ← TOUJOURS libérer après utilisation
🔗getaddrinfo() retourne une liste chaînée — plusieurs résultats possibles (IPv4, IPv6, interfaces différentes). Dans la pratique on itère et on prend le premier qui fonctionne.
🗑️La liste est allouée dynamiquement par le kernel. Appeler freeaddrinfo() est obligatoire — sinon fuite mémoire.
❌Ne jamais utiliser gethostbyname() — obsolète, non thread-safe, IPv4 seulement. getaddrinfo() est son remplacement officiel depuis POSIX 2001.
💡
Hors webserv : getaddrinfo() est LA façon moderne de se connecter à un serveur distant. Dans un client HTTP, tu fais getaddrinfo("api.exemple.com", "443", ...) et tu obtiens directement l'adresse à passer à connect() — sans jamais manipuler manuellement une struct sockaddr_in.
03
getsockname()
Demander au socket son adresse
Ce que ça fait

Après un bind(), le socket est attaché à une adresse. getsockname() permet de récupérer cette adresse depuis le socket lui-même — utile quand tu ne l'as pas stockée, ou quand tu as laissé le kernel choisir.

Cas le plus courant : tu fais bind() avec le port 0 (le kernel choisit un port libre), et tu utilises getsockname() juste après pour savoir quel port a été attribué.

🪪
Analogie

Tu t'installes dans un bureau dont tu n'as pas choisi le numéro. getsockname(), c'est regarder la plaque sur la porte pour savoir où tu es.

discover_port.cpp
Port dynamique 
struct sockaddr_in addr;
addr.sin_family      = AF_INET;
addr.sin_addr.s_addr = htonl(INADDR_ANY);
addr.sin_port        = htons(0);  // 0 = le kernel choisit un port libre

bind(fd, (struct sockaddr*)&addr, sizeof(addr));

// Récupérer le port réellement attribué :
struct sockaddr_in actual;
socklen_t          len = sizeof(actual);
getsockname(fd, (struct sockaddr*)&actual, &len);

std::cout << "Port attribué : " << ntohs(actual.sin_port) << "\n";
🎯Dans Webserv tu connais tes ports (ils sont dans la config) — getsockname() est surtout utile pour les tests et le debug : afficher le port réel d'écoute.
🌐Hors webserv : indispensable dans les serveurs de tests automatisés — on bind sur port 0, on récupère le port, on le communique aux clients de test.
04
setsockopt()
Configurer le comportement du socket
Ce que ça fait

setsockopt() permet de modifier des options du socket au niveau du kernel. La signature : setsockopt(fd, level, optname, &val, sizeof(val)).

Le level indique à quel niveau de la pile réseau s'applique l'option : SOL_SOCKET pour les options génériques du socket, IPPROTO_TCP pour les options spécifiques à TCP.

Dans Webserv, une seule option est réellement indispensable : SO_REUSEADDR.

⚙️
Analogie

C'est le panneau de configuration du socket. Après l'avoir créé, tu peux ajuster son comportement avant de l'utiliser — exactement comme configurer une voiture avant de prendre la route.

server_setup.cpp
SO_REUSEADDR 
int fd = socket(AF_INET, SOCK_STREAM, 0);

int opt = 1;
setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

// Ensuite bind() normalement :
struct sockaddr_in addr;
addr.sin_family      = AF_INET;
addr.sin_port        = htons(8080);
addr.sin_addr.s_addr = htonl(INADDR_ANY);
bind(fd, (struct sockaddr*)&addr, sizeof(addr));
⚠️
Sans SO_REUSEADDR : quand tu arrêtes ton serveur, le port reste en état TIME_WAIT pendant ~60 secondes (le kernel attend des paquets tardifs). Relancer le serveur immédiatement donne bind: Address already in use. Avec SO_REUSEADDR, le port est réutilisable immédiatement. C'est la première ligne à écrire après socket() dans tout serveur.
📋D'autres options existent : SO_RCVBUF/SO_SNDBUF (taille des buffers), TCP_NODELAY (désactiver Nagle), SO_KEEPALIVE (détecter les connexions mortes). Pour Webserv, seul SO_REUSEADDR est nécessaire.
🌐Hors webserv : SO_REUSEADDR est standard dans tout code serveur. Tu le verras dans nginx, Redis, PostgreSQL — c'est une pratique universelle.
📏Le paramètre &opt est un pointeur vers la valeur — 1 = activer, 0 = désactiver. La taille (sizeof(opt)) est obligatoire car les options n'ont pas toutes le même type.
Récap — l'essentiel
Ce que tu dois retenir et utiliser dans Webserv
🔄
htons(port) / htonl(ip)
Toujours convertir port et IP avant de les mettre dans sockaddr. Sans ça, les valeurs arrivent inversées côté réseau.
📖
getaddrinfo() + freeaddrinfo()
Résolution d'adresses moderne, IPv4/IPv6, retourne des structs prêtes à l'emploi. Toujours freeaddrinfo() après.
🪪
getsockname(fd, &addr, &len)
Récupérer l'adresse effective d'un socket après bind(). Utile pour debug et pour les ports dynamiques (port 0).
⚙️
setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &1, ...)
Appel obligatoire juste après socket(). Évite "Address already in use" au redémarrage du serveur.
