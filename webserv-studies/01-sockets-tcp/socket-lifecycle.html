Ce que ça fait techniquement

socket() demande au kernel de créer un point de communication réseau. Il retourne un simple entier — le file descriptor (fd).

AF_INET = protocole IPv4.
SOCK_STREAM = connexion TCP (fiable, ordonné).
0 = protocole par défaut (TCP pour STREAM).

Ce fd n'a encore aucune adresse, aucun port. C'est juste une entrée dans la table des file descriptors du processus — exactement comme open() retourne un fd pour un fichier.

🏗️
Analogie

Tu construis un bâtiment vide. Il existe physiquement, les murs sont posés — mais il n'a pas encore d'adresse postale. Personne ne peut t'y envoyer du courrier pour l'instant.

server.cpp
socket()
// Crée le socket TCP/IPv4 — retourne un file descriptor
int create_server_socket() {
    int server_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (server_fd == -1) {
        std::cerr << "socket() failed: " << strerror(errno) << std::endl;
        return -1;
    }

    // SO_REUSEADDR : évite "Address already in use" au redémarrage
    int opt = 1;
    setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    std::cout << "socket() ok — fd = " << server_fd << std::endl;
    return server_fd;  // souvent 3 (après stdin=0, stdout=1, stderr=2)
}
📌Le fd retourné est typiquement 3 (après stdin=0, stdout=1, stderr=2) si c'est le premier socket créé.
🔗En Unix, tout est fichier. Ce fd peut être passé à read(), write(), close() comme n'importe quel fichier.
⚙️setsockopt avec SO_REUSEADDR est quasi obligatoire en développement pour éviter d'attendre 30s entre deux lancements.

Ce que ça fait techniquement

bind() associe une adresse IP + port au socket. Le kernel sait maintenant que les paquets TCP arrivant sur ce port t'appartiennent.

INADDR_ANY (0.0.0.0) = écoute sur toutes les interfaces de la machine (localhost ET l'IP réseau locale).

htons(8080) convertit le port en network byte order (big-endian), car le réseau a ses propres conventions d'encodage des entiers.

memset(&addr, 0, sizeof(addr)) est crucial — sans ça, le champ sin_zero contient des données aléatoires qui peuvent causer des bugs obscurs.

📮
Analogie

Tu donnes une adresse postale à ton bâtiment. Maintenant les gens savent où t'envoyer du courrier — ils peuvent écrire "Port 8080, Rue INADDR_ANY" sur l'enveloppe.

server.cpp
bind()
int bind_address(int server_fd, int port) {
    struct sockaddr_in addr;

    // Initialiser à zéro — TOUJOURS faire ça avant de remplir sockaddr
    memset(&addr, 0, sizeof(addr));

    addr.sin_family      = AF_INET;       // IPv4
    addr.sin_port        = htons(port);     // host→network byte order
    addr.sin_addr.s_addr = INADDR_ANY;    // 0.0.0.0 = toutes les interfaces

    if (bind(server_fd,
             (struct sockaddr*)&addr,
             sizeof(addr)) == -1) {
        std::cerr << "bind() failed: " << strerror(errno) << std::endl;
        return -1;
    }
    std::cout << "bind() ok — port " << port << std::endl;
    return 0;
}
🔢htons() = Host TO Network Short. Les ports sont sur 16 bits. Le réseau attend du big-endian, ton CPU est peut-être little-endian.
🌐INADDR_ANY signifie que le serveur écoute sur toutes les IP de la machine — parfait pour un serveur local.
⛔Erreur fréquente : "Address already in use". SO_REUSEADDR dans socket() résout ça. Sinon attendre ~30 secondes (TIME_WAIT TCP).

Ce que ça fait techniquement

listen() marque le socket comme passif — il ne se connectera jamais lui-même à quelqu'un, il attend qu'on vienne le voir.

Il crée une file d'attente (backlog) dans le kernel pour les connexions entrantes. Le backlog = 10 signifie que jusqu'à 10 connexions peuvent attendre d'être acceptées.

Si la file est pleine quand un nouveau client arrive, le client reçoit ECONNREFUSED ou sa tentative de connexion est simplement ignorée.

Après listen(), le socket est prêt à recevoir des connexions — mais aucune n'est encore traitée. C'est accept() qui les traite.

🚪
Analogie

Tu ouvres la porte du bâtiment et tu installes une salle d'attente avec 10 chaises (backlog=10). Les gens peuvent maintenant faire la queue — mais personne n'est encore servi.

server.cpp
listen()
// Dans main(), après socket() et bind() :
const int BACKLOG = 10;  // taille de la file d'attente

if (listen(server_fd, BACKLOG) == -1) {
    std::cerr << "listen() failed: " << strerror(errno) << std::endl;
    close(server_fd);
    return 1;
}
std::cout << "listen() ok — backlog = " << BACKLOG << std::endl;
std::cout << "Serveur en écoute sur http://localhost:8080" << std::endl;

// À partir d'ici, le kernel accepte les connexions TCP entrantes
// et les place dans la file. Notre code doit maintenant appeler accept().
📋Le backlog est la file d'attente DANS le kernel, pas le nombre max de connexions. Il représente les connexions établies en attente d'être acceptées.
🔄Après listen(), le TCP handshake (SYN/SYN-ACK/ACK) se fait automatiquement côté kernel. Tu n'as pas à t'en occuper.

accept()
Ce que ça fait techniquement

accept() extrait le premier client de la file d'attente du kernel et crée un nouveau socket dédié à ce client. Il retourne un nouveau fd.

Le socket d'écoute original (server_fd) reste inchangé — il continue d'accepter d'autres clients. C'est ce qui permet de gérer plusieurs connexions.

Mode bloquant (défaut) : si aucun client n'attend, accept() se fige jusqu'à l'arrivée d'un client.

Mode non-bloquant : retourne immédiatement EAGAIN/EWOULDBLOCK si personne n'attend. C'est ce qu'on voudra pour Webserv avec epoll.

🏦
Analogie

Un guichetier prend en charge un seul client de la salle d'attente et lui ouvre une fenêtre dédiée. Le bâtiment reste ouvert pendant ce temps — d'autres clients peuvent arriver et s'asseoir.

server.cpp
accept()
int accept_client(int server_fd) {
    struct sockaddr_in client_addr;
    socklen_t          client_len = sizeof(client_addr);

    // Bloque jusqu'à ce qu'un client arrive (mode bloquant par défaut)
    // Retourne un NOUVEAU fd dédié au client
    int client_fd = accept(server_fd,
                           (struct sockaddr*)&client_addr,
                           &client_len);
    if (client_fd == -1) {
        std::cerr << "accept() failed" << std::endl;
        return -1;
    }

    // client_addr contient l'IP et port du client connecté
    std::cout << "Nouveau client — fd=" << client_fd
              << " ip=" << inet_ntoa(client_addr.sin_addr) << std::endl;
    return client_fd;  // fd dédié à CE client uniquement
}
File DescriptorRôleCréé par
server_fd (ex: 3) La devanture — fixe, attend toujours les connexions. Ne sert qu'à accept(). socket() + bind() + listen()
client_fd (ex: 4, 5, 6…) Le guichet dédié à UN client. recv()/send() se font sur ce fd. Fermé quand la connexion se termine. accept()
⚠️
Le problème du blocking : Dans notre code actuel, si accept() bloque, le serveur ne peut plus rien faire d'autre — ni traiter d'autres connexions, ni faire autre chose. C'est précisément pour ça que Webserv utilise epoll : on vérifie d'abord via epoll_wait() que server_fd a un client en attente (EPOLLIN), puis seulement on appelle accept().

05
recv() + send()
Lire la requête, envoyer la réponse
Ce que ça fait techniquement

recv() lit les données envoyées par le client depuis le buffer réseau du kernel. send() envoie des données vers le client.

Piège critique : un seul recv() ne retourne pas forcément toute la requête HTTP. TCP est un flux d'octets — les données arrivent en fragments. Il faut accumuler dans un buffer jusqu'à trouver \r\n\r\n (fin des headers).

De même, send() peut envoyer moins que demandé. Il faut boucler jusqu'à avoir tout envoyé.

📨
Analogie

Le guichetier lit la lettre du client (recv) et lui remet sa réponse écrite (send). Mais la lettre peut arriver en plusieurs enveloppes séparées — il faut attendre d'avoir tout reçu avant de répondre.

server.cpp
recv() + send()
void handle_client(int client_fd) {
    char buf[4096];

    // recv() retourne le nombre d'octets lus
    // 0 = client a fermé la connexion, -1 = erreur
    int n = recv(client_fd, buf, sizeof(buf) - 1, 0);
    if (n <= 0) { close(client_fd); return; }
    buf[n] = '\0';

    // Affichage de la requête HTTP brute reçue
    std::cout << "--- REQUETE HTTP ---\n" << buf << std::endl;

    // Construction de la réponse HTTP/1.1
    const char* body     = "<h1>Hello World!</h1>";
    const char* response =
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: text/html\r\n"  // ← corrigé : text/html pas text
        "Content-Length: 21\r\n"
        "Connection: close\r\n"
        "\r\n"                          // ← ligne vide obligatoire fin des headers
        "<h1>Hello World!</h1>";

    send(client_fd, response, strlen(response), 0);
    close(client_fd);  // fermer le fd client après réponse
}
🐛
Bug dans le code original : Content-Type: text est invalide — un navigateur ne peut pas interpréter la réponse. La valeur correcte est text/html pour du HTML, ou text/plain pour du texte brut. Le navigateur affichait probablement le HTML comme du texte brut.
📦En Webserv, recv() sera appelé après que epoll_wait() a signalé EPOLLIN sur ce fd — garanti qu'il y a des données à lire.
↔️Le \r\n\r\n entre headers et body est obligatoire en HTTP/1.1. Sans lui, aucun navigateur ne pourra parser la réponse.
🔒Toujours close(client_fd) après la connexion. Sinon : fuite de file descriptors. Le kernel a une limite (typiquement 1024).
Code complet
server.cpp — complet
#include <sys/socket.h>
#include <arpa/inet.h>
#include <sys/types.h>
#include <unistd.h>
#include <cstring>
#include <cerrno>
#include <iostream>

static const int PORT    = 8080;
static const int BACKLOG = 10;

// ── Étape 1 : crée le socket TCP/IPv4 ──────────────────────────
int create_server_socket() {
    int fd = socket(AF_INET, SOCK_STREAM, 0);
    if (fd == -1) { perror("socket"); return -1; }
    int opt = 1;
    setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
    return fd;
}

// ── Étape 2 : associe IP:port au socket ───────────────────────
int bind_address(int fd, int port) {
    struct sockaddr_in addr;
    memset(&addr, 0, sizeof(addr));
    addr.sin_family      = AF_INET;
    addr.sin_port        = htons(port);
    addr.sin_addr.s_addr = INADDR_ANY;
    if (bind(fd, (struct sockaddr*)&addr, sizeof(addr)) == -1)
        { perror("bind"); return -1; }
    return 0;
}

// ── Étape 4 : accepte UN client de la file d'attente ──────────
int accept_next_client(int server_fd) {
    struct sockaddr_in client_addr;
    socklen_t len = sizeof(client_addr);
    int client_fd = accept(server_fd,
                           (struct sockaddr*)&client_addr, &len);
    if (client_fd == -1) { perror("accept"); return -1; }
    std::cout << "[+] client fd=" << client_fd
              << " from " << inet_ntoa(client_addr.sin_addr) << "\n";
    return client_fd;
}

// ── Étape 5 : lit la requête, envoie la réponse ───────────────
void handle_client(int client_fd) {
    char buf[4096];
    int  n = recv(client_fd, buf, sizeof(buf) - 1, 0);
    if (n <= 0) { close(client_fd); return; }
    buf[n] = '\0';
    std::cout << buf;

    const char* response =
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: text/html\r\n"  // ← text/html, pas text
        "Content-Length: 22\r\n"
        "Connection: close\r\n"
        "\r\n"
        "<h1>Hello World!</h1>\n";

    send(client_fd, response, strlen(response), 0);
    close(client_fd);
}

// ── main : assemble les 4 étapes ──────────────────────────────
int main() {
    int server_fd = create_server_socket();
    if (server_fd == -1) return 1;

    if (bind_address(server_fd, PORT) == -1) return 1;

    if (listen(server_fd, BACKLOG) == -1)
        { perror("listen"); return 1; }

    std::cout << "Serveur sur http://localhost:" << PORT << "\n";

    // Boucle principale — séquentiel (un client à la fois)
    // → Prochaine étape : remplacer par epoll + non-blocking
    while (true) {
        int client_fd = accept_next_client(server_fd);
        if (client_fd == -1) continue;
        handle_client(client_fd);
    }
    return 0;
}
💡
Prochaine étape : Ce serveur est séquentiel — il traite un client à la fois et bloque sur accept(). Pour Webserv, on devra passer à epoll : créer un epoll fd via epoll_create1(), enregistrer server_fd + tous les client_fd avec epoll_ctl(), et mettre les sockets en mode non-bloquant avec fcntl(fd, F_SETFL, O_NONBLOCK).

Les 4 étapes du cycle de vie d'un socket serveur
🏗️
socket(AF_INET, SOCK_STREAM, 0)
Crée une prise réseau vide dans le kernel. Retourne un fd. Aucune adresse encore.
📮
bind(fd, &addr, sizeof(addr))
Attache IP + port au socket. Le kernel route les paquets vers ce fd.
🚪
listen(fd, backlog)
Marque le socket passif. Installe une file d'attente. TCP handshake géré par le kernel.
🏦
accept(fd, &addr, &len)
Extrait un client de la file. Retourne un nouveau fd dédié. server_fd reste intact.
