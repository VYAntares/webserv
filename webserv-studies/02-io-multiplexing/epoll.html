epoll
Le surveillant événementiel

epoll est le mécanisme Linux pour surveiller des centaines de file descriptors en O(1). Trois appels système, deux listes internes, un concept central : ne jamais interroger ce qui n'est pas prêt.

01
epoll
Le concept fondamental
Qu'est-ce que epoll ?

epoll est une interface de notification d'événements I/O spécifique à Linux. Son rôle : surveiller un ensemble de file descriptors et dire lesquels sont prêts pour une opération de lecture ou d'écriture.

Contrairement à select() (limité à 1024 fds, copie les sets à chaque appel) et poll() (parcourt linéairement tous les fds — O(n)), epoll maintient une liste interne dans le kernel et retourne directement les fds actifs en O(1).

📋
Analogie

Imagine un hôtel avec 500 chambres. Au lieu de frapper à chaque porte pour demander « vous avez besoin de quelque chose ? » (c'est poll), epoll c'est un système d'appel où chaque chambre appuie sur un bouton quand elle a besoin de service. Tu ne vérifies que les boutons allumés.

Architecture interne — deux listes

Le kernel maintient deux structures pour chaque instance epoll :

  • Interest list (epoll set) — l'ensemble des file descriptors que tu as enregistrés via epoll_ctl(). C'est ta liste de surveillance.
  • Ready list — le sous-ensemble de l'interest list dont les fds sont prêts pour de l'I/O. Cette liste est peuplée dynamiquement par le kernel quand un événement se produit sur un fd.

Quand tu appelles epoll_wait(), le kernel te retourne uniquement la ready list. Tu ne parcours jamais les fds inactifs.

architecture.txt
Schéma mental 
// L'architecture epoll en 3 étapes :

// 1. Créer l'instance epoll (une seule fois)
int epoll_fd = epoll_create1(0);

// 2. Enregistrer les fds à surveiller (interest list)
epoll_ctl(epoll_fd, EPOLL_CTL_ADD, fd, &event);  // ← ajoute fd à la liste

// 3. Attendre que des fds soient prêts (ready list)
int n = epoll_wait(epoll_fd, events, MAX, -1);
// → n contient le nombre de fds prêts
// → events[] contient UNIQUEMENT les fds actifs
⚡ O(1) — epoll ne parcourt pas tous les fds. Le kernel notifie directement les fds actifs.
🧠 Persistant — l'interest list vit dans le kernel. Pas besoin de la reconstruire à chaque appel.
📈 Scalable — gère des milliers de connexions simultanées sans dégradation.

« poll() (or equivalent) must monitor both reading and writing simultaneously. »

— webserv.pdf, IV.1 Requirements
02
epoll_create1()
Créer l'instance epoll
Ce que ça fait

epoll_create1() crée une nouvelle instance epoll dans le kernel et retourne un file descriptor qui la représente.

Ce fd est le « handle » que tu utiliseras dans tous les appels epoll_ctl() et epoll_wait(). Comme tout fd, il doit être fermé avec close() quand tu n'en as plus besoin.

🏗️
Analogie

C'est comme installer un tableau de contrôle vide dans la salle de surveillance. Le tableau existe, mais il n'y a encore aucune caméra branchée dessus.

epoll_create.cpp
Prototype & usage 
#include <sys/epoll.h>

// Prototype
int epoll_create1(int flags);
// Retourne : un fd epoll (>= 0) en cas de succès, -1 en cas d'erreur

// ──────────────────────────────────────────────────
// Usage typique :
// ──────────────────────────────────────────────────

int epoll_fd = epoll_create1(0);
if (epoll_fd == -1) {
    perror("epoll_create1");
    exit(EXIT_FAILURE);
}

// epoll_fd est maintenant une instance epoll vide,
// prête à recevoir des fds via epoll_ctl()

// ──────────────────────────────────────────────────
// Avec le flag EPOLL_CLOEXEC :
// ──────────────────────────────────────────────────

int epoll_fd = epoll_create1(EPOLL_CLOEXEC);
// → le fd sera automatiquement fermé lors d'un exec()
// → équivalent de fcntl(epoll_fd, F_SETFD, FD_CLOEXEC)
🔑 Le paramètre flags est généralement 0 ou EPOLL_CLOEXEC.
📌 epoll_create() (sans le 1) est l'ancienne version qui prend un size hint — ignoré depuis Linux 2.6.8. Utilise toujours epoll_create1().
♻️ N'oublie pas de close(epoll_fd) en fin de programme pour libérer les ressources kernel.
💡
Pourquoi epoll_create1 et pas epoll_create ? L'ancienne epoll_create(int size) ignorait déjà le paramètre size depuis longtemps. epoll_create1() a été ajoutée pour pouvoir passer des flags (comme EPOLL_CLOEXEC) proprement, sans bidouilles.
03
struct epoll_event
La structure de données
Rôle

struct epoll_event est la structure que tu passes à epoll_ctl() pour dire quels événements surveiller, et que tu reçois de epoll_wait() pour savoir quels événements se sont produits.

Elle contient deux champs : events (un bitmask de flags) et data (une union pour stocker des données utilisateur, typiquement le fd).

📝
Analogie

C'est la fiche d'inscription d'une chambre d'hôtel au tableau de surveillance. Elle indique : « surveille cette chambre (fd) pour ces types d'alertes (events) ».

epoll_event.h
Définition 
// Définition de la structure (de <sys/epoll.h>)

typedef union epoll_data {
    void    *ptr;     // pointeur générique (pour passer tes propres données)
    int      fd;      // file descriptor (usage le plus courant)
    uint32_t u32;     // entier 32 bits
    uint64_t u64;     // entier 64 bits
} epoll_data_t;

struct epoll_event {
    uint32_t     events;   // bitmask : EPOLLIN | EPOLLOUT | ...
    epoll_data_t data;     // données utilisateur (souvent .fd = le fd)
};
epoll_event_usage.cpp
Usage concret 
// Exemple 1 : Surveiller un fd en lecture
struct epoll_event ev;
ev.events = EPOLLIN;         // je veux être notifié quand ce fd est lisible
ev.data.fd = client_fd;      // je stocke le fd pour le retrouver dans epoll_wait

// Exemple 2 : Surveiller en lecture ET écriture
ev.events = EPOLLIN | EPOLLOUT;
ev.data.fd = client_fd;

// Exemple 3 : Utiliser le pointeur pour passer des données custom
struct Connection *conn = new_connection(client_fd);
ev.events = EPOLLIN;
ev.data.ptr = conn;        // on stocke un pointeur vers notre struct
// → dans epoll_wait, on récupère conn via events[i].data.ptr
⚠️
Attention : data est une union, pas une struct. Tu ne peux utiliser qu'un seul champ à la fois : soit .fd, soit .ptr, soit .u32, soit .u64. Si tu utilises .ptr pour passer un pointeur custom, tu ne pourras pas utiliser .fd en même temps.
04
epoll_ctl()
Gérer l'interest list
Ce que ça fait

epoll_ctl() est l'appel pour ajouter, modifier ou retirer un file descriptor de l'interest list d'une instance epoll. C'est le seul moyen de dire au kernel « surveille ce fd » ou « arrête de surveiller ce fd ».

Trois opérations possibles, identifiées par le second paramètre :

  • EPOLL_CTL_ADD — Ajouter un fd à l'interest list
  • EPOLL_CTL_MOD — Modifier les événements surveillés pour un fd existant
  • EPOLL_CTL_DEL — Retirer un fd de l'interest list
🔌
Analogie

C'est brancher, rebrancher ou débrancher une caméra sur le tableau de surveillance. ADD branche une nouvelle caméra, MOD change ce qu'elle filme, DEL la débranche.

epoll_ctl.cpp
Prototype 
#include <sys/epoll.h>

int epoll_ctl(int epfd,       // fd de l'instance epoll (retourné par epoll_create1)
              int op,         // opération : EPOLL_CTL_ADD, _MOD, ou _DEL
              int fd,         // le fd cible à ajouter/modifier/retirer
              struct epoll_event *event  // les événements à surveiller (NULL pour DEL)
);
// Retourne : 0 en cas de succès, -1 en cas d'erreur
Point de vue du serveur
EPOLLIN — le client envoie vers le serveur
Client ──── requête HTTP GET / ────► Serveur ← EPOLLIN → recv()
Client ◄─── réponse HTTP 200 OK ──── Serveur ← EPOLLOUT → send()
EPOLLIN
Des données entrent vers moi. Le client m'a envoyé quelque chose — je peux appeler recv() ou accept().
EPOLLOUT
Des données sortent de moi. Le buffer d'envoi a de la place — je peux appeler send() ma réponse.
Cycle complet d'une requête HTTP
① EPOLLIN
Socket d'écoute prêt → accept()
② EPOLLIN
Socket client prêt → recv() la requête
③ EPOLLOUT
Buffer prêt → send() la réponse
④ MOD
EPOLL_CTL_MOD → retour en EPOLLIN
epoll_ctl_examples.cpp
Les 3 opérations 
struct epoll_event ev;

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// EPOLL_CTL_ADD — Ajouter un fd à la surveillance
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
ev.events = EPOLLIN;              // surveiller la lecture
ev.data.fd = listen_fd;
if (epoll_ctl(epoll_fd, EPOLL_CTL_ADD, listen_fd, &ev) == -1) {
    perror("epoll_ctl: EPOLL_CTL_ADD");
    exit(EXIT_FAILURE);
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// EPOLL_CTL_MOD — Modifier les événements surveillés
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Le client a envoyé une requête, maintenant on veut
// écrire la réponse → on passe en EPOLLOUT
ev.events = EPOLLOUT;             // maintenant surveiller l'écriture
ev.data.fd = client_fd;
if (epoll_ctl(epoll_fd, EPOLL_CTL_MOD, client_fd, &ev) == -1) {
    perror("epoll_ctl: EPOLL_CTL_MOD");
}

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// EPOLL_CTL_DEL — Retirer un fd de la surveillance
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Le client se déconnecte → on le retire d'epoll
// puis on ferme le fd
epoll_ctl(epoll_fd, EPOLL_CTL_DEL, client_fd, NULL);  // event peut être NULL pour DEL
close(client_fd);
Opération Effet event peut être NULL ?
EPOLL_CTL_ADD Ajoute fd à l'interest list avec les events spécifiés Non
EPOLL_CTL_MOD Modifie les events surveillés pour un fd déjà enregistré Non
EPOLL_CTL_DEL Retire fd de l'interest list (arrête la surveillance) Oui (depuis Linux 2.6.9)
⚠️
Erreur fréquente : tenter un EPOLL_CTL_ADD sur un fd déjà enregistré retourne EEXIST. Si tu veux changer les événements, utilise EPOLL_CTL_MOD. Inversement, un EPOLL_CTL_MOD sur un fd non enregistré retourne ENOENT.
💡
Bonne pratique : toujours appeler epoll_ctl(EPOLL_CTL_DEL) avant close(). Bien que fermer un fd le retire automatiquement de l'interest list quand c'est le dernier descripteur vers l'open file description, si le fd a été dupliqué (via dup(), fork()), il reste dans l'interest list tant que tous les duplicatas ne sont pas fermés.
05
epoll_wait()
Attendre les événements
Ce que ça fait

epoll_wait() endort le programme à cette ligne — il ne passe pas à la suite. Le kernel met le processus en sommeil (zéro CPU consommé) et attend qu'un événement se produise sur n'importe quel fd de l'interest list.

Dès qu'au moins un fd est prêt, le kernel réveille le processus et remplit le tableau events[] avec uniquement les fds actifs. La valeur retournée nfds indique combien d'entrées ont été écrites dans ce tableau.

😴
Analogie

Le serveur s'endort en attendant. Il ne tourne pas en rond — il dort vraiment, sans consommer de CPU. Dès qu'un client arrive ou envoie des données, le kernel le réveille avec la liste exacte de ce qui s'est passé.

Ce que retourne epoll_wait — le tableau events[]

Tu fournis un tableau events[] vide. Le kernel le remplit avec uniquement les fds prêts au moment où il te réveille. Chaque entrée events[i] contient deux choses :

  • events[i].data.fd — quel fd est prêt (le fd que tu avais enregistré avec epoll_ctl)
  • events[i].events — quel événement s'est produit sur ce fd (EPOLLIN, EPOLLOUT, EPOLLERR…)

Oui, il peut y en avoir plusieurs simultanément. Si 3 clients envoient des données en même temps, nfds = 3 et tu as 3 entrées dans le tableau — chacune correspond à un fd différent. Tu les traites une par une dans le for.

epoll surveille 5 fds — 3 sont prêts ce cycle
Schéma events[] 
// INTEREST LIST — tout ce qu'on a enregistré via epoll_ctl(EPOLL_CTL_ADD)
listen_fd=3   client_fd=5   client_fd=7   client_fd=8   client_fd=9
// ↑ prêts                                       ↑ rien à signaler ce cycle

// EVENTS[] après epoll_wait() → nfds = 3
// Le kernel a rempli 3 entrées, les autres ne sont pas touchées

events[0]  .data.fd = 3   │  .events = EPOLLIN   ← nouveau client qui attend → accept()
events[1]  .data.fd = 5   │  .events = EPOLLIN   ← client a envoyé sa requête → recv()
events[2]  .data.fd = 7   │  .events = EPOLLOUT  ← buffer prêt → send() la réponse
events[3]  .data.fd = —   │  .events = —         ← non rempli (fd=8, fd=9 rien à signaler)
Le paramètre timeout

Contrôle combien de temps epoll_wait() est prêt à attendre avant de rendre la main, même si rien ne se passe.

Valeur Comportement Cas d'usage
-1 Dort indéfiniment jusqu'à un événement. Ne revient jamais « pour rien ». Webserv — le cas standard
0 Regarde instantanément et retourne immédiatement, même si rien n'est prêt. nfds = 0 si rien. Polling rapide sans bloquer
> 0 Attend au maximum N millisecondes. Retourne avant si un événement arrive, ou avec nfds = 0 si timeout. Tâches périodiques (timeouts, heartbeats)
epoll_wait.cpp
Boucle complète annotée 
#define MAX_EVENTS 64

struct epoll_event events[MAX_EVENTS]; // tableau que le kernel va remplir

// La boucle tourne indéfiniment — c'est voulu, c'est un serveur
while (1) {

    // Le programme DORT ici jusqu'à ce qu'un fd soit prêt
    // timeout = -1 : on dort aussi longtemps que nécessaire (0 CPU)
    int nfds = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);
    // ← ici le kernel nous réveille avec nfds événements prêts

    if (nfds == -1) {
        // EINTR = un signal a interrompu l'attente (ex: SIGCHLD)
        // Ce n'est pas une vraie erreur → on recommence simplement
        if (errno == EINTR) continue;
        perror("epoll_wait");
        break;
    }

    // On itère sur les nfds événements prêts — PAS sur tous les fds
    // Si on surveille 1000 fds et que 2 sont prêts : nfds = 2, on boucle 2 fois
    for (int i = 0; i < nfds; i++) {

        int      fd = events[i].data.fd;   // le fd qui a un événement
        uint32_t ev = events[i].events;    // quel type d'événement

        // fd peut être listen_fd, client_fd_1, client_fd_2...
        // On identifie qui est prêt et ce qu'il veut

        if (fd == listen_fd) {
            // Le socket d'écoute est prêt → nouveau client qui veut se connecter
            accept(listen_fd, ...);

        } else if (ev & EPOLLIN) {
            // Un client existant nous a envoyé des données → on lit
            recv(fd, buf, sizeof(buf), 0);

        } else if (ev & EPOLLOUT) {
            // Le buffer d'envoi est prêt → on envoie notre réponse
            send(fd, response, response_len, 0);

        } else if (ev & (EPOLLERR | EPOLLHUP)) {
            // Erreur ou déconnexion → on nettoie
            epoll_ctl(epoll_fd, EPOLL_CTL_DEL, fd, NULL);
            close(fd);
        }
    }
    // Fin du for → on retourne dans epoll_wait → on se rendort
}
😴 Le while dort, il ne tourne pas en rond. epoll_wait(-1) consomme zéro CPU pendant l'attente — le kernel met le processus en sommeil.
📦 events[] ne contient que les fds actifs. Sur 1000 fds surveillés, si 3 sont prêts : nfds=3, le reste du tableau n'est pas touché.
🔑 epoll_fd ≠ client_fd. epoll_fd est la télécommande de surveillance. Les données réelles passent par listen_fd et client_fd.
🚨 EINTR : un signal peut interrompre epoll_wait et retourner -1 avec errno==EINTR. Ce n'est pas une erreur — il faut juste continue.
06
EPOLLIN, EPOLLOUT, ...
Tous les flags epoll
Les flags d'événements

Les flags sont des bitmasks que tu combines avec l'opérateur | dans le champ events de struct epoll_event. Certains sont utilisés à l'entrée (dans epoll_ctl), d'autres à la sortie (dans epoll_wait), certains aux deux.

Flag Direction Description
EPOLLIN entrée/sortie Le fd est prêt en lecture. Des données sont disponibles, ou un nouveau client attend (pour un socket d'écoute).
EPOLLOUT entrée/sortie Le fd est prêt en écriture. Le buffer d'envoi a de la place. Tu peux appeler send() sans bloquer.
EPOLLERR sortie uniquement Erreur sur le fd. Toujours rapporté par epoll même si non demandé. Vérifie avec getsockopt(SO_ERROR).
EPOLLHUP sortie uniquement Hang up. Le pair a fermé sa connexion. Toujours rapporté même si non demandé.
EPOLLRDHUP entrée/sortie Le pair a fermé son côté écriture (half-close). Pratique pour détecter un shutdown propre.
EPOLLET entrée Active le mode edge-triggered. L'événement n'est signalé qu'une fois, lors du changement d'état.
EPOLLONESHOT entrée Désactive le fd après un seul événement. Il faut le réarmer avec EPOLL_CTL_MOD.
flags_examples.cpp
Combinaisons courantes 
// Surveiller la lecture seule (level-triggered, le défaut)
ev.events = EPOLLIN;

// Surveiller lecture + écriture
ev.events = EPOLLIN | EPOLLOUT;

// Edge-triggered en lecture (tu dois lire TOUT jusqu'à EAGAIN)
ev.events = EPOLLIN | EPOLLET;

// Edge-triggered lecture + écriture
// Astuce perf : on enregistre les deux dès le début
// pour éviter des EPOLL_CTL_MOD à répétition
ev.events = EPOLLIN | EPOLLOUT | EPOLLET;

// ONESHOT : un seul événement, puis il faut réarmer
ev.events = EPOLLIN | EPOLLONESHOT;
// après réception d'un événement, le fd est désactivé
// pour le réactiver :
ev.events = EPOLLIN | EPOLLONESHOT;  // re-spécifier les events
epoll_ctl(epoll_fd, EPOLL_CTL_MOD, fd, &ev);

// Détecter la déconnexion propre du pair
ev.events = EPOLLIN | EPOLLRDHUP;
💡
EPOLLERR et EPOLLHUP sont toujours reportés par epoll_wait(), même si tu ne les demandes pas dans epoll_ctl(). Tu n'as pas besoin de les ajouter au bitmask — mais tu dois les vérifier dans ta boucle.
🔑
Quand utiliser EPOLLONESHOT ? Quand plusieurs threads attendent sur le même epoll_fd. Sans EPOLLONESHOT, un même événement pourrait être dispatché à plusieurs threads simultanément (thundering herd). Avec EPOLLONESHOT, un seul thread traite l'événement, puis le réarme.
07
Level-triggered vs Edge-triggered
Les deux modes de notification d'epoll
La question fondamentale

Quand un fd a des données à lire, quand est-ce que epoll_wait() te prévient ? La réponse dépend du mode choisi. C'est la différence la plus subtile et la plus critique d'epoll.

Level-triggered (LT) — le défaut

epoll_wait() te signale un fd tant que la condition est vraie.

Si le buffer du fd contient des données non lues, chaque appel à epoll_wait() te le re-signalera. Tu peux lire petit à petit, epoll continue de te rappeler qu'il reste quelque chose.

C'est le comportement par défaut. Pas besoin de flag supplémentaire.

Edge-triggered (ET) — EPOLLET

epoll_wait() te signale un fd une seule fois quand son état change.

Le kernel te notifie quand de nouvelles données arrivent, pas quand il reste des données. Si tu n'as pas tout lu, epoll ne te re-signalera plus. Les données restent dans le buffer, mais tu ne seras plus prévenu.

S'active avec le flag EPOLLET dans epoll_ctl().

🔔
Analogie

LT = une alarme qui sonne en continu tant que le problème existe. Tu peux être lent, elle te rappellera.
ET = une alarme qui sonne une seule fois au moment où le problème apparaît. Si tu ne réagis pas immédiatement, silence total.

Scénario pas à pas — Un client envoie 2 Ko
1
Enregistrement du fd dans epoll
On ajoute un fd (par ex. un pipe ou un socket client) dans l'instance epoll avec EPOLLIN. Le buffer du kernel associé à ce fd est vide.
2
Le client envoie 2 Ko
2048 octets arrivent dans le buffer du kernel. L'état du fd change : il passe de « vide » à « 2 Ko disponibles ».
Buffer kernel :
2048 octets
3
epoll_wait() retourne le fd
Que ce soit en LT ou en ET, epoll_wait() retourne le fd comme prêt. Les deux modes se comportent pareil ici — c'est un nouvel événement, et il y a des données.
4
On lit seulement 1 Ko (lecture partielle)
On appelle recv(fd, buf, 1024, 0) — on ne lit que 1024 octets. Il reste 1024 octets non lus dans le buffer du kernel.
Buffer kernel :
lu ✓
1024 restants
5
On rappelle epoll_wait()...
C'est ici que les deux modes divergent complètement :
✅ Level-triggered

epoll_wait() retourne le fd immédiatement.

Pourquoi ? Parce que le buffer contient encore des données. En mode LT, la question que se pose le kernel est : « Est-ce qu'il y a des données à lire ? » La réponse est oui → il te signale le fd.

Tu peux lire les 1024 octets restants, et c'est terminé. Aucune perte de données possible.

CYCLE
epoll_wait() → fd prêt
recv() → lit 1024 octets
epoll_wait() → fd encore prêt (il reste des données)
recv() → lit les 1024 restants
epoll_wait() → bloque (buffer vide)
💀 Edge-triggered

epoll_wait() bloque indéfiniment.

Pourquoi ? Parce qu'aucun nouvel événement ne s'est produit. En mode ET, la question du kernel est : « Est-ce que quelque chose de nouveau est arrivé depuis la dernière notification ? » La réponse est non → silence.

Les 1024 octets restants sont toujours dans le buffer, mais tu ne seras jamais prévenu qu'ils sont là. C'est un deadlock.

CYCLE
epoll_wait() → fd prêt
recv() → lit 1024 octets
epoll_wait() → bloque pour toujours ∞
↑ 1024 octets oubliés dans le buffer
Pourquoi parle-t-on de « perte » de données en ET ?

Les données ne sont pas effacées — elles sont toujours dans le buffer du kernel. Le problème est que ton programme ne sait plus qu'elles sont là. epoll_wait() ne le re-signalera pas car il attend un changement d'état (= de nouvelles données), pas la présence de données existantes.

Le seul moyen pour que epoll_wait() re-signale le fd est que le client envoie de nouvelles données. Mais si le client attend une réponse de ta part (qui dépend des octets que tu n'as pas lus), c'est un deadlock mutuel.

La solution — Comment coder correctement en Edge-triggered
Les deux règles obligatoires en ET

Pour éviter le piège ci-dessus, le mode edge-triggered impose deux règles absolues :

  • Règle 1 : Les fds doivent être en mode non-bloquant (O_NONBLOCK via fcntl())
  • Règle 2 : À chaque notification, lire en boucle jusqu'à ce que recv() retourne EAGAIN

EAGAIN signifie « il n'y a plus rien à lire pour l'instant ». C'est le signal que tu as vidé le buffer. Sans O_NONBLOCK, au lieu de retourner EAGAIN, le dernier recv() bloquerait le processus entier en attendant des données qui ne viendront peut-être jamais.

Comparaison de code — LT vs ET

        process_data(buf, n);

edge_triggered.cpp
ET — EPOLLET 
// Enregistrement — avec EPOLLET
struct epoll_event ev;
ev.events  = EPOLLIN | EPOLLET;
ev.data.fd = client_fd;
epoll_ctl(epoll_fd, EPOLL_CTL_ADD,
          client_fd, &ev);

void handle_read_et(int fd) {
    char buf[4096];
    while (1) {
        ssize_t n = recv(fd,
            buf, sizeof(buf), 0);
        if (n == -1) {
            if (errno == EAGAIN)
                break; // buffer vidé
            perror("recv");
            break;
        }
        if (n == 0) {
            close(fd); return;
        }
        process_data(buf, n);
    } // ← boucle jusqu'à EAGAIN
}
Les 3 différences entre le code LT et ET
Différence Level-triggered (LT) Edge-triggered (ET)
1. Flag epoll_ctl EPOLLIN EPOLLIN | EPOLLET
2. Boucle de lecture Un seul recv() while(1) + recv() jusqu'à EAGAIN
3. Mode du fd Bloquant ou non-bloquant Obligatoirement O_NONBLOCK
⚠️
Pourquoi O_NONBLOCK est obligatoire en ET : en mode ET, tu lis en boucle (while(1) { recv(...) }) jusqu'à ce qu'il n'y ait plus rien. Si le fd est en mode bloquant, le dernier recv() — quand le buffer est vide — bloque le processus entier au lieu de retourner -1 avec errno = EAGAIN. Ton serveur est figé, plus aucun client n'est traité.

Quel mode choisir pour webserv ?

Le sujet 42 dit : « le fd du serveur doit être en mode non-bloquant ». Le mode level-triggered est le choix le plus sûr et le plus simple. La plupart des implémentations de webserv à 42 utilisent LT.

Le mode ET est plus performant à grande échelle (moins de réveils inutiles d'epoll_wait), mais la complexité supplémentaire n'apporte pas de valeur réelle pour un projet 42. Utilise LT sauf si tu as une bonne raison de faire autrement.

✅ LT = « il y a des données ? » → le kernel te notifie à chaque epoll_wait() tant que c'est vrai.
⚡ ET = « quelque chose de nouveau ? » → le kernel te notifie une seule fois quand l'état change.
🔑 La seule différence de code : | EPOLLET dans les flags + boucle while + EAGAIN + O_NONBLOCK.
📌 Pour webserv : utilise level-triggered (le défaut). Plus simple, plus sûr, suffisant pour le projet.

Exemple complet
Un serveur echo avec epoll
Ce que fait cet exemple

Un serveur TCP minimaliste qui utilise epoll en edge-triggered pour accepter des connexions et renvoyer en écho tout ce que les clients envoient. Chaque étape est commentée.

echo_server_epoll.cpp
Exemple complet 
#include <sys/epoll.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <fcntl.h>
#include <unistd.h>
#include <cstdio>
#include <cstdlib>
#include <cerrno>

#define MAX_EVENTS 64
#define PORT       8080

// Rendre un fd non-bloquant (obligatoire pour EPOLLET)
void set_nonblocking(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

int main() {
    // ━━━ ÉTAPE 1 : Créer le socket d'écoute ━━━
    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);

    int opt = 1;
    setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    struct sockaddr_in addr = {};
    addr.sin_family      = AF_INET;
    addr.sin_addr.s_addr = INADDR_ANY;
    addr.sin_port        = htons(PORT);

    bind(listen_fd, (struct sockaddr*)&addr, sizeof(addr));
    listen(listen_fd, SOMAXCONN);
    set_nonblocking(listen_fd);

    // ━━━ ÉTAPE 2 : Créer l'instance epoll ━━━
    int epoll_fd = epoll_create1(0);

    // ━━━ ÉTAPE 3 : Enregistrer le socket d'écoute ━━━
    struct epoll_event ev;
    ev.events  = EPOLLIN | EPOLLET;  // lecture, edge-triggered
    ev.data.fd = listen_fd;
    epoll_ctl(epoll_fd, EPOLL_CTL_ADD, listen_fd, &ev);

    struct epoll_event events[MAX_EVENTS];

    // ━━━ ÉTAPE 4 : La boucle événementielle ━━━
    while (1) {
        int nfds = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);

        for (int i = 0; i < nfds; i++) {
            int fd = events[i].data.fd;

            // ─── Nouveau client ? ───
            if (fd == listen_fd) {
                // En ET, on accept() en boucle jusqu'à EAGAIN
                while (1) {
                    struct sockaddr_in client_addr;
                    socklen_t len = sizeof(client_addr);
                    int conn_fd = accept(listen_fd,
                        (struct sockaddr*)&client_addr, &len);

                    if (conn_fd == -1) {
                        if (errno == EAGAIN)
                            break;  // plus de clients en attente
                        perror("accept");
                        break;
                    }

                    set_nonblocking(conn_fd);
                    ev.events  = EPOLLIN | EPOLLET;
                    ev.data.fd = conn_fd;
                    epoll_ctl(epoll_fd, EPOLL_CTL_ADD, conn_fd, &ev);
                    printf("Nouveau client: fd %d\n", conn_fd);
                }
                continue;
            }

            // ─── Données d'un client ? ───
            if (events[i].events & EPOLLIN) {
                char buf[4096];
                // En ET, lire en boucle jusqu'à EAGAIN
                while (1) {
                    ssize_t n = recv(fd, buf, sizeof(buf), 0);
                    if (n <= 0) {
                        if (n == -1 && errno == EAGAIN)
                            break;  // plus rien à lire
                        // n == 0 → déconnexion, n == -1 → erreur
                        epoll_ctl(epoll_fd, EPOLL_CTL_DEL, fd, NULL);
                        close(fd);
                        printf("Client déconnecté: fd %d\n", fd);
                        break;
                    }
                    // Echo : renvoyer ce qu'on a reçu
                    send(fd, buf, n, 0);
                }
            }

            // ─── Erreur ou déconnexion ? ───
            if (events[i].events & (EPOLLERR | EPOLLHUP)) {
                epoll_ctl(epoll_fd, EPOLL_CTL_DEL, fd, NULL);
                close(fd);
            }
        }
    }

    close(listen_fd);
    close(epoll_fd);
    return 0;
}

2️⃣ On crée l'instance epoll avec epoll_create1(0)
3️⃣ On enregistre le socket d'écoute avec EPOLLIN | EPOLLET

FAQ & pièges
Questions fréquentes et erreurs à éviter
Q0 C'est quoi la « clé » qui identifie un fd dans l'interest list ?
La clé est la combinaison du numéro de fd + l'open file description (la structure interne du kernel). C'est pour ça qu'un même fd ne peut être ajouté qu'une fois, mais que deux fds dupliqués (dup()) pointant vers la même open file description peuvent être ajoutés séparément.
Q1 Que se passe-t-il si on enregistre le même fd deux fois ?
epoll_ctl(EPOLL_CTL_ADD) retourne -1 avec errno == EEXIST. En revanche, tu peux ajouter un fd dupliqué (via dup()) avec des événements différents — utile pour filtrer les événements.
Q2 Deux instances epoll peuvent-elles surveiller le même fd ?
Oui. Les événements seront reportés aux deux instances. Mais ça demande une programmation soigneuse pour éviter les race conditions.
Q3 Le fd epoll est-il lui-même pollable ?
Oui. Si une instance epoll a des événements en attente, elle apparaît comme lisible. Tu peux donc imbriquer des instances epoll (ajouter un epoll_fd dans un autre epoll).
Q4 Peut-on ajouter un epoll_fd dans sa propre instance ?
Non. epoll_ctl() retourne EINVAL. Mais tu peux ajouter un epoll_fd dans une autre instance epoll.
Q5 Fermer un fd le retire-t-il automatiquement de l'interest list ?
Oui, mais attention : un fd est un référence vers une open file description. Si le fd a été dupliqué (dup(), fork()), il reste dans l'interest list tant que tous les fds qui pointent vers la même open file description n'ont pas été fermés. C'est pourquoi il est recommandé d'appeler epoll_ctl(EPOLL_CTL_DEL) explicitement avant close().
Q6 Plusieurs événements entre deux appels epoll_wait : combinés ou séparés ?
Combinés. Si un fd reçoit EPOLLIN et EPOLLOUT entre deux appels epoll_wait(), tu recevras un seul événement avec events = EPOLLIN | EPOLLOUT.
Q7 Faut-il toujours lire jusqu'à EAGAIN en edge-triggered ?
Pour les fichiers orientés paquets (datagram socket, terminal en mode canonique) : oui, obligatoirement. Pour les flux (pipe, TCP socket) : tu peux aussi vérifier si recv() retourne moins que demandé — ça signifie que le buffer est vidé. Mais la méthode la plus sûre reste de lire jusqu'à EAGAIN.
Piège #1 — Starvation en edge-triggered

Si un fd a beaucoup de données, la boucle « lire jusqu'à EAGAIN » peut monopoliser le processus et affamer les autres fds. Les connexions avec peu de trafic ne sont jamais traitées.

Solution : maintenir une liste de fds « prêts » dans ton application. Lire une quantité limitée par fd, puis passer au fd suivant (round-robin).

Piège #2 — Cache d'événements obsolète

Si tu caches les résultats de epoll_wait() et que le traitement de l'événement #47 provoque la fermeture du fd de l'événement #13 (qui est dans le même batch), tu risques de traiter un fd déjà fermé.

Solution : avant de traiter un événement, vérifie que le fd est toujours valide. Ou mieux : appelle epoll_ctl(EPOLL_CTL_DEL) puis close(), et marque le fd comme supprimé dans ta structure de données.

Résumé epoll
Trois appels, deux listes, un mécanisme de surveillance I/O complet.
🏗️
epoll_create1(flags)
Crée une instance epoll vide. Retourne un fd. Un seul appel au démarrage.
📝
struct epoll_event
Bitmask d'événements (EPOLLIN, EPOLLOUT, ...) + données utilisateur (.fd ou .ptr).
🔌
epoll_ctl(epfd, op, fd, &ev)
Gère l'interest list. ADD ajoute, MOD modifie, DEL retire un fd.
👀
epoll_wait(epfd, events, max, timeout)
Bloque et retourne les fds prêts. Le coeur de la boucle événementielle.
