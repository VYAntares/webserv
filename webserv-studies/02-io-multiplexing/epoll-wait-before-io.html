La règle d'or
Toujours epoll_wait() avant read/write

Le sujet est catégorique : jamais de recv() ou send() sur un socket sans que epoll_wait() ait confirmé que le fd est prêt. Enfreindre cette règle = 0.

01
La règle
Ce que dit le sujet, mot pour mot

« You must never do a read or a write operation without going through poll() (or equivalent). Calling read/recv or write/send on these descriptors without prior readiness will result in a grade of 0. »

— webserv.pdf, IV.1 Requirements
Ce que ça veut dire

Avant chaque appel à recv(), send(), read() ou write() sur un socket ou un pipe, tu dois avoir reçu la confirmation d'epoll_wait() que ce fd est prêt pour l'opération demandée.

EPOLLIN = le fd est prêt en lecture → tu peux appeler recv().
EPOLLOUT = le fd est prêt en écriture → tu peux appeler send().
Pas d'événement = tu ne touches pas au fd.

Le contrat fondamental

C'est le contrat de base d'un serveur event-driven : tu ne fais rien sans qu'un événement te le dise. Ton serveur est réactif, pas proactif.

Le serveur ne demande pas aux clients s'ils ont des données. Il attend qu'epoll lui dise : « le client 42 a des données prêtes à être lues ». Alors, et seulement alors, il appelle recv().

epoll_wait()
→
EPOLLIN sur fd 7
→
recv(fd 7)
✅
recv(fd 7)
← pas d'epoll_wait avant ❌ = 0
02
Sans epoll_wait()
Ce qui se passe quand on triche
🎯
Le problème central

Sans epoll_wait(), tu tires à l'aveugle. Tu ne sais pas si le fd est prêt. Tu appelles recv() en espérant qu'il y ait des données. Parfois oui, souvent non. Ton code devient un casino.

🔥
Les conséquences

EAGAIN partout, boucles qui tournent dans le vide, données partielles non gérées, CPU à 100% pour rien, et surtout : un serveur qui ne scale pas du tout. C'est l'exact opposé de l'event-driven.

sans_epoll_recv.cpp
❌ Mauvais — recv sans epoll 
// ❌ MAUVAIS : recv() direct après accept()

int client_fd = accept(listen_fd, NULL, NULL);
fcntl(client_fd, F_SETFL, O_NONBLOCK);

// Le client vient de se connecter (TCP handshake OK)
// MAIS il n'a peut-être PAS encore envoyé sa requête HTTP !

char buf[4096];
ssize_t n = recv(client_fd, buf, sizeof(buf), 0);  // ❌ SANS epoll_wait !

if (n == -1 && errno == EAGAIN) {
    // Pas de données encore... et maintenant ?
    // On fait quoi ? On réessaie en boucle ? On attend combien de temps ?
    // → On a aucune idée de QUAND les données arriveront
}
else if (n == 0) {
    // Le client s'est déconnecté avant d'envoyer quoi que ce soit
}
else if (n > 0) {
    // On a eu de la chance... cette fois-ci
    // Mais la requête est-elle complète ? Faut-il relire ?
}

03
Catalogue de bugs
Les 5 scénarios catastrophe
💥 Bug #1 — EAGAIN infini
Tu appelles recv() sur un socket non-bloquant sans epoll. Le client n'a rien envoyé. recv() retourne -1 avec errno = EAGAIN. Tu réessaies. Encore EAGAIN. Tu boucles. CPU à 100% pour absolument rien. C'est du busy-waiting — l'exact opposé de l'event-driven.
💥 Bug #2 — recv() direct après accept()
Un client vient de se connecter. Le TCP handshake est fait, accept() retourne un fd. Tu appelles immédiatement recv(). Mais le client n'a pas encore envoyé sa requête HTTP — la connexion TCP est ouverte, mais le navigateur prépare encore ses headers. recv() retourne EAGAIN. Si tu étais en mode bloquant, ton serveur entier gèle en attendant ce seul client.
💥 Bug #3 — send() sur un buffer TCP plein
Tu construis ta réponse HTTP de 50 KB et tu appelles send(). Le buffer d'envoi TCP du kernel a seulement 16 KB de libre. send() n'envoie que 16 KB et retourne 16384. Les 34 KB restants ? Perdus si tu ne gères pas. Et quand le buffer sera libre ? Tu n'en sais rien sans EPOLLOUT.
💥 Bug #4 — Client déconnecté = SIGPIPE
Le client a fermé sa connexion (il a navigué ailleurs, timeout, etc.). Tu appelles send() sans avoir vérifié avec epoll. Le kernel t'envoie un SIGPIPE. Comportement par défaut : ton serveur se termine. Avec epoll, tu aurais reçu EPOLLHUP ou EPOLLERR et tu aurais proprement fermé le fd.
💥 Bug #5 — Données partielles et requêtes mélangées
Tu appelles recv() à un moment arbitraire. Tu reçois "GET /index.ht" — la requête est coupée au milieu car les données arrivent par paquets TCP. Sans epoll, quand relire la suite ? Tu ne sais pas. Tu relies immédiatement ? Peut-être EAGAIN. Tu attends un temps arbitraire ? C'est du polling. Avec epoll, tu recevras un nouvel EPOLLIN quand la suite arrive.
⚠️
Le point commun de tous ces bugs : sans epoll_wait(), tu n'as aucun moyen de savoir quand un fd est prêt. Tu es condamné à deviner, boucler, ou dormir un temps arbitraire. Tout ça disparaît avec une boucle événementielle correcte.
04
Avec epoll_wait()
Comment faire correctement
Le flux correct pour accept

Le socket d'écoute est enregistré dans epoll avec EPOLLIN. Quand un client se connecte, epoll_wait() te le signale. Alors seulement tu appelles accept(). Puis tu enregistres le nouveau client dans epoll et tu retournes dans la boucle.

Le flux correct pour recv/send

Le client est enregistré avec EPOLLIN. Quand il envoie des données, epoll_wait() retourne EPOLLIN pour ce fd. Tu appelles recv(). Pour la réponse, tu modifies le fd avec EPOLLOUT via epoll_ctl(MOD). Quand le buffer est prêt, epoll te le dit → tu appelles send().

correct_accept.cpp
✅ accept() correct 
// ✅ CORRECT : accept() seulement quand epoll le signale

// Au démarrage : enregistrer le listen_fd
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = listen_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &ev);

// Dans la boucle :
int n = epoll_wait(epfd, events, 64, -1);

for (int i = 0; i < n; i++) {
    if (events[i].data.fd == listen_fd) {
        // ✅ epoll a dit : "quelqu'un frappe à la porte"
        int client_fd = accept(listen_fd, NULL, NULL);
        fcntl(client_fd, F_SETFL, O_NONBLOCK);

        // On enregistre le client dans epoll pour savoir quand il envoie
        ev.events = EPOLLIN;
        ev.data.fd = client_fd;
        epoll_ctl(epfd, EPOLL_CTL_ADD, client_fd, &ev);

        // PAS de recv() ici ! On attend le prochain epoll_wait()
    }
}
correct_recv_send.cpp
✅ recv/send correct 
// ✅ CORRECT : recv() et send() seulement après epoll_wait()

for (int i = 0; i < n; i++) {
    int fd = events[i].data.fd;

    if (events[i].events & EPOLLIN) {
        // ✅ epoll confirme : "ce fd a des données à lire"
        char buf[4096];
        ssize_t bytes = recv(fd, buf, sizeof(buf), 0);

        if (bytes > 0) {
            // Accumuler dans un buffer, parser la requête HTTP
            // Quand la requête est complète → préparer la réponse

            // Maintenant on veut écrire → on demande EPOLLOUT
            struct epoll_event ev;
            ev.events = EPOLLOUT;
            ev.data.fd = fd;
            epoll_ctl(epfd, EPOLL_CTL_MOD, fd, &ev);
        }
        else if (bytes == 0) {
            // Client déconnecté proprement
            epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);
            close(fd);
        }
    }

    if (events[i].events & EPOLLOUT) {
        // ✅ epoll confirme : "le buffer d'envoi est prêt"
        ssize_t sent = send(fd, response_buf, response_len, 0);

        if (sent < response_len) {
            // Envoi partiel → avancer le pointeur, rester en EPOLLOUT
            // epoll_wait() nous re-notifiera quand le buffer est libre
        } else {
            // Tout envoyé → repasser en EPOLLIN pour la prochaine requête
            struct epoll_event ev;
            ev.events = EPOLLIN;
            ev.data.fd = fd;
            epoll_ctl(epfd, EPOLL_CTL_MOD, fd, &ev);
        }
    }

    if (events[i].events & (EPOLLERR | EPOLLHUP)) {
        // Erreur ou déconnexion → on nettoie proprement
        epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);
        close(fd);
    }
}
🔑 accept() — seulement quand EPOLLIN arrive sur le listen_fd. Puis enregistrer le client dans epoll.
📥 recv() — seulement quand EPOLLIN arrive sur un client_fd. Accumuler dans un buffer.
📤 send() — seulement quand EPOLLOUT arrive. Gérer les envois partiels.
🔀 Transition — après recv complet, epoll_ctl(MOD) pour passer de EPOLLIN à EPOLLOUT.
05
L'exception
Fichiers réguliers sur disque

« Regular disk files are exempt. »

— webserv.pdf, IV.1 Requirements
Pourquoi les fichiers sont exemptés

Un fichier sur disque (open("/var/www/index.html")) est toujours prêt. Le kernel sait lire un fichier immédiatement — les données sont sur le disque, pas en attente d'un client réseau distant.

Un read() sur un fichier régulier ne retourne jamais EAGAIN. Il lit les données (ou retourne 0 à la fin du fichier). Pas besoin d'epoll pour ça.

Ce qui est exempté vs ce qui ne l'est pas
  • Exempté — open() + read()/write() sur fichiers .html, .css, .jpg, etc.
  • Exempté — opendir() + readdir() pour les listings de répertoires
  • PAS exempté — recv()/send() sur les sockets clients
  • PAS exempté — accept() sur le socket d'écoute
  • PAS exempté — read()/write() sur les pipes CGI
file_read.cpp
✅ Fichier — pas besoin d'epoll 
// ✅ Les fichiers réguliers sont toujours prêts
// → pas besoin de passer par epoll_wait()

int file_fd = open("/var/www/index.html", O_RDONLY);

char buf[8192];
ssize_t n;

while ((n = read(file_fd, buf, sizeof(buf))) > 0) {  // ✅ direct, sans epoll
    // Accumuler le contenu du fichier dans le buffer de réponse
    response_buffer.append(buf, n);
}

close(file_fd);

// MAIS pour envoyer cette réponse au client via le socket :
// → il FAUT passer par epoll_wait() + EPOLLOUT !
💡
Résumé simple : lire un fichier sur le disque → direct. Envoyer ce fichier à un client via un socket → epoll_wait() + EPOLLOUT d'abord. Le fichier est le contenu. Le socket est le tuyau. Seul le tuyau a besoin d'epoll.
06
Et le CGI ?
Les pipes aussi passent par epoll
Aperçu — on verra les détails plus tard

Le CGI utilise des pipes pour communiquer avec le processus fils (fork() + exec()). Un pipe est comme un socket : c'est un fd qui peut ne pas être prêt. Le processus CGI peut prendre du temps à produire sa sortie.

La règle est la même : le pipe de sortie du CGI doit être enregistré dans epoll via epoll_ctl() en mode non-bloquant. Tu lis le pipe (read()) seulement quand epoll_wait() retourne EPOLLIN dessus. Même logique pour écrire le body POST dans le pipe d'entrée du CGI — attendre EPOLLOUT.

fork() + exec()
→
epoll_ctl(ADD, pipe_fd)
→
epoll_wait()
→
EPOLLIN sur pipe
→
read(pipe_fd)
📌
On détaillera le CGI dans son propre chapitre. Pour l'instant, retiens juste : un pipe CGI suit exactement la même règle qu'un socket client — jamais de read/write sans epoll_wait() avant.
Résumé — La règle d'or
Toujours epoll_wait() avant toute I/O sur un socket ou un pipe.
🚫
Jamais sans epoll
recv(), send(), accept() sur sockets, read()/write() sur pipes — toujours après epoll_wait().
✅
Direct sur fichiers
read()/write() sur fichiers réguliers (HTML, CSS, images) — pas besoin d'epoll, toujours prêts.
🔀
Transition EPOLLIN → EPOLLOUT
Après avoir lu la requête, epoll_ctl(MOD) pour passer en écriture. Puis epoll_wait() avant send().
📌
Pipes CGI inclus
Les pipes suivent la même règle : enregistrer dans epoll, lire/écrire seulement après notification.
