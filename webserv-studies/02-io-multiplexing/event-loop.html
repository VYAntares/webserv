Concept critique — I/O sans errno

La boucle événementielle
sans jamais checker errno

Le sujet interdit de vérifier errno après read/write. Pourtant, le pattern classique edge-triggered repose entièrement sur EAGAIN. Voici comment architecturer un serveur event-driven qui respecte cette contrainte — en laissant epoll orchestrer tout le flux.

01
La contrainte
Ce que le sujet interdit exactement

« Interdiction de vérifier errno pour ajuster le comportement du serveur après une opération read ou write. »

— webserv subject
Ce qui est interdit

Utiliser la valeur de errno pour décider quoi faire après un recv() ou send(). Le cas typique :

  • Tester errno == EAGAIN pour savoir si le buffer est vide/plein
  • Tester errno == EWOULDBLOCK pour re-arm un fd
  • Boucler « tant que EAGAIN » pour drainer un buffer

Tout pattern qui utilise errno comme signal de contrôle pour piloter la machine à états du serveur est interdit.

Ce qui est autorisé

Le retour de la fonction lui-même (n) reste utilisable :

  • n > 0 — des octets ont été lus/écrits
  • n == 0 — le client a fermé la connexion (recv)
  • n <= 0 — erreur réelle → fermer proprement

On décide uniquement sur la valeur de retour, jamais sur errno. C'est epoll qui nous dit quand un fd est prêt — pas errno.

02
Le piège classique
Pourquoi la plupart des tutos ne marchent pas pour nous
Le problème

Tous les tutos epoll montrent le même pattern edge-triggered : boucler sur recv/send jusqu'à EAGAIN. C'est le pattern « drain loop ». Il fonctionne parfaitement — mais il viole la règle du sujet.

Pattern classique ET
Interdit par le sujet 
// ❌ Le "drain loop" classique — INTERDIT
// On boucle jusqu'à EAGAIN pour vider le buffer kernel

void handle_read_INTERDIT(int fd, int epfd) {
    while (true) {
        char buf[4096];
        ssize_t n = recv(fd, buf, sizeof(buf), 0);
        if (n < 0) {
            if (errno == EAGAIN || errno == EWOULDBLOCK)  // ← INTERDIT
                break;   // "buffer vide, on sort"
            close(fd); return;
        }
        if (n == 0) { close(fd); return; }
        request += buf;
    }
}

void handle_write_INTERDIT(int fd, int epfd) {
    while (sent < total) {
        ssize_t n = send(fd, data + sent, total - sent, 0);
        if (n < 0) {
            if (errno == EAGAIN || errno == EWOULDBLOCK)  // ← INTERDIT
                break;   // "buffer plein, on attend"
            close(fd); return;
        }
        sent += n;
    }
}
⚠️
Pourquoi c'est tent&ant ? En Edge Triggered classique, epoll ne te notifie qu'une fois par changement d'état. Donc le réflexe est de boucler jusqu'à EAGAIN pour tout drainer. Mais le sujet dit non. On doit trouver une autre architecture.
03
L'architecture correcte
Laisser epoll orchestrer — un seul appel par événement
Le principe fondamental

Au lieu de boucler jusqu'à EAGAIN, on fait un seul recv() ou send() par événement epoll. On met à jour l'état de la connexion. Si ce n'est pas fini, on ne boucle pas — on laisse epoll nous re-notifier quand le fd est à nouveau prêt.

C'est epoll qui pilote le flux, pas nous. On ne décide jamais « est-ce que le buffer est vide ? » (ce serait checker errno). On dit simplement : « epoll, rappelle-moi quand il y a quelque chose ».

État par connexion — la struct Connection

Pour que ça marche, chaque connexion doit garder son état en mémoire entre deux événements. On ne peut plus tout faire d'un coup dans une seule fonction — il faut un objet qui survit entre les appels.

💡
ev.data.ptr au lieu de ev.data.fd. On stocke un pointeur vers la Connection directement dans l'événement epoll. Comme ça, quand epoll_wait() nous réveille, on retrouve immédiatement l'état de la connexion (requête en cours, réponse, octets envoyés) sans chercher dans une map.
04
handle_read()
Lire sans jamais checker errno
Le principe

epoll_wait() nous a dit EPOLLIN sur ce fd → il y a des données. On fait un seul recv(). Trois cas possibles, tous décidés sur la valeur de retour n :

  • n > 0 — on a reçu des données, on les accumule dans conn->request
  • n == 0 — le client a fermé la connexion (EOF) → on ferme
  • n < 0 — erreur réelle → on ferme

Si la requête n'est pas complète, on ne boucle pas. On sort. epoll nous rappellera quand il y aura plus de données à lire.

handle_read() — conforme au sujet
Zéro errno 
void handle_read(Connection* conn, int epfd) {
    char buf[4096];

    // UN SEUL recv() — epoll nous a dit que le fd est prêt
    ssize_t n = recv(conn->fd, buf, sizeof(buf) - 1, 0);

    if (n <= 0) {
        // n == 0 : client déconnecté
        // n < 0  : erreur réelle
        // Dans les deux cas : fermer proprement
        close_connection(conn, epfd);
        return;
    }

    // Accumuler les données reçues
    buf[n] = '\0';
    conn->request += buf;

    // Est-ce que la requête HTTP est complète ? (\r\n\r\n)
    if (conn->request.find("\r\n\r\n") != std::string::npos) {
        // Requête complète → préparer la réponse
        conn->response = build_response(conn->request);
        conn->sent = 0;

        // Basculer le fd de EPOLLIN → EPOLLOUT
        struct epoll_event ev;
        ev.events  = EPOLLOUT;
        ev.data.ptr = conn;
        epoll_ctl(epfd, EPOLL_CTL_MOD, conn->fd, &ev);
    }
    // Sinon : requête pas finie, on ne fait rien.
    // epoll nous re-notifiera EPOLLIN quand le client enverra la suite.
}
💡
Pas de boucle, pas d'errno. Si le client envoie une grosse requête en plusieurs paquets TCP, on reçoit un EPOLLIN par paquet. Chaque appel fait un seul recv(), accumule dans conn->request, et vérifie si on a le \r\n\r\n final. C'est epoll qui « boucle » pour nous en nous rappelant.
05
handle_write()
Écrire sans jamais checker errno
Le principe

epoll_wait() nous a dit EPOLLOUT → le buffer kernel a de la place. On fait un seul send(). Il envoie ce qu'il peut (peut-être tout, peut-être une partie).

  • n > 0 — on met à jour conn->sent += n
  • Si sent >= total — tout est parti → basculer sur EPOLLIN (keep-alive) ou fermer
  • Si sent < total — pas tout parti → ne rien faire. epoll nous rappellera avec EPOLLOUT
  • n <= 0 — erreur réelle → fermer
handle_write() — conforme au sujet
Zéro errno 
void handle_write(Connection* conn, int epfd) {
    // epoll nous a dit que le fd est prêt → on envoie ce qu'on peut
    ssize_t n = send(conn->fd,
                     conn->response.c_str() + conn->sent,
                     conn->response.size() - conn->sent,
                     0);

    if (n <= 0) {
        // Erreur réelle → fermer proprement
        close_connection(conn, epfd);
        return;
    }

    conn->sent += n;

    if (conn->sent >= conn->response.size()) {
        // Tout est envoyé → fermer ou keep-alive
        close_connection(conn, epfd);

        // Ou, pour keep-alive : rebasculer sur EPOLLIN
        // conn->request.clear();
        // conn->response.clear();
        // conn->sent = 0;
        // epoll_event ev;
        // ev.events  = EPOLLIN;
        // ev.data.ptr = conn;
        // epoll_ctl(epfd, EPOLL_CTL_MOD, conn->fd, &ev);
    }

    // Si sent < total : on ne fait RIEN.
    // On ne boucle pas. On ne teste pas errno.
    // epoll nous rappellera avec EPOLLOUT quand le buffer
    // kernel aura de la place pour la suite.
}
⚠️
EPOLL_CTL_MOD, pas ADD ! Le client_fd est déjà dans epoll (ajouté lors de l'accept). On utilise MOD pour changer les événements surveillés (de EPOLLIN à EPOLLOUT). EPOLL_CTL_ADD sur un fd déjà enregistré retournerait EEXIST.
06
La mécanique derrière
Pourquoi ça fonctionne sans errno
Le buffer kernel — la clé de tout

Chaque socket a deux buffers dans le kernel : un pour la lecture (recv), un pour l'écriture (send). epoll surveille l'état de ces buffers et te prévient :

  • EPOLLIN — le buffer de lecture contient des données → recv() ne bloquera pas
  • EPOLLOUT — le buffer d'écriture a de la place → send() ne bloquera pas

Quand on fait un recv()/send() après une notification epoll, on a la garantie que l'appel va fonctionner — au moins partiellement. Pas besoin de errno pour vérifier.

Le cycle de vie d'une requête — en détail
1
Client se connecte → EPOLLIN sur server_fd → on fait accept() → on crée une Connection → on enregistre client_fd avec EPOLLIN
↓
2
Client envoie des données → EPOLLIN sur client_fd → un recv() → accumule dans conn->request
↓ (répété si la requête arrive en plusieurs paquets)
3
Requête complète (\r\n\r\n trouvé) → on prépare conn->response → EPOLL_CTL_MOD vers EPOLLOUT
↓
4
Buffer d'écriture a de la place → EPOLLOUT → un send() → met à jour conn->sent
↓ (répété si la réponse est grande)
5
sent >= total → tout est envoyé → close() ou bascule vers EPOLLIN (keep-alive)
📦
Analogie — le facteur

Imagine que tu envoies un colis trop gros pour la boîte aux lettres. Au lieu de forcer (« boucler »), tu mets ce qui rentre, et tu laisses le facteur te prévenir quand la boîte est vidée. Le facteur, c'est epoll. Tu ne regardes jamais toi-même si la boîte est pleine (errno).

🔄
Analogie — LT re-notifications

En Level Triggered (pas de EPOLLET), epoll te rappelle tant que la condition est vraie. S'il reste des données à lire ou de la place pour écrire, tu seras re-notifié au prochain epoll_wait(). C'est ça qui remplace la boucle EAGAIN.

💡
Pourquoi Level Triggered est notre allié. En Level Triggered (le mode par défaut, sans EPOLLET), epoll re-notifie tant que des données sont disponibles ou que le buffer a de la place. C'est exactement ce qu'il nous faut : un seul recv()/send() par événement, et epoll fait le reste. En Edge Triggered, on serait obligé de drainer jusqu'à EAGAIN — ce qui est interdit. LT + un appel par événement = la solution parfaite.

Code final
Le serveur complet — zéro errno
main.cpp — serveur event-driven conforme au sujet
Code final complet 
#include <sys/socket.h>
#include <sys/epoll.h>
#include <arpa/inet.h>
#include <unistd.h>
#include <fcntl.h>
#include <csignal>
#include <cstring>
#include <iostream>
#include <map>
#include <string>

static const int PORT       = 8080;
static const int BACKLOG    = 10;
static const int MAX_EVENTS = 64;

// ── État par connexion ─────────────────────────────────
struct Connection {
    int         fd;
    std::string request;
    std::string response;
    size_t      sent;
};

std::map<int, Connection*> connections;

// ── Fermer proprement ─────────────────────────────────
void close_connection(Connection* conn, int epfd) {
    epoll_ctl(epfd, EPOLL_CTL_DEL, conn->fd, NULL);
    close(conn->fd);
    connections.erase(conn->fd);
    delete conn;
}

// ── Construire une réponse HTTP simple ───────────────
std::string build_response(const std::string& request) {
    std::string body = "<h1>Hello World!</h1>\n";
    std::string resp =
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: text/html\r\n"
        "Content-Length: " + std::to_string(body.size()) + "\r\n"
        "Connection: close\r\n"
        "\r\n" + body;
    return resp;
}

// ── Crée le socket serveur ────────────────────────────
int create_server_socket() {
    int fd = socket(AF_INET, SOCK_STREAM, 0);
    if (fd == -1) { perror("socket"); return -1; }

    int opt = 1;
    setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
    fcntl(fd, F_SETFL, fcntl(fd, F_GETFL) | O_NONBLOCK);
    return fd;
}

// ── Bind IP:port ──────────────────────────────────────
int bind_address(int fd, int port) {
    struct sockaddr_in addr;
    memset(&addr, 0, sizeof(addr));
    addr.sin_family      = AF_INET;
    addr.sin_port        = htons(port);
    addr.sin_addr.s_addr = INADDR_ANY;

    if (bind(fd, (struct sockaddr*)&addr, sizeof(addr)) == -1)
        { perror("bind"); return -1; }
    return 0;
}

// ── Accept : crée une Connection, enregistre dans epoll ─
void accept_next_client(int server_fd, int epfd) {
    struct sockaddr_in client_addr;
    socklen_t len = sizeof(client_addr);
    int client_fd = accept(server_fd,
                           (struct sockaddr*)&client_addr, &len);
    if (client_fd == -1) { perror("accept"); return; }

    fcntl(client_fd, F_SETFL, fcntl(client_fd, F_GETFL) | O_NONBLOCK);

    // Créer l'état de la connexion
    Connection* conn = new Connection();
    conn->fd   = client_fd;
    conn->sent = 0;
    connections[client_fd] = conn;

    // Enregistrer dans epoll avec EPOLLIN
    // data.ptr pointe vers notre Connection
    struct epoll_event ev;
    ev.events  = EPOLLIN;         // pas de EPOLLET → Level Triggered
    ev.data.ptr = conn;           // stocker le pointeur, pas le fd
    epoll_ctl(epfd, EPOLL_CTL_ADD, client_fd, &ev);

    std::cout << "[+] client fd=" << client_fd
              << " from " << inet_ntoa(client_addr.sin_addr) << "\n";
}

// ── Lecture : UN recv(), accumule, bascule si complet ──
void handle_read(Connection* conn, int epfd) {
    char buf[4096];
    ssize_t n = recv(conn->fd, buf, sizeof(buf) - 1, 0);

    if (n <= 0) {
        close_connection(conn, epfd);
        return;
    }

    buf[n] = '\0';
    conn->request += buf;

    if (conn->request.find("\r\n\r\n") != std::string::npos) {
        conn->response = build_response(conn->request);
        conn->sent = 0;

        struct epoll_event ev;
        ev.events  = EPOLLOUT;       // bascule lecture → écriture
        ev.data.ptr = conn;
        epoll_ctl(epfd, EPOLL_CTL_MOD, conn->fd, &ev);
    }
}

// ── Écriture : UN send(), met à jour sent, ferme si fini ─
void handle_write(Connection* conn, int epfd) {
    ssize_t n = send(conn->fd,
                     conn->response.c_str() + conn->sent,
                     conn->response.size() - conn->sent,
                     0);

    if (n <= 0) {
        close_connection(conn, epfd);
        return;
    }

    conn->sent += n;

    if (conn->sent >= conn->response.size())
        close_connection(conn, epfd);

    // Si pas tout envoyé : on ne fait rien.
    // EPOLLOUT (Level Triggered) nous re-notifiera.
}

// ── La boucle événementielle ────────────────────────────
int main() {
    signal(SIGPIPE, SIG_IGN);

    int server_fd = create_server_socket();
    if (server_fd == -1) return 1;
    if (bind_address(server_fd, PORT) == -1) return 1;
    if (listen(server_fd, BACKLOG) == -1)
        { perror("listen"); return 1; }

    int epfd = epoll_create1(0);
    if (epfd == -1) { perror("epoll_create1"); return 1; }

    // Enregistrer le listen socket
    struct epoll_event ev;
    ev.events  = EPOLLIN;
    ev.data.fd = server_fd;   // pour le listen, on utilise data.fd
    epoll_ctl(epfd, EPOLL_CTL_ADD, server_fd, &ev);

    std::cout << "Serveur sur http://localhost:" << PORT << "\n";

    struct epoll_event events[MAX_EVENTS];

    while (true) {
        int n = epoll_wait(epfd, events, MAX_EVENTS, -1);  // dort quand rien

        for (int i = 0; i < n; i++) {

            // Cas spécial : le listen socket (on a mis data.fd)
            if (events[i].data.fd == server_fd
                && (events[i].events & EPOLLIN)) {
                accept_next_client(server_fd, epfd);
                continue;
            }

            // Tous les autres fd : on a mis data.ptr
            Connection* conn = (Connection*)events[i].data.ptr;

            if (events[i].events & (EPOLLERR | EPOLLHUP)) {
                close_connection(conn, epfd);
            }
            else if (events[i].events & EPOLLIN) {
                handle_read(conn, epfd);
            }
            else if (events[i].events & EPOLLOUT) {
                handle_write(conn, epfd);
            }
        }
    }

    close(epfd);
    close(server_fd);
    return 0;
}
✅ Zéro errno. Aucun errno n'est testé. Toute la logique repose sur la valeur de retour de recv()/send() et les notifications epoll.
🔄 Un appel par événement. Un seul recv() par EPOLLIN, un seul send() par EPOLLOUT. C'est epoll qui « boucle » pour nous via ses re-notifications LT.
📋 Connection struct. L'état de chaque client survit entre les événements. request s'accumule, sent progresse.
💤 0% CPU au repos. epoll_wait(-1) dort. Plus de busy-wait, plus de boucle EAGAIN.
