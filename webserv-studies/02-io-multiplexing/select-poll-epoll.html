Contexte — I/O Multiplexing

select vs poll vs epoll
Pourquoi choisir epoll ?

Trois mécanismes, trois époques, un seul gagnant pour un serveur web haute performance. Comprendre leurs différences pour savoir pourquoi epoll est le choix évident sous Linux.

01
Le problème
Pourquoi a-t-on besoin de multiplexing ?
Le défi

Un serveur web doit gérer des centaines de connexions simultanées. Chaque connexion est un file descriptor (fd). Le serveur doit savoir lequel est prêt pour une lecture ou une écriture — sans bloquer sur un seul fd et ignorer les autres.

Sans multiplexing, deux options : un thread par connexion (coûteux en mémoire, en context switches) ou boucler en polling actif (brûle du CPU pour rien). Le multiplexing I/O résout ça : un seul thread surveille tout.

🏨
Analogie

Tu gères un hôtel de 500 chambres. Tu dois savoir qui a besoin de room service. Tu as trois stratégies : frapper à chaque porte (select), appeler chaque chambre au téléphone (poll), ou installer un bouton d'appel dans chaque chambre (epoll).

« It must be non-blocking and use only 1 poll() (or equivalent) for all the I/O operations between the clients and the server (listen included). »

— webserv.pdf, IV.1 Requirements
02
select()
L'ancêtre — simple mais limité
Comment ça marche

select() prend trois bitmaps de fds (lecture, écriture, exception) et attend qu'au moins un fd soit prêt. Le kernel parcourt tous les fds du bitmap à chaque appel et modifie les sets en place pour indiquer les fds prêts.

Tu dois reconstruire les sets à chaque itération car select les modifie. Et le nombre max de fds est fixé à FD_SETSIZE (typiquement 1024).

Les problèmes
  • Limite FD_SETSIZE = 1024 — impossible de surveiller plus de fds sans recompiler
  • Copie à chaque appel — les fd_set doivent être reconstruits car select les modifie
  • Parcours linéaire O(n) — le kernel scanne tous les fds, même les inactifs
  • Copie user ↔ kernel — les 3 sets sont copiés à chaque appel système
select_example.c
select() 
#include <sys/select.h>

// Prototype
int select(int nfds,             // fd le plus élevé + 1
           fd_set *readfds,        // fds à surveiller en lecture
           fd_set *writefds,       // fds à surveiller en écriture
           fd_set *exceptfds,      // fds à surveiller pour exceptions
           struct timeval *timeout // timeout (NULL = bloquant)
);

// ──────────────────────────────────────────────────
// Usage typique dans une boucle :
// ──────────────────────────────────────────────────

fd_set master_set, read_set;
FD_ZERO(&master_set);
FD_SET(listen_fd, &master_set);

while (1) {
    read_set = master_set;  // ⚠️ OBLIGATOIRE : reconstruire à chaque tour !

    int n = select(max_fd + 1, &read_set, NULL, NULL, NULL);

    // Parcourir TOUS les fds pour trouver les actifs
    for (int fd = 0; fd <= max_fd; fd++) {  // ⚠️ O(n) — on teste TOUT
        if (FD_ISSET(fd, &read_set)) {
            // fd est prêt en lecture
        }
    }
}
⚠️
Piège classique : oublier de reconstruire le fd_set avant chaque appel à select(). Les sets sont modifiés en place par le kernel — si tu ne les reconstruis pas, tu perds des fds de ta surveillance.
03
poll()
L'amélioration — sans la limite des 1024
Comment ça marche

poll() utilise un tableau de struct pollfd au lieu de bitmaps. Chaque entrée contient le fd, les événements à surveiller (events), et les événements retournés (revents). Pas de limite sur le nombre de fds.

L'avantage sur select : pas de limite FD_SETSIZE et la structure est plus propre. Le problème : le kernel parcourt encore linéairement tout le tableau à chaque appel — O(n).

Mieux que select, mais...
  • Pas de limite de fds — le tableau est dynamique, tu le dimensionnes toi-même
  • events / revents séparés — pas besoin de reconstruire le set, events n'est pas modifié
  • Toujours O(n) — le kernel parcourt tout le tableau, même les fds inactifs
  • Copie user ↔ kernel — tout le tableau est copié à chaque appel
poll_example.c
poll() 
#include <poll.h>

// Prototype
int poll(struct pollfd *fds,    // tableau de structures pollfd
         nfds_t nfds,              // nombre d'entrées dans le tableau
         int timeout               // timeout en ms (-1 = bloquant)
);

struct pollfd {
    int   fd;         // file descriptor
    short events;     // événements demandés (POLLIN, POLLOUT...)
    short revents;    // événements retournés par le kernel
};

// ──────────────────────────────────────────────────
// Usage typique :
// ──────────────────────────────────────────────────

struct pollfd fds[200];
fds[0].fd = listen_fd;
fds[0].events = POLLIN;
int nfds = 1;

while (1) {
    int n = poll(fds, nfds, -1);

    // Parcourir TOUT le tableau pour trouver les actifs
    for (int i = 0; i < nfds; i++) {       // ⚠️ O(n) — on teste TOUT
        if (fds[i].revents & POLLIN) {
            // fds[i].fd est prêt en lecture
        }
    }
}
💡
Le sujet 42 dit « poll() or equivalent ». poll est mentionné par nom dans le PDF, mais on a le droit d'utiliser un « équivalent ». epoll est cet équivalent — en mieux. C'est ce qu'on utilise.
04
epoll
Le gagnant — O(1) et stateful
Comment ça marche

epoll est fondamentalement différent. Au lieu de passer la liste de fds à chaque appel, tu enregistres les fds une seule fois via epoll_ctl(). Le kernel maintient une liste interne (interest list) et une ready list.

Quand tu appelles epoll_wait(), le kernel retourne uniquement les fds actifs. Pas de parcours linéaire, pas de copie de toute la liste, pas de reconstruction. C'est O(1) par rapport au nombre total de fds surveillés.

Les avantages décisifs
  • O(1) par rapport à N — seuls les fds actifs sont retournés, pas parcourus
  • Stateful — l'interest list persiste dans le kernel, pas de reconstruction
  • Pas de copie complète — seuls les événements prêts sont copiés vers l'userspace
  • Pas de limite — autant de fds que la mémoire le permet
  • Edge-triggered — mode EPOLLET pour ne notifier que sur les changements d'état
epoll_example.c
epoll 
#include <sys/epoll.h>

// 1. Créer l'instance (une seule fois)
int epfd = epoll_create1(0);

// 2. Enregistrer les fds (une seule fois par fd)
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = listen_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &ev);

// ──────────────────────────────────────────────────
// 3. La boucle événementielle :
// ──────────────────────────────────────────────────

struct epoll_event events[64];

while (1) {
    int n = epoll_wait(epfd, events, 64, -1);

    for (int i = 0; i < n; i++) {            // ✅ seulement les fds ACTIFS
        if (events[i].data.fd == listen_fd) {
            // nouvelle connexion → accept()
        } else {
            // client prêt → recv() / send()
        }
    }
}
🔔
Pourquoi c'est mieux

Avec select/poll, à chaque tour de boucle tu redonnes toute la liste et tu vérifies tout. Avec epoll, tu enregistres une fois et le kernel te dit directement qui est prêt. Sur 10 000 connexions dont 3 sont actives, select/poll vérifient 10 000 fds. epoll te donne les 3.

📊
En chiffres

Avec 10 000 fds et 10 actifs par tour : select copie ~1.2 KB de bitmaps et scanne 10 000 bits. poll copie ~80 KB de pollfd et scanne 10 000 entrées. epoll retourne juste 10 events — ~120 octets. Le reste vit dans le kernel.

la_difference_cle.c
Le noeud du problème 
// ═══════════════════════════════════════════════════
// La différence fondamentale en une ligne :
// ═══════════════════════════════════════════════════

// select / poll : à chaque tour de boucle
// → "Kernel, voici mes 10 000 fds. Lesquels sont prêts ?"
// → Le kernel copie la liste, la parcourt, retourne tout
// → Tu re-parcours tout pour trouver les 3 actifs

// epoll : UNE seule fois (enregistrement)
// → "Kernel, surveille ce fd." (× 10 000 au démarrage)
// → Ensuite, à chaque tour :
// → "Kernel, qui est prêt ?" → "Voici 3 fds."
// → Zéro copie, zéro parcours inutile
🐌 select/poll avec 10K fds : chaque tour copie et parcourt les 10 000 entrées, même si 3 sont actives.
⚡ epoll avec 10K fds : epoll_wait retourne directement les 3 fds actifs. Le reste vit tranquillement dans le kernel.
📈 Scalabilité : plus tu as de connexions inactives (idle), plus l'avantage d'epoll est énorme.
06
Verdict
Lequel choisir pour webserv ?
Verdict par mécanisme
🪦
select() — Non
Limite à 1024 fds, copie de 3 sets à chaque itération, parcours linéaire. Historique et pédagogique, mais obsolète pour un serveur web. On ne l'utilise pas.
⚠️
poll() — Acceptable mais insuffisant
Supprime la limite des 1024, structure plus propre. Mais toujours O(n) à chaque appel — le kernel parcourt tout le tableau. Utilisable pour un petit serveur, mais ne scale pas.
✅
epoll — Le bon choix
O(1), stateful, edge-triggered disponible. Le kernel fait le tri et ne te retourne que les fds actifs. C'est le mécanisme conçu pour les serveurs haute performance sous Linux. Nginx, HAProxy, Node.js — tout le monde utilise epoll.
webserv_event_loop.cpp
Notre choix pour webserv 
// La boucle événementielle de notre webserv
// Un seul epoll_wait() pour TOUT : listen, clients, CGI pipes

int epfd = epoll_create1(0);

// Enregistrer les sockets d'écoute (un par port configuré)
for (int i = 0; i < num_servers; i++) {
    struct epoll_event ev;
    ev.events = EPOLLIN;
    ev.data.fd = listen_fds[i];
    epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fds[i], &ev);
}

// La boucle principale — UN seul point de blocage
struct epoll_event events[256];
while (running) {
    int n = epoll_wait(epfd, events, 256, 1000);  // ← LE seul point bloquant

    for (int i = 0; i < n; i++) {
        int fd = events[i].data.fd;

        if (is_listen_fd(fd))
            handle_new_connection(epfd, fd);    // accept + epoll_ctl ADD
        else if (events[i].events & EPOLLIN)
            handle_read(fd);                      // recv dans un buffer
        else if (events[i].events & EPOLLOUT)
            handle_write(fd);                     // send la réponse
    }
}
⚠️
Rappel du sujet : epoll est Linux-only. Sur macOS, l'équivalent est kqueue. Sur les anciens Unix, c'est /dev/poll. Pour le projet 42, on code sous Linux — donc epoll est le choix naturel. Les serveurs portables (comme nginx) utilisent une couche d'abstraction pour supporter les deux.

« poll() (or equivalent) must monitor both reading and writing simultaneously. »

— webserv.pdf, IV.1 Requirements
