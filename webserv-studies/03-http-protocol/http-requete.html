03 — Le protocole HTTP

Structure d'une requête
HTTP

Qu'est-ce qu'un protocole ? Comment les couches réseau s'empilent ? Que contient exactement le texte que recv() te donne ? On décortique tout, du paquet TCP jusqu'au dernier header.

01
C'est quoi un protocole ?
Les couches reseau et ou HTTP se situe
Le concept

Un protocole, c'est un contrat entre deux machines. Il definit exactement : quel format de message envoyer, dans quel ordre, et ce que chaque octet signifie. Sans protocole, les deux machines ne se comprennent pas — comme deux personnes qui ne parlent pas la meme langue.

Le reseau fonctionne en couches empilees. Chaque couche fait un seul travail et s'appuie sur la couche d'en dessous. HTTP est tout en haut — il profite du travail de toutes les couches inferieures.

La difference cle entre ces protocoles

Tous tournent sur TCP/IP — mais chacun definit son propre format de message.
HTTP : texte lisible, requete/reponse, stateless (sans memoire entre requetes)
FTP : deux connexions (controle + donnees), binaire ou texte
SMTP : dialogue par commandes (HELO, MAIL FROM, DATA...), pour emails uniquement

Ce que chaque couche fait pour toi
  • Ethernet/Wi-Fi — transmet les bits physiquement. Tu ne t'en occupes jamais.
  • IP — route les paquets de machine en machine. 127.0.0.1 = boucle locale (toi-meme).
  • TCP — garantit que les octets arrivent dans l'ordre, sans perte, sans doublon. C'est TCP qui fait le 3-way handshake (SYN, SYN-ACK, ACK) et les retransmissions. C'est grace a TCP que recv() te donne un flux d'octets propre.
  • HTTP — c'est le protocole que toi tu parses. Ce qui sort de recv(), c'est du texte HTTP brut. A toi de le decoder.
💡
Ton serveur web ne fait PAS de TCP. Le kernel gere TCP pour toi (SYN/ACK, retransmissions, sequencement). Toi, tu appelles socket(), bind(), listen(), accept(), recv(), send() — et le kernel fait tout le TCP en dessous. Ce que tu recois dans recv(), c'est directement le contenu HTTP.
02
Ce que Wireshark montre
Les 10 paquets d'une requete HTTP complete
Le film d'une requete, image par image

Quand tu tapes curl http://localhost:8080, il se passe exactement 10 echanges sur le reseau. Wireshark les capture tous. Voici ce qui se passe — chaque ligne est un paquet TCP reel.

# temps direction protocole contenu role
① 3-way handshake TCP — etablir la connexion
10.000 client → serveur SYN Seq=0, Win=65495 Le client demande a se connecter
20.000 serveur → client SYN, ACK Seq=0, Ack=1 Le serveur accepte et repond
30.000 client → serveur ACK Seq=1, Ack=1 Le client confirme — connexion etablie
② Echange HTTP — la vraie requete
40.000 client → serveur HTTP GET / HTTP/1.1 Le client envoie la requete — c'est ca que ton recv() recoit
50.000 serveur → client ACK Seq=1, Ack=79 Le serveur accuse reception (TCP, pas toi)
60.000 serveur → client HTTP HTTP/1.1 200 OK (text/html) Ta reponse envoyee par send()
70.001 client → serveur ACK Seq=79, Ack=106 Le client accuse reception de la reponse
③ Fermeture TCP — couper proprement
80.001 serveur → client FIN, ACK Seq=106 close(fd) — le serveur lance la fermeture
90.001 client → serveur FIN, ACK Seq=79, Ack=107 Le client ferme aussi de son cote
100.001 serveur → client ACK Seq=107, Ack=80 Dernier ACK — connexion fermee proprement
Ce que le kernel fait (TCP)

Paquets 1-3 (handshake) et 5, 7 (ACKs) et 8-10 (fermeture). Tout ca est gere par accept() et close() — tu n'envoies jamais de SYN ou ACK toi-meme.

Ce que TOI tu geres (HTTP)

Paquet 4 : c'est le texte que recv() te donne. Tu le parses.
Paquet 6 : c'est le texte que tu construis et donnes a send().

⚠️
Tout ca se passe en 1 milliseconde. Les 10 paquets s'echangent en ~0.001 secondes en localhost. Sur un vrai reseau, le handshake prend ~1 RTT (aller-retour reseau). C'est pourquoi Connection: keep-alive est important : eviter de refaire le handshake a chaque requete.
03
Anatomie d'une requete
Decortiquer ce que recv() te donne
Voici exactement ce que ton serveur recoit

Quand un navigateur se connecte a localhost:8080, voici le texte brut que recv() te donne. Chaque ligne se termine par \r\n (retour chariot + saut de ligne). La fin des headers est marquee par une ligne vide (\r\n\r\n).

Requete reelle capturee depuis Firefox
GET / HTTP/1.1\r\n← Request Line Host: localhost:8080\r\n← Header obligatoire User-Agent: Mozilla/5.0 (X11; Ubuntu; Linux x86_64; rv:148.0) Gecko/20100101 Firefox/148.0\r\n Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8\r\n Accept-Language: en-US,en;q=0.9\r\n Accept-Encoding: gzip, deflate, br, zstd\r\n Connection: keep-alive\r\n← Important pour webserv Upgrade-Insecure-Requests: 1\r\n Sec-Fetch-Dest: document\r\n Sec-Fetch-Mode: navigate\r\n Sec-Fetch-Site: none\r\n Sec-Fetch-User: ?1\r\n Priority: u=0, i\r\n \r\n← Ligne vide = fin des headers
Les 3 parties d'une requete HTTP
Partie 1 — Request Line (toujours la premiere ligne)
GET /index.html HTTP/1.1
Methode — que faire (GET, POST, DELETE)
URI — quelle ressource (/, /img/photo.png)
Version — toujours HTTP/1.1 pour webserv
💡
HTTP/1.0 vs HTTP/1.1 — les différences clés
  • Host obligatoire — en HTTP/1.1, le header Host est requis. Sans lui, le serveur doit répondre 400 Bad Request. C'est ce header qui permet le virtual hosting (plusieurs sites sur la même IP/port).
  • Connexion persistante par défaut — en HTTP/1.0, chaque requête ouvrait une nouvelle connexion TCP. En HTTP/1.1, la connexion reste ouverte (Connection: keep-alive est implicite). Le client doit envoyer Connection: close pour demander la fermeture.
  • Transfer-Encoding: chunked — introduit en HTTP/1.1. Permet d'envoyer un body sans connaître sa taille à l'avance. N'existe pas en HTTP/1.0.
Partie 2 — Headers (paires cle: valeur, une par ligne)
Chaque header est sur une ligne : Nom: valeur\r\n. Les headers donnent des meta-informations sur la requete : qui envoie, quels formats sont acceptes, quelle taille fait le body, etc. Seul Host est obligatoire en HTTP/1.1.
Partie 3 — Body (optionnel, apres la ligne vide)
Le body est separe des headers par \r\n\r\n (une ligne vide). Les requetes GET n'ont generalement pas de body. Les requetes POST en ont un — sa taille est donnee par Content-Length ou il est en morceaux via Transfer-Encoding: chunked.
⚠️
Comment savoir quand la requete est finie ? Pas de marqueur de fin pour le body ! Pour GET : la requete est finie apres \r\n\r\n (pas de body). Pour POST : tu lis Content-Length octets apres \r\n\r\n. Pour chunked : tu lis chunk par chunk jusqu'au chunk de taille 0. C'est pour ca que recv() en boucle est necessaire — tu ne recois pas toujours tout d'un coup.
04
Les methodes HTTP
GET, POST, DELETE — ce que webserv doit gerer
GET
Recuperer une ressource. C'est la methode la plus courante — chaque page web, chaque image, chaque CSS est un GET. Pas de body dans la requete.
Exemple : GET /index.html HTTP/1.1
Body requete : non • Body reponse : oui
POST
Envoyer des donnees au serveur. Formulaires, upload de fichiers, soumissions. Le body contient les donnees. Content-Length ou Transfer-Encoding: chunked indique la taille.
Exemple : POST /upload HTTP/1.1
Body requete : oui • Body reponse : oui
DELETE
Supprimer une ressource. Le serveur doit verifier les permissions avant de supprimer. Pas de body dans la requete.
Exemple : DELETE /uploads/photo.jpg HTTP/1.1
Body requete : non • Body reponse : optionnel
💡
Dans le fichier de config, chaque location peut restreindre les methodes autorisees. Par exemple : limit_except GET POST; refusera toute requete DELETE sur cette route avec un 405 Method Not Allowed.
Exemple : une requete POST avec body
Requete POST — formulaire
Ce que recv() donne

Content-Length: 27                                    ← taille du body !

05
Les headers en detail
Ce que chaque header signifie et lesquels tu dois gerer
Headers que tu recois (requete du client)
HeaderRoleImportance pour webserv
Host Le nom de domaine + port demande. Permet de faire du virtual hosting (plusieurs sites sur le meme serveur). Obligatoire
Content-Length Taille du body en octets. Sans ca, impossible de savoir quand le body du POST est fini. Critique pour POST
Transfer-Encoding Si la valeur est chunked, le body arrive en morceaux. Pas de Content-Length dans ce cas. A gerer
Connection keep-alive = garder la connexion ouverte pour d'autres requetes. close = fermer apres la reponse. A gerer
Content-Type Le type MIME du body. Pour un formulaire : application/x-www-form-urlencoded. Pour un upload : multipart/form-data. Pour CGI
User-Agent Le navigateur/client qui envoie. Firefox, Chrome, curl... Informatif, pas critique pour toi. Optionnel
Accept Les types MIME que le client accepte en reponse. Tu peux l'ignorer — envoie ce que tu as. Optionnel
Accept-Encoding Compression acceptee (gzip, deflate, br). webserv n'est pas oblige de compresser. Ignorable
Sec-Fetch-* Headers de securite modernes du navigateur. Informatifs. Tu peux les ignorer. Ignorable
⚠️
client_max_body_size — Dans ta config, cette directive limite la taille du body accepte. Si Content-Length depasse cette limite, tu dois renvoyer 413 Payload Too Large immediatement, sans lire le body. C'est une protection contre les uploads geants.
06
La reponse HTTP
Ce que ton send() doit envoyer
Structure de la reponse

La reponse a la meme structure que la requete : une premiere ligne (status line), des headers, une ligne vide, puis le body. Sauf que la premiere ligne contient le status code au lieu de la methode.

Reponse HTTP — ce que send() envoie
Reponse complete 
HTTP/1.1 200 OK                                   ← status line
Content-Type: text/html                                ← headers de reponse
Content-Length: 22                                     ← OBLIGATOIRE
Connection: close
                                                        ← ligne vide
<h1>Hello World!</h1>                                  ← body (22 octets)
Headers de reponse que tu dois envoyer
HeaderRoleObligatoire ?
Content-Type Le type MIME du body : text/html, image/png, application/json... Determine par l'extension du fichier. Oui
Content-Length La taille exacte du body en octets. Le navigateur en a besoin pour savoir quand il a tout recu. Oui
Connection close ou keep-alive. Si keep-alive, ne pas fermer le fd apres send(). Recommande
Location Pour les redirections (301, 302). Indique la nouvelle URL : Location: /new-page. Si redirection
💡
La reponse est une seule chaine de caracteres. Tu la construis en concatenant la status line + headers + \r\n + body, puis tu donnes tout a send(). C'est du texte brut — pas de magie, pas de serialisation.
07
Les status codes
Les codes que ton serveur doit renvoyer
La regle

Le sujet dit : "Your HTTP response status codes must be accurate." Chaque situation a son code — ne renvoie pas 200 pour tout. Le premier chiffre indique la categorie :

  • 2xx — Succes (tout va bien)
  • 3xx — Redirection (va voir ailleurs)
  • 4xx — Erreur client (la requete est mauvaise)
  • 5xx — Erreur serveur (c'est ta faute)
2xx — Succes
200 OK — la ressource est la
201 Created — POST reussi, ressource creee
204 No Content — DELETE reussi, pas de body
3xx — Redirection
301 Moved Permanently — URL change definitivement
302 Found — redirection temporaire
4xx — Erreur client
400 Bad Request — requete malformee
403 Forbidden — pas de permission
404 Not Found — la ressource n'existe pas
405 Method Not Allowed — methode interdite ici
413 Payload Too Large — body trop gros
5xx — Erreur serveur
500 Internal Server Error — bug dans ton code
502 Bad Gateway — CGI a plante
504 Gateway Timeout — CGI trop lent
⚠️
error_page — Dans la config, tu peux definir des pages d'erreur custom : error_page 404 /custom_404.html;. Si cette directive existe pour un code, ton serveur doit renvoyer le contenu de ce fichier au lieu d'un HTML generique.
