03 — HTTP Protocol — Parsing avancé

Chunked Transfer-Encoding
Quand le client ne connaît pas la taille

Avec Content-Length, le serveur sait combien d'octets lire. Avec Transfer-Encoding: chunked, les données arrivent morceau par morceau, chacun précédé de sa taille en hexadécimal. Le serveur doit les reconstituer.

01
Content-Length
Le cas normal — on connaît la taille
Ce qu'on sait déjà

Quand un client envoie un POST avec un body, il met Content-Length: 102400 dans ses headers. Le serveur sait exactement combien d'octets lire avant de traiter la requête.

C'est le cas qu'on a déjà vu fonctionner avec tshark et notre serveur main3.cpp.

Requête POST avec Content-Length
Cas normal 
POST /upload HTTP/1.1
Host: localhost:8080
Content-Type: application/octet-stream
Content-Length: 102400                    ← le serveur SAIT : 102400 octets à lire

[102400 octets de body]
Ce qu'on a vu dans tshark

L'envoi de 100KB avec curl — TCP fragmente le body en plusieurs paquets, le serveur accumule avec recv() jusqu'à atteindre Content-Length :

#LenInfo
1-3—SYN TCP 3-way handshake
4149POST /upload HTTP/1.1 + headers
632741DATA body partie 1
832741DATA body partie 2
1054DATA body partie 3
1232741DATA body partie 4
134123DATA body partie 5 — total = 102400
15105HTTP 200 OK (réponse du serveur)
17-19—FIN fermeture TCP
💡
Le serveur sait quand s'arrêter : il a lu Content-Length: 102400 dans les headers, donc il accumule les recv() jusqu'à ce que body_received >= 102400. Simple.
02
Le problème
Quand le client ne connaît pas la taille à l'avance
Pourquoi Content-Length ne suffit pas toujours

Parfois le client ne connaît pas la taille totale avant de commencer à envoyer. Exemples concrets :

  • Un script qui génère un rapport ligne par ligne depuis une base de données — combien de lignes ? inconnu
  • Un outil qui streame de la vidéo ou de l'audio en temps réel
  • curl qui lit depuis stdin — il ne peut pas prédire combien de données vont arriver
  • Un proxy qui retransmet sans tout bufferiser d'abord

Pour écrire Content-Length: ???, il faudrait tout générer en mémoire d'abord, compter, puis envoyer. Sur des gigabytes, c'est impossible.

Sans chunked — impossible

Le client devrait :

  • Générer tout le body en mémoire
  • Compter les octets
  • Écrire Content-Length
  • Puis envoyer

Si le body fait 2GB, il faut 2GB de RAM juste pour compter la taille.

Avec chunked — on streame

Le client dit :

  • Transfer-Encoding: chunked
  • Envoie morceau par morceau
  • Chaque morceau avec sa propre taille
  • Chunk de taille 0 = « c'est fini »

Pas besoin de connaître la taille totale. On envoie au fur et à mesure.

03
Le format chunked
Décrypté octet par octet
Règle d'or

La taille de chaque chunk est toujours en hexadécimal, suivie de \r\n. Puis les données du chunk (exactement N octets), suivies de \r\n. Un chunk de taille 0 signale la fin.

Cas normal
Content-Length 
POST /upload HTTP/1.1
Content-Length: 26

abcdefghijklmnopqrstuvwxyz
Chunked
Transfer-Encoding 
POST /upload HTTP/1.1
Transfer-Encoding: chunked

a\r\n
abcdefghij\r\n
10\r\n
klmnopqrstuvwxyz\r\n
0\r\n
\r\n
Décryptage octet par octet
a\r\n
→
Taille du chunk : 0xa = 10 en décimal
abcdefghij\r\n
→
Les 10 octets du chunk + terminateur \r\n
10\r\n
→
Taille du chunk : 0x10 = 16 en décimal
klmnopqrstuvwxyz\r\n
→
Les 16 octets du chunk + terminateur \r\n
0\r\n
→
Chunk de taille zéro = signal de fin
\r\n
→
Terminateur final (fin de la liste de chunks)
⚠️
Piège hexa ! 10 en hexadécimal = 16 en décimal, pas 10. a = 10, ff = 255, 1000 = 4096. Toujours parser avec strtol(str, NULL, 16) en base 16.
04
Sur le réseau
Ce que tshark montre — chunked vs Content-Length
La différence visible dans tshark

Avec Content-Length, tshark sait précisément où finit le body. Avec chunked, tshark doit décoder les chunks pour reconstituer le message. Ce que tshark fait automatiquement — mais ton serveur doit le faire lui-même.

Test avec curl chunked
Terminal 
# Forcer le mode chunked avec curl :
curl -X POST http://localhost:8080/upload \
     -H "Transfer-Encoding: chunked" \
     -H "Content-Type: text/plain" \
     --data-binary "hello world"
Content-Length — dans les headers

Le serveur voit :

  • Content-Length: 102400
  • Puis le body brut, continu
  • Condition de fin : body_received >= 102400
Chunked — dans le body

Le serveur voit :

  • Transfer-Encoding: chunked
  • Puis des chunks avec tailles en hexa
  • Condition de fin : chunk de taille 0
Ce que le serveur reçoit (vue raw)
recv() buffer 
// Ce que recv() retourne quand curl envoie "hello world" en chunked :

"POST /upload HTTP/1.1\r\n"
"Host: localhost:8080\r\n"
"Transfer-Encoding: chunked\r\n"      ← pas de Content-Length !
"Content-Type: text/plain\r\n"
"\r\n"                                   ← fin des headers
"b\r\n"                                  ← 0xb = 11 octets
"hello world\r\n"                       ← le chunk (11 octets)
"0\r\n"                                  ← chunk terminal
"\r\n"                                   ← fin
💡
Le \r\n\r\n ne suffit plus. Dans notre code main3.cpp, on bascule vers EPOLLOUT dès qu'on trouve \r\n\r\n (fin des headers). Mais avec chunked, le body suit les headers et il faut encore accumuler les chunks jusqu'au chunk terminal 0\r\n\r\n.
05
decode_chunked()
Le parser — reconstituer le body
L'algorithme, pas à pas

Le parser chunked répète toujours la même séquence :

  • 1. Lire la ligne de taille (tout jusqu'au prochain \r\n)
  • 2. Parser cette taille en hexa → int chunk_size
  • 3. Si chunk_size == 0 → c'est fini
  • 4. Lire exactement chunk_size octets → les ajouter au body
  • 5. Sauter le \r\n après le chunk
  • 6. Retour à l'étape 1
decode_chunked() — parser complet
C++98 
// Prend le body brut (tout après \r\n\r\n) et retourne le body décodé
std::string decode_chunked(const std::string& raw_body) {
    std::string result;
    size_t pos = 0;

    while (pos < raw_body.size()) {
        // 1. Trouver la ligne de taille (terminée par \r\n)
        size_t crlf = raw_body.find("\r\n", pos);
        if (crlf == std::string::npos)
            break;  // données incomplètes, attendre plus

        // 2. Parser la taille en hexadécimal
        std::string size_str = raw_body.substr(pos, crlf - pos);
        long chunk_size = strtol(size_str.c_str(), NULL, 16);  // base 16 !

        // 3. Chunk de taille 0 = fin
        if (chunk_size == 0)
            break;

        // 4. Avancer après le \r\n de la taille
        pos = crlf + 2;

        // Vérifier qu'on a assez de données
        if (pos + chunk_size + 2 > raw_body.size())
            break;  // chunk incomplet, attendre plus

        // 5. Extraire les données du chunk
        result += raw_body.substr(pos, chunk_size);

        // 6. Sauter les données + le \r\n après le chunk
        pos += chunk_size + 2;
    }

    return result;  // body propre, reconstitué
}
Exemple de décodage pas à pas

Entrée : a\r\nabcdefghij\r\n10\r\nklmnopqrstuvwxyz\r\n0\r\n\r\n

  • Tour 1 : "a" → strtol → 10 → lire 10 octets → "abcdefghij"
  • Tour 2 : "10" → strtol → 16 → lire 16 octets → "klmnopqrstuvwxyz"
  • Tour 3 : "0" → strtol → 0 → fin

Sortie : "abcdefghijklmnopqrstuvwxyz" — le body original, propre, sans métadonnées.

06
Le piège TCP
Les chunks fragmentés entre plusieurs recv()
Le problème qu'on connaît déjà

On l'a vu avec le body normal : TCP coupe n'importe où. Avec chunked, c'est encore pire — TCP peut couper au milieu d'un chunk, au milieu de la taille hexadécimale, ou entre la taille et les données.

TCP fragmente en plein milieu
Danger 
// Les 3 appels recv() retournent :

// recv() #1 :
"POST /upload HTTP/1.1\r\nTransfer-Encoding: chunked\r\n\r\na\r\nabc"
//                                                              ^^^
//                               coupé EN PLEIN MILIEU du chunk ! (3 octets sur 10)

// recv() #2 :
"defghij\r\n10\r\nklmno"
//                  ^^^^^
//      fin du chunk 1 + début du chunk 2 (5 octets sur 16)

// recv() #3 :
"pqrstuvwxyz\r\n0\r\n\r\n"
//  fin du chunk 2 + chunk terminal
⚠️
On ne peut PAS décoder chunk par chunk au fil des recv(). Un chunk peut être réparti sur 2, 3, 10 appels recv(). La taille hexa elle-même peut être coupée ("1" puis "0\r\n"). Le parser doit travailler sur le buffer accumulé complet.
La solution : accumuler, puis décoder

Exactement le même principe que pour Content-Length :

  • Chaque recv() → on accumule dans conn->request
  • Après chaque accumulation, on vérifie si le chunk terminal est arrivé
  • Si oui → on décode tout d'un coup avec decode_chunked()
  • Si non → on attend le prochain EPOLLIN
handle_read() — avec support chunked
Modifié 
void handle_read(Connection* conn, int epfd) {
    char buf[4096];
    ssize_t n = recv(conn->fd, buf, sizeof(buf) - 1, 0);

    if (n <= 0) { close_connection(conn, epfd); return; }

    buf[n] = '\0';
    conn->request += buf;

    // Headers complets ?
    size_t header_end = conn->request.find("\r\n\r\n");
    if (header_end == std::string::npos)
        return;  // headers pas encore complets

    // Détecter le mode : Content-Length ou chunked ?
    bool is_chunked = conn->request.find("Transfer-Encoding: chunked")
                      != std::string::npos;

    size_t body_start = header_end + 4;
    std::string raw_body = conn->request.substr(body_start);

    if (!is_chunked) {
        // Cas normal : vérifier Content-Length
        int cl = parse_content_length(conn->request);
        if ((int)raw_body.size() < cl)
            return;  // body pas complet
    } else {
        // Cas chunked : le chunk terminal est-il arrivé ?
        if (raw_body.find("0\r\n\r\n") == std::string::npos)
            return;  // pas fini, attendre

        // Décoder les chunks → body brut
        conn->decoded_body = decode_chunked(raw_body);
    }

    // Requête complète → basculer vers EPOLLOUT
    conn->response = build_response(conn->request);
    conn->sent = 0;

    struct epoll_event ev;
    ev.events  = EPOLLOUT;
    ev.data.ptr = conn;
    epoll_ctl(epfd, EPOLL_CTL_MOD, conn->fd, &ev);
}
07
Chunked & CGI
Pourquoi le serveur DOIT décoder avant le CGI

« For chunked requests, your server needs to un-chunk them, the CGI will expect EOF as the end of the body. »

— webserv.pdf, IV.3
Le CGI ne comprend pas HTTP

Un CGI (script Python, PHP, etc.) est un programme simple qui lit son stdin jusqu'à EOF. Il ne connaît pas le protocole HTTP. Il ne sait pas ce qu'est un chunk, un \r\n de terminaison, ou une taille en hexa.

Si tu lui envoies b\r\nhello world\r\n0\r\n\r\n, il va interpréter le b et les \r\n comme des données — ce qui est faux.

Flux : Client → Serveur → CGI
Client envoie
→
b\r\nhello world\r\n0\r\n\r\n
chunked
↓ decode_chunked()
Serveur écrit au pipe
→
hello world
body brut
↓ close(pipe) → EOF
CGI lit stdin
→
hello worldEOF
OK
Envoyer le body au CGI
CGI 
// Après fork(), dans le processus parent :

// Le body à envoyer au CGI est le body DÉCODÉ,
// PAS le body brut avec les métadonnées chunked.

std::string body;
if (is_chunked)
    body = conn->decoded_body;    // reconstitué par decode_chunked()
else
    body = raw_body;              // body brut (déjà propre)

// Écrire dans le pipe stdin du CGI
write(pipe_stdin[1], body.c_str(), body.size());
close(pipe_stdin[1]);  // ← fermer = envoyer EOF au CGI
💡
Le close(pipe) est crucial. Le CGI lit son stdin avec une boucle read() jusqu'à EOF. L'EOF n'arrive que quand tous les fd du pipe sont fermés. Si tu oublies le close(), le CGI bloque indéfiniment.
