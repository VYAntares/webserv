03 — HTTP Protocol — Upload

Upload de fichiers
multipart/form-data décrypté

Le sujet dit « clients must be able to upload files ». Les uploads passent par un POST avec un Content-Type spécial : multipart/form-data. Le body contient des « parts » séparées par un boundary. Chaque part a ses propres headers et son contenu binaire.

01
POST & Content-Type
Le client dit ce qu'il envoie

« Clients must be able to upload files. »

— webserv.pdf, IV.1 Requirements
Rappel : le sens du Content-Type

On a vu dans la page MIME que le Content-Type circule dans les deux sens. Pour GET, c'est toi (le serveur) qui produis le Content-Type dans ta réponse — tu dis « je te sers du text/css ».

Pour POST, c'est le client qui t'envoie un Content-Type dans sa requête. Il te dit « je t'envoie du application/json » ou « je t'envoie du multipart/form-data ». Et toi, tu lis ce header pour savoir comment parser le body.

GET vs POST — qui produit le Content-Type
GET
→
Client demande /style.css
←
Content-Type: text/css ← toi
POST
→
Content-Type: multipart/form-data ← lui
←
201 Created ← toi
💡
Le Content-Type du POST te dit comment parser. Sans lire ce header, tu ne peux pas savoir si le body contient username=alice ou des octets binaires d'une image JPEG séparés par des boundaries.
02
Les 3 formats POST
Formulaire, upload, JSON
Quand un client fait POST, il choisit l'un de ces 3 formats

C'est le client qui décide du format. Le navigateur choisit automatiquement selon le type de <form>. Une API JavaScript peut envoyer du JSON. Ton serveur doit lire le Content-Type pour savoir lequel il reçoit.

Format 1 — Formulaire simple
url-encoded 
// Le HTML :
<form action="/login" method="POST">
  <input name="username">
  <input name="password" type="password">
</form>

// Ce que recv() te donne :
POST /login HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Content-Length: 31

username=alice&password=1234&age=25
//       ^^^^^^^^^^^^^^^^^^^^^^^^^^^
//       clé=valeur séparés par &
//       même format qu'une URL query string
Format 2 — Upload de fichier
multipart 
// Le HTML :
<form action="/upload" method="POST" enctype="multipart/form-data">
  <input name="avatar" type="file">
</form>
//                        ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                        c'est le enctype qui déclenche multipart

// Ce que recv() te donne :
POST /upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----abc123
Content-Length: 248

------abc123
Content-Disposition: form-data; name="avatar"; filename="photo.jpg"
Content-Type: image/jpeg

[octets binaires JPEG]
------abc123--
Format 3 — JSON (APIs)
json 
// Envoyé par JavaScript (fetch / XMLHttpRequest) :
POST /api/users HTTP/1.1
Content-Type: application/json
Content-Length: 42

{"username": "alice", "password": "1234"}

// Pour webserv : tu passes le body tel quel au CGI.
// C'est le CGI (Python, PHP) qui parse le JSON.
Format Content-Type Fichiers binaires Parsing Utilisé pour
url-encoded application/x-www-form-urlencoded non simple — split('&') formulaires texte
multipart multipart/form-data oui complexe — boundaries uploads
json application/json non passé au CGI APIs JavaScript
⚠️
Pour webserv, tu DOIS implémenter multipart/form-data — c'est le seul format qui permet les uploads de fichiers. url-encoded et json sont généralement passés tels quels au CGI.
03
Le boundary
Le séparateur inventé par le client
Pourquoi un boundary ?

Un formulaire peut envoyer plusieurs choses en même temps — un fichier image, un nom d'utilisateur, un commentaire. Comment mettre tout ça dans un seul body HTTP ?

La solution : on découpe le body en « parts » séparées par un boundary — une chaîne de caractères unique inventée par le client. C'est comme un séparateur de colonne dans un CSV, mais pour des données binaires.

Le client te dit son boundary dans le header
Header 
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4
//                                  ^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                                  mot-clé   la chaîne choisie par le client
Règles du boundary dans le body
Dans le header Content-Type
→
----abc123
valeur brute
Dans le body (séparateur)
→
------abc123
-- ajouté devant
Dans le body (fin / dernier)
→
------abc123--
-- devant + -- après
💡
Le -- devant est une règle fixe du standard. Le boundary dans le Content-Type c'est ----abc123, mais dans le body chaque occurrence commence par ------abc123 (deux tirets ajoutés). Et le tout dernier a aussi -- à la fin.
Extraire le boundary en C++98

Tu cherches boundary= dans le header Content-Type et tu prends tout ce qui suit. Le boundary peut être avec ou sans guillemets.

extract_boundary()
C++98 
std::string extract_boundary(const std::string& content_type) {
    // Chercher "boundary="
    size_t pos = content_type.find("boundary=");
    if (pos == std::string::npos)
        return "";
    pos += 9;  // strlen("boundary=")

    // Retirer les guillemets si présents
    std::string b = content_type.substr(pos);
    if (b.size() >= 2 && b[0] == '"')
        b = b.substr(1, b.size() - 2);
    return "--" + b;  // ← on ajoute -- tout de suite !
}
04
Anatomie complète
Chaque octet décrypté sur le wire
Un formulaire avec un champ texte + un fichier image

Voici exactement ce que recv() te donne pour un formulaire HTML qui envoie un nom d'utilisateur et un avatar :

Requête multipart complète (annotée)
Wire 
// ─── HEADERS HTTP ───────────────────────
POST /upload HTTP/1.1
Host: localhost:8080
Content-Type: multipart/form-data; boundary=----abc123
Content-Length: 348
                                        ← \r\n\r\n = fin des headers HTTP
// ─── PART 1 : champ texte ───────────────
------abc123\r\n                        ← boundary (séparateur)
Content-Disposition: form-data; name="username"\r\n
\r\n                                    ← fin des headers de la part
alice\r\n                               ← contenu de la part

// ─── PART 2 : fichier ───────────────────
------abc123\r\n                        ← boundary (nouvelle part)
Content-Disposition: form-data; name="avatar"; filename="photo.jpg"\r\n
Content-Type: image/jpeg\r\n             ← MIME du fichier uploadé
\r\n                                    ← fin des headers de la part
[octets binaires de la photo JPEG]\r\n   ← contenu binaire

// ─── FIN ────────────────────────────────
------abc123--\r\n                     ← boundary final (-- après)
Structure d'une part — toujours le même pattern
--boundary\r\n
→
Début de la part
Content-Disposition: form-data; name="champ"
→
Nom du champ (+ filename si fichier)
Content-Type: image/jpeg
→
MIME du fichier (absent pour les champs texte)
\r\n
→
Ligne vide = fin des headers, début du contenu
[contenu : texte ou bytes binaires]
→
Les données de cette part
Content-Disposition

Ce header est toujours présent dans chaque part. Il contient :

  • name="..." — le nom du champ du formulaire HTML
  • filename="..." — le nom original du fichier (seulement si c'est un fichier)

Si filename est présent → c'est un fichier à sauvegarder. Sinon → c'est un champ texte simple.

Content-Type de la part

Chaque part peut avoir son propre Content-Type. C'est le MIME type du fichier uploadé.

  • Photo JPEG → image/jpeg
  • Document PDF → application/pdf
  • Champ texte → pas de Content-Type (implicitement text/plain)

C'est le navigateur qui détecte le type du fichier et l'écrit ici.

05
Le piège binaire
Les bytes d'une image ne respectent rien
⚠️
Le contenu d'un fichier uploadé c'est des bytes bruts. Une photo JPEG contient des octets avec toutes les valeurs possibles, y compris des \0, des \r\n, des -- qui ne signifient rien. Tu ne peux pas utiliser std::string::find("\r\n") naïvement pour chercher la fin du contenu.
Pourquoi chercher \r\n seul ne marche pas
Part 2
→
\xFF\xD8\xFF\xE0...\r\n...\x00\x00...--...\r\n...\xFF\xD9
bytes JPEG : contient des \r\n et -- par hasard
Fin
→
\r\n------abc123
seul vrai séparateur = \r\n + boundary complet
La bonne approche

Pour trouver la fin du contenu d'une part, tu cherches "\r\n--" + boundary dans le buffer — pas juste \r\n. Le boundary complet est la seule séquence garantie de ne pas apparaître dans les données (c'est pour ça qu'il est long et aléatoire).

Trouver la fin du contenu d'une part
C++98 
// boundary = "------abc123" (déjà avec le -- devant)

// FAUX : s'arrête sur un \r\n dans les bytes JPEG
size_t end = body.find("\r\n", start);

// CORRECT : cherche le boundary complet
size_t end = body.find("\r\n" + boundary, start);

// Le contenu de la part = tout entre start et end
std::string content = body.substr(start, end - start);
06
parse_multipart()
Implémentation complète en C++98
Les étapes du parsing
  • 1. Extraire le boundary du header Content-Type
  • 2. Splitter le body sur "--" + boundary
  • 3. Pour chaque part : séparer headers / contenu au \r\n\r\n
  • 4. Parser Content-Disposition → extraire name et filename
  • 5. Si filename existe → sauvegarder le contenu dans le dossier d'upload
  • 6. Répondre 201 Created
Structure Part
C++98 
struct Part {
    std::string name;       // name="username"
    std::string filename;   // filename="photo.jpg" (vide si champ texte)
    std::string mime_type;  // Content-Type de la part
    std::string content;    // contenu brut (bytes)
};
Extraire une valeur d'un header de part
Helper 
// Extraire la valeur entre guillemets après un mot-clé
// ex: extract_value("name=\"avatar\"; filename=\"photo.jpg\"", "filename")
//  → "photo.jpg"
std::string extract_value(const std::string& header,
                           const std::string& key) {
    std::string search = key + "=\"";
    size_t pos = header.find(search);
    if (pos == std::string::npos)
        return "";
    pos += search.size();
    size_t end = header.find('"', pos);
    if (end == std::string::npos)
        return "";
    return header.substr(pos, end - pos);
}
parse_multipart() — le parser complet
Parser 
std::vector<Part> parse_multipart(const std::string& body,
                                     const std::string& boundary) {
    std::vector<Part> parts;
    std::string delim = boundary;        // "------abc123"
    std::string end_delim = boundary + "--"; // "------abc123--"

    size_t pos = body.find(delim);
    if (pos == std::string::npos)
        return parts;
    pos += delim.size() + 2;  // sauter boundary + \r\n

    while (pos < body.size()) {
        // 1. Trouver la fin de la part (prochain boundary)
        size_t next = body.find("\r\n" + delim, pos);
        if (next == std::string::npos)
            break;

        // 2. Séparer headers / contenu au \r\n\r\n
        size_t header_end = body.find("\r\n\r\n", pos);
        if (header_end == std::string::npos || header_end > next)
            break;

        std::string headers = body.substr(pos, header_end - pos);
        size_t content_start = header_end + 4;  // après \r\n\r\n

        // 3. Construire la Part
        Part part;
        part.name     = extract_value(headers, "name");
        part.filename = extract_value(headers, "filename");
        part.content  = body.substr(content_start,
                                        next - content_start);

        // 4. Content-Type de la part (optionnel)
        size_t ct = headers.find("Content-Type: ");
        if (ct != std::string::npos) {
            ct += 14;
            size_t ct_end = headers.find("\r\n", ct);
            part.mime_type = headers.substr(ct,
                ct_end == std::string::npos ? std::string::npos : ct_end - ct);
        }

        parts.push_back(part);

        // 5. Avancer au-delà du boundary
        pos = next + 2 + delim.size();  // \r\n + boundary
        // Vérifier si c'est le boundary final (--)
        if (body.substr(pos, 2) == "--")
            break;
        pos += 2;  // sauter \r\n après le boundary
    }
    return parts;
}
Sauvegarder les fichiers uploadés
Concret 
// Après parsing, traiter chaque part :
for (size_t i = 0; i < parts.size(); i++) {
    if (parts[i].filename.empty()) {
        // Champ texte : parts[i].name = "username"
        //               parts[i].content = "alice"
        continue;
    }

    // Fichier : sauvegarder dans le dossier d'upload
    std::string path = upload_dir + "/" + parts[i].filename;
    std::ofstream ofs(path.c_str(),
                      std::ios::binary);  // ← BINARY !
    ofs.write(parts[i].content.c_str(),
              parts[i].content.size());
    ofs.close();
}

Pièges webserv
Ce qui va casser si tu ne fais pas attention
⚠️
Ouvrir le fichier en mode binaire. std::ofstream ofs(path.c_str(), std::ios::binary). Sans std::ios::binary, le système peut transformer des \n en \r\n (ou l'inverse) et corrompre l'image. Une photo JPEG qui passe par une conversion de retour à la ligne est cassée.
⚠️
Ne pas utiliser strlen() ou c_str() comme taille. Un fichier binaire peut contenir des \0. strlen() s'arrête au premier \0 et tu perds la fin du fichier. Utilise .size() sur le std::string qui, lui, stocke les \0 sans problème.
⚠️
Multipart + chunked = les deux problèmes combinés. Un POST peut arriver avec Transfer-Encoding: chunked ET Content-Type: multipart/form-data. Tu dois d'abord décoder les chunks (decode_chunked()), puis parser le multipart sur le body reconstitué. Deux couches de parsing.
⚠️
Limiter la taille de l'upload. Le sujet webserv permet de configurer client_max_body_size dans le fichier de config. Si le Content-Length dépasse cette limite, répondre 413 Payload Too Large avant de lire tout le body — sinon tu risques de remplir la RAM.
Filename = danger

Le filename vient du client. Il peut contenir ../../../etc/passwd ou d'autres chemins malicieux. Toujours nettoyer le filename : ne garder que le nom de base (après le dernier / ou \), interdire les ...

TCP fragmente tout

Comme pour le chunked, recv() peut te donner les données en morceaux. Le boundary peut être coupé entre deux recv(). Tu dois accumuler dans un buffer et parser seulement quand tu as tout le body (guidé par Content-Length).

💡
CGI et multipart. Si l'upload est destiné à un CGI (ex: un script PHP qui traite le formulaire), tu passes le body tel quel au CGI via son stdin. C'est le CGI qui parse le multipart. Tu n'as pas besoin de parser toi-même dans ce cas — mais tu dois quand même vérifier client_max_body_size.
