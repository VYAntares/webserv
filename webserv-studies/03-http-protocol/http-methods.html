03 — Le protocole HTTP

GET, POST, DELETE
Les méthodes HTTP

Une méthode HTTP, c'est le verbe de la requête. Elle dit au serveur quelle action effectuer sur la ressource désignée par l'URI. C'est le premier mot de la Request Line.

01
parseMethod()
C'est quoi une méthode HTTP ?
Le concept

Le client ne fait pas que "demander des trucs" — il spécifie son intention : lire, écrire, supprimer. Le serveur doit respecter cette sémantique et répondre correctement.

La méthode est le premier mot de la Request Line, avant l'URI et la version HTTP :

Request Line — la méthode est toujours en premier
GET /index.html HTTP/1.1\r\nMETHODE URI VERSION
Les 3 méthodes du sujet webserv

Le sujet ne demande que trois méthodes : GET (lire), POST (envoyer), DELETE (supprimer). Toute autre méthode (PUT, PATCH, HEAD...) doit renvoyer 501 Not Implemented.

💡
Idempotent signifie qu'appeler la même requête 1 fois ou 10 fois produit le même résultat final. GET et DELETE sont idempotents. POST ne l'est pas — chaque appel peut créer une nouvelle ressource ou déclencher un nouvel effet.
02
handleGet()
Lire une ressource
GET

C'est la méthode la plus utilisée sur le web. Quand tu tapes une URL dans ton navigateur → GET. Quand tu cliques sur un lien → GET. Quand une page charge une image → GET. Le client demande une ressource, le serveur la renvoie.

« The GET method requests transfer of a current selected representation for the target resource. GET is the primary mechanism of information retrieval and the focus of almost all performance optimizations. »

— RFC 7231, §4.3.1
Body dans la requête
Aucun (pas de sémantique définie)
Body dans la réponse
Oui (le contenu demandé)
Idempotent ?
Oui — appeler 10x = même résultat
Cacheable ?
Oui — par défaut
Requête GET brute — demande un fichier image
GET /images/logo.png HTTP/1.1\r\n Host: www.example.com\r\n User-Agent: Mozilla/5.0\r\n Accept: image/png,image/*\r\n \r\nPas de body
Ce que ton serveur doit faire avec un GET
  • Parser l'URI → trouver le fichier sur le disque correspondant à la config.
  • Vérifier que le fichier existe et est lisible (stat(), access()).
  • Lire le fichier, construire une réponse 200 avec le bon Content-Type et Content-Length.
  • Si c'est un dossier → soit index.html, soit directory listing si activé dans la config.
Ce que dit la RFC — subtilités importantes
  • La RFC dit qu'un identifiant de ressource n'est pas forcément un chemin de fichier. L'URI peut pointer vers un arbre d'objets, une vue de base de données, ou une gateway vers un autre système. Mais dans webserv, on implémente principalement le mapping fichier → c'est la version simple.
  • Body dans un GET : la RFC dit « a payload within a GET request message has no defined semantics ». Certaines implémentations rejettent un GET avec body. → Dans ton parser, si un GET arrive avec un body, ignore-le — ne crash pas, ne refuse pas la requête.
  • La réponse à un GET est cacheable par défaut — un cache peut la réutiliser pour des requêtes GET ultérieures, sauf indication contraire via Cache-Control.

« A payload within a GET request message has no defined semantics; sending a payload body on a GET request might cause some existing implementations to reject the request. »

— RFC 7231, §4.3.1
HTTP/1.1 200 OK — fichier trouvé
HTTP/1.1 404 Not Found — fichier inexistant
HTTP/1.1 403 Forbidden — pas les droits de lecture
HTTP/1.1 301 Moved Permanently — redirection config
HTTP/1.1 405 Method Not Allowed — GET interdit sur cette route
⚠️
Piège webserv : GET sur un répertoire (URL se terminant par /) doit chercher le fichier index configuré. Si aucun index n'existe et que le directory listing est désactivé → 403. Si le listing est activé → générer une page HTML avec la liste des fichiers.
💡
URI ≠ chemin fichier. La RFC insiste : l'interface HTTP d'une ressource peut être implémentée comme un fichier, mais aussi comme un programme qui s'exécute avec la requête en entrée et envoie la sortie comme représentation. C'est exactement ce que fait le CGI dans webserv : un GET sur /cgi-bin/script.py exécute le script au lieu de servir le fichier.
03
handlePost()
Envoyer des données au serveur
POST

C'est ce qui se passe quand tu soumets un formulaire, uploades un fichier, envoies du JSON. POST envoie un body — le serveur le traite et crée ou modifie quelque chose.

« The POST method requests that the target resource process the representation enclosed in the request according to the resource's own specific semantics. »

— RFC 7231, §4.3.3
Les 4 usages de POST selon la RFC
  • Fournir des données à un processus de traitement (formulaire HTML → script CGI).
  • Publier un message vers un groupe d'articles, un forum, un blog.
  • Créer une nouvelle ressource qui n'a pas encore été identifiée par le serveur.
  • Ajouter (append) des données à la représentation existante d'une ressource.
Body dans la requête
Oui — obligatoire
Body dans la réponse
Oui (confirmation)
Idempotent ?
Non — 2 appels = 2 effets
Cacheable ?
Non — sauf freshness explicite
Requête POST — formulaire simple
POST /login HTTP/1.1\r\n Host: www.example.com\r\n Content-Type: application/x-www-form-urlencoded\r\n Content-Length: 27\r\n \r\nFin des headers username=alice&password=1234← Body
Requête POST — upload de fichier (multipart)
POST /upload HTTP/1.1\r\n Host: www.example.com\r\n Content-Type: multipart/form-data; boundary=----abc123\r\n Content-Length: 198\r\n \r\n ------abc123 Content-Disposition: form-data; name="file"; filename="photo.jpg" Content-Type: image/jpeg (ligne vide) [données binaires du fichier...] ------abc123--← Fin du multipart
Ce que ton serveur doit faire avec un POST
  • Lire le header Content-Length → lire exactement ce nombre d'octets de body.
  • Regarder le Content-Type pour savoir comment interpréter le body.
  • Si c'est un upload → parser le multipart, extraire le fichier, le sauvegarder dans le dossier upload configuré.
  • Si c'est du CGI → passer le body sur stdin du script via un pipe.
  • Répondre 201 Created (ou 200 OK selon le cas).
Ce que dit la RFC — règles critiques pour POST
  • Création → 201 + Location : la RFC dit explicitement : si une ou plusieurs ressources ont été créées suite au POST, le serveur SHOULD envoyer un 201 Created contenant un header Location qui identifie la ressource principale créée. Pas juste un 200.
  • Presque tous les status codes sont possibles en réponse à POST (exceptions : 206 Partial Content, 304 Not Modified, 416 Range Not Satisfiable).
  • Redirection 303 See Other : si le résultat du POST est équivalent à une ressource existante, le serveur peut rediriger le client vers cette ressource avec un 303. C'est le pattern POST-Redirect-GET — ça empêche la re-soumission du formulaire si l'utilisateur recharge la page.
  • POST n'est pas cacheable par défaut. Les réponses POST ne sont cacheables que si elles incluent des informations de fraîcheur explicites — en pratique, presque personne ne le fait.

« If one or more resources has been created on the origin server as a result of successfully processing a POST request, the origin server SHOULD send a 201 (Created) response containing a Location header field that provides an identifier for the primary resource created. »

— RFC 7231, §4.3.3
HTTP/1.1 201 Created — ressource créée (+ header Location)
HTTP/1.1 200 OK — traitement OK (CGI, pas de création)
HTTP/1.1 303 See Other — POST-Redirect-GET (résultat existe déjà)
HTTP/1.1 400 Bad Request — body malformé
HTTP/1.1 413 Content Too Large — body dépasse max configuré
HTTP/1.1 405 Method Not Allowed — POST interdit sur cette route
⚠️
Piège webserv — chunked : Si le client envoie Transfer-Encoding: chunked au lieu de Content-Length, tu dois décoder le format chunked toi-même avant de traiter le body. Il n'y a pas de taille donnée à l'avance — les données arrivent en morceaux dont chacun est précédé de sa taille en hexadécimal.
⚠️
Piège webserv — body size : Ton serveur doit avoir une taille maximale de body configurable (client_max_body_size dans la config). Si le body dépasse → répondre 413 immédiatement sans tout lire.
💡
POST-Redirect-GET en pratique : après un upload réussi, au lieu de répondre 200 avec une page de confirmation, tu peux répondre 303 See Other avec Location: /uploads/photo.jpg. Le navigateur fait alors un GET sur cette URL — si l'utilisateur recharge, il recharge le GET, pas le POST. Ça évite le double-submit.
04
handleDelete()
Supprimer une ressource
DELETE

Le client demande au serveur d'effacer la ressource à l'URI donnée. C'est la méthode la plus simple des trois — pas de body à parser en entrée, juste une action à effectuer.

« The DELETE method requests that the origin server remove the association between the target resource and its current functionality. In effect, this method is similar to the rm command in UNIX: it expresses a deletion operation on the URI mapping of the origin server rather than an expectation that the previously associated information be deleted. »

— RFC 7231, §4.3.5
Body dans la requête
Aucun (pas de sémantique définie)
Body dans la réponse
Optionnel
Idempotent ?
Oui — supprimer 2x = même état final
Cacheable ?
Non — jamais
Requête DELETE brute
DELETE /uploads/photo.jpg HTTP/1.1\r\n Host: www.example.com\r\n \r\nPas de body
Ce que ton serveur doit faire avec un DELETE
  • Résoudre l'URI → trouver le chemin absolu du fichier sur le disque.
  • Vérifier que le fichier existe (stat()).
  • Vérifier les permissions — est-ce que DELETE est autorisé sur cette route dans la config ?
  • Supprimer avec unlink().
  • Répondre selon le résultat (voir les 3 codes de succès ci-dessous).
Ce que dit la RFC — les 3 réponses de succès possibles
  • 202 Accepted — si l'action « will likely succeed but has not yet been enacted ». Utile si la suppression est asynchrone (ex: mise en file d'attente). En pratique dans webserv, tu fais unlink() de façon synchrone → tu n'en auras probablement pas besoin.
  • 204 No Content — si l'action a été effectuée et qu'il n'y a aucune information supplémentaire à fournir. C'est le choix le plus courant.
  • 200 OK — si l'action a été effectuée et que la réponse inclut une représentation décrivant le statut (ex: un message JSON de confirmation).

« A payload within a DELETE request message has no defined semantics; sending a payload body on a DELETE request might cause some existing implementations to reject the request. »

— RFC 7231, §4.3.5
HTTP/1.1 204 No Content — supprimé, pas de body (le plus courant)
HTTP/1.1 200 OK — supprimé avec message de confirmation
HTTP/1.1 202 Accepted — suppression acceptée mais pas encore effectuée
HTTP/1.1 404 Not Found — fichier inexistant
HTTP/1.1 403 Forbidden — pas les droits
HTTP/1.1 405 Method Not Allowed — DELETE interdit sur cette route
⚠️
Piège webserv : Un DELETE sur un répertoire doit-il supprimer récursivement ? Non — sauf si tu l'implémentes explicitement. En général, supprimer un dossier non vide retourne 403 ou 409 Conflict. Reste simple : DELETE ne fonctionne que sur des fichiers.
💡
Body ignoré, pas rejeté. Comme pour GET, la RFC dit qu'un body dans un DELETE n'a pas de sémantique. Si un client envoie un DELETE avec un body, ton parser doit l'ignorer silencieusement — pas crasher, pas renvoyer 400. Le body n'existe pas à tes yeux.
💡
Réponses non cacheables. La RFC dit explicitement que les réponses à DELETE ne sont jamais cacheables. Si un cache a des réponses stockées pour l'URI ciblée, elles sont invalidées automatiquement après un DELETE.

La règle des méthodes autorisées par route

Dans ta config, chaque bloc location peut restreindre quelles méthodes sont acceptées. Si le client envoie une méthode non listée → ton serveur répond 405 Method Not Allowed.

Exemple : une route /uploads peut autoriser POST mais pas GET ni DELETE.

🚫
Méthode inconnue ? Si ton serveur reçoit une méthode qu'il ne reconnaît pas du tout (ex: PATCH, PUT, TOTO...) → 501 Not Implemented. Le sujet ne demande que GET, POST, DELETE — tout le reste est soit 501, soit 405 selon ta configuration.
dispatch logic
C++ 
// Après avoir parsé la request line :

if (method == "GET")
    handleGet(request, response);
else if (method == "POST")
    handlePost(request, response);
else if (method == "DELETE")
    handleDelete(request, response);
else
    sendError(501, response);  // Not Implemented

// Puis dans chaque handler, vérifier si la méthode
// est autorisée sur la route matchée dans la config.
// Si non → sendError(405, response);
💡
Ordre de vérification : d'abord vérifier si la méthode est reconnue (sinon 501), puis vérifier si elle est autorisée sur la route (sinon 405), puis exécuter le handler. Cette distinction est importante — 501 = "je ne connais pas cette méthode", 405 = "je la connais mais elle n'est pas permise ici".
