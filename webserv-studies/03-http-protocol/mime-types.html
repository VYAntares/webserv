03 — HTTP Protocol — Contenu

MIME Types & Content-Type
Étiqueter le contenu

HTTP transporte des octets bruts. Content-Type est l'étiquette qui dit au destinataire « ces octets sont du HTML / du PNG / du JSON ». Sans elle, le navigateur ne sait pas quoi faire des données qu'il reçoit.

01
Le problème
Des octets sans étiquette
Pourquoi le navigateur a besoin de MIME

Quand ton serveur fait send(), il envoie des octets bruts. Des 0 et des 1. Le navigateur reçoit ces octets et doit décider : est-ce du HTML à rendre ? Une image à afficher ? Du CSS à appliquer ? Du JavaScript à exécuter ?

Sans Content-Type, le navigateur devine — et souvent il se plante. Il affiche du texte brut au lieu de rendre une page. Il refuse d'appliquer un CSS. Il bloque un script JavaScript.

Sans Content-Type
Cassé 
HTTP/1.1 200 OK
Content-Length: 45
                          ← pas de Content-Type !

<html><body>Bonjour</body></html>

Le navigateur affiche :
<html><body>Bonjour</body></html>
         ^^^^ texte brut, pas rendu
Avec Content-Type
Correct 
HTTP/1.1 200 OK
Content-Type: text/html       ← étiquette !
Content-Length: 45

<html><body>Bonjour</body></html>

Le navigateur comprend : c'est du HTML
→ il rend la page correctement
02
type/sous-type
Le format MIME décrypté
Structure

Un MIME type suit toujours le format type/sous-type. Le type est la catégorie générale (text, image, application). Le sous-type précise le format exact (html, png, json).

On peut ajouter un paramètre optionnel comme ; charset=UTF-8 pour préciser l'encodage des caractères (utile pour le texte).

Exemples dans une réponse HTTP
Headers 
// Page HTML
Content-Type: text/html; charset=UTF-8
//            ^^^^  ^^^^   ^^^^^^^^^^^^
//            type  sous   paramètre optionnel

// Image PNG
Content-Type: image/png

// Données JSON
Content-Type: application/json

// Fichier inconnu (fallback)
Content-Type: application/octet-stream
Les grandes catégories
  • text/* — contenu textuel lisible (HTML, CSS, JS, plain)
  • image/* — images (PNG, JPEG, GIF, SVG)
  • application/* — données applicatives (JSON, PDF, binaire)
  • multipart/* — contenu en plusieurs parties (uploads)
Le fallback universel

application/octet-stream signifie « c'est des octets binaires, je ne sais pas ce que c'est ». Le navigateur propose généralement de télécharger le fichier au lieu de l'afficher.

C'est le MIME type à utiliser quand l'extension est inconnue ou absente.

04
get_mime_type()
Implémentation en C++98
La map extension → MIME
C++98 
#include <map>
#include <string>

// Construire la map une seule fois (au démarrage)
std::map<std::string, std::string> init_mime_types() {
    std::map<std::string, std::string> m;
    // Texte
    m[".html"] = "text/html";
    m[".css"]  = "text/css";
    m[".js"]   = "application/javascript";
    m[".json"] = "application/json";
    m[".txt"]  = "text/plain";
    m[".xml"]  = "application/xml";
    // Images
    m[".png"]  = "image/png";
    m[".jpg"]  = "image/jpeg";
    m[".jpeg"] = "image/jpeg";
    m[".gif"]  = "image/gif";
    m[".svg"]  = "image/svg+xml";
    m[".ico"]  = "image/x-icon";
    // Application
    m[".pdf"]  = "application/pdf";
    return m;
}

// Variable globale (initialisée une fois)
static std::map<std::string, std::string> mime_types = init_mime_types();
Trouver le MIME type d'un fichier
Utilisation 
// Extraire l'extension d'un chemin
std::string get_extension(const std::string& path) {
    size_t dot = path.rfind('.');
    if (dot == std::string::npos)
        return "";
    return path.substr(dot);  // ".html", ".png", etc.
}

// Résoudre le MIME type
std::string get_mime_type(const std::string& path) {
    std::string ext = get_extension(path);
    if (mime_types.count(ext))
        return mime_types[ext];
    return "application/octet-stream";  // fallback
}

// Utilisation dans build_response() :
// std::string type = get_mime_type("/style.css");
// → "text/css"
// std::string type = get_mime_type("/data.xyz");
// → "application/octet-stream"
Intégration dans build_response()
Concret 
// Avant (on avait tout en dur) :
"Content-Type: text/html\r\n"  // ← toujours HTML

// Après (on détermine dynamiquement) :
std::string build_response(const std::string& path,
                           const std::string& body) {
    std::string type = get_mime_type(path);

    std::ostringstream oss;
    oss << "HTTP/1.1 200 OK\r\n"
        << "Content-Type: " << type << "\r\n"
        << "Content-Length: " << body.size() << "\r\n"
        << "Connection: close\r\n"
        << "\r\n"
        << body;
    return oss.str();
}

MIME & méthodes HTTP
Dans quel sens circule le Content-Type
Le Content-Type circule dans les deux sens

Pour GET, c'est toi (le serveur) qui produis le Content-Type dans ta réponse. Pour POST, c'est le client qui t'envoie un Content-Type dans sa requête — et tu dois le lire pour savoir comment parser le body.

Méthode MIME dans la requête (entrant) MIME dans la réponse (sortant) Qui décide
GET non (pas de body) oui — tu dis ce que tu sers toi, via l'extension du fichier
POST oui — le client dit ce qu'il envoie oui — tu dis ce que tu retournes client pour la req, toi pour la réponse
DELETE non (pas de body) rarement (souvent 204 sans body) —
GET — serveur produit le MIME 
// Client demande :
GET /style.css HTTP/1.1

// Serveur répond :
HTTP/1.1 200 OK
Content-Type: text/css     ← toi

body { color: red; }
POST — client dit ce qu'il envoie 
// Client envoie :
POST /api HTTP/1.1
Content-Type: application/json  ← lui

{"action": "create"}

// Ton serveur lit ce header
// pour savoir comment parser le body
Les 3 Content-Type de POST que ton serveur doit reconnaître
  • application/x-www-form-urlencoded — formulaires simples (username=alice&password=1234)
  • multipart/form-data — uploads de fichiers (parties séparées par un boundary)
  • application/json, text/plain, etc. — passé tel quel au CGI
06
Pièges webserv
Ce qui peut casser à l'évaluation
⚠️
CSS sans le bon MIME = page cassée. Si tu sers style.css avec Content-Type: text/html (ou sans Content-Type), les navigateurs modernes refusent de l'appliquer (protection MIME sniffing). La page s'affiche sans style. Même chose pour JavaScript — un .js servi avec le mauvais MIME type est bloqué.
⚠️
Pages d'erreur sans Content-Type. Quand tu envoies une 404 ou 500 avec du HTML, n'oublie pas Content-Type: text/html. Erreur classique : le body est du HTML mais le navigateur l'affiche en texte brut parce que le header manque.
💡
CGI : ne pas deviner le Content-Type. Quand un CGI (Python, PHP) génère sa réponse, il produit ses propres headers dont Content-Type. Ton serveur doit transmettre tel quel ce que le CGI écrit, pas deviner le type à partir de l'extension du script. Le .py produit peut-être du JSON, du HTML, ou une image — c'est le CGI qui décide.
CGI output — le script décide
CGI 
# Le script Python écrit sur stdout :
Content-Type: application/json\r\n     ← le CGI choisit le MIME
\r\n
{"status": "ok", "id": 42}

# Ton serveur LIT ce Content-Type et le transmet au client.
# Tu ne le remplaces PAS par "text/html" parce que c'est un .py !

# Réponse envoyée au client :
HTTP/1.1 200 OK
Content-Type: application/json       ← transmis tel quel
Content-Length: 26

{"status": "ok", "id": 42}
Résumé — MIME en une page
L'étiquette qui dit au navigateur quoi faire des octets.
🏷️
Format
type/sous-type — ex: text/html, image/png, application/json
🔍
Détermination
Extension du fichier → std::map → MIME type. Fallback : application/octet-stream
🔄
Double sens
GET : toi tu produis le MIME (réponse). POST : le client te dit son MIME (requête) + tu produis le tien (réponse).
⚠️
Pièges
CSS/JS avec mauvais MIME = bloqué. Pages d'erreur sans Content-Type = texte brut. CGI = transmettre son Content-Type, pas deviner.
