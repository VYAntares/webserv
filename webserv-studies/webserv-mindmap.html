Vision globale

Webserv est la construction complète d'un serveur HTTP en C++ 98. Tu vas implémenter le cycle complet : écouter des connexions TCP, parser des requêtes HTTP brutes, router les URLs vers des fichiers ou des scripts CGI, et renvoyer des réponses conformes au protocole — le tout sans jamais bloquer, grâce à un unique appel d'I/O multiplexing.

This project is about writing your own HTTP server. You will be able to test it with an actual browser. HTTP is one of the most widely used protocols on the internet.

— webserv.pdf, Introduction
⚠️

Règle absolue : le programme ne doit jamais crash, même en cas de manque de mémoire. Le sujet précise que si le serveur se termine de manière inattendue, la note est 0. Cela signifie : chaque allocation, chaque syscall, chaque edge-case doit être géré.

01

Sockets & TCP

Le fondement de tout serveur réseau. Un socket est un file descriptor qui permet de communiquer sur le réseau. Comprendre le cycle socket → bind → listen → accept est la première brique indispensable.

Fondation

Le cycle de vie d'un socket serveur

Quatre étapes pour passer d'un fd vide à un serveur qui accepte des clients. Chaque étape est un appel système distinct avec sa sémantique propre. Le socket d'écoute reste permanent ; chaque client produit un nouveau fd dédié.

socket() bind() listen() accept()
man 2 socket man 2 bind man 2 listen man 2 accept
Réseau & Configuration

Adressage & options de socket

Convertir IP:port en sockaddr, résoudre des adresses avec getaddrinfo(), gérer l'endianness réseau avec htons/htonl, et configurer le socket avec setsockopt() — notamment SO_REUSEADDR pour éviter "Address already in use" au redémarrage.

getaddrinfo() freeaddrinfo() htons() htonl() ntohs() ntohl() getsockname() setsockopt() SO_REUSEADDR
man 3 getaddrinfo man 2 setsockopt Beej's Network Guide
Communication

Envoi & réception de données

recv() et send() sont les équivalents réseau de read() et write(). En non-bloquant, ils peuvent renvoyer EAGAIN — c'est normal, cela signifie qu'il faut réessayer quand epoll_wait() le signalera. Les données peuvent arriver fragmentées : il faut accumuler dans un buffer.

recv() send() close() shutdown()
man 2 recv man 2 send
Concept clé

Blocking vs Non-blocking

Par défaut, un socket est bloquant : un appel accept() ou recv() fige le programme tant qu'il n'y a rien. En mode non-bloquant (fcntl(fd, F_SETFL, O_NONBLOCK)), ces appels retournent immédiatement avec EAGAIN si rien n'est disponible. C'est ce qui rend le multiplexing possible.

« Your server must remain non-blocking at all times and properly handle client disconnections when necessary. » — webserv.pdf, IV.1 Requirements
fcntl() F_SETFL O_NONBLOCK
Cours détaillé → man 2 fcntl
Multi-port

Écouter sur plusieurs ports

Le sujet exige que le serveur puisse écouter sur plusieurs ports pour servir du contenu différent. Cela signifie créer plusieurs sockets d'écoute et les enregistrer tous dans le même epoll via epoll_ctl(). Chaque socket d'écoute est lié à un bloc serveur de la configuration.

« Your server must be able to listen to multiple ports to deliver different content. » — webserv.pdf, IV.1 Requirements
02

I/O Multiplexing

Le cœur absolu du projet. Le multiplexing permet de surveiller des dizaines de connexions simultanées avec un seul thread, sans jamais bloquer. Nous utilisons epoll — un unique epoll_wait() pour gérer toutes les I/O, y compris le listen.

It must be non-blocking and use only 1 poll() (or equivalent) for all the I/O operations between the clients and the server (listen included).

— webserv.pdf, IV.1 Requirements
Contexte

Pourquoi epoll plutôt que poll ou select

select() est limité à 1024 fds et recopie les sets à chaque appel. poll() supprime la limite mais parcourt linéairement tous les fds à chaque retour — O(n). epoll (Linux) maintient une liste interne dans le kernel : epoll_wait() retourne directement les fds actifs, en O(1). C'est le choix approprié pour un serveur web sous Linux avec de nombreuses connexions simultanées.

select() poll() epoll EPOLLET EPOLLONESHOT
man 7 epoll man 2 epoll_ctl
Mécanisme central

epoll — Le surveillant événementiel

epoll surveille un ensemble de fds via trois appels : epoll_create1() crée l'instance, epoll_ctl() ajoute/retire/modifie un fd avec ses événements (EPOLLIN, EPOLLOUT, EPOLLERR, EPOLLHUP), et epoll_wait() bloque jusqu'à ce qu'au moins un fd soit prêt — retournant uniquement les fds actifs. Contrairement à poll(), epoll est O(1) : il ne parcourt pas tous les fds à chaque appel.

« poll() (or equivalent) must monitor both reading and writing simultaneously. » — webserv.pdf, IV.1 Requirements
epoll_create1() epoll_ctl() epoll_wait() EPOLLIN EPOLLOUT EPOLLERR EPOLLHUP
man 7 epoll
Piège mortel

La règle d'or : toujours epoll_wait() avant read/write

Le sujet est catégorique : toute opération de lecture ou écriture sur un socket ou pipe DOIT être précédée d'une vérification de disponibilité via epoll_wait(). Appeler recv() ou send() sans confirmation d'epoll entraîne un 0. Seuls les fichiers réguliers sur disque sont exemptés de cette règle.

« You must never do a read or a write operation without going through poll() (or equivalent). Calling read/recv or write/send on these descriptors without prior readiness will result in a grade of 0. Regular disk files are exempt. » — webserv.pdf, IV.1 Requirements
Concept

La boucle événementielle (Event Loop)

L'architecture fondamentale : une boucle infinie qui appelle epoll_wait(), puis itère sur les événements retournés et exécute l'action appropriée (accept() pour le socket d'écoute, recv()/send() pour les clients, read() pour les pipes CGI). Aucune requête ne doit jamais bloquer cette boucle.

« A request to your server should never hang indefinitely. » — webserv.pdf, IV.1 Requirements
🚫

Interdit : vérifier errno après un read ou write pour ajuster le comportement du serveur. Le sujet dit explicitement « Checking the value of errno to adjust the server behaviour is strictly forbidden after performing a read or write operation. »

03

Le protocole HTTP

HTTP est un protocole texte, requête/réponse, sans état. Le client envoie une requête avec une méthode, un chemin, des headers et optionnellement un body. Le serveur répond avec un status code, des headers et un body. Tout le parsing est à faire à la main.

Parsing

Structure d'une requête HTTP

Une requête se compose de trois parties : la request line (GET /index.html HTTP/1.1), les headers (paires clé:valeur séparées par \r\n), et optionnellement un body. La fin des headers est marquée par \r\n\r\n. La taille du body est déterminée par Content-Length ou le Transfer-Encoding: chunked.

RFC 7230 — Message Syntax
Méthodes

GET, POST, DELETE

GET : récupérer une ressource (fichier, page). Pas de body. POST : envoyer des données au serveur (formulaire, upload). A un body. DELETE : supprimer une ressource. Chaque route de la config peut restreindre les méthodes autorisées.

RFC 7231 — Semantics
Réponse

Status codes & headers de réponse

La réponse commence par HTTP/1.1 200 OK\r\n, suivie des headers (Content-Type, Content-Length, Connection, etc.) puis du body. Les status codes doivent être corrects : 200, 201, 204, 301, 400, 403, 404, 405, 413, 500, etc.

Voir le cours complet
« Your HTTP response status codes must be accurate. » — webserv.pdf, IV.1 Requirements
Parsing avancé

Chunked Transfer-Encoding

Quand le client envoie des données en mode "chunked", il n'y a pas de Content-Length. Les données arrivent en morceaux, chacun précédé de sa taille en hexadécimal. Un chunk de taille 0 marque la fin. Le serveur doit recombiner ces morceaux avant de les passer au CGI (qui attend un flux continu terminé par EOF).

« For chunked requests, your server needs to un-chunk them, the CGI will expect EOF as the end of the body. » — webserv.pdf, IV.3
Contenu

MIME Types & Content-Type

Le serveur doit identifier le type de fichier servi et envoyer le bon Content-Type : text/html, image/png, application/json, etc. Sans cela, le navigateur ne sait pas comment interpréter la réponse. Typiquement implémenté via une map extension → MIME type.

Upload

Upload de fichiers (multipart/form-data)

Les uploads passent par un POST avec un Content-Type: multipart/form-data. Le body contient des "parts" séparées par un boundary. Chaque part a ses propres headers (nom du champ, nom de fichier, type MIME) suivis du contenu binaire.

« Clients must be able to upload files. » — webserv.pdf, IV.1 Requirements
Concept critique

Données fragmentées sur un socket

Un recv() ne retourne pas forcément la requête entière d'un coup. Les données TCP arrivent en fragments imprévisibles. Il faut accumuler dans un buffer, détecter \r\n\r\n pour la fin des headers, puis lire exactement Content-Length octets (ou décoder le chunked) pour le body.

04

Fichier de configuration

Le sujet demande un système de configuration inspiré de NGINX. Le serveur prend un fichier en argument et doit parser des blocs serveur et location pour déterminer le comportement de chaque route.

You can take inspiration from the 'server' section of the NGINX configuration file.

— webserv.pdf, IV.3 Configuration file
Parsing

Parser un fichier de config structuré

Implémenter un lexer/parser pour lire des blocs imbriqués (server, location). Gérer les accolades, les directives, les valeurs, les commentaires. Le format n'est pas imposé mais NGINX est la référence suggérée. Validation des valeurs : ports valides, chemins existants, méthodes HTTP connues.

NGINX Beginner Guide NGINX Core Module
Directives

Ce que la config doit supporter

Le sujet liste explicitement : paires interface:port, pages d'erreur par défaut, taille max du body client, et par route : méthodes autorisées, redirections, racine du répertoire, directory listing, fichier index par défaut, upload autorisé avec chemin de stockage, et exécution CGI par extension.

« Directory where the requested file should be located (e.g., if URL /kapouet is rooted to /tmp/www, URL /kapouet/pouic/toto/pouet will search for /tmp/www/pouic/toto/pouet). » — webserv.pdf, IV.3
Routing

Matching de routes (sans regex)

Quand une requête arrive, le serveur doit trouver quel bloc location correspond à l'URI demandée. Le sujet précise explicitement qu'aucune regex n'est requise — un matching par préfixe suffit. Il faut ensuite combiner le root de la location avec le reste du chemin pour trouver le fichier.

05

CGI — Common Gateway Interface

Le CGI permet au serveur d'exécuter des scripts externes (Python, PHP...) et de renvoyer leur sortie au client. C'est le seul cas où fork() est autorisé. La communication se fait via variables d'environnement + stdin/stdout.

Mécanisme

fork() + execve() + pipe()

Le serveur fork() un processus fils, redirige stdin/stdout via pipe() + dup2(), puis execve() l'interpréteur. Le body de la requête est envoyé sur stdin du CGI, et la sortie du CGI (sur stdout) est lue par le serveur via le pipe.

fork() execve() pipe() dup2() waitpid() kill()
man 2 fork man 2 execve man 2 pipe
Protocole

Variables d'environnement CGI

Le serveur doit passer au CGI toute l'information de la requête via des variables d'environnement : REQUEST_METHOD, CONTENT_LENGTH, CONTENT_TYPE, PATH_INFO, QUERY_STRING, SERVER_NAME, SERVER_PORT, SCRIPT_NAME, les headers HTTP préfixés par HTTP_.

« Have a careful look at the environment variables involved in the web server-CGI communication. The full request and arguments provided by the client must be available to the CGI. » — webserv.pdf, IV.3
RFC 3875 — CGI/1.1
Pièges

EOF, chunked, et répertoire de travail

Trois points critiques : les requêtes chunked doivent être dé-chunkées avant d'être envoyées au CGI (il attend EOF, pas des chunks). Si le CGI ne retourne pas de Content-Length, c'est EOF qui marque la fin de sa sortie. Et le CGI doit être exécuté dans le bon répertoire (chdir()) pour que les chemins relatifs fonctionnent.

« The CGI should be run in the correct directory for relative path file access. » — webserv.pdf, IV.3
chdir()
Non-bloquant

Intégrer le CGI dans l'event loop

Le pipe du CGI doit être enregistré dans epoll via epoll_ctl() comme n'importe quel autre fd. Le processus fils peut prendre du temps — il faut un timeout et waitpid(WNOHANG) pour ne pas bloquer. Le pipe de sortie du CGI doit être en mode non-bloquant.

06

Robustesse & résilience

Le sujet insiste lourdement sur la résilience. Le serveur ne doit jamais crash, jamais bloquer, jamais leak. Il doit supporter des stress tests et rester opérationnel en permanence.

Stabilité

Gestion des erreurs exhaustive

Chaque appel système peut échouer. Chaque allocation peut rater. Le programme ne doit jamais crash, même en cas de manque de mémoire. Pages d'erreur par défaut obligatoires si non configurées. Signaux à gérer proprement (SIGPIPE, SIGCHLD pour les CGI).

« Your program must not crash under any circumstances (even if it runs out of memory) or terminate unexpectedly. » — webserv.pdf, General rules
signal() strerror()
Performance

Stress testing

Le sujet demande explicitement de stress tester le serveur. Outils courants : siege, ab (Apache Bench), scripts Python avec requests. Le serveur doit gérer des connexions simultanées massives, des déconnexions brutales, des requêtes malformées, sans jamais tomber.

« Stress test your server to ensure it remains available at all times. Resilience is key. » — webserv.pdf, IV.1 Requirements
Timeout

Timeouts & déconnexions

Un client qui ne finit jamais d'envoyer sa requête ne doit pas bloquer le serveur. Il faut implémenter des timeouts sur les connexions clientes, gérer proprement les déconnexions (EPOLLHUP, recv retournant 0), et fermer les fds pour éviter les fuites.

Compilation

Contraintes C++ 98 & Makefile

Le code doit compiler avec c++ -Wall -Wextra -Werror -std=c++98. Pas de fonctionnalités C++11+. Le Makefile doit contenir les règles $(NAME), all, clean, fclean, re et ne pas relinker inutilement.

Fichiers

Opérations sur le système de fichiers

Pour servir des fichiers statiques, lister des répertoires, vérifier les permissions. Ces opérations sur fichiers réguliers sont exemptées de la contrainte epoll — on peut faire read()/write() directement sur des fichiers disque.

open() read() write() close() stat() access() opendir() readdir() closedir()
Validation

Tester avec NGINX comme référence

Le sujet suggère d'utiliser NGINX pour comparer les headers et comportements de réponse. C'est ta meilleure référence pour les edge-cases HTTP. Utilise aussi telnet pour envoyer des requêtes brutes et vérifier le parsing.

« NGINX may be used to compare headers and answer behaviours (pay attention to differences between HTTP versions). » — webserv.pdf, IV.1 Requirements
