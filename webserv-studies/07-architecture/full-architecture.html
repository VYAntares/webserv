0

Démarrage du serveur

Le point d'entrée. Le programme lit le fichier de configuration, construit toutes les structures en mémoire, puis lance la boucle infinie.

Le programme prend un seul argument : le chemin vers le fichier .conf. Il ignore SIGPIPE pour éviter de crash quand un client se déconnecte pendant un send. Ensuite il parse la config, crée le serveur, et lance la boucle. C'est tout.

main.cpp
int main(int argc, char* argv[]) { signal(SIGPIPE, SIG_IGN); // évite le crash sur write à un client mort Config cfg(argv[1]); // parse tout le fichier .conf Webserv serv(cfg); // crée les sockets + epoll serv.run(); // boucle infinie — ne retourne jamais return 0; }

Le parser de config lit le fichier .conf et construit un arbre de blocs server, chacun contenant ses blocs location. Chaque server a un port, un host, des limites de body, et chaque location définit les méthodes autorisées, le root, l'index, le CGI.

Structures de configuration — ce qu'il faut stocker
struct Location { std::string path; // "/static" std::string root; // "/var/www" std::vector<std::string> methods; // ["GET","POST"] bool autoindex; // listing de dossier std::string index; // "index.html" std::string cgi_ext; // ".py" }; struct ServerBlock { int port; // 8080 std::string host; // "localhost" std::string server_name; // "example.com" size_t max_body; // client_max_body_size std::vector<Location> locations; };
💡

Pourquoi c'est important : toute la logique du serveur dépend de la config. Si ton parser est bancal, les routes ne matcheront pas, les méthodes seront mal filtrées, et le CGI ne se lancera pas. C'est la fondation de tout.

1

Création des sockets serveur

Pour chaque bloc server dans la config, on crée un socket d'écoute. C'est la porte d'entrée par laquelle les clients viendront frapper.

Chaque bloc server { listen 8080; } dans la config crée son propre socket. Le flow est toujours le même : créer le fd, activer la réutilisation d'adresse, l'attacher au port, activer l'écoute, passer en non-bloquant, et l'enregistrer dans epoll.

socket()crée le file descriptor
↓
setsockopt()SO_REUSEADDR — relancer sans attendre que le port se libère
↓
bind()attache le socket au port choisi
↓
listen()active l'écoute, backlog = 10 connexions en file
↓
fcntl()O_NONBLOCK — indispensable pour ne jamais bloquer
↓
epoll_ctl()ADD + EPOLLIN — surveille les connexions entrantes
Server::Server(port) — création du socket
Server::Server(int port) { _fd = socket(AF_INET, SOCK_STREAM, 0); int opt = 1; setsockopt(_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)); fcntl(_fd, F_SETFL, fcntl(_fd, F_GETFL) | O_NONBLOCK); struct sockaddr_in addr; memset(&addr, 0, sizeof(addr)); addr.sin_family = AF_INET; addr.sin_port = htons(port); addr.sin_addr.s_addr = INADDR_ANY; bind(_fd, (struct sockaddr*)&addr, sizeof(addr)); listen(_fd, 10); }

Ensuite on crée une seule instance epoll pour tout le serveur. Chaque socket serveur est enregistré avec EPOLLIN : "préviens-moi quand un client essaie de se connecter".

Webserv::init() — epoll + enregistrement des serveurs
_epfd = epoll_create1(0); for (size_t i = 0; i < cfg.servers().size(); i++) { Server* s = new Server(cfg.servers()[i].port); struct epoll_event ev; ev.events = EPOLLIN; ev.data.fd = s->fd(); epoll_ctl(_epfd, EPOLL_CTL_ADD, s->fd(), &ev); }
socket() setsockopt() bind() listen() fcntl() epoll_create1() epoll_ctl()
⚠️

Piège classique : oublier O_NONBLOCK sur le socket serveur. Si un client initie la connexion TCP puis ne finit pas le handshake, accept() bloquera tout le serveur indéfiniment. Toujours non-bloquant.

2

La boucle principale

Le cœur battant du serveur. epoll_wait() bloque jusqu'à ce qu'un événement se produise, puis dispatche vers le bon handler.

À chaque tour de boucle, epoll_wait() retourne la liste des fds prêts. Pour chaque fd, on regarde : est-ce un socket serveur ? (nouvelle connexion) Est-ce un client avec EPOLLIN ? (données à lire) EPOLLOUT ? (prêt à écrire). C'est un aiguillage — rien de plus.

Webserv::run() — la boucle infinie
void Webserv::run() { struct epoll_event events[64]; while (true) { int n = epoll_wait(_epfd, events, 64, -1); for (int i = 0; i < n; i++) { int fd = events[i].data.fd; // Erreur ou déconnexion brutale ? if (events[i].events & (EPOLLHUP | EPOLLERR)) { _closeClient(fd); continue; } // Nouvelle connexion sur un socket serveur ? if (_isServerFd(fd)) _acceptClient(fd); // Données à lire d'un client existant ? else if (events[i].events & EPOLLIN) _handleRecv(fd); // Prêt à envoyer la réponse au client ? else if (events[i].events & EPOLLOUT) _handleSend(fd); } } }

Quand fd == server_fd : un nouveau client arrive. accept() crée un nouveau fd dédié à ce client. On le met en non-bloquant et on l'enregistre dans epoll pour surveiller ses données.

_acceptClient() — nouvelle connexion
void _acceptClient(int server_fd) { struct sockaddr_in addr; socklen_t len = sizeof(addr); int cfd = accept(server_fd, (struct sockaddr*)&addr, &len); if (cfd == -1) return; // Non-bloquant obligatoire fcntl(cfd, F_SETFL, fcntl(cfd, F_GETFL) | O_NONBLOCK); // Stocker le client _clients.insert(std::make_pair(cfd, Client(cfd))); // Surveiller ses données entrantes struct epoll_event ev; ev.events = EPOLLIN; ev.data.fd = cfd; epoll_ctl(_epfd, EPOLL_CTL_ADD, cfd, &ev); }

Quand EPOLLIN sur un client : le client a envoyé des données. On lit avec recv() et on accumule dans un buffer. On ne parse qu'une fois le marqueur \r\n\r\n trouvé (fin des headers).

_handleRecv() — accumulation des données
void _handleRecv(int fd) { Client& c = _clients[fd]; char buf[4096]; ssize_t n = recv(fd, buf, sizeof(buf) - 1, 0); if (n <= 0) { _closeClient(fd); return; // client déconnecté ou erreur } buf[n] = '\0'; c.recvBuf() += buf; // accumule les morceaux // Requête complète = headers terminés par \r\n\r\n if (c.recvBuf().find("\r\n\r\n") != std::string::npos) { _processRequest(c); // → parse → route → handle → build } // Sinon : on attend le prochain recv (level-triggered renotifie) }

Quand EPOLLOUT sur un client : le socket est prêt à envoyer. On écrit le contenu du sendBuf. Si send() n'envoie qu'une partie (buffer kernel plein), on efface ce qui est parti et on attend le prochain tour.

_handleSend() — envoi de la réponse
void _handleSend(int fd) { Client& c = _clients[fd]; ssize_t n = send(fd, c.sendBuf().c_str(), c.sendBuf().size(), 0); if (n > 0) c.sendBuf().erase(0, n); // retire les bytes envoyés if (c.sendBuf().empty()) _closeClient(fd); // tout envoyé → on ferme }
🔄

Le cycle complet : accept → recv (EPOLLIN) → parse → handler → build response → send (EPOLLOUT) → close. Ce cycle se répète pour chaque client, en parallèle grâce à epoll.

3

Parsing de la requête HTTP

Transformer du texte brut en données structurées. C'est de la chirurgie sur des strings — chaque \r\n compte.

Une requête HTTP c'est du texte avec un format précis. La première ligne contient la méthode, le chemin et la version. Ensuite viennent les headers (clé: valeur), séparés du body par une ligne vide.

Exemple de requête brute
GET /img/photo.png HTTP/1.1\r\n Host: localhost:8080\r\n User-Agent: Mozilla/5.0\r\n Accept: image/png\r\n \r\n ← ligne vide = fin des headers (body ici si POST avec Content-Length)
HttpRequest::_parse() — découper la requête
void _parse(const std::string& raw) { // ── Ligne 1 : méthode + path + version ── size_t eol = raw.find("\r\n"); std::istringstream ss(raw.substr(0, eol)); ss >> _method >> _path >> _version; // ── Headers : "Clé: Valeur" un par ligne ── size_t pos = eol + 2; size_t hend = raw.find("\r\n\r\n"); while (pos < hend) { size_t col = raw.find(':', pos); size_t nl = raw.find("\r\n", pos); _headers[raw.substr(pos, col - pos)] = raw.substr(col + 2, nl - col - 2); pos = nl + 2; } // ── Body : tout après la ligne vide ── _body = raw.substr(hend + 4); }
⚠️

Requêtes fragmentées : TCP ne garantit pas que toute la requête arrive d'un coup. Un recv() peut ne donner que la moitié des headers. Il faut toujours accumuler dans un buffer et ne parser qu'une fois le \r\n\r\n trouvé. Sinon, ton parser crash sur des données incomplètes.

4

Routing — trouver le bon handler

La requête est parsée, maintenant il faut décider qui la traite. Le routeur cherche le bloc location qui correspond le mieux au chemin demandé.

Le routeur parcourt tous les blocs location de la config et garde celui dont le path est le plus long préfixe de l'URL demandée. C'est le "longest prefix match" — exactement comme NGINX. Ensuite il vérifie que la méthode HTTP est autorisée, et décide s'il faut un handler CGI ou un handler de fichier statique.

Router::getHandler() — trouver le bon handler
AHandler* getHandler(const HttpRequest& req, const ServerBlock& cfg) { Location* loc = _findLocation(req.path(), cfg); if (!_methodAllowed(req, loc)) // GET non autorisé ici ? return new ErrorHandler(405); // Method Not Allowed if (_isCgi(req.path(), loc)) // extension .py ou .php ? return new CgiHandler(loc); return new StaticHandler(loc); // sinon : fichier statique } // Longest prefix match — retourne le location le plus spécifique Location* _findLocation(const std::string& url, const ServerBlock& cfg) { Location* best = NULL; size_t best_len = 0; for (size_t i = 0; i < cfg.locations.size(); i++) { if (url.find(cfg.locations[i].path) == 0 && cfg.locations[i].path.size() > best_len) { best = &cfg.locations[i]; best_len = cfg.locations[i].path.size(); } } return best; }
5

Les handlers — traitement concret

C'est ici que le serveur fait vraiment quelque chose : lire un fichier, exécuter un script CGI, ou générer une erreur.

StaticHandler — fichier statique. Le cas le plus courant : servir un fichier du disque. On construit le chemin (root + path), on vérifie qu'il existe avec stat(), on vérifie les permissions, puis on lit le contenu et on le met dans la réponse. C'est 90% du trafic d'un serveur web.

StaticHandler::handle()
void handle(const HttpRequest& req, HttpResponse& res) { std::string path = _root + req.path(); // Le fichier existe ? struct stat st; if (stat(path.c_str(), &st) != 0) { res.setStatus(404); return; // Not Found } if (!(st.st_mode & S_IROTH)) { res.setStatus(403); return; // Forbidden } // Lire le fichier entier std::ifstream f(path.c_str()); std::string body( (std::istreambuf_iterator<char>(f)), std::istreambuf_iterator<char>()); res.setStatus(200); res.setHeader("Content-Type", getMime(path)); res.setBody(body); }
stat() open() read() ifstream

CgiHandler — script dynamique. Pour les fichiers .py ou .php, le serveur ne lit pas le fichier : il l'exécute. On crée un pipe, on fork, le fils redirige stdout vers le pipe et lance le script avec execve(). Le père lit la sortie du pipe et l'utilise comme body de la réponse.

CgiHandler::handle() — fork + pipe + execve
void handle(const HttpRequest& req, HttpResponse& res) { int pipe_out[2]; pipe(pipe_out); pid_t pid = fork(); if (pid == 0) { // ENFANT : redirige stdout vers le pipe dup2(pipe_out[1], STDOUT_FILENO); close(pipe_out[0]); close(pipe_out[1]); // Variables d'environnement HTTP (protocole CGI) setenv("REQUEST_METHOD", req.method().c_str(), 1); setenv("PATH_INFO", req.path().c_str(), 1); setenv("QUERY_STRING", req.query().c_str(), 1); char* args[] = {(char*)_script.c_str(), NULL}; execve(_script.c_str(), args, environ); exit(1); // si execve échoue } // PARENT : lit la sortie du script via le pipe close(pipe_out[1]); // read(pipe_out[0]) → body de la réponse waitpid(pid, NULL, 0); }
pipe() fork() dup2() execve() waitpid()

ErrorHandler — pages d'erreur. Quand quelque chose ne va pas (fichier introuvable, méthode interdite, erreur interne), ce handler génère une page d'erreur. Il cherche d'abord une page custom dans la config. Si aucune n'est définie, il génère du HTML basique par défaut.

ErrorHandler::handle() — réponse d'erreur
void handle(const HttpRequest& req, HttpResponse& res) { res.setStatus(_code); res.setHeader("Content-Type", "text/html"); // Page custom dans la config ? (error_page 404 /custom_404.html;) std::string custom = _cfg.errorPage(_code); if (!custom.empty()) res.setBody(readFile(custom)); else res.setBody("<h1>" + statusText(_code) + "</h1>"); }
6

Construction de la réponse

Le handler a fait son travail. Maintenant il faut emballer le résultat dans le format HTTP et le préparer pour l'envoi.

On assemble la status line (HTTP/1.1 200 OK), les headers (dont Content-Length qui est obligatoire), une ligne vide, puis le body. Le tout est mis dans le sendBuf du client, et on passe le fd en EPOLLOUT pour que epoll nous prévienne quand le socket est prêt à écrire.

HttpResponse::build() — assembler la réponse
std::string build() const { std::ostringstream oss; // Status line oss << "HTTP/1.1 " << _status << " " << statusText(_status) << "\r\n"; // Headers for (auto it = _headers.begin(); it != _headers.end(); ++it) oss << it->first << ": " << it->second << "\r\n"; // Content-Length obligatoire oss << "Content-Length: " << _body.size() << "\r\n"; oss << "\r\n"; // ligne vide = fin des headers oss << _body; return oss.str(); // → client.sendBuf() }
Exemple de réponse HTTP complète
HTTP/1.1 200 OK Content-Type: text/html Content-Length: 42 ← ligne vide = fin des headers <html><body>Hello World!</body></html>
📋

Après le build : on modifie l'événement epoll du client pour passer de EPOLLIN à EPOLLOUT via epoll_ctl(EPOLL_CTL_MOD). C'est ce changement qui dit à la boucle "ce client est en mode envoi maintenant".

7

Fermeture propre

La réponse est envoyée, le client s'est déconnecté, ou une erreur s'est produite. Dans tous les cas, il faut nettoyer proprement dans le bon ordre.

Trois étapes dans cet ordre précis. D'abord retirer le fd d'epoll (pour ne plus recevoir d'événements fantômes). Ensuite fermer le socket. Enfin supprimer l'objet Client de la map. Si tu inverses l'ordre, tu risques un use-after-close.

epoll_ctl(DEL)Retire le fd de la surveillance — plus d'événements
↓
close(fd)Ferme le socket — libère la connexion TCP
↓
clients.erase()Détruit l'objet Client — libère recvBuf et sendBuf
_closeClient() — nettoyage propre
void _closeClient(int fd) { epoll_ctl(_epfd, EPOLL_CTL_DEL, fd, NULL); // 1. retire d'epoll close(fd); // 2. ferme le socket _clients.erase(fd); // 3. libère le Client }
⚠️

Fuite de fd : si tu oublies de fermer un fd, tu atteins la limite système (souvent 1024 par défaut) et plus aucun client ne peut se connecter. Vérifie avec lsof -p $(pidof webserv) que le nombre de fds ouverts reste stable.

Chaque requête suit exactement ce chemin : connexion, lecture, parsing, routage, traitement, réponse, fermeture. Maîtriser ce flux, c'est maîtriser webserv.
